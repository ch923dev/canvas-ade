## 2. Gap list (Jarvis needs ŌåÆ what's missing)

1. **Jarvis tier (D5, new decision).** As a connected-tier client Jarvis can't tidy, spawn
   groups, read layout, or block on terminal completion. Recommendation: **mint Jarvis an
   orchestrator-tier token** ŌĆö it is the user's app-level helper, exactly what the tier is for
   (the app already mints orchestrator for board `'app'`). One decision erases four gaps
   (tidy_canvas, spawn_group, layout/app-model reads, handoff/barriers).
2. **Camera/focus ŌĆö does not exist anywhere.** No MCP tool, no MAIN seam, no `McpCommand`.
   Renderer verbs are all built. Need new tool `focus_viewport({target: boardId | groupId |
   "all"})` ŌåÆ new content-less `McpCommand` ŌåÆ applier calls `focusBoardById`/`fitGroup`/
   `fitAll`. Copy the `tidyBoards` pipeline verbatim (`mcpTidy.ts` ŌåÆ `useMcpCommands.ts:373`);
   un-gated like tidy (viewport-only, reversible by scrolling).
3. **Selective tidy.** `tidy_canvas` is whole-canvas all-or-nothing. Helper mandate wants
   "tidy this group" eventually ŌĆö v1 ships whole-canvas, `scope?: groupId` is a follow-up param.
4. **Browser URL.** `spawn_board` needs a `url` param (browser-type-only, validated
   http/https + localhost) and/or `configure_board` gains a url patch.
5. **spawn_group app-side wiring** (PR-5c) ŌĆö package tool exists; finish the app method.
6. **Per-board move/resize** ŌĆö not needed for v1 (tidy + focus cover the mandate); note as
   future (`move_board`, `resize_board`).
7. **Publish coupling.** Every new tool = `@expanse-ade/mcp` publish (base off
   `feat/canvas-layout`; `main` 0.14.0 is stale) + desktop pin bump + host method ŌĆö the app
   never grows tools it predates (`constants.ts:135-139`).

## 3. External recon ŌĆö how others build Jarvis-style agents (July 2026)

### 3.1 Architecture: pipeline validated; TTS model decision CHANGES

- **Cascaded STTŌåÆLLMŌåÆTTS is the consensus for tool-heavy agents** ŌĆö LiveKit, OpenAI's own
  guidance, and Softcery all favor chained over realtime speech-to-speech for reliable tool
  calling (realtime models still hallucinate tool results; function calls add 400ŌĆō800 ms even
  on fast tools). PLAN.md's pipeline stands.
  (livekit.com/blog/realtime-vs-cascade ┬Ę developers.openai.com/api/docs/guides/voice-agents)
- **Kokoro int8 on CPU: DROP.** Two independent measurements show int8 is *slower* than fp32
  on CPU (sherpa-onnx PR #2460 benchmark: RTF 1.972 int8 vs 1.128 fp32; hexgrad/kokoro#291:
  ~1100 ms vs ~500 ms first audio). Quantized ops fall off the CPU fast path ŌĆö int8 only buys
  disk. **Amended D2: Kokoro fp32/fp16, ORT multithreaded**, sentence-chunked via
  `GenerateWithCallback` (chunks stream out as internal segments finish).
- **Ship Piper en_US medium (~75 MB, RTF 0.114 ŌĆö ~10├Ś faster) as the low-latency/low-end
  fallback voice**, and use pre-rendered WAV fillers (zero-latency acks) regardless of engine.
- Best-in-class local pipelines hit ~500 ms voice-to-voice (KoljaB/RealtimeVoiceChat: LLM first
  token ~220 ms + TTS +80 ms; streams 3ŌĆō10-word *clause fragments* into TTS with a ~10 ms
  sentence-boundary classifier). Takeaway: chunk at the **first clause boundary**, not the
  first period.

### 3.2 Barge-in: AEC assumption CHANGES

- Electron `echoCancellation: true` has open "does nothing on Windows" reports
  (electron#47043, #14325) ŌĆö and Chromium AEC structurally can't cancel audio it doesn't know
  about (raw AudioContext-scheduled PCM is invisible to it).
- **Amended strategy (layered):**
  1. Play TTS through an output path Chromium tracks (media element / standard WebAudio sink)
     to give AEC its best chance; verify per-machine in J2.
  2. **Primary barge-in = transcription-gated**: keep STT+VAD running during playback, but
     interrupt only when a *transcribed real utterance* appears (filters echo, coughs,
     backchannels ŌĆö RealtimeVoiceChat's proven pattern), not raw VAD energy.
  3. Fallback: half-duplex RMS gate (suppress mic while speaking; ~1.5 s cooldown with elevated
     threshold).
  4. Headphone detection ŌåÆ enable aggressive full-duplex.
  - On barge-in: duck/fade Ōēż100 ms, flush TTS queue, cancel LLM stream, truncate the assistant
    transcript at the spoken point.

### 3.3 MCP + voice patterns worth copying

- **VoiceMode** (mbailey/voicemode, 1.3kŌśģ) ŌĆö closest prior art: local whisper.cpp + Kokoro,
  MCP `converse` tool pattern (speak ŌåÆ block-listen ŌåÆ return transcript as tool result).
- **Confirmation binding**: spoken "yes" must bind to a specific pending-action ID (stale-yes
  hazard); LiveKit HITL = propose ŌåÆ finish speaking (`wait_for_playout`) ŌåÆ blocking confirm ŌåÆ
  execute, with idempotency keys. Our on-canvas confirm chip + existing `mcpConfirm` chooser is
  *stronger* than voice-only confirm ŌĆö keep visual confirm for mutations, voice "yes" as an
  input method to the same bound request.
- **Latency masking**: >~800 ms tool calls get a pre-rendered filler ("on itŌĆ”") + working
  earcon; contact-center data shows hangup/frustration climbs past 1 s of dead air.
- **Grounded read-back**: speak results from the tool-result JSON ("Done ŌĆö auth terminal
  spawned, top right"), never from intent. (Already in PLAN.md; validated.)

### 3.4 Semantic targeting ("focus the auth terminal")

- Winning pattern (Talker, Windows Voice Access): give the LLM a compact **workspace manifest**
  each turn (id, type, title, status, group, position summary) and let it resolve referents
  semantically ŌåÆ call tools with concrete IDs. `canvas://app-model` + `canvas://layout` already
  ARE this manifest (orchestrator tier ŌĆö another argument for D5).
- Ambiguity: model asks a one-line disambiguation question, or (later) numbered badges flash on
  candidate boards and the user says "two" (Windows Voice Access pattern).
- Targeted objects should self-indicate (glow/ring) when Jarvis acts on them ŌĆö matches the
  desktop-notifications plan's on-canvas self-indication.

### 3.5 Persona / wake / state UX

- Spoken-style prompt rules (Vapi/ElevenLabs guides): 1ŌĆō2 sentences per point, no
  markdown/bullets, spell numbers/dates for the ear, punctuation controls prosody. Reinforces
  PLAN.md ┬¦3.5's response contract.
- Wake word: sherpa-onnx `KeywordSpotter` (open-vocabulary, 3.3 M params, stack-coherent) is
  the J5 choice per PLAN.md, but recon flags accuracy tuning as finicky (sherpa-onnx#2678);
  openWakeWord is the quality alternative if KWS disappoints. Hotkey + optional KWS stays the
  pragmatic combo.
- Visual state machine: idle / listening / thinking / speaking **+ a fifth state: acting**
  (tool in flight, ideally pointing at the canvas object being manipulated). Earcons on
  listen-start and tool-completion.
- GLaDOS (dnhkng, ~5kŌśģ) ŌĆö persona reference; also demonstrates a "proactive tick" mode (speaks
  when it has something to say) ŌĆö interesting post-v1 idea for build-status callouts.
