## Product brief (what the video must show)

**Expanse** — An infinite, zoomable desktop workspace where your AI coding agents, your live app preview, and your plan all exist on one surface — no tabs, no window juggling, just the work.

### Audience
- Senior software engineers and technical founders who already use AI coding agents (Claude Code, Codex, etc.) daily and feel the fragmentation — terminal in one window, browser preview in another, notes scattered across a third app.
- Developer-tooling enthusiasts who follow products like Linear, Raycast, and Warp and respond to calm, dense, precision-crafted interfaces.
- Early-adopter indie hackers and small-team leads shipping products with AI assistance who are actively looking for a better workflow surface.

### Core features
- **Terminal Board** — A live CLI coding agent runs in a real shell inside a resizable board. Any agentic CLI works — set a launch command and the agent inherits your PATH, profile, and auth. Streamed output, tool-call lines, file-edit diffs with +/- counts, a run-progress sliver across the top edge, pause/interrupt/restart controls, and a follow-up prompt at the bottom.
  - *Why it matters:* Your agent lives on the canvas with everything else. You can glance at its output while watching the browser reload — no alt-tab, no context switch, no losing your place.
- **Browser Board** — A live, responsive preview of your localhost app rendered in a real device frame. One-click viewport switching between Mobile (390px), Tablet (834px), and Desktop (1280px) — true reflow at each breakpoint, not a CSS resize. When the terminal's dev server prints a localhost URL, a single click routes the Browser board to that port.
  - *Why it matters:* You see the real app at the real breakpoint, sitting physically next to the agent that is editing it. Duplicate the board to compare Mobile and Tablet side-by-side.
- **Planning Board** — A whiteboard layer inside a resizable board: sticky notes in four muted tints, freehand pen strokes, arrows that can point at other boards, text, and interactive checklist cards — each with togglable items and a live accent-blue progress bar.
  - *Why it matters:* The plan lives on the same canvas as the code and the preview. Check off a task, see the agent handle the next one, watch the browser reflect the change — all without leaving the surface.
- **Infinite Zoomable Canvas** — Pan and zoom across the whole project. Zoom out to see every board at once in a clean overview (boards collapse to minimal LOD cards below 40% zoom, dot grid fades). Zoom in or double-click to focus one board, which smoothly fills the viewport while other boards dim to 55% opacity. A fit button frames everything in one keystroke.
  - *Why it matters:* The zoom metaphor makes project scale tangible. A senior engineer can orient at a glance and then dive in — the same mental model as a design tool but for a dev workflow.
- **Full View** — Any board can be lifted into a full-screen modal overlay — a live, fully interactive version of that board edge-to-edge, with all its controls intact. Exit with Esc or a click on the scrim. The underlying canvas is preserved; the camera does not move.
  - *Why it matters:* Read a long terminal run at full size. Inspect a mobile preview without squinting. Then dismiss back to the canvas exactly as you left it.

### The cinematic demo moments
- HERO ZOOM-OUT: Start tight on a Terminal board mid-run — agent output streaming, a braille spinner cycling, a 2px blue progress sliver crawling across the top edge. Pull the camera back slowly until two Browser boards (one Mobile, one Tablet side-by-side) and a Planning board with a half-completed checklist all come into frame. The whole project in one breath.
- PORT DETECTION CLICK: Terminal board prints 'Local: http://localhost:5173' during a Vite dev-server startup. A faint connector arrow resolves from the terminal to a Browser board. One click — the browser snaps to the running app, the connected dot turns green. Zero copy-paste.
- LIVE RELOAD LOOP: Agent in the terminal types an edit to a component file, output shows '+ 3 lines  src/Button.tsx'. Cut to the Browser board (Desktop preset, 1280px): the app visibly reloads and the changed component appears. Both boards on screen simultaneously — no switching.
- DUPLICATE BROWSER COMPARISON: Drag-duplicate a Browser board. The copy advances automatically to Tablet preset. Arrange the two boards side-by-side. The same running app renders at 390px on the left and 834px on the right — a real responsive diff, live.
- FULL VIEW EXPAND: Click the maximize icon on the Browser board. The board lifts into a full-screen overlay over a dimmed canvas. Switch from Mobile to Desktop viewport inside full view — the device frame reflows at the real breakpoint. Hit Esc — the canvas returns exactly as it was.
- CHECKLIST PROGRESS: Click into the Planning board. Tick off items in a checklist card — the 3px blue progress bar fills in real time, the done count ticks up, completed items go faint with strikethrough. The planning board is physically adjacent to the terminal and browser it governs.
- FOCUS SNAP: Double-click a single Terminal board. The camera animates (200ms ease) to fit that board in the viewport; all other boards dim to 55% opacity. The board feels enlarged without moving — the canvas did the work.
- EMPTY CANVAS TO FULL PROJECT: Start on the empty-state screen (app mark watermark, ghost-outline add-board buttons). Add a Terminal, then a Browser, then a Planning board in sequence. Each appears on the void canvas. Zoom out to see them arranged as a coherent workspace in under 20 seconds.

### Positioning
**Do:**
- Present it as a finished, premium desktop product ready to ship
- Lead with the problem: AI dev work is fragmented across windows, tabs, and apps — Expanse collapses that into one surface
- Show the spatial metaphor: zoom out = orientation, zoom in = execution — this is the core value proposition
- Emphasize local-first, calm, and focused — the Linear/Raycast vocabulary resonates with the exact audience
- Let the UI speak: the dark void, the single blue accent, the clean board chrome are the brand — show them at full fidelity
- Use language like 'your agents', 'one surface', 'no tabs', 'live side-by-side' — these are the phrases from the validated teaser copy that resonated

**Don’t:**
- Do not name or show any implementation framework or underlying technology
- Do not frame the video as a build-in-public vlog, dev diary, or work-in-progress — it is a product demo of a finished, premium tool
- Do not use generic 'AI app' visual language: no purple gradients, no glassmorphism, no glow, no animated particle backgrounds
- Do not show setup, configuration screens, or installation — demo the live working surface only
- Do not narrate features as a list — let the on-screen action carry the meaning, with minimal voiceover or captions
- Do not compare to competitors by name

### Visual identity
Tokens: void background #0a0a0b, dot grid #202022 at 24px world-space lattice, board surfaces #141416 / #1a1a1d, single accent blue #4f8cff (selected rings, progress sliver, checklist bar, active viewport toggle). Zero hue in all neutrals. Typography: Geist (UI) + Geist Mono (terminal output, type glyphs, meta labels). Board chrome: 8px radius, 1px border-subtle at rest, 1.5px accent ring on selection, compact 34px title bars. Elevation: flat resting shadow only — no glow, no blur. Motion: direct pan (1:1), 200ms cubic-bezier fit/focus camera animations, 80ms braille spinner, 1.2s indeterminate progress sliver loop. Feel: calm, dense, precise — a developer power-tool that recedes until touched, not an AI toy that announces itself.

### Unique angles vs the field
- Spatial coherence over tab switching: the infinite canvas is not a gimmick — it encodes the *relationships* between work items. The Terminal board is physically next to the Browser board it feeds. The Planning board's checklist points at the code it governs. Cursor/Warp operate in a single-tool context; Expanse makes the connections between tools visible and persistent.
- True live preview, not an iframe screenshot: the Browser board renders your real running localhost app at genuine CSS breakpoints via viewport reflow — not a scaled screenshot, not a webview approximation. Switching Mobile to Tablet triggers a real reflow at 390px vs 834px. Figma can mock it; Expanse runs it.
- Agent-agnostic by design: any CLI coding agent that runs in a terminal works. The launch command is free text. Expanse is not a Claude wrapper or a Codex wrapper — it is the surface those agents work on. This is a durable position as the agent landscape keeps changing.
- Port detection as connective tissue: when a dev server prints a localhost URL to the terminal, Expanse reads it and offers a one-click link to the Browser board. No configuration, no port assignment, no injection — detect and connect. This is the moment where the canvas feels alive.
- The zoom metaphor for project orientation: zooming out to see a full project of boards — terminal running, browser live-reloading, checklist being ticked — in one frame is something Cursor, tldraw, Figma, and Warp cannot show. It is the flagship visual that belongs on the product page hero.

## Research by angle

### 1. Video strategy & benchmarks

**Thesis (verbatim from the research agent):** Lead with the spatial moment — zoom-out-from-one-board-to-see-the-whole-project is Expanse's unique visual claim that no competitor can copy. Every other strategic choice flows from protecting and amplifying that moment. The primary cut is a 75-90 second narrated launch film structured as a spatial reveal: start tight on running agent output, pull the camera back until the full canvas fills the frame, then walk through 3-4 demo moments as a single continuous take on a live canvas. The secondary cut is a 15-20 second silent autoplay hero loop extracted from the same source: the zoom-out moment only, looped seamlessly for the product page hero section.

Expanse sits in the rarest slice of developer tool videos: the ones where the interface IS the argument. Linear and Raycast established that calm, dense UI shown without apology is its own credibility signal for a technical audience. The research confirms that 71% of elite SaaS launch videos use an abstract or kinetic hook, but the 29% that inject UI immediately do so because the product's visual language is itself the differentiation — that is Expanse's position. The spatial zoom metaphor is not just a demo moment; it is the only metaphor that encodes Expanse's core value proposition (orientation + execution on one surface) in a single shot. Every benchmark product — Warp, Cursor, Figma, Linear — lives in a single context window. Expanse's flagship visual is the one thing they cannot show.

**Key recommendations**
- PRIMARY CUT: A 75-90 second narrated launch film. Not a voiceover explainer — a single calm sentence of copy at the start and end, with sound design and UI action carrying the middle. Structure: 0-5s cold open on a live terminal board mid-run (streaming output, braille spinner cycling, blue progress sliver); 5-15s slow camera pull-back — the spatial reveal, two Browser boards and a Planning board with a half-completed checklist entering frame; 15-60s three demo moments in sequence (port-detection click, live-reload loop, checklist progress tick) — no cuts between them, the camera just pans across the canvas; 60-75s final pull-back to show the whole project, static, then a single line of copy and a CTA. This is the product page hero film, the Product Hunt video, and the launch tweet asset.
- SECONDARY CUT: A 15-20 second silent autoplay hero loop. Extract the zoom-out moment (frames 5-15s of the primary cut), slow it to 15 seconds, render seamlessly looping, muted. This sits as the product-page background video above the fold — a living preview that communicates the spatial metaphor before the user reads a single word. No captions needed; the UI speaks. Keep it at 720p/1080p, hardware-accelerated, under 3MB.
- SOCIAL TERTIARY CUTS: Three 15-30 second vertical (9:16) clips, each isolating one demo moment from the primary cut: (1) the port-detection click + browser snap; (2) the terminal edit + browser reload side by side; (3) the checklist progress fill + adjacent terminal. These are for LinkedIn, X, and YouTube Shorts distribution at launch. No narration — burned-in minimal caption lines only.
- FIRST-3-SECONDS RULE: Open on the terminal board already running — braille spinner mid-cycle, output text mid-line, the progress sliver at ~30% completion. Never start on the app mark, a logo animation, or an empty canvas. The viewer's eye should immediately read 'this is a live coding agent at work' before any words appear. This is the immediate UI-injection approach (the 29% who skip abstraction) and it is correct here because Expanse's UI is visually novel — dark void, single blue accent, board chrome — and the target audience will recognize it as premium instantly.
- THE SPATIAL REVEAL IS THE HERO MOMENT: The camera pull-back at seconds 5-15 is the only shot in the video that needs to be cinematically perfect. Use a smooth cubic-bezier ease (matching Expanse's own 200ms camera animations), slow enough that the viewer registers each board entering frame as a distinct 'oh' moment. The Terminal → Browser (Mobile) → Browser (Tablet) → Planning board sequence should feel like a breath expanding. This is the shot that goes on the product page. No competitor can show this shot.
- AUDIO STRATEGY: Minimal, non-distracting. A single ambient track — sparse, electronic, in the Linear/Raycast vocabulary (think Nils Frahm or Jon Hopkins instrumental texture, not generic 'productivity' music). Layer subtle UI sound design on key moments: the port-detection click resolving (a single soft 'connect' tone), the browser reload flash, the checklist tick (a satisfying, muted click). No voiceover in the primary cut except optionally one opening line ('Your agents. Your preview. Your plan. One surface.') and a closing CTA line.
- KINETIC TYPOGRAPHY AS COPY DELIVERY: Instead of voiceover, use minimal on-screen text appearing in Geist — the same font as the product UI — at key demo moments. 3-5 words max per card, timed to appear and dissolve within 1.5 seconds. Examples: 'Local: http://localhost:5173' appears on screen as the terminal prints it, then 'One click.' appears as the browser snaps. This technique (used by Linear, Raycast, and Framer) lets the copy physically inhabit the product world rather than narrate over it.
- WHAT TO NEVER SHOW: No setup screens, no configuration, no 'add your first board' tutorial flow. No mouse cursor hunting for menus (the cursor should move with magnetic, hyper-eased bezier motion — never raw screen-recorded jitter). No loading spinners that suggest slowness. No split-screen of the 'old way' (scattered terminal + browser tabs) unless as a 2-second flash used to establish the problem. Never show the Electron runtime, the underlying framework, or the word 'beta'.
- CTA PLACEMENT: End on the canvas, zoomed out to see the full project, static for 2 seconds. A single line appears: 'Expanse — one surface for everything.' Below it, the URL and a 'Join the waitlist' or 'Download' command. Style the CTA as a terminal-style line (Geist Mono, lowercase, subtle cursor blink) — this technique, used by Linear and Raycast, makes the CTA feel like it belongs to the product world, not a marketing layer placed over it.
- PRODUCTION TIER: Aim for the $8K-$15K cinematic tier if outsourcing, or build it in-house using screen recording (OBS/ScreenStudio at 4K, then composited in After Effects/DaVinci). The most important investment is NOT production budget but the canvas state itself: the demo canvas must be built to pixel-perfection — a curated project with the right boards, real agent output (a plausible but clean coding session), a real localhost app rendering in the Browser board at the right breakpoints, and a Planning board checklist at ~50% completion. The authenticity of the content is the premium signal, not the camera work.

**Pitfalls to avoid**
- OVER-NARRATING THE UI: The target audience reads UI at a glance. If the video explains what each board does in voiceover while showing it on screen, it signals distrust of the viewer's intelligence. Let the labels, the terminal output text, the browser URL bar, and the checklist items do the reading. The video's job is pacing and sequencing, not explanation.
- GENERIC 'AI PRODUCTIVITY' VISUAL LANGUAGE: Purple gradients, glowing particles, abstract neural networks, and 'AI typing on its own' tropes will immediately signal to the Linear/Raycast audience that this is a consumer AI toy, not a precision power tool. Every frame should look like it could be a Linear or Warp screenshot. The dark void and single blue accent are the brand — protect them.
- SHOWING THE WRONG CANVAS STATE: The demo canvas must look like a real project in mid-flight, not a showcase environment. A terminal with 'Hello World' output, a browser showing a placeholder page, and a planning board with 'TODO: add features' will read as fake. The terminal should show a plausible coding agent working on a real-looking React component refactor. The browser should show a real-looking app UI (can be a purpose-built demo app). The checklist should have 5 items, 3 checked, 2 remaining — authentically mid-task.
- STARTING WITH THE EMPTY CANVAS: The 'empty canvas → add boards' sequence from the demo moments brief is compelling as a secondary demo clip, but not as the launch video opener. The research is clear: videos that open on already-running product outperform those that start at setup or onboarding. Open on the work, not the setup.
- SHAKY OR JITTERY CURSOR: Raw screen-recorded mouse movement reads as amateur and signals instability. Every cursor motion in the video must use smooth, physics-defying easing (ScreenStudio handles this automatically; in AE, use exponential ease on all cursor keyframes). The cursor is a performance signal — it tells the viewer whether the interface responds at thought-speed.
- RUNNING TOO LONG: The Warp 2.0 launch video is 20 minutes and requires a committed viewer. The research consensus for developer tools at the awareness stage is 60-90 seconds maximum. A 90-second cut that leaves the viewer wanting more is more valuable than a 3-minute cut that covers every feature. The long walkthrough belongs on a documentation page or YouTube, not on the product page or Product Hunt.
- MISMATCHED AUDIO TONE: A generic 'tech startup' beat (four-on-the-floor kick, risers, drop) is as wrong as silence with no sound design. Study the audio of Linear's page (sparse, textural, receding) and Raycast's launch content. The audio should feel like the product: precise, unhurried, and slightly cold. Avoid anything that sounds like a productivity app commercial.
- SHOWING IMPLEMENTATION DETAILS: Even one frame that reveals 'React Flow', 'Electron', 'node-pty', or a console.log mentioning the tech stack violates the positioning guardrail and hands ammunition to competitors framing the product as 'just an Electron app with React Flow'. Audit every frame of terminal output and browser devtools visibility before publishing.
- NEGLECTING THE HERO LOOP SEPARATE FROM THE LAUNCH FILM: These serve different jobs. The launch film is watched once with intent. The hero loop is seen passively by every page visitor, often for less than 5 seconds. Treating them as the same asset (just trimmed) results in a hero loop that is too eventful and a launch film that opens too slowly. Build the 15-second loop as its own edit with its own rhythm — centered entirely on the zoom-out reveal.
- SKIPPING THE SOUND DESIGN ON KEY MOMENTS: The port-detection click resolving, the browser reload flash, and the checklist tick are Expanse's three most satisfying micro-interactions. In the video, each should have a subtle, satisfying sound cue — not a UI 'ding' but a tonal, designed sound that matches the calm/dense vocabulary. These sounds create the subconscious 'this feels fast and reliable' impression that words cannot.

### 2. Scripting, storyboard & on-screen text

**Thesis (verbatim from the research agent):** A calm, precision-paced product reveal in the tradition of Linear / Raycast launch videos — no narrator explaining features, the spatial zoom-out IS the opening argument, VO sparse and philosophical, on-screen callouts do the technical work, CTA earns its moment by landing after the hero moment not before it.

A software walkthrough video script is built in three layers simultaneously: the beat sheet (narrative logic), the shot list (what the screen shows, second by second), and the VO/caption layer (the words, and when they appear). For a 60–120 second premium developer-tool demo like Expanse, the craft disciplines are: write VO at 130–150 WPM and treat every word as load-bearing; use the beat-sheet method to sequence features as a single rising workflow rather than a list; script feature reveals around a problem beat that precedes every solution beat; use kinetic callouts to add the *outcome* of what the screen already shows (not a redundant label); and end on a CTA that is the natural next interaction, not an interruption. The spatial zoom-out is Expanse's structural unfair advantage — it is both the opening hook AND the closing payoff, giving the script a circular arc that feels complete without feeling rehearsed.

**Key recommendations**
- WORD RATE: Write VO at 130–150 WPM (roughly 2.2–2.5 words/second). At 60 seconds that is 130–150 words of spoken copy max. At 90 seconds: 195–225 words. Read aloud with a stopwatch before locking. Dense technical terms slow delivery — budget 10–15% extra time for them.
- BEAT SHEET FIRST, SCRIPT SECOND: Write a 6–8 beat sheet (one sentence per beat, no screen detail) before you touch VO copy. Each beat = a problem/tension moment followed by a resolution/reveal moment. Only after beats are sequenced should you write VO sentences and shot descriptions.
- SHOW THE INTERFACE IN THE FIRST 3 SECONDS: Research confirms 85% of high-performing software demos surface the core UI within 3 seconds. For Expanse: open on the Terminal board mid-run — do not start with a title card or abstract animation. The screen IS the argument.
- NEVER NARRATE WHAT IS VISIBLE: VO should add the *why* and the *outcome*, not describe the action. If the screen shows the camera pulling back to reveal all boards, the VO does not say 'here you can see all your boards' — it says 'the whole project, one breath.' Let the image carry the literal; let the words carry the meaning.
- FEATURE REVEAL STRUCTURE (one per beat): (1) Establish the tension/friction in one sentence. (2) Show the resolution via screen action — no VO during the action itself, 1–2 seconds of silence for cognitive processing. (3) Drop a 2–4 word kinetic callout that names the outcome. (4) Resume VO for the bridge to the next beat. This pattern trains viewer attention: silence = something is about to click.
- PACING RULE — THE VISUAL HOLD: Before any complex click or state change, hold the frame for 0.5–1 second. 86% of analyzed top demo examples use intentional visual holds before complex interactions. It reads as confidence, not drag. Never let a screen loading state dictate your edit — cut around it or use a freeze-frame.
- KINETIC CALLOUT RULES: (a) No more than 4–5 words per callout. (b) Appears ON or immediately adjacent to the element it labels — not at the bottom of the screen. (c) Enters after the action completes, not before. (d) Use one consistent type style (e.g., Geist Mono, uppercase, accent blue) so callouts read as a system, not decoration. (e) Kinetic entry: slide-in 150ms or fade 100ms — match Expanse's 80–200ms motion vocabulary. Never spin, bounce, or glow.
- VOICEOVER vs PURE VISUAL: For a 60-second hero/launch video, design the sound-off experience first. VO should be a layer that deepens the premium feel, not a crutch that explains what the screen can't show. Concretely: if the video makes sense watched silently with captions, the VO is polishing; if it only makes sense with VO, the edit has failed.
- FEATURE SEQUENCING — WORKFLOW CHAIN NOT FEATURE LIST: Do not cut from feature A to feature B as isolated demos. Chain them: the terminal prints a URL → the browser board connects → you zoom out and see both → you check a planning card → the whole canvas is now live. 85% of top demos bridge features into continuous workflows rather than isolating them. The chain makes each beat earn the next.
- CTA PLACEMENT AND COPY: The CTA should arrive as the natural continuation of the last beat — not as a tonal gear-shift. The screen returns to the full canvas overview (the same visual from the hero moment), then the CTA appears on the void. Copy: one verb, one outcome, no friction language. 'Get early access' outperforms 'Request a demo' for this audience. Reinforce visually with the accent blue (#4f8cff) so the CTA inherits the product's own visual language.
- DURATION TARGETS BY USE CASE: Hero/launch page autoplay: 45–75 seconds (no VO or sparse VO, caption-first). Feature walkthrough / email embed: 90–120 seconds (VO + callouts). Full product tour: 2–3 minutes (VO-led, beat sheet required). For Expanse's initial launch, 60–75 seconds is the target — long enough to show all 5 demo moments, short enough to never drag.

**Pitfalls to avoid**
- NARRATING THE VISIBLE: The most common mistake in developer-tool demos. If your VO says 'and here you can see the terminal board showing live output' while the screen shows exactly that, you have wasted 2 seconds and insulted the viewer's intelligence. VO speaks to what the viewer cannot see: the outcome, the meaning, the before-state that is now gone.
- FEATURE-LIST STRUCTURE: Cutting from Terminal → Browser → Planning as isolated demos with their own mini intros destroys narrative momentum. Each feature should be the *consequence* of the previous one. The workspace is a system; show it behaving as one.
- LEADING WITH THE EMPTY STATE OR SETUP: Opening on an empty canvas, an install screen, or a configuration dialog signals 'this is work to get started.' For Expanse specifically, open mid-run — agent output streaming, progress sliver live, the work already happening.
- GENERIC AI VIDEO AESTHETIC: Purple gradients, glowing particle effects, generic 'neural network' imagery. This is the opposite of Expanse's brand. Every visual choice should reinforce the void-dark, one-accent-blue palette. If a transition feels like a Canva template, cut it.
- LETTING THE ZOOM-OUT OVERSTAY: The hero zoom-out is the strongest moment in the video. Use it in the first 10 seconds as the opening argument and in the final 5 seconds as the closing visual. Do not hold it for longer than 3–4 seconds either time — it is a reveal, not a screensaver.
- OVER-ANIMATING CALLOUTS: Callouts that bounce, spin, or scale with spring physics draw attention to themselves rather than to the UI element they are labeling. The callout is a pointer, not a performer. Fade in 100ms, hold, fade out 100ms.
- SHOWING THE CTA TOO EARLY: A CTA before the viewer has seen the core value proposition is spam. For Expanse, the CTA earns its place only after the HERO ZOOM-OUT is shown at least once and at least two feature beats have landed.
- RECORDING LIVE WITH REAL DATA: Live screen recordings show typos, loading spinners, off-brand tab titles, and notification banners. For the final video, use a purpose-built demo environment with clean data, no browser chrome, and pre-staged state. ScreenStudio or equivalent handles the recording layer.
- MISMATCHED VO ENERGY: A calm, precise product like Expanse needs a VO delivery that matches — measured pace, low affect, no exclamation. An enthusiastic 'and then you can ALSO do this!' VO on a Linear-aesthetic UI creates cognitive dissonance. If in doubt, go slower and quieter than feels comfortable.
- SKIPPING THE READ-ALOUD CHECK: Every draft of the VO script must be read aloud with a stopwatch before it is considered final. What reads as 90 seconds in your head is frequently 115 seconds spoken. Cut to fit the edit, not the other way around.

### 3. Screen-recording tooling

**Thesis (verbatim from the research agent):** The Expanse demo is not a tutorial — it is a cinema-style product reveal. The infinite canvas pan/zoom is the hero moment, not a click-by-click walkthrough. That means the ideal recorder must: (1) stay invisible during the actual in-app camera movement (no post-zoom fighting the app's native zoom), (2) capture the void #0a0a0b background and dot-grid at full fidelity with zero compression artifacts, and (3) export at 60fps with motion blur so the 200ms cubic-bezier fit animations feel intentional. The strategy is to split the job — use a clean full-screen or single-window capture at maximum quality for the live pan/zoom moments, then let the recorder's auto-zoom handle close-up board interactions (click highlights, checklist ticks, button presses). Screen Studio on macOS is the unanimous first-choice for this split because its auto-zoom fires on clicks while leaving the canvas camera entirely alone. On Windows (where Expanse is the dev machine), FocuSee or Rapidemo cover the auto-zoom layer, but the canvas-motion shots should be OBS or ScreenFlow-style raw capture that you then cut around.

Screen Studio (macOS) is the single strongest choice for Expanse's flagship product demo — it delivers automated cinematic zoom on clicks, full-retina 4K 60fps window capture, motion-blurred cursor smoothing, and near-zero post-production overhead. For the infinite-canvas pan/zoom hero shots specifically, however, the in-app camera IS the zoom: no recorder should be fighting it with a second zoom layer. The right workflow is clean high-resolution capture (Screen Studio window mode, OBS Display Capture, or ScreenFlow) for the sweeping canvas moments, and auto-zoom tooling only for the close-up board interactions (checklist ticks, port-detection click, full-view expand). ScreenFlow is the macOS backup with the deepest editing control. On Windows, FocuSee and Rapidemo are the closest functional equivalents to Screen Studio. OBS is the free raw-capture baseline on both platforms. Tella and Camtasia are collaboration/e-learning tools that are overweight for a tight premium product reveal. CleanShot X is screenshot-and-annotation, not video production.

**Key recommendations**
- PRIMARY (macOS): Screen Studio — record the entire Expanse window in window-capture mode at 4K 60fps. Let its auto-zoom handle board-level interactions (the port-detection click, checklist toggles, full-view expand). For the HERO ZOOM-OUT and FOCUS SNAP moments, disable auto-zoom or set a wide zoom threshold — the Expanse canvas camera IS the cinematic move; a second recorder zoom on top will look double-zoomed and jittery.
- HERO SHOTS WORKFLOW: Record the canvas pan/zoom sequences in Screen Studio with auto-zoom OFF (or threshold set very high so it never fires during a canvas gesture). Then record the close-up interaction sequences (checklist, terminal output, port-detection click) with auto-zoom ON at moderate intensity. Cut between them in DaVinci Resolve or ScreenFlow's timeline.
- WINDOWS PRIMARY (dev machine): FocuSee — cross-platform, one-time $69.99 license, genuine auto-zoom + cursor highlight + 4K 60fps. Use it for the interaction close-ups. For the canvas motion shots on Windows, use OBS with Display Capture (not Window Capture — OBS Window Capture has documented quality regression on macOS; on Windows it's fine but Display Capture cropped to the Expanse window is more reliable for HiDPI).
- EDITING + POLISH: ScreenFlow ($169 one-time, macOS) is the best timeline editor if you are cutting multiple clips together — it natively handles Retina 2x assets, has zoom/pan effects for any shots that need a post-production push-in, and exports ProRes for a lossless master before your final H.264/H.265 encode.
- AVOID SUBSCRIPTION OVERHEAD FOR A ONE-TIME DEMO: If the demo is a single 60-90 second hero video, Screen Studio's free-record/pay-to-export model means you can test everything before committing to the $9/month annual plan ($108/year). Export once, cancel. Or buy ScreenFlow + Screen Charm (one-time $79) as a permanent cross-tool stack with no recurring cost.
- FOR THE DOT-GRID / VOID BACKGROUND FIDELITY: Record at native Retina resolution (3840×2160 effective on a 2x display), then export at 2x or downscale with Lanczos in OBS/ScreenFlow. The #202022 dot grid at 24px world-space is extremely fine — any lossy encode at low bitrate will macro-block it. Use CRF 15 or lower in H.264, or Apple ProRes 422 LT as the master.
- CURSOR HIGHLIGHTS: Screen Studio, Screen Charm ($79 one-time, macOS), and FocuSee all add click-ripple highlights automatically. For the LINEAR/RAYCAST aesthetic of Expanse, keep highlights minimal — a faint white ring, not a colored burst. Screen Studio lets you disable or desaturate click effects to match the single-blue-accent brand.
- FRAME RATE: 60fps is non-negotiable for the canvas pan/zoom sequences. The Expanse 200ms cubic-bezier fit animation at 30fps will stutter visibly. All recommended tools support 60fps export; on Tella it requires the Premium tier ($19/month).

**Pitfalls to avoid**
- DO NOT let the recorder's auto-zoom fight the Expanse canvas camera. When you pan/zoom the Expanse canvas, the app's own cubic-bezier camera animation is the cinematic effect. If auto-zoom fires simultaneously, you get a double-zoom that looks broken. Disable auto-zoom (or set a very high click-distance threshold) for all canvas-gesture sequences; re-enable it only for board-level click interactions.
- DO NOT record at 30fps. The Expanse 200ms fit-animation and 80ms braille spinner are both sub-300ms — at 30fps (33ms/frame) these animations render as 6–9 frames. At 60fps they render as 12–18 frames and the easing curve is visible. 30fps will make the UI feel janky even if the underlying code is smooth.
- DO NOT use a compressed codec as your recording format. The #0a0a0b void background and #202022 dot grid are near-identical dark values with fine 24px periodicity. Any DCT-based codec at a normal streaming bitrate (8-12 Mbps) will produce visible macro-blocking in dark uniform areas. Record to ProRes 422 LT, lossless H.264 (CRF 0-15), or UT Video Codec. Compress only for the final export.
- DO NOT rely on OBS Window Capture on macOS. There are documented quality regressions in OBS's macOS Window Capture source (blurrier than Display Capture). On macOS, use Display Capture with a crop filter to isolate the Expanse window, or use Screen Studio/ScreenFlow which handle Retina window capture natively.
- DO NOT show a desktop with system chrome, taskbar, notification badges, or other windows in frame. Record the Expanse window full-screen or use a dedicated recording user account with all notifications silenced, a plain dark wallpaper (#0a0a0b), and the Dock/taskbar hidden. Any stray OS chrome breaks the 'finished premium product' illusion.
- DO NOT use cursor click effects that introduce non-brand colors. Camtasia's default yellow ring, FocuSee's default colored highlights, and Screen Studio's default blue ring all need to be checked against the Expanse accent blue (#4f8cff). A mismatched click color (e.g., orange or red) reads as a bug in a design-forward demo. Set click highlights to white at low opacity or to #4f8cff.
- DO NOT ignore audio. Even a purely visual product demo benefits from light ambience or a single calm music bed (no SFX, no synthetic voice). Tella and ScreenFlow both include licensed music libraries. A silent 90-second video of UI interactions without audio guidance will lose attention faster than the same video with a minimal score.
- DO NOT use Cursorful for the Expanse demo in its current state. As of mid-2025 it is a Chrome extension — it can only record browser tabs, not the Electron desktop app. The desktop app version was scheduled for Q4 2025 but there is no confirmed release. Verify before using.

**Pitfalls to avoid**
- Remotion renders via headless Chromium, not a canvas engine — heavy per-frame calculations (particle systems, complex physics) will be slow. - Screen recording on Windows with a native WebContentsView layer: OBS Game Capture or Windows Graphics Capture will capture the Electron window including the native view correctly. QuickTime and some browser-based recorders will not. Test your recording setup before the shoot session.
- Do not use Remotion for the actual live-app demo moments (port detection click, live reload, browser viewport switch). These must be real screen recordings — technical users will immediately recognize fake UI animation. Remotion's role is overlays, titles, and callout arrows on top of real recordings, not replacing them.
- DaVinci Resolve's free version does not support noise reduction via the AI tools panel (Magic Mask etc.) — but the primary need here is color grading, not AI tools, so this is not a blocker. Fairlight (audio) in the free version is fully capable for voiceover cleanup.
- Remotion's Business Source License (BSL) is commercial-use-free for companies under $1M ARR with 3 or fewer people. If Expanse has 4+ team members or exceeds the ARR threshold at the time of video production, a company license ($50-200/month) applies. Confirm license tier before shipping a commercial video.
- Jitter exports Lottie but not MP4 directly from all plans — verify export format support against the plan tier before committing to it for final deliverables.
- After Effects expressions and motion blur are technically superior for some compositing tasks, but the 40-100 hour learning curve investment cannot be recouped on a single product video. If professional motion graphics are needed, hire a freelancer for the After Effects layer rather than learning it yourself.
- CapCut's AI tools (auto-captions, trending transitions, style presets) will actively pull the Expanse video toward generic social-media aesthetics — the opposite of the calm, dense, premium tone required. Even if CapCut is used for social cuts, turn off all AI style suggestions and work from scratch.
- Avoid recording at 1x pixel density on a standard 1080p display — the dark board surfaces and fine 1px border-subtle strokes will compress poorly and look muddy. - Keep total VO script under 180 words for a 90-second video. Target a 130-WPM delivery pace (calm, never rushed). Each demo moment should carry at most one sentence of narration — the rest is action and music.

### 6. Technical capture plan for the app

**Thesis (verbatim from the research agent):** Record at true native monitor DPI (2560x1440 or 3840x2160) so every pixel the OS compositor renders maps 1:1 to what OBS captures — no scaling artifacts, no blurring of Geist Mono terminal glyphs or dot-grid lines. Deliver a downscaled 1920x1080 master using Lanczos in DaVinci Resolve, which turns the over-sampled pixels into razor-sharp text with zero sharpening pass needed. The live WebContentsView panels (native OS overlay, not HTML) are captured via OBS Display Capture on the primary monitor at full resolution using Windows Graphics Capture (WGC) — the only method that composites native views correctly without tearing. All camera motion is recorded at real-time 60fps with in-app smooth easing — no post speed-ramps needed for canvas pan/zoom — but deliberate pacing and a scripted deterministic demo script ensure every take is identical.

Expanse presents two hard capture problems that most screen-recording advice ignores: (1) the Browser Board is a native OS-layer WebContentsView that sits above all HTML and is invisible to window-scoped capture methods, and (2) the terminal redraws on a separate GPU compositor path from the React canvas, creating a two-layer compositing race that flickers under BitBlt/DXGI capture. Both are solved by using OBS Display Capture (full monitor, WGC) rather than Window Capture, at the monitor's native resolution, recording to a near-lossless intermediate (CQP 16, 10-bit HEVC), then delivering a Lanczos-downscaled 1920x1080 H.264 master. The demo itself must be fully scripted and deterministic — a real Claude Code agent session pre-staged so every output, every port-detection event, and every checklist tick is known in advance — so that the operator can achieve five clean matching takes in a single session.

**Key recommendations**
- Use OBS Display Capture (Windows Graphics Capture, full primary monitor) — never Window Capture — because the Browser Board's WebContentsView is a native OS surface that paints above HTML and is invisible to per-window capture APIs. WGC composites the full DWM frame, capturing every layer. Confirm by checking that the Browser Board renders in a test OBS preview before any serious recording.
- Set OBS Base (Canvas) Resolution = your monitor's native pixels — e.g. 2560x1440 on a 1440p display or 3840x2160 on a 4K. Set Output (Scaled) Resolution = same as Base. Never let OBS downscale during capture; downscale only in post (DaVinci Resolve, Lanczos). This is the single biggest lever for sharp terminal text.
- Record intermediate files as near-lossless: HEVC (H.265) encoder, CQP 16, 10-bit color depth, 4:2:0 (or 4:4:4 if NVENC supports it on your GPU), 60fps. Use NVENC if available (RTX-class GPU), x265 otherwise. Format: MKV container so a crash mid-take does not corrupt the file. Expect ~8–15 GB per 3-minute take at 1440p.
- Deliver the final video from DaVinci Resolve at 1920x1080, Lanczos resize filter, H.264 (x264 slow preset, CRF 16–18 for a local master; CRF 23 for web upload). Do NOT add any sharpening — the over-sampled Lanczos downscale from 1440p to 1080p already produces sharper-than-native text. Adding a sharpen pass will halate Geist Mono terminal glyphs.
- Disable Windows Hardware-Accelerated GPU Scheduling (HAGS) before recording: Settings > System > Display > Graphics > Change default graphics settings > Hardware-accelerated GPU scheduling OFF. HAGS causes encoder freezes in OBS with NVENC under high VRAM load, which is exactly what a live Chromium WebContentsView imposes.
- Set the monitor DPI to exactly 100% (96 DPI) for the recording session. At 125% or 150%, some DPI-unaware sub-processes in the capture chain introduce a fractional pixel scale that softens glyphs. Expanse itself should be DPI-aware and render crisply at 100%; the canvas dot grid and Geist Mono will be pixel-perfect.
- Run Expanse in a custom frameless full-screen mode during recording: use the existing keyboard-shortcut to maximise the Expanse window to fill the entire monitor, then set Windows taskbar to auto-hide and trigger Focus Assist (Do Not Disturb) so no notification toasts, badge counts, or tray pop-ups intrude. The recording frame should be pure void #0a0a0b from edge to edge.
- For canvas pan/zoom: use Expanse's built-in 200ms cubic-bezier camera animations at real-time 60fps — do NOT record at 30fps and speed-ramp in post. Speed-ramped 30fps interpolated to 60fps produces judder on the canvas grid lines and board borders. The in-app easing is already cinematic; record it faithfully. If a move needs to feel slower, add a deliberate pause before triggering it, not a post speed-ramp.
- For terminal output: do NOT run a live agent session during the hero take. Pre-stage a scripted replay using charmbracelet/vhs or a bash script that pipes pre-captured output into a named pipe connected to the terminal board's PTY. This gives frame-accurate, repeatable output with known timing, identical braille spinner phase, and the exact diff line counts (+3 lines src/Button.tsx) every take.
- Capture the Browser Board live, not as a screenshot. The whole point of the port-detection and live-reload demo moments is that the app visibly reloads inside the device frame in sync with the terminal output. Stage the running Vite dev server on localhost:5173 with a simple demo component (a button that changes color on a prop) so the change is visually obvious at small board size. The reload flash is a feature, not a bug — it proves liveness.

**Pitfalls to avoid**
- PITFALL: Using OBS Window Capture instead of Display Capture. Window Capture uses BitBlt or a per-HWND WGC hook that sees only the HTML BrowserWindow content. The Browser Board's WebContentsView is a separate native OS surface — it will appear as a black rectangle or be missing entirely in Window Capture. Always use Display Capture (full monitor, WGC) for Expanse.
- PITFALL: Recording with OBS's Output Resolution set lower than the monitor's native pixels. OBS will apply its own bilinear or Lanczos downscale in the capture path, which is lower quality than doing the same downscale in DaVinci Resolve on the lossless intermediate. Keep Base = Output = native resolution.
- PITFALL: Leaving HAGS enabled. The Browser Board runs a live Chromium renderer; combined with NVENC encoding in OBS, this can exhaust VRAM and cause the encoder to freeze mid-take. Disable before every recording session.
- PITFALL: Running a live Claude Code agent session for the terminal output during hero takes. Agent sessions are non-deterministic: rate-limit messages, tool-call retries, different diff line counts, and unexpected error output will ruin takes unpredictably. Use vhs or a pre-staged named-pipe replay for all terminal content.
- PITFALL: Performing canvas pan/zoom slowly and then speeding up the footage in post. Speed-ramping 60fps to a different effective rate creates judder on the canvas dot grid and smooth board borders at interpolated frames. The correct approach is to move the camera slowly and deliberately at real-time, using the in-app 200ms easing, and record it faithfully.
- PITFALL: Windows display scaling set above 100% during recording. At 125% or 150%, there is a fractional-pixel mismatch between what Expanse renders internally and what OBS captures from the DWM compositor output. This softens the Geist Mono terminal glyphs in a way that is not recoverable in post. Set display scaling to 100% for recording sessions.
- PITFALL: Notifications and tray pop-ups. Even with Focus Assist enabled, some apps (Teams, Windows Defender, Windows Update) can generate full-screen toasts that appear on the primary capture monitor. Before recording, verify Focus Assist is active AND manually dismiss any pending notification in the system tray.
- PITFALL: Expecting the MKV intermediate to be small. A 3-minute take at 2560x1440, 60fps, CQP 16 HEVC will be 8–15 GB. Ensure the recording drive has at least 100 GB free before the session. Record to an NVMe SSD — a spinning disk cannot sustain the write rate for 60fps near-lossless at 1440p.
- PITFALL: The LOD snapshot path appearing during takes. Expanse detaches live WebContentsViews during pan/zoom and below ~40% zoom, showing a capturePage snapshot instead. If the demo camera moves through the 40% threshold during a hero shot, the Browser Board will visibly switch from a live Chromium frame to a static bitmap. Pre-plan camera positions so the hero zoom-out stops at ~45% zoom, keeping all boards above the LOD cutoff threshold.
- PITFALL: Tearing during browser reloads. When the Vite dev server reloads, the WebContentsView briefly shows a white flash before re-rendering. At 60fps and 1440p this lasts 1–3 frames (~17–50ms) and is visible. Stage the demo app to reload with a dark background-color: #0a0a0b so the flash matches the Expanse void rather than a jarring white frame.

2. Expanse has two GPU rendering paths competing in the same frame: the React canvas (Chromium's HTML compositor) and the WebContentsView (a separate Chromium renderer process). This is an inherent property of the architecture and not fixable in OBS. The mitigation is to cap OBS at 60fps (not higher) and accept that the rare 1-frame seam will be invisible after Lanczos downscale and H.264 encode.

3. The terminal board's xterm renderer uses WebGL (the webgl addon) which has its own vsync path. Under OBS display capture, xterm redraws are captured correctly because WGC operates on the DWM compositor output post-vsync. However, if OBS is set to 60fps and xterm's internal render loop runs at a different cadence (e.g. the braille spinner animates at ~12fps on an 80ms cycle), there will be frames where the spinner appears mid-update. This is cosmetically acceptable and cannot be eliminated without patching xterm's rAF timing.

**Pitfalls to avoid**
- Recording at less than native resolution — the Expanse dot grid (#202022 at 24px lattice) and 1px board borders will alias or disappear at anything below 1:1 pixel capture. Always record at the native display resolution and scale down in post, never upscale.
- Using OBS for the primary capture — OBS frame-timing jitter makes the 200ms camera ease-animations look stuttery at 30fps. Use Cleanshot X (macOS) or ShareX (Windows) for the demo recording.
- Starting the video with the empty canvas or a setup screen — the brief says never show configuration. Open on the Terminal board already mid-run with output streaming.
- Letting any social cut run past its platform ceiling without testing: X/Twitter at 140s gets preview-clipped; PH at 45s risks auto-cutoff at 60s; LinkedIn completion rates collapse after 90s. Trim ruthlessly.
- Not testing the landing-page loop over a 4G mobile throttle (Chrome DevTools Network: 'Fast 4G'). If it buffers past 2 seconds, re-encode at a lower bitrate or consider a static poster-image fallback for mobile.
- Using a YouTube link in X/Twitter posts — native video uploads get significantly more reach than linked YouTube thumbnails in the feed. Always upload the video file directly.
- Captions using a thin or small typeface on the dark void background — white Geist at 22pt minimum is the floor. Below that, captions blur on retina displays and become illegible on phone screens at thumbnail scale.
- Exporting a lossy compressed file and then re-exporting from it — always export lossless from DaVinci first, then run the single HandBrake compression pass. Re-compression introduces blocking artifacts especially visible on the dark void surfaces.
- Using glassmorphism, gradients, glow, or purple/pink accent colors in any caption overlay or B-roll graphic — these signal 'generic AI product' to the exact developer audience Expanse is targeting. Every overlay element must use the Expanse token set: void background, single blue #4f8cff accent, Geist type.
- Framing any copy or caption as a work-in-progress, alpha, or 'shipping soon' signal — Product Hunt hunters specifically reward products that demo as finished and ready. Every frame and every caption card should read as a shipping product.
- Not uploading an SRT file separately to YouTube in addition to burned-in captions — the SRT file is how YouTube indexes the caption text for search ranking; burned-in captions are invisible to the crawler.
- Positioning the YouTube title with a vague keyword like 'AI workspace' — target the specific search queries senior engineers actually run: 'Claude Code terminal setup', 'AI coding agent workflow', 'local dev preview responsive', and use the exact phrasing in the title and first line of the description.
