Confirmed the critical data-flow gap: `initiator` is a **flattened display string** (a script URL or a type word like "parser"/"script"/"Redirect"), not a structured `requestId` reference. This directly undermines the proposal's Part B claim that `initiator` yields "the triggering request ŌåÆ directed edges." I have enough to write the review.

Here is the adversarial feasibility review. Findings are numbered, severity-tagged, and grounded in the actual code.

---

**1. [major] Part B's call-graph edges from `initiator` are NOT computable as claimed.** The proposal states (┬¦5 "What it consumes") that `initiator` gives you "the triggering request ŌåÆ directed edges." It does not. In `preview/index.ts` the field is typed `initiator?: string`, and `initiatorOf` (`previewOsrNetwork.ts:197`) flattens the CDP initiator object to a **display string** ŌĆö "a script url or a type word (parser/script/preload/other)" or the literal `'Redirect'`. It carries no `requestId`, so you cannot resolve a `requestWillBeSent` back to the specific record that triggered it. CDP's raw `Initiator` object does carry a `stack` with script URLs (and sometimes a `requestId` for `preflight`/`SignedExchange`), but that structured data is discarded before it reaches the store. **Fix:** either scope down the "calls" edges to coarse page/loader grouping (which *is* computable ŌĆö see #2), or extend `initiatorOf` + `NetRecord` in MAIN to preserve a structured initiator (script-url + optional triggering requestId) ŌĆö that is new MAIN capture work the proposal explicitly claims it avoids ("zero new CDP plumbing"). The "PAGE ŌåÆ POST /api/session" edges in the ASCII sketch are buildable from `loaderId`/`navBoundary`; the **requestŌåÆrequest** edges are not.

**2. [minor] Page/document grouping IS computable ŌĆö keep that claim.** `loaderId` (`requestId===loaderId ŌćÆ main document`, per the source comment), `frameId`, and `navBoundary` are all present on `NetRecord`. Grouping requests under a navigation/document is sound. Only the finer requestŌåÆrequest initiator edges (#1) are not. No fix needed; just don't overstate.

**3. [major] The id-lineage pass (engine 4) requires bodies for BOTH endpoints of every edge, making it far heavier than framed.** "An id in call A's *response* later appearing in call B's *URL/request*" needs A's **response body** parsed (gated, 5MB, lazy) AND B's request URL/post-data. Today bodies are fetched **only on an explicit per-row Load-body click** ŌĆö there is no bulk body store. To discover lineage across a session you must fetch+parse the response body of *every* candidate producer request, which is exactly the unbounded auto-fetch the proposal's own perf section (┬¦"Large-payload") forbids ("never auto-pretty-print an unbounded body"; "body arrives only on an explicit Load body click"). The privacy opt-in (┬¦"Privacy") is described as gating *whether* sampling happens, but does not resolve the cost/architecture contradiction: lineage inherently needs many bodies, not one. **Fix:** state plainly that lineage requires a MAIN-side bounded body-sampling subsystem (cap N bodies, cap bytes each, behind the opt-in + `isForeignSender`), and that URL-side id-matching (ids that appear in *URLs/query*, which are body-free) is the only part shippable without it. Move full body-based lineage firmly into P3 with that subsystem as an explicit dependency, not a "pure `lib/` pass."

**4. [minor] File-size pressure is overstated; the new-file conclusion still stands.** The proposal cites `OsrNetworkDetail.tsx` 423 and `OsrNetworkPanel.tsx` 596 against a "max-lines: 700" cap. The lint rule (`eslint.config.mjs:262`) uses `skipBlankLines:true, skipComments:true`, so it counts **code lines only** ŌĆö the real budgets are larger than the raw `wc -l` figures imply. The decision to add `JsonView.tsx` as a new file is still correct (a non-trivial virtualized tree would blow any of these), but the specific "423/596 leave no room" framing is imprecise. No change to the plan; tighten the justification.

**5. [minor] `prettyBody` is called at TWO sites, not "three converging call sites."** ┬¦2 says three (`:106`, `:249`, and an implied third); the grep shows exactly two live call sites in `OsrNetworkDetail.tsx` (lines 106/108 and 249/250) plus the definition. The migration list in ┬¦4 ("two call sites") is correct; the ┬¦2 "three converging" count is wrong. Minor, but it's a factual claim in the audit. **Fix:** say two.

**6. [minor] `NetTab` comment is misquoted, weakening the "pre-built scaffold" argument.** The proposal asserts the single-member union was "pre-built scaffolding for a second view." The actual comment (`osrNetworkStore.ts:12-15`) says the Assets/Downloads tabs were *removed* (superseded by the Project Library) and the union was "kept so the store shape + the header tab affordance stay intact" ŌĆö i.e. it's a *vestige*, not deliberate scaffolding for Data Flow. The mechanical claim (extending the union is cheap, ephemeral, no schema bump) is true and fine. Just drop the "intentional scaffold" spin ŌĆö it's a post-hoc rationalization of dead code.

**7. [minor] "No virtualization needed under the 5MB cap" (P0 + Decision 3) is risky for the stated row counts.** The proposal's own thresholds say ">1,000 visible rows = virtualization mandatory" and "Ōēź10,000 = mandatory," yet P0 ships with no virtualizer relying on "the 5MB cap + Load-body gate make naive render acceptable for typical bodies." A 5MB JSON body can easily exceed 100k tokens/rows ŌĆö well past the proposal's own "mandatory" line. *However*, the flat-row model with **default-collapse-to-depth-2** (also in P0) caps *visible* rows at the top two levels regardless of body size, which genuinely does bound the initial DOM. So P0 is defensible **only because of default-collapse**, not because of the 5MB cap. **Fix:** make the P0 acceptance criterion explicit that default-collapse (not the byte cap) is what bounds DOM, and gate "expand all" / deep expansion behind the P1 virtualizer (expanding a 50k-element array in P0 with no virtualizer is the freeze the audit's own H5 warns about).

**8. [minor] Virtualization inside `.bb-stage` works, but the resize/maximize interaction needs a re-measure contract.** The custom virtualizer ("measure container, start = floor(scrollTop/rowH)") is fine inside the clipped/resizable panel ŌĆö the panel is a normal flex sibling (not an overlay), so it has a real measurable box. The risk the proposal doesn't call out: the panel is **drag-resizable `[0.15,0.85]`** and dockable bottom/right, so container height changes continuously during a drag. The virtualizer must recompute `count` on a `ResizeObserver`, not once on mount, or rows clip/blank mid-resize. Uniform row height makes this cheap; just specify it. Not a blocker ŌĆö a correctness note for P1.

**9. [minor] Big-number raw-source display depends on `prettyBody` being abandoned, which P0 already does ŌĆö consistency check passes, but note the Raw-mode source dependency.** The "display the raw source substring for >2^53 integers" feature requires the **original body string**, which the viewer does consume (`BodyState{body}`). Good. But the proposal also says Raw mode should "re-indent from the original source string (not a re-`stringify`)." A faithful re-indent of arbitrary source while preserving big-number literal text is a small tokenizing pretty-printer, not a one-liner ŌĆö it's more work than "re-indent." Flag the effort: this is part of why `osrJson.ts` is non-trivial and unit-tested, which the plan does account for. No fix, just don't under-scope `osrJson.ts`.

**10. [pass ŌĆö no action] Security invariant #1 (no innerHTML) is satisfied by the flat-row/token-to-`<span>` model.** Each token rendered as a React child text node with class-based coloring, never `dangerouslySetInnerHTML`, is the correct and only acceptable approach, and matches the existing `.bb-net-kv` precedent. The proposal's P0 acceptance test ("assert no `dangerouslySetInnerHTML` in the rendered tree") is the right guard. Correct as written.

**11. [pass ŌĆö no action] Invariants #2/#3/#4/#5 (5MB cap, lazy/user-initiated, IPC guard, ephemeral, no heavy deps) are honored for Part A.** The viewer reuses the existing `getOsrNetBody` path (`isForeignSender`-guarded), the existing `bodies` cache, holds view state in React component state (ephemeral, no schema bump), and vendors a ~80-line virtualizer rather than pulling `react-json-view`/`react-window`. This is consistent with the perfect-freehand/Mermaid vendoring doctrine. Part A is clean. The invariant pressure is entirely in **Part B** (#3 body-sampling, plus the new board type's schema bump in P3 which the proposal does correctly flag as an ADR 0007 cost).

**12. [minor] Schema-inference "required-iff-present-in-every-sample" is only sound on UN-truncated samples.** Invariant #2's `truncated:true` bodies (the 5MB-capped ones) will have missing trailing fields that are present-but-cut, which would be mis-inferred as "optional" or absent. The monoid merge must **exclude truncated bodies from presence/required computation** (or down-weight them), or the inferred schema lies exactly on the large payloads. The proposal's own M9 truncation-tolerance note covers the *viewer* but not the *schema engine*. **Fix:** add "truncated samples are shape-only (types yes, required/presence no)" to engine 2's contract.

**Net:** Part A (the viewer "fix") is feasible and security-clean as specified ŌĆö the only corrections are the miscounted call sites (#5), overstated line-budget framing (#4), and the default-collapse-vs-cap reasoning (#7). Part B (the "vision") has one genuine **major** correctness hole ŌĆö `initiator` cannot yield requestŌåÆrequest edges (#1) ŌĆö and one **major** architecture/cost contradiction ŌĆö body-based lineage and schema inference need a MAIN body-sampling subsystem that contradicts the lazy/single-body model (#3), plus the truncation-soundness gap (#12). Part B's P2 *body-free inventory* (route-template collapsing from `url`/`method`/`status`, grouping by `loaderId`) is the genuinely shippable, invariant-safe core; the schema/ER/lineage tiers should be re-scoped around the body-sampling dependency rather than described as pure `lib/` passes over data "already in the store."
