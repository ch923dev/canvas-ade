=== COMPLETENESS GAP LIST (prioritized) ===

The proposal is thorough on the JSON-tree mechanics and the data-flow vision, but it under-specifies the **body-type matrix**, several **error/edge states**, and the **interaction with existing panel chrome**. Below, P0 = a senior reviewer would block on it; P1 = should be in scope before P2/P3; P2 = acknowledge-and-defer is acceptable.

**P0 — blocking gaps**

1. **Non-JSON body types are named (M6) but never designed.** NDJSON, SSE (`text/event-stream`), GraphQL (JSON envelope with `errors[]`/`data`), `multipart/form-data`, `application/x-www-form-urlencoded`, and XML/HTML are all bucketed into "fall back to raw" — which re-inflicts every H-class eyesore on body types that are common in the exact apps this targets. *Rec: define a `detectBodyKind(mime, body)` dispatcher in `osrJson.ts` (the same module that already owns `looksJson`); minimum-viable: NDJSON/SSE → split into per-record/per-event sub-trees reusing `JsonView`; form/urlencoded → the existing `.bb-net-kv` table; GraphQL → tree with `errors[]` surfaced first. XML/HTML can stay raw in P0 but must be an explicit, labeled decision, not silent fallthrough.*

2. **The truncated-body case is asserted as solved by "Raw fallback" but the >5MB tree is left unspecified.** M9's fix is "parse-fail → Raw," but a 5MB *valid-prefix* body that is merely cut at the cap will often still `JSON.parse`-fail, dumping the user back into the unreadable `<pre>` precisely on the largest, most-needing-structure payloads. *Rec: add a streaming/lenient tokenizer path that builds the flat-row model up to the truncation point and renders a real `…(truncated at 5MB)` terminal row, rather than all-or-nothing parse. Make "tree of a truncated body" a P0 acceptance criterion, not a P1 nicety.*

3. **Empty / 4xx-5xx / error-response bodies have no design.** The proposal optimizes the happy 200-with-JSON path. There is no treatment of: empty body (204/304), HTML error pages returned with a JSON `Accept`, a 422/500 JSON error envelope (which the user most wants to read fast), or `loadingFailed` records where `getResponseBody` itself errors. *Rec: specify the four states — empty ("No body"), body-fetch-failed (distinct from truncated), non-2xx JSON (still a tree, but surface the status prominently), and HTML-in-JSON-slot — each with its own affordance. The detail pane should never render a blank or a raw error string with no label.*

4. **Duplicate keys and big-integer precision are half-covered.** Big-number raw-source display is well-specified, but **duplicate object keys are silently lost** the moment any path goes through `JSON.parse` (P0's flat-row build, schema inference, *and* the Raw "re-stringify" alternative all collapse `{"a":1,"a":2}`). A network inspector showing a different payload than the wire is a correctness bug. *Rec: the flat-row tokenizer must be source-driven (walk the original string, not `JSON.parse` output) so duplicate keys and key order survive; add a "duplicate key" badge. This also subsumes the big-number fix and makes the Raw mode lossless by construction.*

5. **Request payloads are mentioned (the `BodyBar` request-payload site) but not differentiated.** The proposal swaps both call sites to the same `<JsonView>`, yet request bodies have distinct shapes — form-encoded posts, multipart uploads with binary parts, GraphQL operation+variables — that the response-tuned viewer will mis-handle. *Rec: confirm `detectBodyKind` runs on request bodies too, and that multipart request parts (file uploads) degrade to a parts-list, not a base64 dead-end.*

**P1 — should be scoped before the data-flow tiers**

6. **WebSocket message JSON is entirely unaddressed.** `WsRecord.frames[]` is captured (16KB cap) and the inventory sketch shows `~frames`, but there is no per-frame JSON viewer — WS frames are overwhelmingly JSON, and today they hit the same raw `<pre>`. *Rec: route each text frame through `JsonView` in the WS detail subtab; this is near-free once the viewer exists and is a glaring omission for a "data flow" tool.*

7. **Keyboard + screen-reader accessibility is specified for the tree but not the whole surface.** ARIA tree + arrow-key contract + `aria-activedescendant` is good, but missing: focus order *between* the row list, the subtabs, the search box, and the new tree; an SR-announceable live region for "loaded N rows / truncated / N matches"; and whether `Ctrl/Cmd+G` collides with any existing panel/browser-board shortcut. *Rec: add a focus-management + shortcut-collision section spanning the panel, not just the tree component; verify the search keymap against the Browser board's existing key handling.*

8. **Theming for non-JSON value affordances is undefined.** Option A's palette covers JSON token types, but says nothing about how NDJSON record separators, SSE event/`id:`/`retry:` fields, form-key/value pairs, the duplicate-key badge, the truncation row, or the PII/`[reveal]` chip are themed within the one-accent rule. *Rec: extend the palette spec to every affordance the body-type matrix introduces, and confirm none of them reach for `--ok/--warn/--err` decoratively (the exact tension the proposal flags for Option B).*

9. **Data-flow handling of auth headers/secrets is named but the high-cardinality-route failure mode is under-tested.** Scrubbing is well-designed; route-template collapsing has an "editable example escape hatch," but there's no specified behavior when collapsing is *wrong* in the dangerous direction — e.g. a real REST resource like `/api/v1` vs `/api/v2` collapsed to `/api/{param}`, or query-param-keyed routes (`?page=`, GraphQL single-endpoint `POST /graphql` carrying 50 operations) producing one mega-node. *Rec: specify GraphQL-operation-name as the grouping key for single-endpoint APIs, and add a guardrail/acceptance test for under-collapse (v1/v2) and over-collapse (1000 ids), since a wrong inventory is worse than none.*

**P2 — performance & chrome interaction (acknowledge; some defer-OK)**

10. **Thousands of captured requests is bounded at capture but not at render.** `MAX_RECORDS=1000`/board is the ring cap, but the proposal's perf work is entirely about *one big body*; the **row table itself** at 1000 rows × multiple boards, plus the data-flow graph at hundreds of nodes, is never virtualized or perf-budgeted. The vision explicitly aggregates "thousands of captured requests" (G1) yet the cap is 1000/board. *Rec: state the row-list virtualization status and the graph node-count budget (with the focus-subgraph default as the mitigation), and reconcile the "thousands" framing against the 1000-ring reality.*

11. **Interaction with the existing dock / resize / full-view is assumed, not specified.** The proposal says the viewer "reuses dock/resize/full-view for free," but the new tree introduces its own scroll container, virtualizer (which needs a measured height), and a maximize affordance for H4 — all of which fight the panel's existing `[0.15,0.85]` drag-resize and the two-dock responsive column-hiding. Open questions: does the virtualizer re-measure on dock switch and on drag-resize (continuous reflow vs. settle)? Does the tree's own maximize duplicate or conflict with the panel's full-view control? Does the right-dock's column-hiding apply to the data-flow tab? *Rec: add an explicit "viewer ↔ panel chrome" contract: virtualizer height = observed `.bb-net-dbody` size via ResizeObserver, re-measure on dock/resize/full-view transitions (debounced to the existing settle), and pick ONE maximize affordance (reuse the panel's, don't add a second).*

12. **No mixed-content / encoding edge cases.** `Content-Encoding: gzip/br` (does `getResponseBody` return decoded?), non-UTF-8 charsets, and BOM-prefixed JSON (`\uFEFF` breaks the `/^\s*[{[]/` heuristic) are unhandled. *Rec: strip BOM before the `looksJson` test; document the CDP decode assumption; treat undecodable charsets as the binary dead-end with a clear label.*

**Cross-cutting note:** gaps 1, 2, 4, and 5 all point at the same root cause — **the design is built on `JSON.parse`/`JSON.stringify` semantics, but a wire-faithful network inspector must be source-string-driven** (duplicate keys, big ints, truncation, key order, non-JSON kinds). Adopting a single lenient, source-preserving tokenizer in `osrJson.ts` as the P0 spine (instead of `JSON.parse` + a separate big-number patch) collapses four separate gaps into one architectural decision and is the highest-leverage change to the proposal.
