## Full refutation detail

### U1. Provider response body fully buffered before size cap (unbounded memory)
- **Location:** `src/main/llmService.ts:201`  ·  **Slice:** M-LLM-MEM  ·  **Verdict:** CONFIRMED (out of scope)  ·  **Severity if it were in scope:** Low
- **Why not in the queue:** CONFIRMED-but-out-of-scope. Faulty path: llmService.ts:186 `const res = await deps.fetch(...)` → line 201 `return parseResponse(config.provider, await res.json())` buffers the entire body into memory and JSON.parses it; the only cap (sanitizeSummary `.slice(0, MAX_OUTPUT_CHARS)`, summaryLoop.ts:66) runs AFTER parse on the already-extracted text. Grep of llmService.ts for content-length/headers/response-size = "No matches found" — confirms no receive-time guard. The unboundedness is acknowledged on purpose: summaryLoop.ts:30-37 ("a local/openrouter endpoint with no server-side max_tokens ... can return a multi-megabyte completion") and the BUG-016 test summaryLoop.test.ts:316-358. Bounding factors: provider gating at llmService.ts:174/178 (opt-in + key/baseUrl required), loopback enforcement at lines 76-77, and the 30s abort at lines 27 + 183-190 + 203 cap the worst case well short of a real main-process OOM. Manifests, if at all, only as transient main-process memory inflation when a user's own non-conforming local LLM server streams a huge completion during a Tier-2 summarize — a refinement, not a defect.

### U2. file:writeText coerces a non-string body to empty string and reports success (silent truncation)
- **Location:** `src/main/fileIpc.ts:146-155`  ·  **Slice:** R-FILETREE  ·  **Verdict:** UNCONFIRMED (out of scope)  ·  **Severity if it were in scope:** Low
- **Why not in the queue:** REFUTED — unreachable in shipped app. fileIpc.ts:148 types args as {path:string;text:string}; the only caller FileBoard.tsx:270 passes textRef.current, which derives from `const [text,setText]=useState('')` (FileBoard.tsx:98), updated only via setText(content) from `await window.api.file.readText(path)` (FileBoard.tsx:230,236 — readText returns Promise<string> per fileIpc.ts:88) and CodeMirror onChange={setText} (FileBoard.tsx:608) — both always strings. preload index.ts:532-533 declares `writeText: (path: string, text: string)`. No non-string ever reaches args.text. The same defensive coercion is an established house pattern (clipboardIpc.ts:43: `deps.writeText(typeof text === 'string' ? text : '')`), confirming it is intentional normalization, not an oversight. No repro possible without inventing a non-existent caller.

### U3. Auto-update enable not coupled to signing-secret presence (unsigned-auto-update RCE invariant unenforced)
- **Location:** `.github/workflows/production.yml:95`  ·  **Slice:** CONFIG-SUPPLY  ·  **Verdict:** UNCONFIRMED  ·  **Severity if it were in scope:** Low
- **Why not in the queue:** Refutation: (1) production.yml:95 `ENABLE_AUTO_UPDATE: ${{ vars.AUTO_UPDATE == '1' && '1' || '' }}` is the ONLY ENABLE_AUTO_UPDATE assignment (grep over .github/ confirms) — factually as candidate states. (2) The real invariant is compiler-enforced, not workflow-enforced: ADR 0008 docs/decisions/0008-packaging-signing-and-auto-update-gate.md:28-31 "In an unsigned build the constant is false and esbuild dead-code-eliminates the entire updater path out of the bundle ... The security invariant is therefore enforced by the compiler, not by convention" and :43-44 "there is no window in which an unsigned build can auto-update: the code that would do so does not exist in unsigned bundles." Backed by autoUpdate.ts:63 `if (!enabled || !isPackaged) return`. (3) The secret-coupling the candidate demands is documented as deliberate operator procedure, not an automatic invariant: releasing.md:69 "## Enabling auto-update (do this LAST, after signing is verified)", releasing.md:85-86 "set repo variable AUTO_UPDATE = 1 ... after confirming signing works", and ADR 0008:41 "To go signed + auto-updating: add the signing secrets, then set AUTO_UPDATE=1 *after* verifying signing." (4) Unreachable now: no CSC_LINK/WIN_CSC_LINK secrets exist (Phase 5 packaging is the still-open release blocker per CLAUDE.md Status), and staging.yml never sets ENABLE_AUTO_UPDATE at all. This is a defense-in-depth hardening nit on a deliberately-manual operator switch, not a confirmed defect.

### U4. SCA gate (pnpm audit) runs only on PRs — absent from staging (main push) and production (release) builds
- **Location:** `.github/workflows/production.yml:29-33`  ·  **Slice:** CONFIG-SUPPLY  ·  **Verdict:** UNCONFIRMED (out of scope)  ·  **Severity if it were in scope:** Low
- **Why not in the queue:** Refuted by `.github/dependabot.yml` lines 8-11 (npm ecosystem, `schedule: interval: weekly`) + its header comment lines 2-4 ("Pairs with the `pnpm audit` SCA step... audit flags advisories, dependabot opens the bump PRs that clear them"): the locked dependency tree IS continuously re-checked against the advisory database by scheduled Dependabot, not solely by the per-PR `pnpm audit` in `.github/workflows/pr.yml` lines 32-33. `.github/workflows/codeql.yml` lines 10-12 (`schedule: cron '0 6 * * 1'`) confirms the repo's deliberate weekly scheduled-scan model for newly-published advisories against unchanged code. The candidate's claim that the production check job (`production.yml` lines 28-33) and staging check job (`staging.yml` lines 26-32) lack a `pnpm audit` step is factually correct, but the inference that this leaves the locked tree with "zero SCA gate" / "never re-checked" before release is false given the weekly Dependabot SCA. No reproduction of a known-vulnerable dependency shipping unflagged is possible because GitHub's lockfile-driven security alerts surface such an advisory regardless of CI workflow steps.
