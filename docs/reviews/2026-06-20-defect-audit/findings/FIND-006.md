# FIND-006 — Cap-rejected command task re-queues with no re-pump and hangs when all cap slots are held by completed (still-open) worker boards

| | |
|---|---|
| **Severity** | Medium |
| **Category** | concurrency · liveness |
| **Status** | CONFIRMED (independently re-verified) |
| **Primary location** | `src/renderer/src/canvas/boards/command/useCommandDispatch.ts:188-197` |
| **Discovery slice** | R-COMMAND (run 2) |

## Summary
On a MAIN spawnGroup cap rejection, runDispatch sets the task back to 'queued' and DELIBERATELY does not pump (comment: 'WAITS for a real settle to pump it'). But the only pump triggers are: a command-task settle (done/failed, lines 187/195), confirmConfig (229), retry (244), and a 'gone' failure (263). MAIN's cap (mcpLifecycle.ts `tracked.size`) counts orchestrator-spawned boards that remain tracked AFTER a command task settles to done — they are only released by MAIN's idle reaper (after idleTtlMs) or a user close. The renderer's inFlightSlots only counts routing/executing tasks, so renderer thinks 0 in-flight while MAIN is at cap=4. A newly dispatched task is then cap-rejected → re-queued → no pump. When MAIN's reaper later frees a slot (or the user closes a board), there is NO renderer subscription that re-pumps, and no other command task is in flight to fire a settle-pump. The re-queued task stays 'queued' forever until the user happens to dispatch/retry something else.

## Trigger
Dispatch and let 4 orchestrator tasks complete to 'done' (their worker boards stay open/tracked in MAIN past settle). Dispatch a 5th task while the 4 boards are still tracked but idle: MAIN rejects with a cap error, the task re-queues. The MAIN reaper eventually reaps the idle boards (freeing slots) but emits no renderer pump; the 5th task remains stuck 'queued' with no in-flight task to ever trigger a pump.

## Evidence / concrete faulty path (code-grounded)
Concrete stuck-state repro (code-argument trace): (1) Dispatch+confirm 4 terminal-only tasks; each completes to `done` (useCommandDispatch.ts:186) and its worker terminal stays live on the canvas → MAIN `tracked` holds 4 entries (mcpLifecycle.ts:128/190, removed only by closeBoard/reconcile/reapIdle). (2) Renderer now sees inFlightSlots=0 (commandDispatch.ts:102-104 filters status==='routing'||'executing'). (3) Dispatch a 5th task → confirmConfig pumps (useCommandDispatch.ts:229) → canDispatch passes 0+1<=4 → setTaskStatus 'routing' → runDispatch → api.spawnGroup. (4) MAIN: `if (tracked.size + memberCount > cap) throw … cap reached` = 4+1>4 → throws (mcpLifecycle.ts:170-174). (5) runDispatch: `if (isCapError(err)) { useCommandStore.getState().setTaskStatus(id, 'queued') }` with NO pumpRef.current?.() (useCommandDispatch.ts:191-192). (6) No in-flight command task remains, and pump is only invoked at lines 187/195/229/244/263 (grep-verified) — none watches MAIN slot-freeing or the reaper. (7) Reaper (mcp.ts:153-155, REAP_INTERVAL_MS=60s; reaps after 60s silence + 5min TTL) eventually frees MAIN slots but emits no renderer pump. Result: the 5th task is stuck `queued` until the user manually dispatches/retries/configures something else. Refutation checks failed: no closeBoard-on-done in the renderer; no board-list subscription re-pumps (CommandBoard.tsx has no useEffect); the unit test useCommandDispatch.test.tsx:171-184 asserts only the re-queue, never recovery.

## Verifier reasoning (why CONFIRMED; scope & severity)
The candidate's mechanism is accurate and I reproduced the faulty path by code argument. Renderer slot accounting (commandDispatch.ts:101-105 `inFlightSlots`) counts ONLY tasks in `routing`/`executing`; a `done` task contributes 0. MAIN's cap (mcpLifecycle.ts:170 `tracked.size + memberCount > cap`) counts ALL spawned worker boards, and a `tracked` entry is removed only by closeBoard (user close), reconcile (user-closed board gone from mirror), or reapIdle — which only reaps after ~6 min of dormancy (MCP_IDLE_ACTIVITY_MS=60s output-silence to count idle + MCP_IDLE_TTL_MS=5min, mcpRegistry.ts:13-34). Nothing in the renderer closes a worker board when its command task settles to `done` (verified: no close/reap call in canvas/boards/command/* except the `gone` failure path and Cancel-dialog). So after 4 terminal-only tasks settle to `done`, renderer inFlightSlots=0 but MAIN tracked.size=4. A 5th dispatch passes the renderer's canDispatch (0+1<=4 true, commandDispatch.ts:117) → pump sets routing → runDispatch → spawnGroup → MAIN rejects (4+1>4, mcpLifecycle.ts:170-174 throws 'cap reached') → runDispatch catches isCapError and sets the task back to `queued` with NO pump (useCommandDispatch.ts:191-192). I exhaustively grepped pumpRef.current?.() and it fires at ONLY lines 187, 195, 229, 244, 263 — a command-task settle (done/failed), confirmConfig, retry, and a `gone` board failure. None subscribes to MAIN slot-freeing, the reaper, or canvas board changes (CommandBoard.tsx has no useEffect/pump call; it only recomputes the display `pool`). With no in-flight command task, no settle-pump can ever fire; the MAIN reaper later frees the slot but emits no renderer pump. The 5th task stays `queued` indefinitely. The code comment at lines 189-190 ('WAITS for a real settle to pump it') encodes the flawed assumption that the slots are held by in-flight tasks — false when they are held by MAIN `tracked` entries from already-`done` workers. The unit test at useCommandDispatch.test.tsx:171-184 confirms the re-queue-without-recovery behavior (asserts only the `queued` state). This is a genuine correctness/logic defect, not a perf/a11y/style item, so it is in scope. Rated Medium (not High): it is a narrow window requiring all slots held by completed-but-still-open worker boards plus a fresh dispatch landing in that window; the task remains visibly `queued` (a recovery affordance), and any subsequent dispatch/retry/configure fires a pump and self-heals it; no crash, data loss, or security weakening.

## Fix direction (audit only — NOT applied)
When a cap-rejected task re-queues, register it to be re-pumped when a slot frees (pump on worker settle / board close), so it cannot hang indefinitely when all cap slots are held by completed-but-still-open worker boards.
