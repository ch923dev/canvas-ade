# BUG-014: Schema drift-guard test hardcodes literal `9`, so it cannot detect the drift it guards

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/projectStore.ts`, `src/main/projectStore.test.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The test meant to enforce the MAIN↔renderer schema lock-step asserts the on-disk fresh-doc fields against the **hardcoded literals** `schemaVersion: 9` / `minReaderVersion: 9` rather than against the imported `boardSchema.SCHEMA_VERSION`/`MIN_READER_VERSION`. Because MAIN's `projectStore.ts` also hardcodes 9, the test stays green even though the renderer constant is now 10 — the guard is silently ineffective. Asserting two copies of the same stale literal can never catch divergence. This is **why BUG-013 shipped undetected**.

## Where
- `src/main/projectStore.test.ts:61, 78, 81` — `expect(onDisk.schemaVersion).toBe(9)` / `expect(onDisk.minReaderVersion).toBe(9)` (literal assertions; the file never imports `boardSchema`)
- `src/main/projectStore.ts:31, 37` — MAIN also hardcodes 9
- The correct cross-module guard pattern already exists in-repo: `src/main/llmModels.lockstep.test.ts` imports BOTH MAIN + renderer constants and asserts deep-equality "so a one-sided edit fails CI"; `assetExtsParity.test.ts` (S11b) does the same for `ASSET_EXTS`
- ADR `docs/decisions/0007-schema-forward-compat.md:64-65` documents the prior v8→v9 occurrence of this drift class

## How it triggers
Bump `boardSchema.SCHEMA_VERSION` (renderer) without bumping `projectStore.ts` (MAIN). The suite stays green because it only checks the MAIN literal equals 9, not that MAIN equals the renderer's current value. (Currently live: renderer 10 vs MAIN 9, test green.)

## Verification evidence
High confidence; could not refute. `grep import.*boardSchema` across `src/main/**/*.test.ts` returns **zero** matches — no cross-module schema-version guard exists anywhere. The additive nature of v10 lowers the blast radius but doesn't refute the claim, which is about *test ineffectiveness*: the test is provably unable to catch the drift, and the drift is concretely present while the test is green.

## Suggested fix direction
Make `projectStore.ts` import the authoritative versions from the renderer's `boardSchema` (or, mirroring `llmModels.lockstep.test.ts`, keep the MAIN duplicate but add a `src/main` test that imports BOTH `boardSchema.SCHEMA_VERSION`/`MIN_READER_VERSION` and the `projectStore` constants and asserts equality). Then assert the on-disk fresh-doc values against the imported renderer constant rather than a literal, so a one-sided bump fails CI. Separately bump MAIN's `SCHEMA_VERSION` 9→10 to clear the current live drift (BUG-013).
