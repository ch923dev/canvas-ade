# BUG-009: `write_result` stores worker summary/refs verbatim with no length bound

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/main/mcpOrchestrator.ts`, `@expanse-ade/mcp` package source (`src/server/tools/writeResult.ts`)
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`writeResult` copies `result.summary`/`result.refs` straight into the MAIN-side results Map via `registry.recordResult` with **no length/array-size cap**. The board-snapshot IPC (`MAX_FIELD_LEN=256`) and the audit log (`MAX_LONG=100_000`) both cap untrusted input, but the `write_result` payload is uncapped at both the package schema (`z.string().optional()`, `z.array(z.string())` — no `.max`) and the orchestrator. Under the project's lethal-trifecta model (worker output is tainted), an adversarial/buggy worker can push a multi-MB summary/refs blob into MAIN memory for its own board, re-emitted uncapped through `canvas://board/{id}/result`.

## Where
- `src/main/mcpOrchestrator.ts:519-529` — `writeResult` copies `summary`/`refs` verbatim → `registry.recordResult`
- `src/main/boardResults.ts:21-23` — `results.set(id, result)` (uncapped store)
- `@expanse-ade/mcp` `dist/index.js:475-477` — tool schema `summary: z.string().optional()`, `refs: z.array(z.string()).optional()` (no `.max`)
- Contrast capped siblings: `boardRegistry.ts:153-182` (`MAX_FIELD_LEN`/`MAX_BOARDS`), `auditLog.ts:56-83` (`MAX_SHORT`/`MAX_LONG`)

## How it triggers
A worker-tier session calls `write_result` with a multi-megabyte `summary` or a huge `refs` array; the orchestrator records it unbounded into `boardResults`' Map (one entry per board, overwritten on re-write, pruned only on project open/save/switch).

## Verification evidence
High confidence; not refuted. **Severity stays Low** (verifier did not raise it): growth is bounded in *count* — one entry per board, overwritten on re-write, pruned via `pruneBoardResults` (BUG-035) on every project open/save/switch — so it can't accumulate unboundedly; the cap is ~one fat entry per live board (≤500). It's a self-board write (a worker can only bloat its **own** token-bound board), runtime-only (never persisted to `canvas.json`), on a single-user desktop. So it's a defense-in-depth consistency gap / minor self-DoS + memory-amplification vector — exactly Low.

## Suggested fix direction
Mirror the sibling untrusted-input caps. Cheapest/most authoritative at the package schema (`writeResult.ts`): `.max(…)` on `summary` (e.g. `MAX_LONG=100_000` like the audit log) and bound `refs` with `.max(N)` on the array + per-element `.max(256)`. Add a belt-and-suspenders clamp in `mcpOrchestrator.ts:524-528` (slice summary, slice+cap refs) so MAIN doesn't rely solely on the external package. Add a unit test asserting an oversized payload is truncated.
