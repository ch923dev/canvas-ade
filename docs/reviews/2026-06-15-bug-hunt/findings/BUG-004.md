# BUG-004: `did-navigate-in-page` never clears the `failed` latch → SPA stuck on `load-failed`

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/renderer/src/canvas/hooks/usePreviewEvents.ts`, `src/renderer/src/canvas/boards/useOffscreenPreview.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`did-navigate` sets `e.failed = true` on an error response code (0 or ≥400). The **only** place `e.failed` is cleared is `registerLoadLatch`'s `did-start-navigation` for a main-frame load. An in-page navigation (hash/`pushState`) does **not** fire `did-start-navigation`, so after a page commits a ≥400 response (failed latched) and then client-side-routes to a working view, the latch stays `true`: the next `did-finish-load` is suppressed and `registerCrashReadyGate` keeps `ready` false — the native view can stay hidden / stranded on `load-failed` for content the SPA has actually rendered. Only a full main-frame reload recovers it.

## Where
- `src/main/preview.ts:457-480` — `did-navigate` sets `e.failed = true` on error code; emits terminal `did-fail-load` for ≥400
- `src/main/preview.ts:481-490` — `did-navigate-in-page` handler re-emits only `did-navigate`; touches neither `e.failed` nor `e.ready`
- `src/main/preview.ts:235-239` — `registerLoadLatch` `did-start-navigation` (the sole latch reset; main-frame only)
- `src/main/preview.ts:242, 290-294` — `did-finish-load` gated on `!latch.failed`; `registerCrashReadyGate` returns without `ready` when `isFailed()`
- `src/renderer/src/canvas/hooks/usePreviewEvents.ts:102-111` & `useOffscreenPreview.ts:43-48` — `did-navigate` updates url/back/fwd but never changes status

## How it triggers
A previewed SPA whose dev server returns a 4xx/5xx on the initial document (carrying the SPA shell), then routes via `history.pushState` to a working route. The board doesn't promote back to `connected` and the native layer isn't re-shown until a full main-frame reload.

## Verification evidence
Confirmed (medium confidence). Three-step trace: (1) `did-navigate(404)` → `e.failed=true` → renderer `load-failed`; (2) `did-finish-load` for the 404-bodied page → `registerCrashReadyGate` sees `isFailed()` → returns without `ready`, view stays hidden, no `connected` emit; (3) `did-navigate-in-page` → emits only `did-navigate` → status stays `load-failed`, `e.failed`/`e.ready` unchanged. **Narrow & partly by-design** (treating a 4xx-bodied document as failed is arguably correct, and this app previews the user's own localhost which usually returns 200 + catch-all route) — hence Low; a maintainer could legitimately mark WONTFIX with a documented rationale.

## Suggested fix direction
In the `did-navigate-in-page` handler, if the in-page nav commits a non-error in-app route, clear the failure symmetrically with the main-frame path: reset `e.failed = false` and emit a recovery signal so the renderer lifts `load-failed`; re-promote `ready`/visibility in `registerCrashReadyGate`. Mirror any change in **both** consumers (`usePreviewEvents.ts`, `useOffscreenPreview.ts`). Alternatively, if a 4xx-bodied page is intentionally "failed," document the decision and decline.
