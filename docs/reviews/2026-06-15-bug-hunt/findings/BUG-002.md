# BUG-002: `handoff_prompt` / `wait_for_*` barriers can never settle for a live agent

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/mcpOrchestrator.ts`, `src/main/mcpRegistry.ts`, `src/renderer/src/store/boardStatus.ts`, `src/renderer/src/store/terminalRuntimeStore.ts`, `src/renderer/src/canvas/boards/terminal/useTerminalSpawn.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`awaitHandoffSettled` resolves only when `deriveStatus(board)` leaves `'running'`. But a live terminal's status is **permanently `'running'`** for the whole life of the PTY â€” there is no per-task `runningâ†’idle` transition. So after a handoff write lands, the await never sees idle: it blocks the full `handoffTimeoutMs` (default `MCP_IDLE_TTL_MS` = 5 min) and records `'timed_out'`, returning whatever (usually empty) result `readResult` holds. The same dependency (`isSettled = status !== 'running'`) breaks the package barriers `wait_for_idle` / `wait_for_all`.

## Where
- `src/main/mcpOrchestrator.ts:82-109` â€” `awaitHandoffSettled` (resolves only on non-`running`)
- `src/main/mcpOrchestrator.ts:454-474` â€” `handoffPrompt` (audits `timed_out`, returns empty `readResult`)
- `src/main/mcpRegistry.ts:181-183` â€” `deriveStatus` returns the renderer mirror bucket, `'running'` for a live session
- `src/renderer/src/store/boardStatus.ts:41-42` â€” terminal â†’ `terminalRunning ? 'running' : 'idle'`
- `src/renderer/src/store/terminalRuntimeStore.ts:21` + `terminalState.ts:33` â€” `isRunning = state === 'running'`
- `src/renderer/src/canvas/boards/terminal/useTerminalSpawn.ts` â€” the only non-`running` setStates are `spawn-failed`/`exited`/`spawning`/`idle`(mount-time only); none ties to an agent *finishing a command*. `awaiting-input` is RESERVED and never emitted.

## How it triggers
Orchestrator calls `handoff_prompt` on any normal terminal whose agent (e.g. `claude`) stays alive after finishing the task. The board's `running` bucket stays `true` until the PTY exits/spawn-fails. `awaitHandoffSettled`'s immediate check returns null (running), the status stream only re-emits `running`, and only the one-shot backstop fires â†’ `'timed_out'`.

## Verification evidence
High-confidence end-to-end trace. The status pipeline has no code path that flips a started, live terminal back to a non-running bucket (`pty.ts:111` `PtyState` has no per-command idle). The green orchestrator tests only pass because they **seed** `status:'idle'` before the call (`mcpOrchestrator.test.ts:651`) or **manually fabricate** `board.status='idle'` + `emitStatus({status:'idle'})` (685-686) â€” production never produces that transition for a live agent, so the tests don't exercise the real path.

## Suggested fix direction
Give the orchestrator a **per-task completion signal distinct from PTY liveness** (non-prescriptive):
- (a) Settle `awaitHandoffSettled` on a recorded `write_result` for the board (a worker reporting its own result is the real "done" marker), or
- (b) Emit a genuine `runningâ†’idle/awaiting` transition from PTY output quiescence (the reserved `awaiting-input` state), or
- (c) Document `handoff_prompt` as fire-then-wait-for-`write_result` and key the barriers off result presence.
