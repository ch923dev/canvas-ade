# BUG-008: Audit-append failure after the PTY write turns a successful dispatch into an error → retry re-runs the command

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/mcpOrchestrator.ts`, `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
In `runGatedWrite` the nonce is consumed and `writeToPty` lands the command **before** the final `'dispatched'` audit. The injected audit closure deliberately **re-throws** on an append failure. So if the audit JSONL write fails (ENOSPC/EACCES on userData) right after the command was written into the shell, `runGatedWrite` rejects, the dispatch tool returns an error even though the command already executed, and the trailing `completed` audit is skipped. A client treating the error as "did not run" and retrying issues a fresh nonce and **re-executes** the command — the nonce guard does not prevent re-execution across separate tool calls.

## Where
- `src/main/mcpOrchestrator.ts:218-228` — `writeToPty` (irreversible side effect) precedes…
- `src/main/mcpOrchestrator.ts:234` — `await audit('dispatched', …)`
- `src/main/index.ts:308-318` — injected audit closure: `.catch((err) => { console.error(...); throw err })`
- `src/main/auditLog.ts:153-164` — `append()` rejects the caller's promise on `appendFile`/`mkdir` failure
- `src/main/dispatchGuard.ts:32-37` — `issue()` mints a fresh UUID per call; no cross-call idempotency
- `src/main/mcpOrchestrator.ts:445-473` — `handoffPrompt`: a throw at line 445 skips `awaitHandoffSettled` and the trailing `completed`/`closed`/`timed_out` audit

## How it triggers
MCP dispatch with the audit log dir unwritable (disk full, permissions): the command is written to the PTY, then `audit('dispatched')` rejects and propagates → the tool reports failure for an already-executed command, with the forensic line missing.

## Verification evidence
Medium confidence; mechanism could not be refuted. No test exercises an audit mock that *rejects after* a successful PTY write (all are `async () => {}`). **Severity-capping mitigations** (not refutations): the trigger is narrow (audit dir must fail precisely on the post-write append), and any retry re-runs through the full gate including a **mandatory human confirm** — so re-execution isn't silent/automatic. Still, the ran-but-reported-failure inconsistency, the missing forensic line, and a human re-approving (reasonably believing the first attempt failed) are real.

## Suggested fix direction
Make the **post-write** audit non-fatal to the dispatch outcome: after `writeToPty` lands, treat a `'dispatched'`/`'completed'` append failure as a logged forensic-gap warning, but let `runGatedWrite`/`handoffPrompt` still resolve with the realised result so the client sees success and doesn't retry. Keep the re-throw only for **pre-write** audit branches (rejected/denied/failed) where no side effect occurred — e.g. a flag distinguishing pre- vs post-side-effect audits in the `index.ts` closure, or have `runGatedWrite` swallow specifically the `audit('dispatched')` rejection after a successful write.
