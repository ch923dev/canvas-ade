# BUG-010: Title-only rename burns a budgeted LLM summarize with byte-identical input

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/memoryEngine.ts`, `src/main/summaryLoop.ts` (+ `memoryEngine.test.ts`, `summaryLoop.test.ts`)
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`boardFingerprint()` includes the board title for terminal/browser boards, justified by an in-code comment claiming "title IS the opening line of `summaryLoop.boardContent`'s LLM prompt (BUG-018)." That premise is **factually wrong now**: `boardContent` emits the literal constant `'Terminal board.'` / `'Browser preview board.'` and never interpolates the title. So a title-only rename changes the fingerprint → arms an intent → runs the budgeted `runSummarize` LLM call → with byte-identical prompt text, producing the same body. This is exactly the wasted autonomous spend + egress the design intends to avoid. (The markdown heading legitimately needs updating on rename, but it's done via a full LLM round-trip instead of a cheap heading swap.)

## Where
- `src/main/memoryEngine.ts:61, 73` — `title: str(b.title)` in the fingerprint (with the now-false BUG-018 justification comment, lines 18-21/59-60/72-73)
- `src/main/summaryLoop.ts:143, 152` — `boardContent` returns constants, **no** title
- `src/main/summaryLoop.ts:237-250` — `buildSummarizeInput` builds from `boardContent` only
- `src/main/summaryLoop.ts:507-637` — `doIntent`: **no** input-equality / cached-summary dedup before `runSummarize`
- Contradictory tests: `summaryLoop.test.ts:125-137` (title EXCLUDED from prompt) vs `memoryEngine.test.ts:96-100` (rename DOES change fingerprint, with the false comment)
- Git: `c28ddbb` (PR #44) removed title from `boardContent`; `e2255b5` (PR #48, BUG-018) re-added title to `boardFingerprint` only, citing the already-deleted interpolation

## How it triggers
Open a project with the summary loop active **and an LLM provider/API key configured**, rename a terminal or browser board (title only, no content). After the debounce the loop fires `runSummarize`, consuming the per-day budget for an identical summary.

## Verification evidence
Confirmed; could not refute. The path is unconditionally reachable for a configured provider (`index.ts:391-393` wires `onIntent → summaryLoop.onIntent`); `doIntent` has no dedup. **Severity corrected Medium→Low:** egress only happens when the user opted in with a key (default no-key path = no spend), it's bounded by the per-day budget, and the input is byte-identical/benign (no secret/data-leak class). It's wasted spend + one redundant egress per rename.

## Suggested fix direction
Make the two halves agree and skip the redundant call. Either **(a)** drop `title` from `boardFingerprint` for terminal/browser (`memoryEngine.ts:61,73`) so a pure rename no longer arms an intent — and fix both contradictory comments + the BUG-018 tests to the title-excluded contract (the smaller, lockstep-restoring change, consistent with `summaryLoop.test.ts`); or **(b)** if the heading must track renames, special-case a title-only change in `doIntent` to rewrite the `# <title>` heading from the cached body without re-summarizing.
