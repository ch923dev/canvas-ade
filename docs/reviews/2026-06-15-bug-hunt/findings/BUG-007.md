# BUG-007: Idle-reaper never reaps a quiescent MCP terminal (the `idle` bucket never occurs)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/mcpLifecycle.ts`, `src/main/mcpRegistry.ts`, `src/main/mcpOrchestrator.ts`, `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`reapIdle` reaps a tracked board only when its derived status is `'idle'` (or undefined) for `idleTtlMs`. But a live terminal is permanently `'running'` (same root cause as BUG-002), so an MCP-spawned terminal that finished its work and is sitting idle at a shell prompt is **never** seen as idle and is never reaped — defeating the runaway-swarm idle-reap for exactly the board type the swarm spawns. Only boards that *exit* (or browser/planning, which read idle/static) are reaped.

## Where
- `src/main/mcpLifecycle.ts:110-161` — `reapIdle`; line 128-131: `const idle = status===undefined||status==='idle'; if(!idle){rec.idleSince=null;continue}`
- `src/main/mcpLifecycle.ts:119-122` — documents the T3.4 "swarm doesn't accrete dormant boards" goal this defeats
- `src/main/mcpRegistry.ts:181` — `deriveStatus` returns `'running'` for any live terminal (see BUG-002)

## How it triggers
`spawn_board('terminal')` via MCP; the agent finishes and the shell stays alive. The reaper interval runs every `REAP_INTERVAL_MS`; `statuses.get(id) === 'running'` forever, so `rec.idleSince` is cleared every sweep and `t - idleSince >= idleTtlMs` never becomes true. Cap-budget reclaim for terminals then relies on the user manually closing the board or the agent exiting.

## Verification evidence
High confidence — flagged `duplicate` of BUG-002's root-cause chain (the never-occurring `idle` bucket), confirmed via the same `deriveStatus`/`boardStatus` trace. This is the *consequence* in the lifecycle reaper, distinct enough to fix as its own card but sharing BUG-002's fix.

## Suggested fix direction
Introduce a real quiescence signal for the reaper instead of the coarse running/idle bucket. MAIN already tracks `lastActivityAt` per PTY session (`pty.ts`) and an `IDLE_AFTER_MS` threshold (`summaryLoop.ts`/`recapFacts.ts`); expose that activity-staleness to the lifecycle reaper (a session-level "no output for N ms" predicate, separate from the human-facing `running` pill) so `reapIdle` measures dormancy by output silence. Keep the human pill unchanged. Add a reaper test that drives the **real** `deriveStatus` + a live-session lookup (not a manual `setStatus` override) for a stale terminal.
