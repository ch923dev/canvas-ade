# BUG-003: Recap-consent modal (& ConfirmModal) don't detach live previews → native view paints over the dialog

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Modal.tsx`, `src/renderer/src/canvas/RecapConsentModal.tsx`, `src/renderer/src/canvas/ConfirmModal.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`RecapConsentModal` (and `ConfirmModal`) render on the shared `Modal` primitive but never register a `previewStore.setMenuOpen` token. The preview manager only demotes a live native `WebContentsView` to its HTML snapshot on camera motion, `nodeGesture || menuOpen`, or static chrome-zone occlusion — a modal opening triggers **none** of these. Because a native view paints **above all HTML** (the locked occlusion invariant), a live Browser board overlapping the centered dialog renders on top of the dialog *and its scrim*: the user can't read/click the consent copy where the preview covers it, and clicks on the covered scrim hit the native view instead of dismissing. This is the inverse of the known recap-consent occlusion class (preview-over-modal, not modal-over-canvas).

## Where
- `src/renderer/src/canvas/RecapConsentModal.tsx:49-58` — renders on `Modal` (zIndex 1000, centered), **no** `setMenuOpen`
- `src/renderer/src/canvas/ConfirmModal.tsx:52-61` — same root cause (zIndex 10000, no `setMenuOpen`)
- `src/renderer/src/canvas/boards/usePreviewManager.ts:758, 780` — `nodeGesture || menuOpen` is the only popover-driven detach trigger
- `src/renderer/src/canvas/SettingsModal.tsx:61-66` — the peer modal that **does** register `setMenuOpen` (the correct pattern)
- `src/renderer/src/canvas/AppChrome.tsx:84` — renders the consent modal when `askRecap && projectDir !== null`

## How it triggers
Open a project whose `canvas.json` has a Browser board near viewport center, with recap consent still `'undecided'`. On load the Browser board's native view becomes live (auto-eligibility, cap ~4, no gesture needed) while the consent effect resolves `getConsent()==='undecided'` and shows the centered modal. The live preview paints over the consent dialog.

## Verification evidence
High-confidence; verifier could not refute. A center-of-viewport board overlaps neither `chromeExclusionZones` reservation (only the top-center dock band + top-right cluster), so `shouldDemoteForOcclusion` returns `false`; `isLiveEligible` has no modal awareness; `reconcile` attaches new eligible boards on load up to `MAX_LIVE=4` (integration test `usePreviewManager.integration.test.tsx:208` asserts a freshly-added board is `live:true`). Esc still dismisses (independent listener) and the preview rarely covers the *entire* dialog — but any overlap occludes copy/buttons, which is the defect.

## Suggested fix direction
Make the shared `Modal` primitive register a `useId`-keyed `previewStore.setMenuOpen(token, true)` on mount / `false` on unmount (mirroring `SettingsModal.tsx:61-66`). Fixing it once in `Modal.tsx` covers **all** modals built on it (RecapConsentModal, ConfirmModal, SettingsModal, and any future one) so overlapping live native views detach to their clippable HTML snapshot while a modal is open. The known few-frame async-detach lag still applies but is acceptable/pre-existing.
