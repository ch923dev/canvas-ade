# BUG-011: Milestone text truncated before `redactSecrets` → boundary-straddling secret prefix can leak

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/agentTranscript.ts`, `src/main/summaryLoop.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`extractMilestones` slices each turn's text to `maxTextChars` (default 600) **before** `redactSecrets` runs (later, in `summaryLoop.buildRecapInput`). Several redaction rules are length-gated (hex/base64 require ≥40 chars; `sk-`/`gh*_`/`Bearer` require ≥16-20). If a high-entropy token begins near offset 600, the slice can cut it below the matching rule's threshold, so the surviving prefix no longer matches any pattern and is sent verbatim to the user's LLM as part of the "secret-scrubbed" recap egress.

## Where
- `src/main/agentTranscript.ts:49` — `text.slice(0, cap)` (cap=600) truncates **before** any redaction; no redact call in this file
- `src/main/summaryLoop.ts:420-425` — `redactSecrets` runs on the already-truncated `m.text`
- `src/main/summaryLoop.ts:93-117` — length-gated patterns (`sk-` ≥16, `gh*_` ≥20, `Bearer` ≥16, hex/base64 ≥40)
- `src/main/index.ts:366-382` — `getAgentMilestones` calls `extractMilestones(…, { maxTextChars: 600 })`

## How it triggers
A claude transcript turn whose text places a long API key / hex blob so that fewer than ~40 (or <16-20 for prefixed tokens) of its characters fall before the 600-char cap; the prefix survives the slice, fails the length-gated redaction, and egresses.

## Verification evidence
Confirmed (medium confidence); order-of-operations is code-provable (truncation strictly precedes the only redaction pass). Concrete repro: a 51-char `sk-proj-<44 random>` key starting at byte ~590 leaves ~10 chars (`sk-proj-XX`) after `slice(0,600)`, failing the `{16,}` requirement → the `sk-` prefix egresses. **Marginal impact** (caps severity, not refuted): only a sub-threshold *prefix* leaks (not a usable credential), the scrubber is documented best-effort, egress is to the user's own consent-gated provider, the transcript is trusted-user-only (not attacker-controllable), and it needs accidental boundary alignment.

## Suggested fix direction
Apply redaction **before** length-sensitive truncation: run `redactSecrets` on the full milestone text inside `extractMilestones` (or on `textFromContent` output before `slice(0, cap)`), so secrets are matched at full length and replaced with the short `[redacted]` token before the 600-char cap. Keep the existing `buildRecapInput` call as belt-and-suspenders, or move it entirely. No schema/contract change needed.
