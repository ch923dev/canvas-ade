# BUG-016: Model backdrop scenes never re-check buffer size in the loop → a dpr change reblurs the animation

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/backdrop/scenes/blossomRiver.ts`, `auroraNight.ts`, `starfieldNebula.ts`, `sunsetOcean.ts`, `snowfallRidge.ts`, `rainyWindow.ts`, `cityLights.ts`, `mistyPines.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The eight model-based backdrop scenes only call `ensureSize()` in `start()`/`renderStill()` and from the `ResizeObserver` callback. Their rAF `tick()` paints directly and never calls `ensureSize()`. The `ResizeObserver` fires only on a change to the canvas element's CSS box, **not** on a change to `window.devicePixelRatio`. `ensureSize()` is the only place dpr is re-read and the buffer rebuilt. So if dpr changes without a CSS-size change, the running scene keeps rendering into the stale-resolution buffer (then CSS-scaled), appearing blurry/aliased until an unrelated element resize. The two **ambient** scenes (`drift`, `current`) call `ensureSize()` every frame and self-correct — so the behavior is inconsistent across the registry.

## Where
- `src/renderer/.../scenes/blossomRiver.ts:590-597` — `tick()` calls only `paint(now/1000)`, no `ensureSize()`; `:563-571` `ensureSize` (sole dpr re-read, `Math.min(dpr, 1.5)`); `:578-588` `ResizeObserver` (element box only)
- Same omission verbatim: `auroraNight.ts:292-299`, `starfieldNebula.ts:309-316`, `sunsetOcean.ts:300-307`, `snowfallRidge.ts:313-320`, `rainyWindow.ts:260-267`, `cityLights.ts:332-339`, `mistyPines.ts:477-484`
- Contrast: `drift.ts:123-131` and `current.ts:198-207` call `ensureSize()` before painting each frame

## How it triggers
Select any model-based scene backdrop, then move the app window from a 1× display to a HiDPI/Retina (2×) display (or change OS display scaling) **without** resizing the canvas element's CSS box. The animation keeps the old buffer resolution and renders blurry; `drift`/`current` would not.

## Verification evidence
High confidence; not refuted. Cosmetic (rendering sharpness), no crash/data-loss → Low.

## Suggested fix direction
Make the model scenes' running loop dpr-aware like the ambient scenes already are: call `ensureSize()` inside `tick()` before `paint()` (it early-returns when w/h are unchanged, so rebuild cost is incurred only on an actual dpr change). Alternatively, mirror the `TerminalBoard` pattern at the host (`BackdropLayer.tsx`) with a `matchMedia('(resolution: Xdppx)')` watcher that re-runs `ensureSize` on the active handle — but per-tick `ensureSize` is the lighter, registry-consistent fix. Rebuilding re-seeds from `mulberry32(SEED)`, so composition stays deterministic.
