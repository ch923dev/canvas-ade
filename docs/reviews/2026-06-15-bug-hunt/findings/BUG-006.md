# BUG-006: Project-dir guard allows ANY absolute path → arbitrary-location dir/file creation

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`isUnsafeProjectDir` rejects only non-string / empty / non-absolute / `..`-containing paths. It does **not** constrain the target to a user-approved location. `project:create` then `mkdirSync(dir, {recursive})` + writes a fresh `canvas.json` + scaffolds `<dir>/.canvas/` for **any** absolute path. The guard's stated purpose (defense-in-depth against a compromised renderer, BUG-033/M-6) fails within its own threat model: a compromised-but-main-frame renderer (which `isForeignSender` allows) can make MAIN create directories and a fixed-shape file at an arbitrary absolute location (e.g. a startup folder, `~/.config`).

## Where
- `src/main/projectIpc.ts:41-53` — `isUnsafeProjectDir` (only `..` + absoluteness checks; no allowlist/containment)
- `src/main/projectIpc.ts:217-233` — `project:create` handler (proceeds after the guard)
- `src/main/projectStore.ts:149-185` — `createProject` → `mkdirSync(dir,{recursive})` + `writeProject` unconditionally
- `src/main/canvasMemory.ts:87-99` — `ensureScaffold` writes `<dir>/.canvas/memory`, `audit`, `.gitignore`
- Test oracle: `projectIpc.test.ts:11-33` confirms `'C:\Users\x\proj'` / `'/home/x/proj'` are accepted

## How it triggers
A compromised/synthetic same-process caller invokes `project:create` (or `project:open`) with an absolute, `..`-free path pointing anywhere writable — the guard passes and MAIN creates dirs + a `canvas.json` there.

## Verification evidence
High confidence; could not refute the create path. **Scope narrowed by the verifier:** `project:open`'s scaffold runs only *after* a readable envelope-valid `canvas.json` already exists at the target, so the *open* path is not a clean arbitrary-creator — but `project:create` has no such gate. Impact is genuinely **bounded** (the written `canvas.json` is a fixed shape, not attacker-controlled; the scaffold writes an empty tree + a `.gitignore` only-if-absent; `createProject` reuses-if-exists rather than clobbering; attacker-controlled asset bytes only go under the already-vetted `getCurrentDir()`). So the ceiling is empty-dir/fixed-file creation at an arbitrary absolute location — Low, and `dir` is classified trusted-user input by the locked design (Round-3 SEC-1) — hence this is hardening-parity, not a primary exploit.

## Suggested fix direction
Complete the defense-in-depth goal: derive approved roots from the OS folder-dialog result and the recents list, and reject any `project:create`/`project:open` `dir` not equal-to-or-under one of them (or require create flows to originate from a same-tick dialog selection). Fail closed (`{ok:false, error:'invalid path'}`). Add a unit test asserting an absolute, non-approved path is rejected.
