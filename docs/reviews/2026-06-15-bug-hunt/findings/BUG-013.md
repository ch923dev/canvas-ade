# BUG-013: MAIN `SCHEMA_VERSION` stuck at 9 while renderer `boardSchema` is 10 (lock-step violated)

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/projectStore.ts` (+ `projectStore.test.ts`)
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`boardSchema.ts` was bumped to `SCHEMA_VERSION = 10` (commit `6a50eaa`, v10 agentKind/monitorActivity), but the MAIN mirror in `projectStore.ts` (`SCHEMA_VERSION = 9`, `MIN_READER_VERSION = 9`) was **not** bumped in lock-step ŌĆö a direct violation of the BUG-024 contract documented at `projectStore.ts:25-31`. `createProject()` therefore writes a fresh `canvas.json` stamped `schemaVersion: 9` instead of 10. Impact is contained (the renderer migrates 9ŌåÆ10 on load and re-stamps on first save; no data loss/crash), but every freshly created project carries a one-version-stale on-disk marker until its first autosave, and external tooling (MCP, scripts) reading `canvas.json` before that save sees the wrong version. `minReaderVersion` staying 9 is correct **by coincidence** (v10 is additive per ADR 0007), not by lock-step intent.

## Where
- `src/main/projectStore.ts:31, 37` ŌĆö `SCHEMA_VERSION = 9`, `MIN_READER_VERSION = 9` (mirror, not bumped)
- `src/main/projectStore.ts:25-31` ŌĆö the self-documented lock-step contract (comment still says "(9)")
- `src/main/projectStore.ts:178-185` ŌĆö `createProject` stamps `schemaVersion: SCHEMA_VERSION` (=9) into the fresh doc
- `src/renderer/src/lib/boardSchema.ts:45` ŌĆö `SCHEMA_VERSION = 10`; `:63` `MIN_READER_VERSION = 9`; `:466` additive `9ŌåÆ10` identity migration
- Git: `git show 6a50eaa --stat` lists `boardSchema.ts` but **not** `projectStore.ts`

## How it triggers
Create a brand-new project (Welcome ŌåÆ Create). MAIN writes `canvas.json` with `schemaVersion:9`. Read the file before the renderer's first autosave (~1s) ŌĆö it shows `9` though the running app is a v10 build.

## Verification evidence
High confidence; could not refute. Drift is live (`git grep` confirms renderer 10 vs MAIN 9). The existing test `projectStore.test.ts:72-84` hardcodes `expect(onDisk.schemaVersion).toBe(9)` ŌĆö it asserts the **stale** value, so it passes and codifies the bug (see BUG-014). The only mitigant ŌĆö renderer self-heal via the additive identity migration ŌĆö is what keeps this Low, not what refutes it. This drift class has shipped before (v8ŌåÆv9 per ADR 0007 ┬¦"#126 missed MAIN").

## Suggested fix direction
Bump `projectStore.ts:31` `SCHEMA_VERSION` to 10 (`MIN_READER_VERSION` stays 9 ŌĆö additive per ADR 0007) and update the lock-step doc comment to reference 10. Update assertions in `projectStore.test.ts` to expect 10. To prevent recurrence, add a parity check (see BUG-014) so the MAIN mirror can't silently drift again ŌĆö it's now been missed at both the v9 and v10 bumps.
