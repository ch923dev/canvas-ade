# BUG-015: `parkTerminal()` IPC voided without `.catch()` → unhandled promise rejection

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Canvas.tsx`, `src/renderer/src/canvas/hooks/useBoardActions.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
On keyboard/`deleteKeyCode` removal of a terminal board, `void window.api.parkTerminal(intent.id)` is fired with no `.catch()`. `parkTerminal` is `ipcRenderer.invoke('pty:park', id): Promise<boolean>`, which rejects if the main handler throws or is gone (teardown race, window closing). With no rejection handler and no global renderer `unhandledrejection` handler in production, it surfaces as an unhandled rejection. The same unguarded call exists on the ⋯-menu/palette delete path.

## Where
- `src/renderer/src/canvas/Canvas.tsx:469` — `if (removed?.type === 'terminal') void window.api.parkTerminal(intent.id)` (no `.catch()`)
- `src/renderer/src/canvas/hooks/useBoardActions.ts:83` — same pattern on the ⋯-menu/palette delete
- `src/preload/index.ts:189` — `parkTerminal = (id) => ipcRenderer.invoke('pty:park', id): Promise<boolean>`
- Contrast the **guarded** sibling: `Canvas.tsx:282-306` wraps `window.api.memory.*` in try/catch, commented "keeps an unexpected rejection from surfacing as an unhandled promise"; tested regressions BUG-031/BUG-065 in `SettingsModal.test.tsx` simulate `'IPC channel gone'` and assert no `unhandledrejection`

## How it triggers
Delete a terminal board (Delete/Backspace or the ⋯ menu) at a moment the main `pty:park` handler throws or has been torn down (window closing / main reload) → the invoke rejects with no catch.

## Verification evidence
Confirmed (medium confidence); could not refute. In normal use `park()` cannot throw (foreign sender → false; `parkCore` fully guarded), so the only trigger is the teardown/channel-gone race — but that path is genuinely reachable and unguarded, and the codebase's own stated discipline (BUG-031/BUG-065 + the `memory.*` catches) treats this class as a bug worth a `.catch()` + regression. The MAIN `process.on('unhandledRejection', crashShutdown)` does **not** apply (renderer rejection doesn't crash MAIN) — so the impact is renderer console noise on a closing window, not a crash. *(Note: the sibling `detachAllPreviews` no-`.catch()` was **unconfirmed** — its handler is guarded and never removed; see `unconfirmed.md`.)*

## Suggested fix direction
Add a `.catch()` to both call sites (`Canvas.tsx:469`, `useBoardActions.ts:83`) that swallows/logs the rejection, mirroring the adjacent `memory.*` guards and the BUG-031/BUG-065 pattern. Optionally extract a tiny shared `fireAndForget(p)` helper so the discipline is uniform across the unguarded `pty:*`/`preview:*` invokes (this would also close the unconfirmed `detachAllPreviews` item). A jsdom regression asserting no `unhandledrejection` on a rejected `parkTerminal` matches the established convention.
