# BUG-001: Recap-map watcher crashes the whole app on a destroyed-but-non-null `mainWindow`

- **Severity:** High
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
After the main window is closed, `mainWindow` holds a **destroyed but non-null** `BrowserWindow` (it is assigned once and never nulled). The recap-map `fs.watch` `onChange` callback reads `mainWindow?.webContents` — optional chaining only guards `null`/`undefined`, so the destroyed window's `.webContents` getter **throws** `"Object has been destroyed"` *before* the downstream `!wc.isDestroyed()` guard can run. The throw escapes a bare `setTimeout` (no try/catch) as an `uncaughtException` → `crashShutdown(1)` force-quits the entire app.

## Where
- `src/main/index.ts:532-543` — `watchRecapMap` `onChange` callback (`const wc = mainWindow?.webContents` then `!wc.isDestroyed()`)
- `src/main/index.ts:125-128` — window `closed` handler (closes previews but never `mainWindow = null` and never `stopRecapWatch()`)
- `src/main/index.ts:683` — `process.on('uncaughtException', err => crashShutdown(1, err))`
- `src/main/agentRecapMap.ts:201` — `onChange` invoked from a bare `setTimeout(...)` with no try/catch
- Contrast the **correct** pattern at `src/main/index.ts:614-620` (`flushRenderer`): `if (!win || win.isDestroyed()) return` *before* touching `.webContents`.

## How it triggers
**macOS (steady state):** `window-all-closed` does not quit on darwin (`index.ts:651`), so closing the window leaves the app alive with the recap `fs.watch` still armed and `mainWindow` destroyed-but-non-null. The next `session-map.jsonl` write (any consented Claude SessionStart append) fires the debounced `onChange` → line 541 throws → whole-app crash-exit(1). **Windows/Linux:** a narrower race in the ~200 ms debounce window between the `closed` event and `before-quit`'s `shutdown()` disposing the watcher.

## Verification evidence
Adversarial verifier confirmed (high confidence), reproduced by code-path trace:
1. `mainWindow` is `let … = null` (index.ts:72), assigned once (index.ts:104); a grep for `mainWindow =` shows it is **never** re-nulled. The `closed` handler (125-128) only disposes previews.
2. The codebase itself documents that `.webContents` on a destroyed window throws (index.ts:614-618), and `index.flushRenderer.test.ts:78-88` models a destroyed window whose `webContents` getter throws, asserting the guard must check `isDestroyed()` *first*.
3. The recap `onChange` (532-543) uses the wrong guard order: optional chaining can't stop the getter from throwing on a non-null destroyed window.
4. `onChange` runs from a bare `setTimeout` (agentRecapMap.ts:201) with no try/catch → async throw → `uncaughtException` → `makeCrashHandler` (quit.ts:41-47) fires fire-and-forget `shutdown()` (kills live PTYs) + `app.exit(1)`. `recapWatcher?.track` (line 539) does not touch `mainWindow`, so it cannot mask the throw.

## Suggested fix direction
In the `onChange` callback, use the canonical destroyed-window guard before touching `.webContents`: `const win = mainWindow; if (!win || win.isDestroyed()) return;` then read `win.webContents` (mirror `flushRenderer` at index.ts:618). Additionally **null `mainWindow` in the `closed` handler** (index.ts:125) so the destroyed-but-non-null steady state disappears for *every* `() => mainWindow` consumer, and/or call `stopRecapWatch()` on window close so the watcher can't fire while no window exists.
