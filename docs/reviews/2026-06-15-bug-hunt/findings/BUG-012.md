# BUG-012: `removeBoard` never reclaims a board's `idleOnMountIds` entry (BUG-033 class, un-fixed delete path)

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/store/canvasStore.ts` (+ `canvasStore.test.ts`)
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`duplicateBoard` and `markRestoredIdle` add terminal board ids to the module-scoped `idleOnMountIds` Set. The **undo()** path was hardened against accumulating dead UUIDs (BUG-033 sweep), but the ordinary **delete** path — `removeBoard` — never calls `idleOnMountIds.delete(id)`. So in the common "duplicate a terminal, then change my mind and delete it" (or delete a restored terminal) flow — without undoing — the clone/restored UUID stays in the Set for the whole session. The Set grows monotonically (one dead UUID per such delete): a session-lifetime memory leak of exactly the kind the BUG-033 comment describes, but only the undo path was fixed.

## Where
- `src/renderer/src/store/canvasStore.ts:582-625` — `removeBoard` (no `idleOnMountIds.delete(id)` anywhere)
- `src/renderer/src/store/canvasStore.ts:364` — `markRestoredIdle` (add), `:649` — `duplicateBoard` (add)
- `src/renderer/src/store/canvasStore.ts:833-836` — `undo()` reclaim (the only delete; parks to `parkedIdleIds`)
- `src/renderer/src/store/history.ts:6` — `HISTORY_LIMIT=50` (the deleted snapshot eventually evicts from `past`, but eviction triggers no reclaim)

## How it triggers
Add/restore a terminal so its id is in `idleOnMountIds` (duplicate a terminal, or load a project with terminals), then delete it via the normal close/delete (`removeBoard`) rather than Ctrl+Z. Repeat across a session — each leaks one UUID.

## Verification evidence
High confidence; could not refute. Full mutation inventory of the Set verified: adds at 364/649, deletes at 357 (`clearIdleOnMount`, only on explicit Start), 362 (fresh load), 835 (undo sweep). **No** mutation in `removeBoard` and **no** GC sweep reconciling the Set against present boards. After a never-undone delete, the snapshot ages out of the 50-entry `past` rail (genuinely unreachable) but nothing reclaims the entry. UUIDs are never reused, so entries are **inert** (can't re-match a future board) — hence Low — but the Set is unbounded for the session and the BUG-033 hardening is incomplete. Test gap: `canvasStore.test.ts` covers undo/redo idle reclaim but has no `removeBoard` idle-reclaim test.

## Suggested fix direction
Reclaim `idleOnMountIds` entries when a board becomes truly unreachable, **without** eagerly deleting inside `removeBoard` (that would resurrect a started agent on undo — the BUG-012/respawn class). Mirror the parked-flag discipline: on `removeBoard`, move the deleted idle id to `parkedIdleIds` (so undo can re-promote it), and extend pruning so parked ids unreachable from **both** the past and future rails are dropped (or GC `idleOnMountIds`/`parkedIdleIds` against `present ∪ past ∪ future` after history-rail eviction). Add a `canvasStore.test.ts` case: duplicate → `removeBoard` directly (no undo) → exhaust history → assert `isIdleOnMount(cloneId)` is reclaimed.
