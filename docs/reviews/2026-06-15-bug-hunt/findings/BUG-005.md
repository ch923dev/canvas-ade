# BUG-005: OSR blocked-scheme URL silently no-ops → board hangs on "Connecting…" forever

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/previewOsr.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
In OSR (offscreen) preview mode, when a Browser board's URL is a non-`http(s)` scheme, `ensureOsr` still creates the hidden offscreen `BrowserWindow` (a live renderer) but `if (isAllowedPreviewUrl(url)) { void wc.loadURL(url) }` simply **skips** the load and emits **no** lifecycle event. The renderer (`useOffscreenPreview`) calls `openOsrPreview` unconditionally and immediately sets status `'connecting'`, then waits for a MAIN event to leave that state. Since none is emitted for the blocked scheme, the board is stranded on the spinner indefinitely while holding an idle `about:blank` offscreen renderer until unmount. The **shipping native path does not have this bug** — `preview:open`/`preview:navigate` emit a synthetic `did-fail-load` (errorCode -1, "blocked scheme") on the rejected branch; the OSR path omitted that `else` entirely.

## Where
- `src/main/previewOsr.ts:266-269` — `if (isAllowedPreviewUrl(url)) { void wc.loadURL(url) }` with **no `else`**
- `src/main/preview.ts:605-616` & `693-703` — the correct native path (`e.failed=true` + emit synthetic `did-fail-load`)
- `src/renderer/src/canvas/boards/useOffscreenPreview.ts:24-29` — `!url` early-return only catches the empty string; sets `'connecting'` with no scheme check
- `src/renderer/src/lib/boardSchema.ts:685` — validates only `typeof b.url === 'string'` (no scheme enforcement)

## How it triggers
With `VITE_PREVIEW_OSR` enabled, set/persist/paste a Browser board URL with a well-formed non-`http(s)` scheme (`file:///C:/x`, `data:text/html,…`, `myapp://x`). A hand-edited/migrated `canvas.json` or a non-URL-bar writer (MCP push, terminal push-to-preview) can set such a `board.url` without the `commitUrl` `isHttpUrl` gate. The board spins on "Connecting…" forever + leaks an idle offscreen renderer until unmount.

## Verification evidence
High confidence; could not refute. The native `usePreviewEvents` resolver that would otherwise rescue the state is disabled in OSR mode (`BrowserPreviewLayer.tsx:17` returns null), so the slim listener in `useOffscreenPreview` is the only consumer — and it never fires for a blocked scheme. There is no renderer timeout fallback out of `'connecting'`. **Scope:** flag-gated experimental spike (`VITE_PREVIEW_OSR` off by default, PR #151), so default production builds don't hit it — which lowers severity but doesn't refute; any OSR-enabled build reproduces it deterministically and the native path proves the intended behavior.

## Suggested fix direction
Mirror the native path's `else` branch in `ensureOsr`: on a rejected scheme, set `e.failed=true` and emit a synthetic `did-fail-load` (errorCode -1, "blocked scheme") so `useOffscreenPreview` transitions to `load-failed`. Optionally do an early scheme check **before** `new BrowserWindow` to avoid the idle-renderer allocation, and/or add a renderer-side `isHttpUrl` guard before patching `'connecting'`.
