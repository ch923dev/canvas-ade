# Unconfirmed Candidates

These were flagged in discovery but the independent adversarial verifier could NOT reproduce or concretely demonstrate them. Listed for transparency; **none are in the active work queue.** Several are noted as cheap optional hardening — do them only if you want defense-in-depth, not because there's a live defect.

| Location | Category | Why it was raised | Why verification failed (refutation) |
|----------|----------|-------------------|--------------------------------------|
| `src/main/preview.ts:505-523` (`attach`) | correctness | Reattach flips `setVisible(true)` before `setBounds` → a stale-position ghost-flash on settle | `setVisible` and `setBounds` are two synchronous, non-yielding native calls in the same JS task within one IPC handler — no `await`/rAF/paint can interleave them, so the first composite already carries the fresh rect. The detach-side hide-before-remove guards a *different*, async hazard. (Optional: reorder bounds-before-reveal as harmless cleanliness.) |
| `src/main/preview.ts:827-841` (`debugCrashView`) | concurrency | `SIGKILL` of the renderer pid could kill a second same-origin preview board | Each preview uses a unique session `partition: preview-<id>` — a hard Chromium renderer-process boundary; cross-partition SiteInstances are never coalesced, so the pid is per-view-unique. Also E2E-only (`CANVAS_E2E`-gated). |
| `src/renderer/.../boards/useOffscreenPreview.ts:73-80` | error-handling | `new ImageData(rgba, w, h)` throws if buffer length ≠ w·h·4 (no validation) | The sole producer (`previewOsr.ts` `emitFrame`) reads `getSize()` + `toBitmap()` synchronously from one immutable `NativeImage`; `toBitmap()` is tightly-packed 4 B/px BGRA (no stride), so `buffer.length === w·h·4` is framework-guaranteed. The DPR-supersample path that could break it is deferred/unimplemented; feature is flag-gated off. |
| `src/renderer/.../boards/terminal/useTerminalWebgl.ts:28-36` | resource-leak | Freed WebGL slot stranded when the chosen waiter's retry declines | Retry-bail is only reachable during a *global* LOD transition that suspends all terminals and clears the whole `wantWebgl` queue; the board re-attaches via the suspend effect on zoom-in. The per-board null-ref bail is unreachable (detach removes from queue before nulling). Slot count always decremented correctly. |
| `src/renderer/.../boards/TerminalBoard.tsx:648-659` (`onDrop`) | correctness | File-drop onto an idle/exited terminal pastes into a dead PTY with no live guard | The `idle` case is unreachable: the opaque `zIndex:2` Start overlay is a *sibling* of the well, so the well's `onDrop` never fires (drop events don't bubble sideways). The `exited` case is a benign silent no-op identical to every other paste path (`isLive` is an async-disposal guard, not a lifecycle guard). |
| `src/renderer/.../boards/planning/usePlanningPointer.ts:241-266` | data-integrity | Lost update: placing a note while an empty note is focused clobbers the auto-prune (BUG-023 closure form) | The joint state (placement tool armed + empty element still focused) is structurally unreachable — every path that arms the note/check tool first blurs the empty element (committing its prune as an isolated step), and each placement forces `setTool('select')`. No single `onWellPointerDown` runs both commits off one stale closure. (Optional: convert to live-read `commit((cur)=>…)` form for consistency.) |
| `src/renderer/.../boards/PlanningBoard.tsx:128-131` (`measuredRef`) | resource-leak | `measuredRef` Map accumulates entries for deleted elements & `__ghost__` alt-drag copies | Real structurally but inert: the map is only ever read via `.get(liveId)`, never iterated wholesale; UUID keys preclude collisions; it's a per-component `useRef` GC'd on board unmount, so growth is bounded to one board's session (single-digit MB worst case). No reachable harmful consequence. |
| `src/renderer/.../canvas/BoardNode.tsx:285` + `Canvas.tsx:575` (`detachAllPreviews`) | error-handling | `void window.api.detachAllPreviews?.()` without `.catch()` → unhandled rejection | The handler is synchronous and fully guarded (`isForeignSender` + destroyed-window early-returns), registered once and never removed, so it doesn't throw/reject in steady state. The only residual reject is a teardown race → a console warning during process death. It's the uniform, accepted fire-and-forget idiom across ~20 preview callsites; flagging two is inconsistent. (Contrast BUG-015, where the same pattern on `parkTerminal` *can* reject because the handler isn't equivalently guarded.) |
| `src/renderer/.../canvas/AppChrome.tsx:52-72` | correctness | Recap consent prompt not re-evaluated after the decision changes in SettingsModal (effect keyed only on `projectDir`) | No reachable stale state: the `RecapConsentModal` scrim (`zIndex 1000`) is mutually exclusive with the Settings gear (`zIndex 50`), so the two consent-writing surfaces can never be open at once; the one-time per-project nudge is decoupled-by-design. Re-evaluating would risk popping the consent modal over Settings (a regression). |

**Note on the two "no-`.catch()`" items:** BUG-015 (`parkTerminal`) is **confirmed** because its main handler path has a genuinely reachable reject (teardown race) and the codebase's own discipline treats that class as a bug; the `detachAllPreviews` variant above is **unconfirmed** because its handler is guarded and the idiom is uniform. If you adopt a shared `fireAndForget()` helper for BUG-015, routing all ~20 preview callsites through it would also close this one as a side effect.
