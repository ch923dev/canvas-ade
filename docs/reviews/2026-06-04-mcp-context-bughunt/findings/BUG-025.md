# BUG-025: Audit log wired AFTER MCP server starts → early dispatch escapes the trail

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/index.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T4.1
- **Collides with:** BUG-026
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
The MCP server (`startMcpServer`) is started at line 168 with an `audit` closure that calls `getAuditLog()?.append(e)`. `registerAuditHandler` — which sets the module-level `registered` variable that `getAuditLog()` returns — is called at line 253, roughly 85 lines later. Any MCP dispatch tool call (handoff_prompt, assign_prompt, interrupt, relay_prompt) that fires between lines 168 and 253 during app boot will silently drop the audit entry (`getAuditLog()` returns `null`, the `?? Promise.resolve()` short-circuits). The JSONL trail will have a gap with no error.

## Where
`src/main/index.ts:188`

## How it triggers
An MCP client connects over loopback and calls a dispatch tool (e.g. `handoff_prompt`) during the narrow boot window between the MCP server binding and `registerAuditHandler` executing. In practice the renderer hasn't loaded yet, but the HTTP server is live and an external client can connect immediately.

## Verification evidence
The ordering flaw is structurally confirmed in `origin/feat/mcp-integration:src/main/index.ts`:

- Line 160: `mcp = await startMcpServer({..., audit: (e) => getAuditLog()?.append(e).then(()=>{}) ?? Promise.resolve(), ...})`
- Line 193: `registerAuditHandler(ipcMain, () => mainWindow, createAuditLog({...}))`

`registerAuditHandler` sets the module-level `registered` variable in `auditIpc.ts` (line 17-29). `getAuditLog()` returns `registered`, which is `null` until `registerAuditHandler` is called. The `?.append(e) ?? Promise.resolve()` pattern silently discards audit entries when `registered === null`.

After `await startMcpServer(...)` resolves, the HTTP server is live and accepting connections (it is already bound). Any MCP dispatch tool call arriving in the gap between line 160 and line 193 would have its audit entries silently dropped. All dispatch paths (`handoffPrompt`, `dispatchPrompt`, `relayPrompt`) call `registry.audit(...)` on every branch — including rejected/denied ones.

**However, severity is lower than claimed (High → Medium):** The gap cannot result in an unaudited PTY write because:
1. `createWindow()` is called at line 195, AFTER `registerAuditHandler` at line 193 — so `mainWindow` is `null` throughout the entire gap. `requestConfirm` (mcpConfirm.ts line 55) immediately returns `{ approved: false }` when `getWin()` returns `null`, making the confirm gate fail-closed. No write is authorized without a confirm.
2. The MCP server port is not published anywhere during startup (no `writeFileSync` of port, no env var, no IPC). An external client cannot discover the ephemeral port during the boot window.
3. The gap is extremely narrow — a few synchronous IPC registrations separating lines 160 and 193, on the order of microseconds.

The actual impact is: if an external client somehow knew the port and connected during this microsecond window, the tool call would be rejected (board not found / confirm denied), and THOSE rejection audit entries would be silently dropped. The JSONL trail would have a gap, but no PTY write would have occurred. The forensic completeness of the audit log is the real victim, not authorization bypass.

## Suggested fix direction
Move `registerAuditHandler(...)` to immediately before `startMcpServer(...)` (or pass the `AuditLog` instance directly into `startMcpServer` so the closure captures a non-null value). Since `createWindow()` is already after `registerAuditHandler`, the fix is a 3-line reorder: create the `AuditLog` instance first, call `registerAuditHandler`, then call `startMcpServer` with the audit closure. This eliminates the gap entirely without touching the confirm-gate or security model.
