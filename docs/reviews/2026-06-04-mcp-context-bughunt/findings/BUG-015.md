## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) Silent summary loss when a slow in-flight call fails while a second intent is pending — `src/main/summaryLoop.ts:149`

- **Where:** `src/main/summaryLoop.ts:149`
- **Sev/cat:** Low / correctness
- **What:** The inFlight guard at line 149 drops any second intent for the same boardId while a first call is running. If the first call fails (no-provider → return at line 169, readProject failure → return at line 155, board deleted → return at line 160), the second intent is silently discarded. The memoryEngine fingerprint map was already updated to the new content, so no future debounce will fire unless the content changes again. The board summary therefore remains stale even though a second summarise was warranted.
- **Trigger:** Sequence: (1) boardId 'X' fires an intent while the LLM provider returns a budget-exceeded or no-provider error; (2) another content change to board X arrives before the first async call resolves (possible at 45 s debounce with a slow network), arming a second intent; (3) the second intent reaches line 149 while the first call's await is suspended; (4) first call returns early at line 169; summary file not written. The second intent was already dropped.
- **Evidence:** Concrete code-path argument:

1. `memoryEngine.ts:153` — `fingerprints.set(id, fp)` runs at `observe()` time, immediately when the second content change arrives. The fingerprint is advanced to the new value at this point, regardless of any in-flight call.

2. `memoryEngine.ts:155` — `armDebounce(id)` is called, scheduling the second intent to fire in 45s.

3. `summaryLoop.ts:149` — When that second debounce fires, `if (inFlight.has(boardId)) return` silently drops it if the first call is still running.

4. `summaryLoop.ts:181` — `inFlight.delete(boardId)` only runs in the `finally` block AFTER the first call resolves/fails.

5. After the second intent is dropped and the first call fails (e.g., slow network hang → provider error at line 169, or slow budget check), no re-arm occurs. `fingerprints` already has the new value, so future `observe()` calls with the same content see `prev === fp` and do NOT call `armDebounce`. The board summary is permanently stale for that content snapshot.

The viable trigger path is: first LLM call is slow (network hang, e.g., provider timeout > 45s), content changes during that window arming a second debounce, second debounce fires while first await is still suspended, first eventually returns error. The "no-provider" (synchronous-ish) and "readProject failure" paths are NOT practically vulnerable because they resolve in milliseconds — the `inFlight` entry is cleared long before the 45s second debounce fires.

No existing test covers the failure case: `summaryLoop.test.ts:220-247` only checks that the first write succeeds on concurrent same-board intents; it does not test the dropped-second-intent-after-first-failure scenario.
- **Fix direction:** After `inFlight.delete(boardId)` in the `finally` block, check whether content has changed (compare a re-read fingerprint against the stored one, or maintain a `pendingRetry` flag set when an intent is dropped at line 149). If a retry is warranted, re-arm the debounce for that boardId. Alternatively, instead of silently dropping at line 149, enqueue a "retry after in-flight" flag that the `finally` block checks before clearing.

### (2) `inFlight` guard is per-boardId but not per-project, allowing cross-project duplicate writes after a rapid project switch — `src/main/summaryLoop.ts:149`

- **Where:** `src/main/summaryLoop.ts:149`
- **Sev/cat:** Low / concurrency
- **What:** The `inFlight` Set is keyed only by `boardId`. If the same board id exists in two different projects (nanoid-space makes this unlikely but not impossible, and synthesized/test ids may collide), or if the user switches projects and the new project happens to have a board with the same id, the guard on line 149 blocks the second intent even though it is for a legitimately different board in a different project. Conversely, after a project switch the old in-flight entry keeps blocking new intents for any board that shares the id.
- **Trigger:** 1. Open project A with board id `abc123`. Trigger a summarize intent (slow network). 2. Switch to project B, which also contains board `abc123`. The detector emits a new intent for that board; the guard on line 149 silently drops it because the old entry is still in `inFlight`.
- **Evidence:** Watertight code-path argument:

1. `createSummaryLoop` (summaryLoop.ts:142) creates `inFlight = new Set<string>()` scoped to the closure for the app's lifetime. The `SummaryLoop` interface (lines 136-139) exposes only `onIntent` — no `reset()`.

2. On project switch, `project:open` (projectIpc.ts:110) calls `memoryEngine.reset()` which clears the engine's fingerprints and timers. But `summaryLoop` has no reset path — `inFlight` is never cleared.

3. The guard at summaryLoop.ts:149 (`if (inFlight.has(boardId)) return`) is keyed purely on `boardId`, not `(projectDir, boardId)`.

4. Sequence that triggers the silent drop:
   - Project A, board `abc123`: `onIntent` fires, `inFlight.add('abc123')`, slow `runSummarize` await begins.
   - User switches to Project B (also has board `abc123`): `memoryEngine.reset()` runs — clears engine state but `inFlight` still has `'abc123'`.
   - `memoryEngine.observe(projectB_doc)` baselines (no emit: `primed=false` after reset).
   - User edits Project B's board `abc123` → `memoryEngine.observe` detects change → `armDebounce` fires → `onIntent({boardId:'abc123'})` is called.
   - `inFlight.has('abc123')` is true → **silently dropped (line 149)**.

5. There is also a converse defect: the stale Project A in-flight call calls `deps.getCurrentDir()` at line 152 AFTER the switch, so it now gets Project B's dir and writes its summary into Project B's `.canvas/memory/` — a cross-project write contamination. Neither scenario is guarded.

6. Existing tests (summaryLoop.test.ts:219-248) only test same-project concurrent calls; no test covers cross-project switching with an in-flight call.

Nanoid makes accidental same-ID collision between real projects rare, but test/synthesized IDs (e.g. short deterministic IDs in unit/e2e fixtures) make this a real practical scenario. The structural defect is unconditional regardless of collision probability.
- **Fix direction:** Add a `reset()` method to the `SummaryLoop` interface that clears `inFlight`, and call it from `registerProjectHandlers` alongside `memoryEngine.reset()` on project open/switch. Alternatively, key `inFlight` on `${dir}:${boardId}` using the dir captured at intent-emit time rather than the one read lazily inside the async body.
