# BUG-006: summaryLoop project-dir TOCTOU → board summary written to the wrong project

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/summaryLoop.ts`
- **Roadmap:** none
- **Collides with:** BUG-014, BUG-015, BUG-016, BUG-017
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
The call to `deps.getCurrentDir()` on line 152 captures the current project dir before the `await runSummarize(...)` on line 163. If the user closes and opens a different project while the LLM call is in-flight (which can take up to 30 s), `dir` still holds the old project's path. After the await completes the code on lines 171–177 writes memory files (writeBoard, writeIndex, writeProject) into the OLD directory even though a completely different project is now open. The `inFlight` guard only prevents duplicate calls for the same boardId — it does not protect against a project switch.

## Where
`src/main/summaryLoop.ts:153`

## How it triggers
1. Open project A, edit board B, trigger a summarize intent. 2. Before the 30-second LLM call completes, close project A and open project B. 3. The LLM response arrives; the loop writes board-B.md + MEMORY.md + project.md into project A's `.canvas/` directory (stale `dir`). Board B may not even exist in project B.

## Verification evidence
The TOCTOU race is directly readable in the code with no upstream guard:

1. `/src/main/summaryLoop.ts` line 152: `const dir = deps.getCurrentDir()` — snapshot taken.
2. Lines 163–168: `await runSummarize(...)` — async LLM call; `DEFAULT_TIMEOUT_MS = 30_000` ms confirmed in `/src/main/llmService.ts` line 26.
3. Lines 171–177: `createCanvasMemory(dir)` + `writeBoard` / `writeIndex` / `writeProject` — all use the stale `dir`. No post-await re-check of `deps.getCurrentDir()` exists anywhere in the function.

Project-switch path (`/src/main/projectIpc.ts` line 101 `project:open`): calls `setCurrentDir(r.dir)` (line 96, via `remember`) and `memoryEngine.reset()` (line 110). `memoryEngine.reset()` cancels only **pending debounce timers** (`timers.delete(id)` + `clearTimeout`) — it has no knowledge of, and no handle to, the already-running `summaryLoop.onIntent` Promise. The `inFlight` Set guards only duplicate calls for the **same boardId**, not project switches.

Concrete scenario: user opens project A, edits board B (debounce fires → `onIntent` starts → `dir = "/projects/A"` captured → `runSummarize` awaits 30 s). User opens project B during that window; `currentDir` is now `/projects/B`. Await resolves; `createCanvasMemory("/projects/A")` writes board-B.md, MEMORY.md, and project.md into `/projects/A/.canvas/memory/` — now stale/wrong prose for project A's memory context.

## Suggested fix direction
After the await resolves (line 169), add a guard: `if (deps.getCurrentDir() !== dir) return` before any write. This covers the race with zero new state.
