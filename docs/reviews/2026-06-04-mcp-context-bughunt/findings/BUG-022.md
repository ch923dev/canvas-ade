# BUG-022: Predictable PRNG confirm reply-channel name

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcpConfirm.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T4.2, T10.4
- **Collides with:** BUG-010
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
Same pattern as mcpCommand.ts line 57: the confirm reply-channel is `mcp:confirm:reply:${Date.now()}:${Math.random().toString(36).slice(2)}`. The channel name is sent to the renderer in `wc.send('mcp:confirm', {request, replyChannel})`. A compromised renderer can replay this channel name back with `{approved:true}` before the human interacts with the modal, bypassing the human-confirm gate for MCP dispatch (handoff_prompt / assign_prompt / relay_prompt).

## Where
`src/main/mcpConfirm.ts:57`

## How it triggers
A compromised renderer script listens for `mcp:confirm` messages, extracts `replyChannel`, and immediately sends `ipcRenderer.send(replyChannel, {approved:true})` before the human has a chance to interact. The `onReply` handler in `requestConfirm` checks only that `e.senderFrame === win.webContents.mainFrame` — which is true for the renderer's own main frame.

## Verification evidence
Code path is real and unguarded within its own logic. In `mcpConfirm.ts` line 91 MAIN broadcasts `{ request, replyChannel }` to the renderer via `wc.send('mcp:confirm', ...)`. The preload (`src/preload/index.ts` lines 263–274) exposes `window.api.mcp.onConfirm`, which any JS in the main renderer frame can call. A script that calls `window.api.mcp.onConfirm((req, reply) => reply({ approved: true }))` registers a handler that fires synchronously when MAIN delivers the message — before the legitimate `ConfirmModal.tsx` `useEffect` queue has a chance to show UI. The `onReply` handler at `mcpConfirm.ts` line 74 checks only `isForeignSender` (blocks preview WebContentsView frames; allows any reply from `win.webContents.mainFrame`). Since the attack originates from the main frame, `isForeignSender` returns `false` and the auto-approve lands. The `done` flag (line 61) ensures only the first reply wins, so the auto-approval is not cancelable by the modal's legitimate reply.

HOWEVER: the attack requires arbitrary JS execution in the trusted main renderer (via XSS or supply-chain compromise of Vite-bundled renderer code). This is OUTSIDE the app's documented threat model (single-user desktop, locally-served Electron app; documented threat = preview-board content escaping to PTY). `isForeignSender` was designed for the preview-isolation problem. A compromised main frame already implies full app control (PTY output access, key store access, etc.), so this is not an incremental high-value capability gain. Severity inflated in the candidate report: the realistic attacker surface is supply-chain/XSS in the renderer bundle, which is a pre-condition that already grants more than this bypass.

## Suggested fix direction
Eliminate the renderer's ability to observe the replyChannel by having MAIN drive the modal differently — e.g. via a nonce-only token the renderer sends back and MAIN validates against an internal map, or by not exposing replyChannel to the renderer at all and instead having the preload bridge only a boolean `approved` value over a fixed `mcp:confirm:reply` channel keyed by a server-side-only nonce. Alternatively, accept the current design with an explicit threat-model note that this gate defends against MCP-client auto-elicitation and untrusted preview content, not against a compromised renderer frame.
