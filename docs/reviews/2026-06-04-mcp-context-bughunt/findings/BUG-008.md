### (1) handoffPrompt await-idle loop uses stale board reference after board is closed mid-wait — `src/main/mcpOrchestrator.ts:403`

- **Where:** `src/main/mcpOrchestrator.ts:403`
- **Sev/cat:** Medium / correctness
- **What:** In `handoffPrompt`, the await-idle loop at line 403-411 calls `registry.listBoards().find((b) => b.id === boardId) ?? board` where `board` is the board object captured at line 292 BEFORE the PTY write. If the board is closed by the user (or by a concurrent `reapIdle`) during the polling wait, `listBoards().find(...)` returns `undefined`, and the code falls back to the stale `board` object (which has `type: 'terminal'` and the original status). `deriveStatus` with the stale board falls back to the PTY session map — and if the session was killed when the board closed, `sessionById.get(board.id)` returns `undefined`, which maps to `'idle'`, causing the loop to exit. This is behaviorally correct by accident, but if the session map also retains the id after close (depending on timing), the loop could falsely continue polling. More critically, after the loop exits, `registry.readResult(boardId)` at line 412 reads a result for a board that is now closed — it will return `{ present: false }` (the empty shell) rather than the actual result, silently returning a false-empty result to the caller without error.
- **Trigger:** A user manually closes a terminal board while `handoffPrompt` is awaiting it to go idle. The orchestrator is writing to the board's PTY and waiting. The board closes, `listBoards()` no longer includes it, the fallback uses the stale object, the loop exits (session gone = idle), and `readResult` returns an empty shell.
- **Evidence:** Concrete test reproduction at Z:/canvas-ade-context-followup/src/main/mcpOrchestrator.ts.

Exact code path (all line numbers from the file):

1. Line 292: `const board = registry.listBoards().find((b) => b.id === boardId)` captures the board object. For any terminal board that has been publishing its live status, `board.status === 'running'` (always a concrete value — `buildBoardSnapshot` in `boardStatus.ts` always sets a bucket; the mirror sanitizer only drops *invalid* buckets, not valid ones like `'running'`).

2. Lines 403–411: Poll loop. When the board closes mid-wait, `registry.listBoards().find(...)` returns `undefined`. The `?? board` fallback provides the stale object with `status: 'running'`.

3. Line 149 of `deriveStatus`: `if (board.status) return board.status` fires immediately — returns `'running'` without consulting the session map. The loop thus sees `'running'` on every iteration and runs to the full `handoffTimeoutMs` deadline (default = `MCP_IDLE_TTL_MS` = 5 minutes).

4. Line 412: After the 5-minute wait, `registry.readResult(boardId)` → `results.get(boardId) ?? { present: false }` (from `boardResults.ts` line 17) returns the empty shell because `write_result` was never called by the closed board.

5. Lines 415–423: `audit(..., status: 'completed', outputs: '{"present":false}', ...)` — the failed operation is silently recorded as completed.

Written a scratch test to verify: with `board.status='running'`, board removed from mirror after first sleep, `handoffTimeoutMs=100ms`, `handoffPollMs=5ms` — `sleepCount` ended at >5 (loop ran to deadline, not early-exit), result was `{ present: false }`, and the completed audit contained `"present":false`. Test passed green. (Test file deleted after run.)

The candidate's claim that the loop might "exit immediately" (via session-map path) is wrong for the normal case: `deriveStatus` short-circuits at `if (board.status)` and never reaches the session-map branch, so the loop ALWAYS stalls for the full timeout when the board closes while running.
- **Fix direction:** In the poll loop (lines 403–411), treat a board absent from `listBoards()` as equivalent to `'idle'` rather than falling back to the stale capture. Replace the `?? board` fallback with an explicit `undefined`-check: if `listBoards().find(...)` returns `undefined`, break out of the loop (board gone = no longer running). Optionally, also surface a distinct error/audit entry to distinguish "timed out" from "board closed mid-dispatch", rather than always logging `status: 'completed'`.

### (2) handoffPrompt deadline loop: board that remains 'running' forever causes silent timeout with 'completed' audit status — `src/main/mcpOrchestrator.ts:403`

- **Where:** `src/main/mcpOrchestrator.ts:403`
- **Sev/cat:** Medium / correctness
- **What:** The await-idle loop at lines 403-411 is bounded by `handoffTimeoutMs` (default = `MCP_IDLE_TTL_MS` = 5 minutes). When the deadline is exceeded, the loop exits silently — `break` is never executed (the `while` condition `now() < deadline` becomes false). Execution falls through to `registry.readResult(boardId)` which may return a partial or empty result. The audit entry at line 415 records `status: 'completed'` regardless of whether the timeout was reached. There is no way for the MCP client to distinguish a true completion from a timeout.
- **Trigger:** An agent running in the target terminal board takes longer than `handoffTimeoutMs` (5 minutes default) to finish. The orchestrator exits the wait loop and returns whatever `readResult` has at that moment, logging `completed` in the audit trail.
- **Evidence:** Code path in `Z:/Canvas ADE` (remote branch `origin/feat/mcp-integration`) at `src/main/mcpOrchestrator.ts` lines 402–423:

```
const deadline = now() + handoffTimeoutMs        // line 403
while (now() < deadline) {                        // line 404
  const status = deriveStatus(...)               // line 405
  if (status !== 'running') break                // line 408
  await sleep(handoffPollMs)                     // line 409
}
const result = registry.readResult(boardId)      // line 412 — unconditional
await registry.audit({                           // line 414 — unconditional
  type: 'handoff_prompt',
  ...
  status: 'completed',                           // line 420 — always 'completed'
  ...
})
return result                                    // line 424
```

When the target board stays in `'running'` past `handoffTimeoutMs` (default = `MCP_IDLE_TTL_MS` = 5 minutes), the `while` condition `now() < deadline` becomes false and the loop exits silently with no `break`. The code falls straight through to `readResult` (which returns `EMPTY_RESULT = { present: false }` when no result was written) and then writes an audit entry with `status: 'completed'` unconditionally. There is no post-loop guard (e.g. `if (now() >= deadline) { audit 'timed_out'; throw/return }`) and no variable tracking the exit reason.

The `AuditInput.status` field is a plain `string` with no constraining union type, so a `'timed_out'` value is structurally legal but simply never emitted.

The two existing `await-idle` tests (`dispatchedBeforeWait` test and the `'await-idle: waits while the target is running'` test) both inject a `sleep` mock that sets `board.status = 'idle'` on the first call, so they exclusively test the normal-exit (`break`) path. The deadline-expiry path has zero test coverage.

The bug is a watertight code-path argument: the only way the loop can exit is via `break` (status not `'running'`) or the condition becoming false (deadline exceeded). The second case is completely unhandled downstream.
- **Fix direction:** After the while loop, check whether the deadline was exceeded (e.g. `const timedOut = now() >= deadline`) and emit a separate audit entry with `status: 'timed_out'` (or include a `detail: 'deadline_exceeded'` marker) before falling through to — or instead of — the `'completed'` audit. The caller/MCP client can then inspect the audit trail or the returned `BoardResult.present` flag to distinguish a true completion from a timeout. The `AuditInput` type already supports free-form `status` strings, so no type changes are needed.

### (3) handoffPrompt: handoffTimeoutMs defaults to MCP_IDLE_TTL_MS (5 minutes) — await-idle can block an MCP tool call for 5 minutes — `src/main/mcpOrchestrator.ts:172`

- **Where:** `src/main/mcpOrchestrator.ts:172`
- **Sev/cat:** Low / correctness
- **What:** Line 172: `const handoffTimeoutMs = opts.handoffTimeoutMs ?? MCP_IDLE_TTL_MS`. `MCP_IDLE_TTL_MS` is 5 minutes (line 24). A `handoff_prompt` call to a board that is stuck in `running` state (e.g., an agent that never goes idle) will block the calling MCP tool for up to 5 minutes before timing out and returning a stale/empty result. During this time, the MCP HTTP connection is held open. Multiple concurrent stuck handoffs can exhaust HTTP connection resources.
- **Trigger:** Call `handoff_prompt` to a terminal board running an agent that never emits an 'idle' status (e.g., a long-running agent, or a board whose status update is stuck). The tool call blocks for 5 minutes. Multiple such calls stack up, holding HTTP connections open.
- **Evidence:** The behavior is real and verifiable by code path:

1. `mcpOrchestrator.ts:172` — `handoffTimeoutMs = opts.handoffTimeoutMs ?? MCP_IDLE_TTL_MS` where `MCP_IDLE_TTL_MS = 5 * 60 * 1000` (line 24). No caller in `mcp.ts` (the production wiring at line 34: `buildOrchestrator(registry, { idleTtlMs: IDLE_TTL_MS })`) passes `handoffTimeoutMs`, so production always uses the 5-minute default.

2. `mcpOrchestrator.ts:403-411` — the await-idle poll loop runs until `now() >= deadline` (up to 5 minutes) if the board's status never leaves `running`.

3. The MCP transport (canvas-ade-mcp dist/index.js + @modelcontextprotocol/sdk webStandardStreamableHttp.js:499-559) uses SSE (`text/event-stream`) — `return new Response(readable, …)` is returned immediately but the HTTP/SSE connection IS held open until the tool's async handler resolves and `send()` pushes the result onto the stream. The tool-call async handler (`registerHandoffPrompt` at dist/index.js:345-348) directly awaits `orchestrator.handoffPrompt(…)`, so the SSE stream for that call stays alive for the full timeout.

4. The code comment at mcpOrchestrator.ts:401-402 explicitly acknowledges the limitation: "M5 replaces this interim poll with real attention signalling."

5. However, the "connection exhaustion" severity claim is inflated: the server binds to 127.0.0.1 only (mcp.ts, canvas-ade-mcp dist/index.js:558), the app is single-user/single-desktop (CLAUDE.md: "no multiplayer"), the spawn cap is 4 boards (MCP_SPAWN_CAP), so at most 4 concurrent stuck `handoff_prompt` calls could stack. Node.js HTTP has no low hard connection limit. The real impact is a blocked MCP client (Claude Code typically serializes tool calls) waiting up to 5 minutes on a stuck board, not resource exhaustion. Severity is Low, not Medium/High.
- **Fix direction:** Pass a shorter `handoffTimeoutMs` (e.g. 60s) in the `buildOrchestrator` call in `mcp.ts`, independent of `MCP_IDLE_TTL_MS` which controls idle-reap TTL (a different concept). The two constants serve different purposes and should not share a value. Until M5 ships real attention signalling, document the timeout as caller-configurable and add a log warning when the deadline expires so the operator knows the board is stuck.

### (4) handoffPrompt await-idle poll uses stale board snapshot on timeout — `src/main/mcpOrchestrator.ts:403`

- **Where:** `src/main/mcpOrchestrator.ts:403`
- **Sev/cat:** Low / concurrency
- **What:** At line 403-411, the await-idle loop polls `registry.listBoards()` looking for `boardId`. If the board is closed/deleted during the wait, `listBoards().find(...)` returns `undefined`, and the code falls back to the captured `board` snapshot (line 405: `?? board`). The snapshot's `status` field may be stale (e.g. still `running` from before the board was closed). `deriveStatus` with the snapshot and a fresh `sessionMap()` will then read the PTY session map — if the session was also removed, it returns `idle`, breaking out of the loop. But if by coincidence a NEW terminal with the same id was spawned (possible since the orchestrator does not prevent id reuse on `addBoard`), the loop might never break if the new session enters `running`.
- **Trigger:** Board is deleted while handoffPrompt is in the await-idle loop, and a new board with the same UUID is created and starts running, causing the poll to loop until `handoffTimeoutMs` expires (default 5 minutes). The MCP HTTP connection is held open for the full timeout.
- **Evidence:** Confirmed via code-path analysis and inline simulation. In `handoffPrompt`, `board` is captured at line 292 from `listBoards()` before the await-idle loop. At that point, for an actively-running terminal, the mirror carries `status: 'running'`. Inside the loop (lines 404–411), `registry.listBoards().find(b => b.id === boardId) ?? board` falls back to the stale `board` object if the board is deleted from the mirror. `deriveStatus` at line 149 checks `if (board.status) return board.status` first — since `board.status === 'running'` on the stale snapshot, it returns `'running'` unconditionally, bypassing both the PTY session map check and any deletion signal. The loop then stalls until `handoffTimeoutMs` expires (default `MCP_IDLE_TTL_MS = 5 * 60 * 1000` ms). Simulation confirmed: `deriveStatus(capturedBoard_with_status_running, emptySessionMap)` returns `'running'`. The id-reuse embellishment is a red herring — `crypto.randomUUID()` makes collision ~10^-37 per pair and there is no explicit id-reuse path — but the core stall fires without it. No existing test covers board-deleted-during-await-idle. Fix direction: inside the loop, treat `listBoards().find(...) === undefined` as an early-exit condition (board gone → assume idle/gone, break immediately) rather than falling back to the stale snapshot. File: `src/main/mcpOrchestrator.ts`, lines 403–411.
- **Fix direction:** In the await-idle loop, check if `listBoards().find(b => b.id === boardId)` returns `undefined` and break immediately (board deleted = treat as idle/completed) rather than falling back to the stale snapshot. This eliminates the 5-minute stall on board deletion without requiring the id-reuse scenario.
