# BUG-013: registerLlmHandlers silently drops injectedDeps.budget

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/llmIpc.ts`
- **Roadmap:** none
- **Collides with:** BUG-001, BUG-011, BUG-012
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
At lines 73–77 the `deps` object is constructed as `{ ...(injectedDeps ?? defaultDeps()), keyStore, budget: injectedDeps?.budget ?? createBudgetStore(...) }`. The spread of `injectedDeps` includes whatever `budget` was in the injected deps object, but then the explicit `budget:` key immediately overwrites it: `injectedDeps?.budget ?? createBudgetStore(...)`. This means `injectedDeps.budget` IS used (because `injectedDeps?.budget` is evaluated first). However, the `keyStore` key in the spread (from `injectedDeps` if it carries one) is also overwritten by the explicit `keyStore` assignment. So if a caller passes `injectedDeps` with a custom `keyStore`, that custom keyStore is silently discarded and replaced with either `createKeyStore(userDataDir, encryptor)` or `NOOP_KEY_STORE`. The integration test at line 25 passes `injectedDeps = { fetch, env }` without a keyStore, so the override is harmless there, but any future caller that injects a custom keyStore in `injectedDeps` will find it silently ignored — a latent correctness trap that violates the least-surprise contract of dependency injection.

## Where
`src/main/llmIpc.ts:73`

## How it triggers
Pass `registerLlmHandlers(ipcMain, getWin, dir, { fetch, env, keyStore: myCustomStore }, encryptor)`. The `myCustomStore` in `injectedDeps` is spread into `deps` first, then immediately overwritten by the explicit `keyStore: encryptor ? createKeyStore(...) : NOOP_KEY_STORE` assignment. The caller's store is silently ignored.

## Verification evidence
Reading `Z:/Canvas ADE/.claude/worktrees/audit-main/src/main/llmIpc.ts` lines 70–77 confirms the exact pattern: the function-scoped `keyStore` (line 70) is always constructed from `encryptor`, then spread-then-overridden at line 75. Any `keyStore` property on `injectedDeps` is spread in at line 74 but immediately clobbered.

The `ProviderDeps` type (`llmService.ts` lines 108–117) includes `keyStore?: Pick<KeyStore, 'getKey'>` as a valid optional field — TypeScript permits a caller to pass `injectedDeps = { fetch, env, keyStore: myStore }` with no compiler warning, yet `myStore` will be silently discarded.

The override is intentional (the line 71 comment: "Resolution uses the SAME store the IPC writes to") because `llm:setKey` / `llm:clearKey` / `llm:status` all use the function-scoped `keyStore` variable directly (lines 101, 107, 114) — if `deps.keyStore` could be the injected one, the IPC write channels and the summarize deps would be out of sync.

No current caller passes a `keyStore` in `injectedDeps`: `index.ts` line 189 passes `undefined`; all test callers pass `{ fetch, env }` only (confirmed by `llmIpc.integration.test.ts` lines 25–88). So the bug does not fire today.

The defect is a latent DI contract violation: the function signature accepts `injectedDeps?: ProviderDeps` with no documentation or type-level restriction preventing a caller from injecting a custom `keyStore` — yet that field is silently ignored. A future test or refactor that injects a keyStore for isolation will get the function-scoped store instead, with no error or warning.

## Suggested fix direction
Document the override contract explicitly in the function signature JSDoc (e.g., "Note: `injectedDeps.keyStore` is always ignored — the keyStore is constructed from `encryptor` to keep IPC write channels consistent") and/or add a TypeScript `Omit<ProviderDeps, 'keyStore'>` type on the `injectedDeps` parameter so the type system enforces the contract. No runtime behavior change needed since no current caller is affected.
