## Summary

### (1) ensureScaffold does not call mem.ensureScaffold() before writeFileAtomic.sync in setCommitOptIn — `src/main/canvasMemory.ts:80`

- **Where:** `src/main/canvasMemory.ts:80`
- **Sev/cat:** Low / correctness
- **What:** setCommitOptIn (line 125) creates only the root .canvas/ directory (mkdirSync(root, { recursive: true })), then writes the .gitignore directly. It does NOT create memoryDir or auditDir. If setCommitOptIn is called before ensureScaffold() on a fresh project, .canvas/ is created and .gitignore is written, but .canvas/memory/ and .canvas/audit/ are not created. A subsequent call to writeIndex/writeProject will then call mkdirSync(memoryDir, { recursive: true }) lazily — so this is not a crash, but the dirs exist in an inconsistent partial state (only root + gitignore, no memory/ or audit/) until the next write. This can cause test or tooling confusion when inspecting the tree.
- **Trigger:** Caller invokes mem.setCommitOptIn(true) on a fresh project before ever calling mem.ensureScaffold().
- **Evidence:** Code path confirmed: `setCommitOptIn` (canvasMemory.ts:125-128) calls only `mkdirSync(root, { recursive: true })` then writes the gitignore. It never creates `memoryDir` (.canvas/memory/) or `auditDir` (.canvas/audit/). The trigger scenario — calling `setCommitOptIn` before `ensureScaffold` on a fresh project — is exercised in the existing test at line 137 (`m.setCommitOptIn(true)` with no prior `ensureScaffold`), but that test only asserts on `isCommitted()` and the gitignore content, not directory structure.

However, the practical impact is minimal for two reasons: (1) `setCommitOptIn` has zero production callers in the current codebase — only test calls exist; in production the flow always goes through `scaffoldProjectMemory` (projectIpc.ts:108,176; projectStore.ts:95,104), which calls `ensureScaffold` before any Settings UX could invoke `setCommitOptIn`. (2) All write methods (`writeIndex`, `writeProject`, `writeBoard`) lazily call `mkdirSync(memoryDir, { recursive: true })` before writing, so a subsequent write self-heals `memoryDir`. Only `auditDir` is never lazily created anywhere — but `auditDir` is also never written by any code path other than `ensureScaffold` in the current codebase.

The inconsistency is real (partial scaffold: root + gitignore but no memory/ or audit/) but has no observable consequence in the current production code. It is a future-caller hazard: if a Settings IPC handler is wired to call `setCommitOptIn` on a not-yet-scaffolded project dir, `auditDir` would be absent until the next `ensureScaffold()` or `writeBoard/writeIndex/writeProject` call.
- **Fix direction:** Add `mkdirSync(memoryDir, { recursive: true })` and `mkdirSync(auditDir, { recursive: true })` inside `setCommitOptIn` before writing the gitignore, matching `ensureScaffold`'s behavior; or extract a shared private `ensureDirs()` helper called by both.

### (2) No `ensureScaffold()` call before writing board memory; writes can silently fail if `.canvas/memory/` does not yet exist — `src/main/summaryLoop.ts:152`

- **Where:** `src/main/summaryLoop.ts:152`
- **Sev/cat:** Low / correctness
- **What:** The `onIntent` handler creates a `CanvasMemory` instance (line 171) and calls `mem.writeBoard`, `mem.writeIndex`, and `mem.writeProject` directly. `writeBoard` calls `mkdirSync(memoryDir, { recursive: true })` before writing, so directory creation is handled there. However, `writeIndex` and `writeProject` also each call `mkdirSync`. The `.canvas/.gitignore` is only written by `ensureScaffold()`, which is called at project open (in `scaffoldProjectMemory`). If `scaffoldProjectMemory` failed silently at project open (it swallows all errors), the `.gitignore` is never written. The summary loop has no recovery path to recreate it — a user who opens a project on a temporarily full disk will get memory files but no `.gitignore`, causing the raw AI summaries to be tracked by git contrary to the stated default-private contract.
- **Trigger:** Project is opened on a disk that is briefly full (causing `ensureScaffold` to fail silently). Disk space is freed. A board is edited; the summarize loop fires and writes summaries successfully. The `.gitignore` is absent, so `.canvas/` contents are visible to `git status` and may be committed unintentionally.
- **Evidence:** Code-path argument (no speculation):

1. `summaryLoop.ts:171` — `const mem = createCanvasMemory(dir)` creates a fresh instance; no `ensureScaffold()` is called anywhere in the `onIntent` handler.

2. `canvasMemory.ts:88-113` — `writeBoard`, `writeIndex`, and `writeProject` each call `mkdirSync(memoryDir, { recursive: true })` before writing, which creates `.canvas/memory/` but does NOT write `.canvas/.gitignore`.

3. `canvasMemory.ts:80-87` — `.gitignore` is written ONLY by `ensureScaffold()`, guarded by `if (!existsSync(gitignore))`.

4. `canvasMemory.ts:147-156` / `projectStore.ts:95,104` / `projectIpc.ts:108,176` — `scaffoldProjectMemory` (the sole caller of `ensureScaffold`) swallows ALL errors silently: if ENOSPC or EACCES occurs at project open, `.gitignore` is never written and no retry path exists.

5. The test at `canvasMemory.test.ts:71` explicitly asserts "creates memory/ on writeBoard even without ensureScaffold", which confirms that `writeBoard` intentionally skips the gitignore step.

Concrete failure sequence: disk is briefly full at project open → `scaffoldProjectMemory` throws ENOSPC, is caught and swallowed, `.canvas/.gitignore` is never written → disk space freed → user edits a board → summary loop fires → `writeBoard`/`writeIndex`/`writeProject` each succeed (now there's space), creating `.canvas/memory/*.md` with no `.gitignore` present → `git status` shows the memory files as untracked/tracked, breaking the "default-private" contract documented in `canvasMemory.ts:7`.
- **Fix direction:** In `summaryLoop.ts` `onIntent`, after `const mem = createCanvasMemory(dir)`, call `mem.ensureScaffold()` (wrapped in try/catch to preserve best-effort semantics). This is cheap — `ensureScaffold` is idempotent and checks `existsSync` before writing — and provides the missing recovery path for a scaffold that failed at open.
