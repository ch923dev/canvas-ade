## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) Audit log sequence chaining breaks on concurrent append+restart — seq can reset to 1 — `src/main/auditLog.ts:139`

- **Where:** `src/main/auditLog.ts:139`
- **Sev/cat:** Low / concurrency
- **What:** The sequence is initialized by `initSeq()` which reads the file once and computes `max(existing seqs) + 1`. This promise is cached in `nextSeq`. If two `AuditLog` instances are created over the same file (e.g. if `registerAuditHandler` is called twice, or across process restarts with overlapping windows), both will seed from the same file state and start at the same `nextSeq`, producing duplicate seq numbers. More critically: if `appendFile` fails (disk full, permissions), the `nextSeq` chain advances past the failed entry, creating a seq gap without a record — the audit trail has a gap that looks like a normal gap, not a failure.
- **Trigger:** (1) `appendFile` throws at line 144 (disk full). `nextSeq` was already advanced to `seq+1` at line 140, so the next call gets seq+1. The entry at seq is missing from the file but the sequence continues — the gap is invisible in the audit trail. (2) The app restarts while an append is in flight; the new instance seeds `initSeq()` from the file and may assign the same seq as the in-flight write.
- **Evidence:** Code-path argument confirmed by a minimal reproduction test (which passed green with the observed gap):

**Claim 1 (seq gap on appendFile failure) — CONFIRMED.**

In `Z:/canvas-ade-integration/src/main/auditLog.ts` lines 139-144:

```ts
const reservation = initSeq()
nextSeq = reservation.then((seq) => seq + 1)  // line 140: UNCONDITIONAL advance
const seq = await reservation                  // line 141
const entry = shapeAuditEntry(input, seq, now())
await mkdir(opts.dir, { recursive: true })
await appendFile(path, JSON.stringify(entry) + '\n', 'utf8')  // line 144: can throw
```

The `nextSeq` chain is advanced at line 140 before `appendFile` is even called (line 144). If `appendFile` throws (ENOSPC, permissions), the exception propagates to the caller but `nextSeq` is already pointing to `seq + 1`. The next successful `append` call gets `seq + 1`, skipping `seq` entirely.

The scratch reproduction test (now deleted per instructions) confirmed: after a second-call ENOSPC failure, the file contained seqs `[1, 3]` and the gap at `2` is undetectable by an audit reader — it appears identical to a restart-induced gap. Output: `Sequences written to file: [ 1, 3 ]`, `Next seq after failed write: 3`.

**Claim 2 (two instances / restart overlap) — NOT CONFIRMED for production.**

`createAuditLog` is called exactly once at `index.ts:253` per process lifetime and `registerAuditHandler` simply overwrites the module-level `registered` reference. There is no code path in production that creates two concurrent `AuditLog` instances over the same file. Process restarts are safe: `initSeq()` reads `max(seqs)+1` from the on-disk file, so a clean restart picks up where the last successful write left off.

**Severity re-calibration:** The gap only occurs on disk-write failure (uncommon) and is indistinguishable from a normal restart gap — the forensic value of successfully-written entries (nonce/type/targetId/prompt) is intact. Real-world impact is low.
- **Fix direction:** Wrap the `appendFile` call in a try/catch inside `append`; on failure, reset `nextSeq` back to `Promise.resolve(seq)` so the failed sequence number is retried rather than skipped. Alternatively, only advance `nextSeq` after a successful write (chain the advance off the `appendFile` promise instead of off `reservation`).

### (2) Concurrent audit `append` calls serialize sequence numbers but can interleave file writes — `src/main/auditLog.ts:139`

- **Where:** `src/main/auditLog.ts:139`
- **Sev/cat:** Low / concurrency
- **What:** The sequence-reservation chain (`nextSeq = reservation.then(seq => seq + 1)` at line 140) ensures each caller gets a unique, monotonically increasing sequence number. However, a second concurrent `append` call chains off the promise that resolves to `seq + 1` — a value which resolves as soon as the first `initSeq()`/seed promise resolves, **not** after the first `appendFile` completes. This means the second caller's `appendFile` can start before the first caller's `appendFile` finishes: the file writes may be interleaved, and the JSONL line for `seq+1` can physically precede or overlap the line for `seq` in the file.
- **Trigger:** Two MCP dispatch tools fire simultaneously (e.g. two `handoff_prompt` calls queued while the human confirm modal is being serviced). Both enter `append` concurrently, both reserve their seq numbers correctly, but then both issue `appendFile` whose completion order is indeterminate. The resulting JSONL may have out-of-order lines, or in the worst case, a partial line from one write interleaved mid-line with another on OSes that don't guarantee `appendFile` atomicity for large writes.
- **Evidence:** Code-path proof: In `createAuditLog` (`src/main/auditLog.ts`), the `append` method at lines 139-144 does:

```
const reservation = initSeq()                  // grab tail promise (P_n)
nextSeq = reservation.then((seq) => seq + 1)   // advance tail to P_{n+1}
const seq = await reservation                   // wait for seq value
...
await appendFile(path, ...)                     // write to file
```

When two callers A and B enter concurrently, B chains off `nextSeq` (P1 = P0.then(s=>s+1)), so B awaits P1, not A's `appendFile`. P1 resolves one microtask after P0, which is long before A's `appendFile` I/O completes. Both `appendFile` calls are in-flight simultaneously.

The comment at lines 135-138 explicitly claims this prevents "interleaved writes", but the `appendFile` is NOT chained into `nextSeq` — only the seq-number arithmetic is serialized. B's write can begin (and the OS can service it) before A's write completes.

**What actually happens:**
1. Seq uniqueness: Preserved — A gets seq=1, B gets seq=2. No duplicate seqs.
2. Character-level data corruption: Unlikely in practice — each JSONL line is well under OS `O_APPEND` atomic-write boundaries (~4KB on Linux, larger on Windows), so individual line bytes won't interleave.
3. Physical line ordering vs seq ordering: Real risk — B's line (seq=2) can physically land in the file before A's line (seq=1). The `read()` function returns by file position (`.slice(-limit).reverse()`), not by seq, so the caller sees seq=2 as "newer" than seq=1 even though seq=1 was logically earlier.

The misleading code comment ("can't interleave writes") is evidence the author intended serialization but did not implement it — the fix is to chain `appendFile` into the tail promise so each I/O completes before the next begins. The existing concurrent-append test (line 87-104) only checks that both entries exist and have distinct seqs; it does NOT assert physical line order matches seq order, so the test does not catch this defect.
- **Fix direction:** Chain `appendFile` into the `nextSeq` tail promise: `nextSeq = reservation.then(async (seq) => { /* build+write entry */ await appendFile(...); return seq + 1; })` so each write completes before the next caller's write begins. This also eliminates the separate `await reservation` / `await appendFile` split.
