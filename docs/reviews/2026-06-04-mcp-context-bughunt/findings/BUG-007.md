### (1) Stale-closure race: provider/model state reset if user edits before status resolves â€” `src/renderer/src/canvas/SettingsModal.tsx:36`

- **Where:** `src/renderer/src/canvas/SettingsModal.tsx:36`
- **Sev/cat:** Low / correctness
- **What:** The `useEffect` on mount fires `window.api.llm.status()` and unconditionally overwrites `provider`, `model`, `baseUrl`, and `hasKey` when the promise resolves (line 37-40). There is no cancellation mechanism. If the user immediately changes the provider/model in the UI before the IPC round-trip completes, the status response silently clobbers whatever the user typed, reverting them to the persisted values with no warning.
- **Trigger:** User opens SettingsModal on a slow IPC path (e.g. main-process contention), immediately clicks a different provider or types a model override, and the `llm.status` promise resolves ~300ms later. The user's edits are erased.
- **Evidence:** Code path at Z:/canvas-ade-integration/src/renderer/src/canvas/SettingsModal.tsx lines 35-42: the mount effect calls `window.api.llm.status().then(s => { setProvider; setModel; setHasKey; setBaseUrl })` with zero cancellation mechanism (no `cancelled` ref, no `isMounted`, no cleanup return in the effect). The `.then()` callback will unconditionally fire and overwrite any state the user has typed.

The IPC handler at Z:/canvas-ade-integration/src/main/llmIpc.ts line 87 is synchronous (`(e): LlmStatus =>`, not `async`), so it performs a file read (readLlmConfig) and returns immediately â€” the round-trip is microtasks + IPC overhead, typically well under 10ms. This makes the practical race window very narrow: a user would have to interact before the first microtask batch completes after mount. The scenario described (300ms slow IPC on main-process contention) is theoretically possible but requires unusual load.

The defect IS real and unguarded. The existing tests in SettingsModal.test.tsx all call `await waitFor(() => expect(llm.status).toHaveBeenCalled())` or `await screen.findByLabelText(...)` before any interaction, so they test only the resolved-then-interact path and do not cover the race. No upstream guard, type constraint, or validation can prevent the clobber.

Severity downgraded from High to Low: the synchronous IPC handler makes the race window extremely narrow under normal conditions. The user would have to interact within a single event-loop turn after modal mount, which is practically impossible without artificial main-process contention. The harm when triggered (reverting user edits) is real but easily recovered (just re-type the change).
- **Fix direction:** Add a boolean `cancelled` flag in the effect (set to `true` in the cleanup return), guard the `.then()` body with `if (!cancelled)`. Pattern matches the `prose`-fetch guard already used elsewhere in the codebase.

### (2) setConfig success not guarded before calling setKey â€” partial-save if setConfig itself returns ok:false â€” `src/renderer/src/canvas/SettingsModal.tsx:66`

- **Where:** `src/renderer/src/canvas/SettingsModal.tsx:66`
- **Sev/cat:** Medium / correctness
- **What:** The `save` function calls `window.api.llm.setConfig(...)` and awaits it, but then unconditionally falls through to call `window.api.llm.setKey(...)` without checking whether the setConfig result was `ok:true`. If `setConfig` returns `{ ok: false }` (e.g. forbidden sender or an IPC-level failure that doesn't throw), the key is still sent to main despite the config not being saved, and the modal calls `onClose()` at line 77, silently treating a partial failure as success.
- **Trigger:** Main returns `{ ok: false, reason: 'forbidden' }` from `llm:setConfig` (e.g. frame-guard fires) while a key is present in the field. setKey will still be invoked and the modal closes as though save succeeded.
- **Evidence:** In SettingsModal.tsx lines 61-65, `await window.api.llm.setConfig(...)` discards its return value â€” there is no assignment and no `if (!r.ok)` guard. Execution unconditionally falls through to the `if (key.trim())` block at line 66, then to `onClose()` at line 77.

The main-side handler `llmIpc.ts` line 124 has a concrete non-throwing `{ ok: false }` path: `if (guard(e)) return { ok: false, reason: 'forbidden' }`. This returns a resolved value (not a rejection), so the renderer's `catch` block at line 78 is NOT reached. The `await` resolves to `{ ok: false, reason: 'forbidden' }`, which is silently discarded.

Concrete code path:
1. `llm:setConfig` IPC handler fires frame-guard â†’ returns `{ ok: false, reason: 'forbidden' }`.
2. Renderer `await` resolves normally; the discarded value is never inspected.
3. `key.trim()` non-empty â†’ `llm:setKey` is invoked (the guard also fires there, returning `{ ok: false }`, so the key is not stored â€” but `setKey` returning `{ ok: false, reason: 'forbidden' }` hits the `if (!r.ok)` branch and shows an error message, meaning the user DOES see an error in this specific fork).
4. If `key` is empty: `setKey` is skipped entirely and `onClose()` is called at line 77 â€” modal closes as if save succeeded when config was never written.

The worst-case is the empty-key fork: user saves only provider/model, `setConfig` returns `{ ok: false }`, no error is shown, modal closes silently with config not persisted.

The existing test suite (SettingsModal.test.tsx) only covers `setConfig` throwing (line 103: `mockRejectedValue`) and `setKey` returning `{ ok: false }`, but has NO test for `setConfig` returning `{ ok: false }` (a resolved-but-failed response).
- **Fix direction:** Capture the `setConfig` result and guard: `const cr = await window.api.llm.setConfig(...); if (!cr.ok) { setError('Could not save settings â€” please try again.'); return; }`. Add a test: mock `setConfig` to resolve `{ ok: false, reason: 'forbidden' }`, assert onClose is NOT called and the error alert is shown.

### (3) Clear-key does not close modal on success, but also does not update hasKey when provider has just been changed â€” `src/renderer/src/canvas/SettingsModal.tsx:87`

- **Where:** `src/renderer/src/canvas/SettingsModal.tsx:87`
- **Sev/cat:** Low / correctness
- **What:** The `clear` function calls `clearKey({ provider })` using the *current* React state `provider`. If the user first changes the provider dropdown (say, from openrouter to anthropic) and then clicks 'Clear key' before saving, the `clearKey` call targets the newly selected provider even though the persisted config may still be `openrouter`. The visual indicator `hasKey` (which tracks the *loaded* status) is then set to false for the locally-selected provider, but the backend's active config may still have a key for the old provider. Additionally, on success the modal stays open (no `onClose()`) â€” this is by design per the comment, but creates an inconsistency where Save always closes while Clear does not.
- **Trigger:** User opens modal (provider=openrouter, hasKey=true shown), changes provider dropdown to anthropic (no key), clicks 'Clear key'. The clear operation runs against anthropic's (empty) key store entry instead of openrouter's. Then hasKey is reset to false for the wrong context.
- **Evidence:** Concrete code-path trace:

1. Modal mounts â†’ `llm:status` IPC (llmIpc.ts:87) returns `provider='openrouter', hasKey=true` (from `keyStore.hasKey(config.provider)` where config = persisted config). State: `provider='openrouter', hasKey=true`.

2. User changes dropdown â†’ `onProvider('anthropic')` (SettingsModal.tsx:52-55) calls `setProvider('anthropic')` and `setModel(...)`. Critically, `hasKey` is NOT updated â€” it remains `true` (still reflecting openrouter's key status). No IPC re-fetch happens for the new provider's key status.

3. User clicks "Clear key" â†’ `clear()` (line 87) executes `window.api.llm.clearKey({ provider: 'anthropic' })` (line 91), targeting the newly-selected provider.

4. Backend handler (llmIpc.ts:112-116) calls `keyStore.clearKey('anthropic')`. In llmKeyStore.ts:71-73, if `data['anthropic'] === undefined` the function returns early â€” the anthropic key entry is empty so nothing is written. The openrouter key is untouched in storage.

5. Frontend then calls `setHasKey(false)` and `setKey('')` (lines 92-93). Now `hasKey=false` but the persisted openrouter key still exists in the keystore.

6. If the user now clicks Save without re-opening the modal, `llm:setConfig` persists `provider='anthropic'` â€” when `llm:status` is next called, it will return `hasKey: keyStore.hasKey('anthropic')` which is `false`, consistent with what the UI shows. However, the openrouter key silently remains in storage (orphaned but not harmful).

Secondary confirmed issue: `onProvider` never fetches `hasKey` for the newly-selected provider, so `hasKey` is always stale after a dropdown change â€” it shows the initial-load provider's key presence, not the currently-selected one. This is the root desynchronization that enables the bug.

The damage is UI inconsistency (misleading `hasKey` indicator) and a no-op clear call rather than actual deletion of the wrong key. No key material is destroyed for the original provider.
- **Fix direction:** In `onProvider`, after setting provider state, call `window.api.llm.status()` (or a lighter `llm:hasKey` IPC) to refresh `hasKey` for the newly-selected provider. Alternatively, gate the "Clear key" button to only be enabled when `provider` matches the persisted active provider (from the initial status load), preventing clearing an unsaved provider selection.

### (4) Key sent to MAIN even when key.trim() is all-whitespace but not truly empty â€” `src/renderer/src/canvas/SettingsModal.tsx:66`

- **Where:** `src/renderer/src/canvas/SettingsModal.tsx:66`
- **Sev/cat:** Low / security
- **What:** The guard at line 66 is `if (key.trim())`, which correctly skips an empty key. However, `key.trim()` is also passed directly as the key value at line 67: `key: key.trim()`. This means a key that was entered with surrounding spaces is trimmed before being sent. This is not a security flaw per se, but the guard and the value passed differ only in that the state holds untrimmed input. The more subtle issue: if a user accidentally pastes a key with a newline embedded (e.g. `sk-abc
def`), `key.trim()` passes the outer guard (non-empty after trim), but the key sent to MAIN includes the embedded newline since `trim()` only removes leading/trailing whitespace.
- **Trigger:** User pastes an API key that inadvertently contains an embedded newline character in the middle (e.g. from a wrapped copy-paste). `key.trim()` does not remove it; the key stored via safeStorage contains the embedded newline, causing all provider requests to fail with an auth error that may be hard to diagnose.
- **Evidence:** Code path verified across three files:

1. SettingsModal.tsx:66-67 â€” `if (key.trim())` guard passes for `'sk-abc\ndef'` because `trim()` only strips leading/trailing whitespace; the embedded `\n` survives. The value `key.trim()` = `'sk-abc\ndef'` is sent to MAIN.

2. llmIpc.ts:105-110 â€” MAIN passes `a.key` directly to `keyStore.setKey(a.provider, a.key)` with no sanitization.

3. llmKeyStore.ts:64-68 â€” `setKey` encrypts the key as-is (`encryptor.encryptString(key)`) and persists it to disk. The embedded newline is stored in safeStorage.

4. At use-time (llmService.ts:172-185), `buildRequest` injects the key directly into HTTP headers (`'x-api-key': key` / `Authorization: Bearer ${key}`). When `deps.fetch()` processes these headers, Node.js's Headers API throws `TypeError: "sk-abc\ndef" is an invalid header value` (confirmed by running: `node -e "new Request('https://example.com', {headers:{'x-api-key':'sk-abc\ndef'}})"` â†’ throws). This throw is caught by `runSummarize`'s catch block (line 216-220) and surfaced as `{ ok: false, reason: 'provider-error', message: '<undici-internal error>' }` â€” an opaque, non-diagnostic message that gives the user no indication the key value itself is malformed.

The persistence aspect makes this worse: once the newline-containing key is stored via safeStorage, it will cause this opaque provider-error on every subsequent summarize call until the user manually clears and re-enters the key. There is no guard in `llmIpc.ts` or `llmKeyStore.ts` that would reject or sanitize the embedded newline before storage.

The bug is real and reproducible. Severity is Lower than the candidate's "Medium" â€” this is not a security/injection vulnerability (Node.js fetch rejects the header; the key never leaves the app in a dangerous way), but a UX/input-validation defect causing persistent opaque failures.
- **Fix direction:** In `save()` at SettingsModal.tsx, replace `key.trim()` with `key.replace(/\s+/g, '')` (or at minimum `/[\r\n]/g`) before the guard and the IPC call, so all whitespace (including embedded newlines and tabs) is stripped from the key before storage. Alternatively add a validation step that shows a user-visible error if the trimmed key still contains non-printable/whitespace characters.

### (5) Scrim pointer-down closes modal while save is in-flight despite busy guard being inside the dialog only â€” `src/renderer/src/canvas/SettingsModal.tsx:104`

- **Where:** `src/renderer/src/canvas/SettingsModal.tsx:104`
- **Sev/cat:** Low / correctness
- **What:** The scrim `onPointerDown` handler at line 104 correctly checks `if (!busy) onClose()`. However, `busy` is a React state variable. The guard reads the value captured in the current render's closure, which is correct for React event handlers. The issue is that clicking the scrim while `busy=true` silently does nothing â€” there is no visual feedback that the click was swallowed, no cursor change, and no toast/indicator. The user may think the click failed and click again repeatedly, queuing up multiple ignored close attempts without knowing the save is in progress.
- **Trigger:** User clicks Save (busy becomes true), immediately clicks the scrim (dark overlay). The click is silently ignored. No spinner, no 'savingâ€¦' label on the Save button.
- **Evidence:** Code path is confirmed as described. At line 104-106, the scrim `onPointerDown` silently no-ops when `busy=true`: `if (!busy) onClose()`. The `styles.scrim` object (lines 196-203) has no `cursor` property, so when busy the scrim still shows the default arrow cursor. The `styles.primary` and `styles.ghost` objects both set `cursor: 'pointer'` unconditionally (lines 247, 257) â€” the browser's native `disabled` attribute handles cursor change on the buttons themselves, but the scrim is not a button and gets no feedback at all. The buttons at lines 178, 182, 185 are correctly `disabled={busy}` and DO provide partial visual feedback (greyed out). However, the scrim (the dark overlay covering the rest of the app) has no cursor change, no opacity change, and no `pointer-events: none` or other indicator when busy. A user who clicks the scrim area â€” not the buttons â€” while a save is in flight gets a completely silent rejection. No test in SettingsModal.test.tsx covers scrim-click-during-busy feedback. The functional guard itself (closure captures current `busy`) is correct React behavior; this is purely a UX/feedback gap on the scrim element.
- **Fix direction:** Add `cursor: busy ? 'not-allowed' : 'default'` (or `'wait'`) to the scrim's inline style when busy, so the user sees a visual cue that the modal is locked. Optionally also change the scrim background opacity slightly when busy to reinforce the locked state. The functional guard at line 105 is correct and should not change.
