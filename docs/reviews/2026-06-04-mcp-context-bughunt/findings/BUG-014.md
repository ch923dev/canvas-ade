## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) MEMORY.md rebuilt from a stale doc snapshot when two boards are summarized concurrently — `src/main/summaryLoop.ts:176`

- **Where:** `src/main/summaryLoop.ts:176`
- **Sev/cat:** Low / correctness
- **What:** When two concurrent onIntent calls for different boards both succeed, each independently calls mem.writeIndex(buildMemoryIndex(r.doc, (id) => mem.readBoard(id) !== undefined)). Each captures r.doc from deps.readProject() before either write completes. The hasSummary callback does a live filesystem read so it will see the sibling's board file if that write finished first, but the last caller to execute writeFileAtomic.sync wins and the intermediate winner's MEMORY.md write is silently overwritten. Because writeFileAtomic.sync is synchronous and Node.js is single-threaded, the writes don't corrupt each other — but if call A finishes its writeBoard then is preempted at the await boundary while call B completes fully (writeBoard + writeIndex), then A resumes and writes a second MEMORY.md using its own r.doc snapshot, that final MEMORY.md may omit boards added or removed between the two readProject() calls.
- **Trigger:** Two boards change content within the same 45-second debounce window (common during initial project setup), both intents fire near-simultaneously, both LLM calls succeed roughly at the same time.
- **Evidence:** Concrete code-path argument in summaryLoop.ts:

1. `inFlight` guard (line 149) is keyed by `boardId` — it only blocks a second intent for the SAME board. Two different boards (e.g. `board-A` and `board-B`) run fully concurrently with no mutual exclusion.

2. Each call captures `r = deps.readProject(dir)` (line 154) as its own snapshot before hitting `await runSummarize(...)` (line 163) — the async yield point.

3. After the await, line 176 executes:
   `mem.writeIndex(buildMemoryIndex(r.doc, (id) => mem.readBoard(id) !== undefined))`
   `r.doc` is the snapshot from step 2. `buildMemoryIndex` iterates `boardsOf(r.doc)` to produce the board-list lines. The `hasSummary` callback is a live disk read, but the *board list itself* (which board IDs appear in MEMORY.md at all) is fully determined by the stale `r.doc`.

4. Scenario: Call A reads `r.doc` (2 boards). A project save then adds board-C. Call B reads `r.doc` (3 boards). Both LLM calls complete. B finishes first: writes `board-B.md` + correct MEMORY.md with 3 boards. A finishes second: writes `board-A.md` + MEMORY.md from its 2-board snapshot — silently overwrites B's correct 3-board MEMORY.md. Board-C is omitted.

5. The existing test suite covers only same-boardId concurrency (the `in-flight guard` suite, line 219-248). There is no test for different-boardId concurrent intents producing stale-snapshot writeIndex conflicts.

6. `writeFileAtomic.sync` is synchronous so no byte-level corruption, but last-writer-wins with stale data is confirmed.

Mitigating factors (informing severity): MEMORY.md is passive display/MCP-read context only and explicitly never action-triggering (design contract). The stale state is self-healing — the next summarization cycle for any board rewrites the index from `readProject()` again. The race window is narrow (two LLM calls must return near-simultaneously). Net effect: a transient incorrect omission of recently-added boards from MEMORY.md, correcting itself on next intent.
- **Fix direction:** After `await runSummarize` succeeds, re-read the project doc (a second `deps.readProject(dir)` call) immediately before building and writing the index, so the board list reflects any saves that occurred during the LLM round-trip. Alternatively, serialize writeIndex calls behind a per-dir async mutex so the last write always uses the freshest snapshot.

### (2) `buildMemoryIndex` reads `r.doc` (the pre-summarize snapshot) to compute which boards have summaries, potentially producing a stale MEMORY.md after concurrent intents — `src/main/summaryLoop.ts:176`

- **Where:** `src/main/summaryLoop.ts:176`
- **Sev/cat:** Low / correctness
- **What:** After writing a board summary (`mem.writeBoard`), the code on line 176 calls `buildMemoryIndex(r.doc, (id) => mem.readBoard(id) !== undefined)`. The doc `r` was read from disk at line 154, before the `await runSummarize`. If another `onIntent` for a different boardId has concurrently completed and written its summary between the two awaits, `buildMemoryIndex` will see the correct new summary via `mem.readBoard`, but the board list is sourced from the stale `r.doc`. If a board was added or removed in the project between when `r` was read and now, the MEMORY.md will list the wrong set of boards.
- **Trigger:** 1. Board X has been recently deleted from the project, but the summarize intent for old board Y (debounced 45 s ago) fires and reads the doc at T1. 2. Between T1 and the write at T2, the project has been saved without board X. 3. MEMORY.md is written from the stale `r.doc` and still lists the deleted board X.
- **Evidence:** Code path confirmed in Z:/canvas-ade-integration/src/main/summaryLoop.ts:

1. Line 142: `inFlight` is keyed by `boardId` — only prevents duplicate calls for the SAME board. Two `onIntent` calls for DIFFERENT boardIds (e.g., board A and board B) proceed concurrently.

2. Line 154: `const r = deps.readProject(dir)` — reads canvas.json from disk into `r.doc`. This is a synchronous snapshot (confirmed by projectStore.ts:55-61 which calls `readFileSync`).

3. Lines 163-168: `await runSummarize(...)` — async network call; yields the event loop, allowing a second `onIntent` for a different boardId to proceed past its own line 154 read (or for the project file to be rewritten on disk by a canvas save).

4. Line 176: `buildMemoryIndex(r.doc, ...)` — uses the pre-await `r.doc` snapshot to enumerate the board list. `hasSummary` correctly reads fresh from disk, but the **board enumeration** is from the stale snapshot.

Concrete race (the reported trigger): 
- Intent for board Y fires, reads doc at T1 (boards: [X, Y, Z])
- User deletes board X → project saves updated canvas.json (boards: [Y, Z]) 
- Intent Y's `runSummarize` completes at T2
- Line 176 writes MEMORY.md enumerating boards [X, Y, Z] — board X (deleted) still appears

No guard prevents this. The existing test (summaryLoop.test.ts:219-248) only tests same-boardId dedup, not cross-board concurrent staleness. The `readProject` call is not repeated after the await boundary.

Note: `buildProjectRollup` on line 177 also uses `r.doc`, so the project-level board counts share the same staleness window.
- **Fix direction:** Re-read the project doc after `await runSummarize` completes (a second `deps.readProject(dir)` call before lines 176-177) so the board list used for `buildMemoryIndex` and `buildProjectRollup` is current. Alternatively, hold a global serialization lock (one intent at a time) to eliminate the concurrent-write window entirely, though that adds latency.
