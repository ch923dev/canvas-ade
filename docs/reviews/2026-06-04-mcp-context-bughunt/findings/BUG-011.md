### (1) llm:summarize swallows unvalidated IPC input — null/missing `text` field reaches the provider as `undefined` — `src/main/llmIpc.ts:82`

- **Where:** `src/main/llmIpc.ts:82`
- **Sev/cat:** Low / error-handling
- **What:** The `llm:summarize` handler at line 81 receives `input: SummarizeInput` from the renderer, but the IPC argument is `unknown` at runtime. If the caller sends `{}` or `{ system: 'x' }` (no `text` field), `input.text` is `undefined`. `chatMessages` at llmService.ts:42 does `msgs.push({ role: 'user', content: input.text })`, sending `content: undefined` to the provider. `JSON.stringify` serializes `undefined` as `null` in an array, so the actual JSON body is `"content": null` — a shape most providers reject with an HTTP 400/422 error.
- **Trigger:** Any renderer-side caller (or a future automated summarize path) invokes `llm:summarize` without the `text` property. The provider returns a 400 error; `res.text()` is interpolated into the error message and sent back over IPC. Under the mock seam the mock returns `[mock] undefined` without error, masking the bug in tests.
- **Evidence:** The IPC handler at llmIpc.ts:81 accepts `input: SummarizeInput` with no runtime validation. If a caller sends `{}` or `{ system: 'x' }`, `input.text` is `undefined` at runtime. The code path: `runSummarize` → `provider.summarize(input)` → `buildRequest` → `chatMessages(input)` pushes `{ role: 'user', content: input.text }` (llmService.ts:42). `JSON.stringify({ role: 'user', content: undefined })` serializes as `{"role":"user"}` (the `content` key is omitted entirely), producing a malformed body that most providers reject with HTTP 400/422. The mock path (`[mock] ${input.text}` → `[mock] undefined`) succeeds silently, masking the defect in all current tests. HOWEVER: the bug is currently dormant — there are zero renderer-side callers of `window.api.llm.summarize` in the codebase (SettingsModal only calls status/setKey/clearKey/setConfig). The only code path that calls `runSummarize` is `summaryLoop.ts`, which calls it directly (bypassing IPC) and always provides a valid `text` string via `buildSummarizeInput` (line 85: `text.length > 0 ? text : 'Empty board.'`). The bug is a latent defect that fires the moment a renderer-side caller is added without a non-empty `text` guard. Severity downgraded from Medium to Low given zero current exposure and TypeScript compile-time enforcement protecting typed callers.
- **Fix direction:** Add a runtime guard at the top of the `llm:summarize` handler: `if (typeof input?.text !== 'string' || input.text.length === 0) return { ok: false, reason: 'provider-error', message: 'invalid input: text is required' }`. This closes the gap for any future renderer caller without touching the internal summary loop path.

### (2) llm:summarize does not validate that input.text is a non-empty string — undefined text is sent to the provider — `src/main/llmIpc.ts:81`

- **Where:** `src/main/llmIpc.ts:81`
- **Sev/cat:** Low / correctness
- **What:** The `llm:summarize` handler passes its second IPC argument directly to `runSummarize(config, input, deps)` with no runtime validation. If the renderer sends `{ system: 'x' }` (no `text` key), `input.text` is `undefined`. `runSummarize` calls `provider.summarize(input)`, which calls `buildRequest`, which calls `JSON.stringify({ … messages: chatMessages(input) })`. Inside `chatMessages`, `msgs.push({ role: 'user', content: input.text })` serialises `content: undefined` — JSON.stringify drops undefined object values — so the sent request body contains `{ role: 'user' }` with no `content`. Most provider APIs treat this as a 400 error; the budget counter has already been incremented at that point (tryConsume ran before the network call), so the call is charged against the daily cap even though no useful work was done. If `input` itself is `null` or not an object, the handler throws synchronously (TypeError accessing `input.text`), which becomes an untyped IPC rejection.
- **Trigger:** Call `ipcRenderer.invoke('llm:summarize', { system: 'prompt' })` (no `text` key), or `ipcRenderer.invoke('llm:summarize', null)`. In the first case a malformed request is sent to the LLM provider and the budget counter is consumed. In the second case the handler throws a TypeError that propagates as an untyped IPC error.
- **Evidence:** Traced fully in Z:/Canvas ADE/.claude/worktrees/audit-main/src/main/llmIpc.ts (line 81) and llmService.ts.

Code path for `{ system: 'x' }` (no `text` key):
1. llmIpc.ts:81 — handler receives `input` typed as `SummarizeInput` but no runtime validation; passes directly to `runSummarize`.
2. llmService.ts:209 — `getProvider` returns a live provider if a key is configured (no input inspection).
3. llmService.ts:214 — `deps.budget.tryConsume(cap)` fires and increments the counter BEFORE the network call.
4. llmService.ts:217 — `provider.summarize(input)` called with `input.text === undefined`.
5. llmService.ts:83 — `JSON.stringify({ model, messages: chatMessages(input) })` where `chatMessages` (line 42) pushes `{ role: 'user', content: undefined }`; JSON.stringify silently drops the `undefined` key, producing `{ "role": "user" }` in the body.
6. The provider HTTP layer returns a 400; this is caught at lines 218–223 and returned as `{ ok: false, reason: 'provider-error', message: '...' }`. Budget is not refunded (documented, fail-closed).

Partial refutation of the null-input claim: the description says `null` input "throws synchronously becoming an untyped IPC rejection." This is incorrect — the TypeError from accessing `null.text` inside `provider.summarize(input)` (whether the mock lambda at line 162 or the real provider) is caught by the try/catch at lines 216–222 and returns a well-typed `SummarizeResult`. No untyped rejection occurs.

The real confirmed part: missing `text` field causes a malformed HTTP request (no `content` in the user message) AND wastes a daily budget slot, with no input guard at either the IPC handler or `runSummarize` entry point. No existing test covers this scenario.
- **Fix direction:** Add a runtime guard at the top of the `llm:summarize` handler (or inside `runSummarize`) that checks `typeof input?.text === 'string' && input.text.length > 0`, returning `{ ok: false, reason: 'provider-error', message: 'invalid input' }` before budget is touched.
