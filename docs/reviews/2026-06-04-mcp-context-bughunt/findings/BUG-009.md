## Summary

### (1) closeBoard: cap slot not freed when removeBoard command fails (throws before tracked.delete) — `src/main/mcpOrchestrator.ts:247`

- **Where:** `src/main/mcpOrchestrator.ts:247`
- **Sev/cat:** Medium / correctness
- **What:** At line 247, if `registry.sendCommand({type:'removeBoard',...})` returns `{ok:false}` the method throws before reaching `tracked.delete(boardId)` at line 249. The PTY has already been drained (line 244) — the process is killed — but the board id remains in `tracked`. This means (1) the cap slot is permanently consumed, and (2) `reapIdle` will try to close the same board again on the next sweep, calling `drainPty` on an already-dead PTY session.
- **Trigger:** MCP calls `close_board` (directly or via `reapIdle`). The renderer window is destroyed or the 2-second ACK timeout fires during the `removeBoard` roundtrip. The PTY is killed but the `tracked` entry survives, burning the cap slot and triggering repeated reap attempts.
- **Evidence:** Code-path argument with line cites:

`closeBoard` (lines 241–250, `/src/main/mcpOrchestrator.ts`):
1. Line 244: `await registry.drainPty(boardId)` — always resolves, PTY is killed.
2. Line 246: `const ack = await registry.sendCommand({ type: 'removeBoard', id: boardId })`
3. Line 247: `if (!ack.ok) throw new Error(...)` — throws before reaching line 249.
4. Line 249: `tracked.delete(boardId)` — NEVER executed on the throw path.

`tracked` is the cap budget map. Two search results confirm `tracked.delete` is only called at line 186 (reconcile, board absent from live mirror) and line 249 (closeBoard success). Neither fires when `sendCommand` returns `{ok:false}`.

`reconcile()` (lines 182–188) drops a tracked entry only when `!live.has(id)`. But if `sendCommand({type:'removeBoard'})` returns `{ok:false}`, the renderer rejected the removal — the board is still in `registry.listBoards()`, so `live.has(id)` is true and reconcile never prunes it.

`reapIdle` (line 273): `for (const id of reapable) await this.closeBoard(id)` — sequential, no try/catch; the throw from `closeBoard` propagates out of `reapIdle`. In `mcp.ts` line 40: `void orchestrator.reapIdle().catch(() => {})` — the error is swallowed. On the next timer tick, `reconcile()` still won't prune the stuck id (board is live), `idleSince` is already set and past TTL, so the id re-enters `reapable` and `closeBoard` is called again — indefinitely.

Concrete trigger: the renderer window is destroyed or a 2-second ACK timeout fires during the `removeBoard` roundtrip. `drainPty` has already killed the PTY. The board remains tracked, burning the cap slot and causing repeated failed reap attempts every sweep (~60 s interval).

No existing test covers the `removeBoard`-fails-during-reapIdle scenario. The "throws when renderer rejects removeBoard" test (line 218) tests `closeBoard` in isolation and only confirms the throw; it does not check the state of `tracked` afterwards, nor does any reapIdle test use a failing `sendCommand`.
- **Fix direction:** Wrap `tracked.delete(boardId)` in a `finally` block (or delete before throwing) so the cap slot is freed regardless of whether `sendCommand` succeeds. Separately, catch errors per-id in the `reapIdle` loop (`for (const id of reapable) { try { await this.closeBoard(id) } catch { /* log, continue */ } }`) so a single failing close does not abort the entire reap sweep and does not cause an uncaught rejection to escape `reapIdle`.

### (2) reapIdle: closeBoard failure inside the reap loop is unhandled — reaper aborts mid-sweep — `src/main/mcpOrchestrator.ts:273`

- **Where:** `src/main/mcpOrchestrator.ts:273`
- **Sev/cat:** Medium / correctness
- **What:** `reapIdle` calls `await this.closeBoard(id)` for each reapable id in a sequential for loop (line 273). `closeBoard` throws when the `removeBoard` command fails. An uncaught throw from the first reap causes the entire `reapIdle` call to reject, leaving all subsequent reapable ids in `tracked` (not reaped) and `idleSince` still set past the TTL. Every future sweep will attempt to reap the same stuck id first, fail, and again skip the rest — making the reaper effectively broken for multiple simultaneous reapable boards.
- **Trigger:** Two or more MCP-spawned boards go idle past the TTL at the same time, and the renderer fails to ack the first `removeBoard` (e.g., window is reloading). The second board is never reaped in that sweep, and subsequent sweeps keep failing on the first board.
- **Evidence:** Scratch test confirmed the exact failure mode. Code path: `reapIdle()` at `src/main/mcpOrchestrator.ts:273` runs `for (const id of reapable) await this.closeBoard(id)` with no try/catch. `closeBoard` at line 247 throws `new Error('close_board failed: ...')` when `ack.ok` is false. The throw propagates out of `reapIdle`, abandoning all remaining ids in `reapable`. The `.catch(() => {})` at `mcp.ts:40` silences the rejection but does NOT resume the loop. The failing board stays in `tracked` with `idleSince` past TTL because `tracked.delete(boardId)` at line 249 is unreachable after the throw — so every future sweep retries the same stuck board first, fails again, and skips all others. Scratch test output: `threwDuringReap: true`, `removedIds: [id1 only]`, `id2` never attempted. The test's assertions for the buggy behavior all passed.
- **Fix direction:** Wrap each `await this.closeBoard(id)` call in a per-iteration try/catch inside the reapIdle loop. On failure, log a warning and continue to the next id rather than propagating the rejection. Consider only including successfully-closed ids in the returned array.
