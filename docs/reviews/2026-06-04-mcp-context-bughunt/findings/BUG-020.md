# BUG-020: dispatchGuard outstanding-nonce set grows unbounded on denied/failed dispatch

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcpOrchestrator.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T4.3
- **Collides with:** BUG-002, BUG-008, BUG-009, BUG-021
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
In `dispatchGuard.ts`, `issue()` adds a nonce to the `outstanding` Set; `consume()` removes it. In every dispatch method (handoffPrompt, dispatchPrompt, relayPrompt, interrupt), a nonce is issued at step (3) before the confirm. If the confirm is denied (step 4), the nonce is recorded in the audit trail but never passed to `consume()` — it stays in the `outstanding` Set forever. Over a long session with many denied dispatches, the `outstanding` set grows without bound (one entry per denied dispatch). Similarly, if `guard.consume(nonce)` returns false (replay guard at step 5), the nonce was never in the set (or already consumed), but for the deny path the nonce IS in the set and is leaked.

## Where
`src/main/mcpOrchestrator.ts:170`

## How it triggers
An MCP session that issues many `handoff_prompt` requests that the human repeatedly denies. Each denial leaves one UUID in the `outstanding` set. Over hours/days the set grows proportionally to the number of denials. There is no TTL, GC, or max-size on `outstanding`.

## Verification evidence
Exact code path: `dispatchGuard.ts:34` — `issue()` unconditionally calls `outstanding.add(nonce)`. In every dispatch method (handoffPrompt line 340, dispatchPrompt line 479, relayPrompt line 609, interrupt line 698), a nonce is minted before the human confirm. The deny branch (`!approved`) records an audit entry and then throws (e.g. handoffPrompt line 357: `throw new Error('handoff_prompt: dispatch denied by the human gate')`). The `guard.consume(nonce)` call is at line 362 — AFTER the throw — and is therefore unreachable on the deny path. `dispatchGuard.ts` has no `clear()`, no TTL, no size cap, and no other removal path beyond `consume()` (which calls `outstanding.delete()`). The `DispatchGuard` interface exposes only `issue()` and `consume()`. The guard instance is process-scoped (created once in `buildOrchestrator`, line 169, kept for the process lifetime). Each denied dispatch leaks one UUID (~36 bytes) into the `outstanding` Set permanently. The existing test suite (`dispatchGuard.test.ts`) covers issue/consume/replay but has no test for the deny-path leak. Practical severity is low: at typical human-driven denial rates, accumulation is negligible (millions of denials needed for measurable memory pressure), but the leak is structurally unbounded with no mitigation in place.

## Suggested fix direction
Call `guard.consume(nonce)` (or add a dedicated `guard.revoke(nonce)` that calls `outstanding.delete()`) in the deny branch before throwing, so denied nonces are removed from the set. Alternatively, add a `clear()` or max-size eviction to `DispatchGuard` and call it on orchestrator teardown.
