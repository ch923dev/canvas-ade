# BUG-003: Provider HTTP error body echoed into IPC `message` → latent secret/data leak

- **Severity:** Medium
- **Category:** security
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/llmService.ts`
- **Roadmap:** none
- **Collides with:** BUG-001
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
When a provider returns a non-2xx response, the error is constructed as `${config.provider} HTTP ${res.status}: ${await res.text()}` (line 183). This raw response body — which may contain API error detail, request IDs, or in misconfiguration scenarios partial key material echoed back by a provider's error payload — is stored in `SummarizeResult.message` and transmitted over IPC to the renderer. The renderer can then surface or log this string.

## Where
`src/main/llmService.ts:183`

## How it triggers
An OpenAI/Anthropic API key is partially invalid (e.g. truncated). The provider responds with `{"error": {"message": "Incorrect API key provided: sk-abc...", "type": "invalid_request_error"}}`. The full response body including the truncated key is packed into `message` and returned to the renderer via `llm:summarize`.

## Verification evidence
The code path is watertight:

1. `llmService.ts:183`: `throw new Error(\`${config.provider} HTTP ${res.status}: ${await res.text()}\`)` — raw provider HTTP response body (no sanitization) is placed verbatim into the Error message. A provider like OpenAI returns `{"error":{"message":"Incorrect API key provided: sk-abc...","type":"invalid_request_error"}}` as the body on a 401, so the truncated key is in the string.

2. `llmService.ts:218-223` (`runSummarize`): catch block does `message: err instanceof Error ? err.message : String(err)` → the full raw body string enters `SummarizeResult.message`.

3. `llmIpc.ts:81-85`: `ipcMain.handle('llm:summarize', ...)` returns the `SummarizeResult` directly; the `message` field is transmitted to the renderer verbatim.

4. `preload/index.ts:215-216`: `summarize: (input) => ipcRenderer.invoke('llm:summarize', input)` — the renderer can receive and read `result.message`.

The design comment in `llmIpc.ts:6` explicitly states "no key/secret is logged or surfaced", but the provider's own error response body is never stripped.

**Currently dormant**: Grep of `src/renderer` shows zero existing renderer code that calls `llm:summarize` or surfaces its `message` field. `summaryLoop.ts:169` discards errors with `if (!result.ok) return`. So there is no active UI/log exposure today — but the IPC channel is fully wired and any future renderer consumer of this channel would receive potentially key-bearing error text. The bug is real and the code path is confirmed; the risk is latent not active.

## Suggested fix direction
In `llmService.ts` line 183, strip or truncate the raw response body before embedding it in the error: either omit the body entirely (`throw new Error(\`${config.provider} HTTP ${res.status}\`)`), or sanitize it by truncating to a safe length and stripping anything matching a key pattern. The `message` field in `SummarizeResult` should carry only a safe, opaque error string — not raw provider response content.
