# BUG-027: memory:readBoards instantiates a new CanvasMemory on every IPC call

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/projectIpc.ts`
- **Roadmap:** none
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
The `memory:readBoards` handler at line 217 calls `createCanvasMemory(dir)` on every invocation. While not an immediate correctness bug, `createCanvasMemory` includes path construction, scaffold checks, and file-system access setup. More importantly, the handler iterates over caller-supplied `ids` with no bound on how many ids can be submitted. A renderer (or a compromised preload) can submit an array of thousands of ids, causing thousands of `readBoard` calls (each doing `existsSync` + `readFileSync`) per IPC call, and creates a fresh memory object each time. There is no limit on the size of the `ids` array.

## Where
`src/main/projectIpc.ts:217`

## How it triggers
The renderer (or any frame that passes the foreign-sender check) calls `window.api.memory.readBoards(Array.from({length: 10000}, (_, i) => 'id' + i))`. Each element goes through `SAFE_ID.test(id)` (cheap) but also `existsSync(file)` for every id. With 10,000 ids, this is 10,000 synchronous filesystem stats on the IPC thread, blocking other IPC handlers.

## Verification evidence
The handler at `main` branch `src/main/projectIpc.ts` line 217 does iterate over a caller-supplied `ids: string[]` with no length cap. The full path:

1. `guard(e)` checks sender is the main-frame renderer — passes for the real renderer
2. `createCanvasMemory(dir)` — construction is pure `path.join`, zero I/O, so the "overhead per call" claim is overstated
3. For each `id` in `ids`: `readBoard(id)` calls `safeBoardId(id)` (regex, cheap) then `readMd(paths.board(id))` which calls `existsSync(file)` synchronously — one syscall per id
4. With 10,000 valid-format ids (matching `/^[A-Za-z0-9_-]+$/`, ≤64 chars), this is 10,000 synchronous `existsSync` calls on the IPC thread

The legitimate caller (`Canvas.tsx:237`) passes `useCanvasStore.getState().boards.map(b => b.id)` — naturally bounded to real board count. The theoretical attack requires a compromised renderer (XSS in a planning board or dev-tools abuse) calling `window.api.memory.readBoards(Array.from({length:10000}, (_,i) => 'id'+i))`, which is single-user desktop threat territory.

The bug report's claim that "analogous `sanitizeSnapshot`/`sanitizeConnectors` in `boardRegistry.ts` apply explicit `MAX_BOARDS`/`MAX_CONNECTORS` caps" is FALSE — `git grep MAX_BOARDS` returns nothing; that defensive parallel does not exist in this codebase. So the relative severity is lower than presented.

The no-cap defect is real but exploitable only by a compromised renderer against a single-user desktop app (no remote/web attack vector). Severity should be Low, not Medium.

## Suggested fix direction
Add `const MAX_IDS = 256` (or similar, matching the practical board-count ceiling) and early-return `{}` if `ids.length > MAX_IDS` before constructing `mem`. No need to change `createCanvasMemory` — it does no I/O at construction.
