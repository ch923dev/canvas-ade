# BUG-019: boardMemory: no board-ID length cap on filename construction

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/boardMemory.ts`
- **Roadmap:** none
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
readBoardSummary validates the board id only with a regex (SAFE_ID) but applies no upper-length bound. canvasMemory.ts applies the same regex AND a MAX_ID_LEN=64 cap via safeBoardId(). boardMemory.ts has no such cap, so an agent-controlled id of e.g. 100 000 valid alphanum characters passes the regex, reaches join(root, `board-${id}.md`), and the resulting path string is passed to existsSync then readFileSync. On Windows this causes ERROR_FILENAME_EXCED_RANGE (path >32767 chars); on Linux it fails with ENAMETOOLONG at the kernel. Both are caught by readDoc's try/catch (graceful empty), but the path construction itself and the filesystem syscall are unnecessary and the divergence from canvasMemory.ts's own defence is a latent inconsistency.

## Where
`src/main/boardMemory.ts:57`

## How it triggers
MCP agent (or any caller) passes a syntactically valid but arbitrarily long board id (e.g. 'a'.repeat(100000)) to the canvas://board/{id}/summary resource URI.

## Verification evidence
The code path is unambiguous. In `boardMemory.ts` line 57, `readBoardSummary` guards with `SAFE_ID.test(id)` (regex `/^[A-Za-z0-9_-]+$/`) but no length cap. A caller passing `'a'.repeat(100_000)` passes the regex, enters `readDoc(join(root, 'board-' + id + '.md'))`, and the resulting ~100 016-char path is handed to `existsSync` then `readFileSync`. On Windows this triggers `ERROR_FILENAME_EXCED_RANGE` (path > 32 767 chars); on Linux it triggers `ENAMETOOLONG`. Both are caught by `readDoc`'s `try/catch` at line 43, returning `{ present: false, text: '' }` — so there is no crash or data disclosure, just a wasted syscall.

The divergence from `canvasMemory.ts` is explicit: that file defines `const MAX_ID_LEN = 64` (line 26) and uses it in `safeBoardId()` (line 28), which gates every write and read through the memory engine. `boardMemory.ts`, which is the MCP-facing read path (`canvas://board/{id}/summary` → published `@expanse-ade/mcp` dist/index.js line 218-220 → `orchestrator.boardSummary` → `registry.readSummary` → `readBoardSummary`), omits this cap entirely.

The MCP resource handler in the published package (dist/index.js line 218-220) performs only a truthiness check (`if (!id) throw`) — no length validation — so an agent-controlled oversized ID flows unimpeded into the filesystem call.

Severity is correctly noted as a latent inconsistency / unnecessary syscall with graceful fallback, not a security boundary violation (path traversal is genuinely prevented by the charset). Downgrading to Low: no crash, no data exposure, no privilege escalation — just a wasted OS call and an API-surface inconsistency.

## Suggested fix direction
Add `const MAX_ID_LEN = 64` to `boardMemory.ts` and include `id.length <= MAX_ID_LEN` in the guard at line 57, mirroring `canvasMemory.ts`'s `safeBoardId()` exactly.
