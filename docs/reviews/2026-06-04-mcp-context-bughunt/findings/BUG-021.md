## Summary

### (1) relayPrompt does not re-verify connector presence after confirm wait — TOCTOU on authorization — `src/main/mcpOrchestrator.ts:549`

- **Where:** `src/main/mcpOrchestrator.ts:549`
- **Sev/cat:** Low / security
- **What:** In `relayPrompt`, the orchestration connector is verified at step (1) (line 554-570) before any confirm. However after the mandatory confirm wait at step (4) (line 612-626), the connector is not re-checked. A connector could be deleted from the canvas (removing the cable between source and target) during the confirm wait. After a human approves based on 'this cable authorizes the relay', the relay executes even though the cable no longer exists.
- **Trigger:** User A calls relayPrompt, the confirm modal shows 'Relay source→target authorized by cable X'. While the modal is open, user B (or the same user via canvas interaction) deletes the cable. User A approves. The relay executes without a live authorization cable, violating the invariant 'the cable IS the authorization' stated at line 554.
- **Evidence:** Concrete code-path argument:

1. `relayPrompt` checks `registry.listConnectors()` at line 556–560 (reads `connectorMirror`, a mutable module-level `let` in `boardRegistry.ts` line 46).
2. The function then `await`s `registry.confirm(...)` at line 612, which calls `requestConfirm()` in `mcpConfirm.ts` — an unbounded async wait with NO default timeout (`DEFAULT_TIMEOUT_MS = undefined`, line 35 of `mcpConfirm.ts`).
3. While the confirm is pending, `registerBoardRegistryHandler`'s `ipcMain.on('mcp:boards', ...)` (boardRegistry.ts line 158) can fire at any time, overwriting `connectorMirror = sanitizeConnectors(connectors)` (line 169). This happens whenever the renderer pushes a canvas state update — e.g., when the user deletes the cable on the canvas.
4. After `approved` resolves at line 616, the code proceeds directly to nonce consumption (line 629) and PTY write (line 642). There is NO re-check of `listConnectors()`, no saved snapshot of `cable` that would be re-validated, and no mutex/lock anywhere in the orchestrator.
5. The comment at line 554 explicitly declares "The cable IS the authorization", making this a violated invariant, not just a latent gap.
6. No existing test covers this scenario — all relay tests use a static `listConnectors: () => opts.connectors ?? []` mock that never mutates during the confirm await.
- **Fix direction:** After `await registry.confirm(...)` resolves with `approved: true`, re-call `registry.listConnectors().find(...)` with the same predicate and reject (audit `status: 'rejected'`) if the cable is no longer present before proceeding to nonce consumption and PTY write. Alternatively, capture the `cable.id` before the confirm and re-verify it by ID post-confirm.

### (2) relayPrompt does not verify that sourceId is the authenticated caller's own board — `src/main/mcpOrchestrator.ts:577`

- **Where:** `src/main/mcpOrchestrator.ts:577`
- **Sev/cat:** Low / security
- **What:** `relayPrompt(sourceId, targetId, text)` accepts an arbitrary `sourceId` parameter from the MCP tool call. It checks that a connector `sourceId → targetId` exists, but does not verify that the calling MCP worker token is actually bound to `sourceId`. An orchestrator-tier token (which has full access) can call `relayPrompt` with any sourceId to exploit any existing orchestration connector, even cables intended for other agent pairs.
- **Trigger:** MCP client with orchestrator-tier token calls `relay_prompt` with `sourceId=board_A, targetId=board_B` where the cable A→B was set up for a different agent workflow. The call is authorized because the cable exists, even though the caller is not board_A's agent.
- **Evidence:** Code path confirmed:

1. `registerRelayPrompt` (`dist/index.js` line 415) registers `relay_prompt` under the orchestrator-only gate (`ctx.tier === "orchestrator"` block, line 445) — so worker tokens cannot call it. That is the first defense.

2. However, `registerRelayPrompt` does NOT receive `ctx` (compare with `registerWriteResult` at line 373 which correctly receives `ctx` and calls `orchestrator.writeResult(ctx.boardId, …)` to bind the write to the caller's own board). `relay_prompt`'s handler is `async (args) => orchestrator.relayPrompt(args.sourceId, args.targetId, args.prompt)` — `sourceId` is entirely caller-supplied and is never compared to the token's `ctx.boardId`.

3. Inside `mcpOrchestrator.ts` `relayPrompt` (lines 556-571), the only authorization check is `registry.listConnectors().find(c => c.kind === 'orchestration' && c.sourceId === sourceId && c.targetId === targetId)`. If the cable exists, the relay proceeds regardless of which caller is holding the orchestrator token.

4. `mcp.ts` line 30-33 mints exactly ONE orchestrator token (`boardId: 'app'`), and worker tokens are tier `'worker'` and therefore cannot reach `relay_prompt` at all. This substantially limits the practical attack surface — exploitation requires the single orchestrator token.

5. However, the `mintBoardToken` API fully supports minting multiple orchestrator-tier tokens (`tier: 'orchestrator'`). If the system ever issues orchestrator tokens to external agents or multiple parties, any holder can pass an arbitrary `sourceId` to exploit any cable not intended for them. The design comment "the cable IS the authorization" conflates route-existence with caller-identity.

The pattern-level defect is confirmed: `sourceId` in `relay_prompt` is not bound to `ctx.boardId` (unlike `write_result`). In the current single-orchestrator-token deployment the exploitability is low, so severity should be downgraded from Medium to Low.
- **Fix direction:** Pass `ctx` into `registerRelayPrompt` (same pattern as `registerWriteResult`), then add a guard at the top of the handler: if `ctx.boardId !== args.sourceId` throw an unauthorized error — or alternatively assert `ctx.boardId === 'app'` (the only current orchestrator identity). The analogous correct pattern is already present in `writeResult` at dist/index.js line 385.
