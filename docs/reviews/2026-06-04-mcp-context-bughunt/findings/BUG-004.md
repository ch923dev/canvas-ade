- **Severity:** Medium
- **Category:** data-integrity
- **Status:** CONFIRMED (adversarially verified)

- **Roadmap:** none
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) Tampered/float calls value bypasses cap enforcement — `src/main/llmBudget.ts:56`

- **Where:** `src/main/llmBudget.ts:56`
- **Sev/cat:** Low / data-integrity
- **What:** The read() validator accepts any `number >= 0` for `calls`, including non-integer floats (e.g. 0.5) and very large floats. `state.calls + 1` on a float like 0.5 yields 1.5, which can stay below the cap indefinitely — or a persisted value of `cap - 0.5` passes the `>= cap` check (false) and allows one extra call. Additionally, a value of `Infinity` passes `typeof p.calls === 'number' && p.calls >= 0`, and `Infinity >= cap` is true so the check blocks, but then the written value `Infinity + 1` is still `Infinity`, wedging the budget permanently at cap regardless of a new day (the `stored.day !== today` branch resets only when the stored day differs — a corrupted `day` could still mismatch, but the float/Infinity scenario on the current day does not reset).
- **Trigger:** Manually edit `llm-budget.json` to set `calls` to a fractional value (e.g. `0.5`) or to `Infinity`. On the next `tryConsume(200)` call, `0.5 >= 200` is false so it increments to `1.5`, consuming an extra slot. With `Infinity`, `Infinity >= cap` is true so the budget is frozen forever until the day flips.
- **Evidence:** The validator at line 56 of `src/main/llmBudget.ts` is confirmed missing `Number.isFinite()` and `Number.isInteger()` guards.

**Infinity scenario (real, low practical impact):** `JSON.parse('{"calls":1e309}')` produces `Infinity` in Node.js (verified). A manually crafted or externally corrupted `llm-budget.json` with `"calls": 1e309` passes the validator (`typeof Infinity === 'number'` is true, `Infinity >= 0` is true), stores `Infinity` in state, and causes `tryConsume` to permanently return `false` for the remainder of the calendar day (`Infinity >= cap` → true, no write, day-reset at line 79 only fires when `stored.day !== today`). The normal WRITE path cannot produce this state because `JSON.stringify({calls: Infinity})` → `{"calls":null}`, which then fails the `p.calls >= 0` check (null >= 0 is false). So this is a tampered-file-only attack on a local userData path.

**Float scenario (minimal impact, not as described):** Starting from `calls: 0.5`, `tryConsume(200)` increments to 1.5, 2.5, ..., 200.5 — still exactly 200 calls allowed before cap. The claim of "indefinitely below cap" is wrong; the claim of "one extra call at cap-0.5" is partially correct (the cap boundary shifts by 0.5 so the effective ceiling is 200.5 not 200.0), meaning at most one extra call if the injected starting point is crafted as `cap - epsilon`. This is negligible.

**Existing tests** cover normal integer flows and corruption (garbage JSON → reset to null → 0) but have no test for float or Infinity inputs. No upstream guard in `llmIpc.ts` or `llmService.ts` sanitizes the file content before it reaches the validator.

**Attack path:** Edit `~/.../userData/llm-budget.json` to `{"day":"2026-06-04","calls":1e309}` → launch app → budget frozen all day. Requires local file access.
- **Fix direction:** Add `Number.isFinite(p.calls) && Number.isInteger(p.calls)` to the validator at line 56, replacing the bare `p.calls >= 0` check. This rejects both float and Infinity inputs, causing a graceful reset to 0 with the existing warn+null path.

### (2) Persisted `calls` value is trusted without an upper-bound cap, allowing overflow denial-of-service — `src/main/llmBudget.ts:56`

- **Where:** `src/main/llmBudget.ts:56`
- **Sev/cat:** Medium / data-integrity
- **What:** The budget file `calls` field is read back and trusted as-is. There is no upper-bound check (e.g. `calls <= DEFAULT_MAX_CALLS_PER_DAY * 10` or `Number.MAX_SAFE_INTEGER`). A value of `Number.MAX_SAFE_INTEGER` or `2^53` written to disk (e.g. by a corrupted write or an adversarial file) passes validation and `state.calls >= cap` will be true for any reasonable cap, permanently blocking all LLM calls for the current day without triggering the day-reset path (because the stored `day` still matches today).
- **Trigger:** Write `{"day": "2026-06-04", "calls": 9007199254740991}` to `llm-budget.json` on the current day. Every subsequent `tryConsume(200)` call returns false (budget exceeded) and no reset occurs until midnight.
- **Evidence:** Probe test written to src/main/llmBudget.overflow.test.ts and run via `pnpm vitest run llmBudget.overflow` — 1/1 passed, confirming the DoS.

Code path (all line numbers in Z:/canvas-ade-integration/src/main/llmBudget.ts):
- L56: validation `typeof p.calls === 'number' && p.calls >= 0` — Number.MAX_SAFE_INTEGER (9007199254740991) satisfies both predicates. No upper-bound check, no Number.isFinite guard.
- L79: day-reset guard `if (!stored || stored.day !== today)` — with adversarial file carrying today's date, condition is false; poisoned value propagates.
- L90: `if (state.calls >= cap) return false` — `9007199254740991 >= 200` is true; tryConsume returns false for every call for the rest of the calendar day.

Trigger: write `{"day":"2026-06-04","calls":9007199254740991}` to userData/llm-budget.json (local user-writable path on Windows: %APPDATA%\Roaming\<appName>\llm-budget.json). No reset until midnight. Tier-2 brain silently falls back to Tier-1 for the full day.

No upstream guard exists in llmService.ts (L214 passes cap straight through) or summaryLoop.ts (L167 constructs the store with no pre-sanitization). The existing test suite (8 tests) does not cover this path.
- **Fix direction:** In the `read()` function, add an upper-bound check alongside the existing `>= 0` guard: e.g. `p.calls >= 0 && Number.isFinite(p.calls) && p.calls <= DEFAULT_MAX_CALLS_PER_DAY * 10`. Values that fail the new check should fall through to the `console.warn` + `return null` path (same as malformed shape), which causes `current()` to reset calls to 0 for today.
