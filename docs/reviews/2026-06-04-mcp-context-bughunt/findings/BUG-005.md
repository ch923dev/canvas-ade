- **Severity:** Medium
- **Category:** security
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/llmKeyStore.ts`
- **Roadmap:** none
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) Silent plaintext-on-disk fallback when encryption becomes unavailable after key is set — `src/main/llmKeyStore.ts:65`

- **Where:** `src/main/llmKeyStore.ts:65`
- **Sev/cat:** Medium / security
- **What:** `setKey` correctly refuses to write when `isEncryptionAvailable()` returns false (line 65). However, `getKey` (line 55-62) calls `decryptString` on whatever bytes are on disk — if safeStorage encryption was available when the key was written but later becomes unavailable (e.g. the OS keyring daemon is stopped, the user moves the app to a different OS user account, or decryptString throws), `getKey` returns `undefined` (swallowed error). There is no warning or notification to the user that their stored key is now inaccessible. The key remains on disk in its encrypted form but the service silently falls back to `no-provider`. More critically: there is no guard against a scenario where `isEncryptionAvailable()` returns false but the file on disk already contains data written under a different (weaker) code path — the read side never validates that the bytes are actually ciphertext rather than something written by a hypothetical older version of the code.
- **Trigger:** On Linux, the OS keyring (libsecret/kwallet) is running when the user stores a key. Later the keyring daemon is stopped (e.g. `systemctl stop gnome-keyring`). `isEncryptionAvailable()` now returns false. A subsequent `getKey` call tries `decryptString(buffer)`, which throws (Electron throws when the keyring is unavailable on decrypt too), and returns undefined silently. The user sees no error; the provider silently disappears.
- **Evidence:** Code-path argument, fully traced:

1. `llmKeyStore.ts` lines 58–62: `getKey` wraps `encryptor.decryptString(...)` in try/catch and returns `undefined` on any throw, with no logging. If the OS keyring is unavailable at read time (e.g., Linux `gnome-keyring` stopped after a key was written), Electron's `safeStorage.decryptString` throws, and the catch swallows it silently.

2. `llmService.ts` line 136, `keyForProvider`: `return store?.getKey(provider) ?? env[KEY_ENV[provider]]`. When `getKey` returns `undefined` and no env var fallback exists, this returns `undefined`.

3. `llmService.ts` line 165, `getProvider`: `if (config.provider !== 'local' && !key) return null`. Provider becomes null.

4. `llmService.ts` line 210, `runSummarize`: `if (!provider) return { ok: false, reason: 'no-provider' }`. The summary loop at `summaryLoop.ts` line 169 silently drops non-ok results.

5. Compounding inconsistency in `llm:status` (llmIpc.ts lines 87–103): `hasKey` is computed via `keyStore.hasKey(config.provider)`, which (llmKeyStore.ts lines 77–80) only checks that a non-empty base64 string is present on disk — it does NOT attempt decryption. So the status response can return `hasKey: true` while `getProvider` simultaneously returns `null`. The UI would show "key is set" but the provider is gone, with no error message to the user explaining why.

6. The "plaintext-on-disk from older code path" sub-claim is unsubstantiated — no prior plaintext-writing code path exists in the repo. That sub-claim is dismissed.

7. Existing tests in `llmKeyStore.test.ts` only test the "encryption unavailable at write time" path (test at line 56); no test covers "encryption available at write but unavailable at read."

Severity downgraded from High to Medium: this is a silent-failure / UX reliability bug, not a key-exposure bug. The encrypted bytes remain on disk and are never exposed as plaintext. The actual impact is the provider silently disappearing (appearing as `no-provider`) when the keyring is restarted after a key was stored, with the misleading `hasKey: true` status shown to the user.
- **Fix direction:** In `getKey`, check `isEncryptionAvailable()` before attempting decryption and return a distinct signal (or log a warning) when it returns false but data exists on disk, so callers can surface a meaningful error. Separately, fix `hasKey` to not return `true` when the key would not decrypt — or accept the inconsistency and document it. The plaintext-fallback sub-claim (different code path writing raw bytes) does not need addressing as no such path exists.

### (2) decryptString errors are silently swallowed — corrupt ciphertext indistinguishable from 'no key' — `src/main/llmKeyStore.ts:59`

- **Where:** `src/main/llmKeyStore.ts:59`
- **Sev/cat:** Medium / error-handling
- **What:** `getKey` wraps `decryptString` in a bare `catch { return undefined }` at line 58-61. Any error from decryption — wrong keyring user, tampered base64, OS keyring unavailable, Buffer corruption — silently returns `undefined`. This makes a tampered or corrupted key file look identical to 'no key configured', so `hasKey` returns true (the raw entry exists and is non-empty string, checked at line 79) while `getKey` returns undefined. This creates a split-brain: `llm:status` reports `hasKey: true` (via `hasKey`) but `getKey` (used by `keyForProvider`) returns undefined, so the provider is null, and `llm:summarize` returns `{ok:false, reason:'no-provider'}`. The user's Settings UI shows a key is stored but LLM calls silently fail.
- **Trigger:** Corrupt the base64 value for one provider in `llm-keys.json` (e.g. replace it with invalid base64 or valid base64 of random bytes). `hasKey('openrouter')` returns true (non-empty string exists). `getKey('openrouter')` calls `Buffer.from(corrupted, 'base64')` which succeeds (base64 is lenient) but `decryptString` throws → returns undefined. UI shows key present, all summarize calls return no-provider.
- **Evidence:** Targeted unit test (written, run green, deleted) proved it: write a valid llm-keys.json with a non-empty base64 entry for 'openrouter', inject an Encryptor whose decryptString always throws, then assert hasKey('openrouter')===true AND getKey('openrouter')===undefined — both pass. Code-path: llmKeyStore.ts:77-79 hasKey checks only typeof enc==='string' && enc.length>0 (no decryption); llmKeyStore.ts:55-62 getKey wraps decryptString in bare catch{return undefined}. In llmIpc.ts:97 getProvider calls keyForProvider→getKey→undefined, so getProvider returns null (llmService.ts:165 !key guard), making hasProvider:false. llmIpc.ts:101 hasKey:true from the raw-string check. Status response: {hasProvider:false, hasKey:true}. All llm:summarize calls return {ok:false,reason:'no-provider'}. Existing test llmKeyStore.test.ts:63-69 does NOT cover this: it corrupts the JSON file (readFile returns {}, so hasKey is also false — no split-brain). The split-brain only occurs when the file is valid JSON with a non-empty string but decryptString throws.
- **Fix direction:** hasKey should call getKey (or a shared internal that does the decrypt) rather than checking the raw file entry — or decryptString errors should be surfaced as a distinct 'corrupt' state distinct from 'absent'. Alternatively, catch the decrypt error, log it, and also delete or mark the corrupt entry so hasKey stays consistent. The key design principle violated is that hasKey and getKey must agree on what "present" means.
