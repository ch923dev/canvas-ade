# BUG-028: mcpSmoke temp dir not cleaned on unexpected throw

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcpSmoke.ts`
- **Roadmap:** none
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
In the memory smoke block (lines 644-678), a temp directory `root` is created with `mkdtempSync`. The cleanup `rmSync(root, ...)` and `__setMemoryDirForTest(null)` appear in two branches: when `emptyMem === null` (SKIP path, line 648-651) and after the full test (line 662-663). However, if `orch.readMemory()` at line 647 throws an unexpected error (not a `null` return, but a real transport exception), the error propagates up to the outer `try/catch` at line 418, bypassing the `rmSync` cleanup and leaving the temp dir on disk. The `__setMemoryDirForTest(null)` override also stays set, poisoning subsequent smoke probes in the same process.

## Where
`src/main/mcpSmoke.ts:645`

## How it triggers
An MCP transport error (e.g. the server drops the connection, Session not found -32001) occurs during `orch.readMemory()` in the memory block. The error propagates past the block's non-try-caught scope, `rmSync` is never called, and `overrideDir` in `boardMemory.ts` remains pointing to a now-leaked temp path for the rest of the process lifetime.

## Verification evidence
Code path is watertight. In `origin/integration/mcp-on-main:src/main/mcpSmoke.ts`:

1. `readMemory()` (lines 279-291) catches errors and returns `null` ONLY for JSON-RPC code -32602 or messages matching `/resource .*not found/i`. Any other exception (transport drop, -32001 Session not found, network reset, JSON parse error of a non-standard payload, etc.) is **re-thrown** (`throw e`, line 289).

2. The memory smoke block (lines 644-678) calls `mkdtempSync` (line 645) and `__setMemoryDirForTest(root)` (line 646) with **no try/finally wrapper**. Cleanup (`__setMemoryDirForTest(null)` + `rmSync(root, ...)`) appears only in two specific branches: the null/SKIP path (lines 649-650) and the happy-path else branch (lines 662-663).

3. If `orch.readMemory()` at line 647 throws any non-suppressed error, execution jumps directly to the outer `catch (err)` at line 1211. No cleanup runs: the `mkdtempSync` temp dir is left on disk, and `__setMemoryDirForTest` remains set to `root` — poisoning any subsequent code in the same process that reads the memory dir override. The same applies to the second `readMemory()` call at line 658 and the `readSummaryDoc` calls at lines 659-661.

4. The outer catch (lines 1211-1213) only logs and sets `code=1`; it has no finally and performs no cleanup.

Concrete trigger: a `-32001 Session not found` error from the MCP server (documented MCP transport error) would not match `-32602` or `/resource .*not found/i`, would re-throw, and would hit this path. A plain `fetch` network error (ECONNREFUSED) would also re-throw.

## Suggested fix direction
Wrap the memory smoke block content in a try/finally: set a `let root: string | null = null` before the block, assign after mkdtempSync, and in the finally call `__setMemoryDirForTest(null)` and `if (root) rmSync(root, { recursive: true, force: true })`. This is smoke-test-only code (not production path), so the impact is limited to leaked temp dirs and a poisoned test-seam override in a dying process — hence downgraded to Low.
