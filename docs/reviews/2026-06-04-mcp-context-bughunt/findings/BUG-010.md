# BUG-010: MCP confirm request has no timeout → tool call hangs indefinitely

- **Severity:** Medium
- **Category:** resource-leak
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcpConfirm.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T4.2
- **Collides with:** BUG-022
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
`DEFAULT_TIMEOUT_MS` is explicitly set to `undefined` with the rationale 'a human takes time'. This means if a human never dismisses the modal (e.g. walks away, or the modal crashes/fails to render), the `requestConfirm` promise never resolves, and any MCP tool awaiting it (handoff_prompt, assign_prompt, relay_prompt, interrupt) blocks its HTTP response indefinitely. The MCP HTTP server's own request timeout (if any) is the only backstop — but this is not documented and may not exist.

## Where
`src/main/mcpConfirm.ts:35`

## How it triggers
A user starts a handoff_prompt dispatch, the confirm modal is shown, the user walks away or the modal fails to render correctly (e.g. because of a renderer crash that does NOT fire the render-process-gone event), or a denial-of-service condition where the modal is shown but the renderer never replies. The `requestConfirm` promise hangs permanently, holding the MCP HTTP connection open.

## Verification evidence
Code path: `index.ts:185` calls `requestConfirm(ipcMain, () => mainWindow, req)` with no `opts` argument. In `mcpConfirm.ts:58`, `timeoutMs = opts.timeoutMs ?? DEFAULT_TIMEOUT_MS` resolves to `undefined` (line 35). At line 84, `timeoutMs === undefined` → `timer = null` → no timeout fires. The two escape hatches (lines 88-89) listen for `webContents` `destroyed` and `render-process-gone` events, which only fire if the renderer process crashes/exits. A frozen renderer (UI deadlock, hung React event loop, or modal fails to register its reply handler) keeps the process alive — neither event fires — so the `Promise` at line 59 never resolves. The MCP HTTP layer (`@ch923dev/canvas-ade-mcp/dist/index.js`) uses bare `express` + `node:http createServer` with no `server.timeout`, `headersTimeout`, or `requestTimeout` set, and Node 18+'s default `server.timeout` is 0 (disabled). The SSE GET connection (`StreamableHTTPServerTransport`) stays open indefinitely awaiting the tool result. Concrete path: user triggers `handoff_prompt` → `mcpOrchestrator.ts:344` awaits `registry.confirm(...)` → `requestConfirm` promise never resolves → MCP HTTP handler never sends a response → SSE connection held open forever (resource leak). The existing test suite (`mcpConfirm.integration.test.ts`) covers `destroyed`/`render-process-gone`/send-throws but has no test for the frozen-renderer scenario where neither lifecycle event fires and no IPC reply ever arrives.

## Suggested fix direction
Add a reasonable wall-clock timeout (e.g. 5–10 minutes) as the default in `DEFAULT_TIMEOUT_MS`, or pass `timeoutMs` at the injection site in `index.ts:185`. A frozen renderer is the gap the current escape hatches cannot cover; a fallback timeout is the only backstop. The timeout should be generous enough for a human to respond but finite enough to prevent permanent connection holds.
