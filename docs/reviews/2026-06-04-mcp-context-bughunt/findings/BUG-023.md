# BUG-023: mcp.ts env TTL accepts zero/negative → idle-reap silently disabled

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcp.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T3.4
- **Collides with:** none
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
The expression `Number(process.env.CANVAS_MCP_IDLE_TTL_MS) || MCP_IDLE_TTL_MS` uses a JavaScript falsy-or fallback, which means any value that coerces to a falsy number (0, NaN, or an empty string) falls back to the default. However, a legitimate attacker or misconfigured env var can set `CANVAS_MCP_IDLE_TTL_MS=1` (truthy but 1 ms), making the TTL 1 millisecond. More importantly, a negative number like `-1` is truthy and passes, making the condition `t - rec.idleSince >= idleTtlMs` (`t - x >= -1`) always true on the very first sweep — causing every MCP-spawned board to be reaped immediately after its first idle classification, regardless of actual idle duration. This breaks the intended TTL semantic. Similarly `CANVAS_MCP_REAP_INTERVAL_MS=0` (falsy) falls back to the default, but `CANVAS_MCP_REAP_INTERVAL_MS=1` creates a 1-ms interval that hammers the process event loop. No validation floor is applied.

## Where
`src/main/mcp.ts:5`

## How it triggers
Set `CANVAS_MCP_IDLE_TTL_MS` to a negative integer (e.g. `-1`) in the environment before launching the app. Every MCP-spawned board will be closed immediately on the first reap sweep. Set `CANVAS_MCP_REAP_INTERVAL_MS=1` to create a hot polling loop.

## Verification evidence
The defect is real but the severity is overstated. Code confirmed in `Z:/Canvas ADE` git history (commit 2100022, file `src/main/mcp.ts` lines 5-6):

```ts
const IDLE_TTL_MS = Number(process.env.CANVAS_MCP_IDLE_TTL_MS) || MCP_IDLE_TTL_MS
const REAP_INTERVAL_MS = Number(process.env.CANVAS_MCP_REAP_INTERVAL_MS) || 60_000
```

The `||` fallback only fires for falsy values (0, NaN, empty string). A negative integer like `-1` is truthy and passes through as `IDLE_TTL_MS = -1`. This value is forwarded without a floor to `buildOrchestrator(registry, { idleTtlMs: IDLE_TTL_MS })` in `mcpOrchestrator.ts` line 166 (`opts.idleTtlMs ?? MCP_IDLE_TTL_MS` — the `??` guard only catches undefined/null, not negative).

In `reapIdle()` (lines 251-274 of mcpOrchestrator.ts): first sweep sets `idleSince = t` and continues; second sweep evaluates `t - rec.idleSince >= idleTtlMs` which becomes `~0ms >= -1` = always true, so every tracked board is reaped on the second sweep. Verified with Node.js: `Number('-1') || 300000 = -1`; `0 >= -1 = true`.

Similarly `CANVAS_MCP_REAP_INTERVAL_MS=1` produces a 1ms setInterval (confirmed: `Number('1') || 60000 = 1`).

**However, severity must be downgraded:** These env vars require process-level access before app launch — on this single-user Electron desktop app, that's equivalent to already having user-level code execution. No network/remote exploit path exists. The impact is limited to: (a) MCP-spawned boards prematurely closed (DoS of MCP feature, not data loss — boards are just removed from canvas); (b) hot poll loop wastes CPU. The comment explicitly calls these dev/smoke hooks ("Env-overridable so the live smoke can drive a fast reap"). No existing test validates range bounds. Fix: add `Math.max(1, ...)` floor on both constants.

## Suggested fix direction
Add a positive floor: `const IDLE_TTL_MS = Math.max(1, Number(process.env.CANVAS_MCP_IDLE_TTL_MS) || MCP_IDLE_TTL_MS)` and similarly for `REAP_INTERVAL_MS`. Optionally warn when the parsed value is negative/zero.
