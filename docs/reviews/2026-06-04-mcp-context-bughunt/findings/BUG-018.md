## Summary

### (1) Board title changes never trigger re-summarization for terminal and browser boards — `src/main/memoryEngine.ts:52`

- **Where:** `src/main/memoryEngine.ts:52`
- **Sev/cat:** Low / correctness
- **What:** boardFingerprint() excludes the title field from the fingerprint for both 'terminal' (includes only launchCommand/cwd/port) and 'browser' (includes only url/viewport) boards. The comment only justifies excluding geometry/selection fields. However summaryLoop.ts boardContent() DOES include the title in the text fed to the LLM (e.g. 'Terminal board "${title}".'), so a title rename changes the model input and would produce different summary prose — but the fingerprint stays identical, the debounce never fires, and the cached board-<id>.md summary remains stale with the old title forever.
- **Trigger:** User renames a terminal or browser board after at least one observe() baseline has been set. No intent is ever emitted; the stale summary persists in .canvas/memory/ until a non-title content field (launchCommand, url, etc.) also changes.
- **Evidence:** Concrete code-path argument:

1. `summaryLoop.boardContent()` (lines 46-53, 54-61) includes `b.title` as the first line of the LLM prompt: `Terminal board "${title}".` and `Browser preview board "${title}".`.

2. `memoryEngine.boardFingerprint()` (lines 52-58 terminal, 59-68 browser) hashes only `{t, launchCommand, cwd, port}` and `{t, url, viewport}` respectively — `title` is absent for both types.

3. A title rename produces an identical fingerprint → `armDebounce()` is never called → no `SummarizeIntent` emitted → `summaryLoop.onIntent()` never fires → `.canvas/memory/board-<id>.md` stays stale with the old title.

4. The test in `memoryEngine.test.ts` line 69-71 explicitly proves this: it passes `title: 'Renamed'` alongside geometry overrides and asserts the fingerprints are equal. The test passes as written (confirmed: 19/19 tests green via `pnpm vitest run memoryEngine`).

5. The docstring (lines 13-17) lists per-type fields for the fingerprint without mentioning board-level `title`; the comment for `previewSourceId` exclusion (lines 60-63) explicitly justifies it as "never reaches the summary input" — no such justification exists for excluding `title`, which DOES reach the summary input.

6. Planning boards have the same omission (board-level `title` absent from fingerprint, line 81), but for terminal/browser the title is the entire opening line of the LLM prompt, making the staleness more visible.

The omission is inconsistent with the stated design principle ("fingerprint mirrors the SUMMARY INPUT") and the `previewSourceId` exclusion rationale.
- **Fix direction:** Add `title: str(b.title)` to the terminal and browser fingerprint objects in `boardFingerprint()`. Also add a test asserting that a title-only rename of a terminal/browser board IS detected. The planning board-level title could also be included for completeness, but its impact is lower since checklist/note content dominates the LLM input.

### (2) First observe() after reset() sets a new baseline but does not emit; a content-identical re-open never emits — `src/main/memoryEngine.ts:154`

- **Where:** `src/main/memoryEngine.ts:154`
- **Sev/cat:** Low / correctness
- **What:** After reset(), primed is set to false. The next observe() re-baselines all boards (sets fingerprints, primed = true) without emitting. This is documented. However, if a project is closed and re-opened with the same canvas.json content, no intent will fire for any board. Since reset() is called on project switch, the second open of the same project has no outstanding summaries for unchanged boards — this is intentional. But if the .canvas/memory/ board files were deleted externally (e.g. by the user or a GC pass) between sessions, the engine will not re-summarize unchanged boards even though their summary files are now absent, leaving an empty memory dir until the next content change.
- **Trigger:** User deletes .canvas/memory/*.md manually, then re-opens the project without editing any board content. The engine baselines and never fires. The memory dir stays empty until a content change triggers a new summarize.
- **Evidence:** The code path is watertight. In `memoryEngine.ts` line 154, `if (!primed) continue` silences all emit on the first observe after a reset. `reset()` (line 168) clears `fingerprints` and sets `primed = false`. On both `project:open` (projectIpc.ts lines 110–114) and `project:current` (lines 178–180), the sequence is: `reset()` → `observe(doc)` (baseline, no emit) → return. There is no code anywhere that reads existing `.canvas/memory/board-*.md` files at open time and compares them to the current boards to schedule re-summarization for boards whose summary files are absent. `canvasMemory.readBoard()` is only called inside `summaryLoop.ts:onIntent` (line 176) while building the index after a summary run — it is never called at project-open time to detect missing files. `scaffoldProjectMemory` only ensures the `.canvas/` directory exists; it does not inspect individual board files. The trigger scenario (user deletes `board-*.md` files then re-opens the project without editing any board) leaves the memory dir empty indefinitely, because the engine baselines off the current doc content and sees no delta. The only recovery path is an actual content change to each board after re-open.
- **Fix direction:** At project open (after `reset()` + `observe()`), scan the memory dir for each board in the doc; for any board whose `board-<id>.md` file is absent, call `armDebounce(id)` after priming — or expose a `rehydrate(boardIds)` method on the engine that arms intents for explicitly listed boards, skipping the primed guard.
