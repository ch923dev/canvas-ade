## Summary
3 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) llm:setConfig accepts arbitrary baseUrl with no URL validation — SSRF / internal-network egress — `src/main/llmIpc.ts:128`

- **Where:** `src/main/llmIpc.ts:128`
- **Sev/cat:** High / security
- **What:** The `llm:setConfig` IPC handler (line 118-137) writes `a.baseUrl` verbatim into `LlmConfig` with no URL validation. The persisted value is then read back by `readLlmConfig` (which also passes it through unchanged at line 57) and used as the HTTP endpoint in `buildRequest` (llmService.ts line 77-78). A renderer-side caller can therefore point the LLM egress at any URL: `file://`, `http://localhost:2379` (etcd), `http://169.254.169.254/` (AWS IMDS), or any internal service. The `local` provider is the intended use-case, but nothing restricts it to `local` only or validates the scheme/host.
- **Trigger:** Renderer (or a compromised extension running in the renderer) calls `ipcRenderer.invoke('llm:setConfig', { provider: 'local', model: 'm', baseUrl: 'http://169.254.169.254/latest/meta-data/' })`. The next `llm:summarize` call sends the board content to the metadata endpoint and may exfiltrate the response (or credentials returned) as a provider-error message.
- **Evidence:** Complete code path verified with no refutation found:

1. `llmIpc.ts:128-134` — `llm:setConfig` writes `a.baseUrl` verbatim into `LlmConfig` with zero URL validation; the only guard is the frame-identity check (`guard(e)`) which passes for the legitimate renderer.

2. `llmConfig.ts:57` — `readLlmConfig` passes `baseUrl` back as-is after only a `typeof ... === 'string'` check — no scheme or host restriction.

3. `llmService.ts:71-78` — `buildRequest` for `provider === 'local'` assigns `base = config.baseUrl` (line 73) and constructs `url: \`${base}/chat/completions\`` (line 78). No validation intervenes.

4. `llmService.ts:177` — `deps.fetch(req.url, {...})` — Node global fetch in MAIN executes the HTTP call to the attacker-supplied URL.

5. `llmService.ts:183` — on non-2xx, `await res.text()` is embedded in the error message which propagates all the way back to the renderer as `{ ok: false, reason: 'provider-error', message: '...' }` — a data-exfiltration vector.

No existing test, type constraint, or runtime check validates the scheme or host of `baseUrl`. The `preview.ts` scheme allowlist (Bug #32) covers the `WebContentsView` preview board and is entirely separate from this LLM egress path. The foreign-sender guard only prevents cross-origin renderer frames from invoking the handler; the main renderer frame is unconditionally trusted.

Minimal repro (no test needed — the path is unambiguous):
```
ipcRenderer.invoke('llm:setConfig', { provider: 'local', model: 'm', baseUrl: 'http://169.254.169.254/latest/meta-data' })
// then:
ipcRenderer.invoke('llm:summarize', { text: 'x' })
// → { ok: false, reason: 'provider-error', message: 'local HTTP 200: <IMDS response body>' }
```
- **Fix direction:** Validate `baseUrl` in `llm:setConfig` (and/or in `writeLlmConfig` / `readLlmConfig`) against a strict allowlist: require `http:` or `https:` scheme, and restrict the host to `localhost` / `127.0.0.1` / `::1` (loopback-only), since `local` is the LM Studio / Ollama use-case. Reject any other scheme or host with `{ ok: false, reason: 'invalid-baseUrl' }` before writing to disk.

### (2) baseUrl is not validated as a URL or restricted to http/https schemes when read from config file — `src/main/llmConfig.ts:57`

- **Where:** `src/main/llmConfig.ts:57`
- **Sev/cat:** Low / security
- **What:** `readLlmConfig` at line 57 accepts any string as `baseUrl` with only a type check (`typeof p.baseUrl === 'string'`). This value is stored and later used as the base for the HTTP endpoint. A `file://` or `data:` URL in the config file would reach `deps.fetch` as the target URL. While `fetch` in Node.js 22 does not support `file://` URLs (it throws), other scheme strings that look URL-like but aren't could produce confusing error messages. The fix in `llm:setConfig` should also be applied at read-time.
- **Trigger:** The `llm-config.json` is manually edited or written by an external process to set `baseUrl: 'file:///etc/passwd'`. When `runSummarize` is called with the `local` provider, `deps.fetch` is called with `file:///etc/passwd/chat/completions` as the URL. Node 22 fetch throws `TypeError: fetch is not a function for file: URLs`, which is caught as a `provider-error` and the error message (potentially containing path details) is returned over IPC.
- **Evidence:** Code path is confirmed watertight: `readLlmConfig` line 57 (`typeof p.baseUrl === 'string' ? p.baseUrl : undefined`) passes any string scheme through. The value reaches `getProvider` (llmService.ts:169, truthy-only guard), then `buildRequest` (line 71-76, `base = config.baseUrl`), then `deps.fetch(${base}/chat/completions, ...)` (line 177). No scheme validation exists at any layer, including in `llm:setConfig` (llmIpc.ts:118-136) contrary to the candidate's framing.

Two concrete outcomes verified via Node 22.17.0:
1. `file://` URLs: `fetch` throws `TypeError: fetch failed` (cause: "not implemented... yet..."), caught by `runSummarize` as `{ok:false, reason:'provider-error', message:'fetch failed'}`. No path content leaks.
2. `data:` URLs: `fetch` returns HTTP 200 OK (Node 22 supports `data:` scheme). The body is the data content + "/chat/completions" appended. `JSON.parse` fails and the error message includes the first ~10 chars of the data content (e.g., `Unexpected token 'S', "SECRET_KEY"... is not valid JSON`), which is returned as `provider-error.message` over IPC to the renderer.

However: the attack vector requires write access to `app.getPath('userData')/llm-config.json` — the user's own local app data directory. In a single-user desktop app this is the user's own filesystem. The candidate's description of "externally edited config file" is the only realistic trigger. The data-URL info-leak only surfaces content the attacker already wrote, limiting the practical impact to essentially nil.
- **Fix direction:** Add scheme validation at both read-time (`readLlmConfig`) and write-time (`llm:setConfig` handler): accept only strings that pass `new URL(s)` without throwing and whose protocol is `http:` or `https:`. Reject (return `undefined`) at read-time; return `{ok:false, reason:'invalid-url'}` at write-time. Note: `llm:setConfig` currently has NO validation (the candidate's claim it has a "fix" is incorrect — both sites are unguarded).

### (3) Unvalidated `baseUrl` from config is used as a URL prefix without sanitization — `src/main/llmService.ts:71`

- **Where:** `src/main/llmService.ts:71`
- **Sev/cat:** Medium / security
- **What:** In `buildRequest` at line 71-73, `config.baseUrl` is used directly as the base URL for the `local` provider: `url: \`${base}/chat/completions\``. The `baseUrl` is read from `llm-config.json` which is written by the `llm:setConfig` IPC handler. While the IPC handler is frame-guarded, `baseUrl` is accepted as any string (line 57 of llmConfig.ts: `typeof p.baseUrl === 'string'`). A value like `http://internal-host/path?injected=` or `file:///etc/passwd` would be used as-is. The `fetch` call in Node/Electron MAIN can reach any URL including internal network addresses (SSRF) and potentially `file://` URLs.
- **Trigger:** A renderer (after bypassing the frame guard, or via a legitimate Settings UI call) sets `baseUrl` to `http://169.254.169.254/latest/meta-data/` (AWS instance metadata). Every `runSummarize` call with the `local` provider POSTs to the instance metadata endpoint, exfiltrating cloud credentials in the request body (the LLM text to summarize).
- **Evidence:** Concrete code path:
1. `llmIpc.ts:118-135` — `llm:setConfig` IPC handler accepts `a.baseUrl?: string` with no URL validation. The frame guard (`isForeignSender`) only blocks non-main-frame senders; the main renderer (legitimate Settings UI) can pass any string.
2. `llmConfig.ts:57` — `readLlmConfig` only checks `typeof p.baseUrl === 'string'`; no scheme or host allowlist.
3. `llmService.ts:71-73` — `buildRequest` for `provider === 'local'` uses `base = config.baseUrl` directly: `url: \`${base}/chat/completions\``.
4. `llmService.ts:177` — `deps.fetch(req.url, { method: 'POST', ... })` issues the real HTTP call from MAIN with no intercept.

There is NO scheme allowlist, NO host allowlist, and NO URL parsing/validation anywhere on the `baseUrl` path. Node's undici blocks `file://` (verified: `fetch('file:///C:/Windows/win.ini')` throws "fetch failed"), but `http://` to arbitrary internal addresses works fine (verified: outbound HTTP fetch from Node succeeds). Setting `baseUrl` to `http://169.254.169.254/latest/meta-data/` would cause every `runSummarize('local')` call to POST the summarized text to the instance metadata endpoint — a real SSRF on cloud-hosted deployments. On desktop, the attack surface is narrower (requires the user or an XSS to call `llm:setConfig`) but the missing validation is a concrete gap. The frame guard does not close it because the Settings UI itself is the attack vector.
- **Fix direction:** Add a URL validation step in `llmConfig.ts:readLlmConfig` and/or `llmIpc.ts:llm:setConfig`: parse with `new URL()`, then enforce an allowlist of schemes (`http:` and `https:` only) and optionally restrict to loopback/LAN if only localhost LLM servers are intended. Reject (return null / `{ ok: false, reason: 'invalid-baseUrl' }`) on any non-conforming value. Mirrors the `preview.ts` pattern which already does `new URL(rawUrl)` validation before use.
