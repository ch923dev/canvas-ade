- **Category:** security

## Summary
2 related defects in this file/area, grouped as one assignable unit (they collide on the same file). Each is detailed below.

### (1) llm:setKey does not validate provider is a known ProviderName before passing to key store — `src/main/llmIpc.ts:105`

- **Where:** `src/main/llmIpc.ts:105`
- **Sev/cat:** Low / security
- **What:** The IPC handler at line 105 accepts `a: { provider: ProviderName; key: string }` but performs no runtime validation that `a.provider` is a member of the known `ProviderName` union. A renderer (or a Browser board that somehow reaches this channel) could send an arbitrary string as `provider`. `keyStore.setKey(a.provider, a.key)` stores it in the JSON file under that arbitrary key. On read-back, `JSON.parse` would return it as a property of the `KeyFile` object. While this doesn't immediately escape the MAIN process, it pollutes the key file with unknown providers and could cause unexpected behavior if the provider string contains characters that affect JSON parsing, file path construction, or future iterations.
- **Trigger:** Invoke `llm:setKey` with `{ provider: '__proto__', key: 'x' }`. The key file gains a `__proto__` property. On next `readFile`, the `JSON.parse` result is a plain object (prototype pollution via JSON.parse is not possible in V8 for most engines, but `__proto__` as a JSON key IS parsed and returned as an own property). The real risk is with arbitrary string providers being stored and later iterated.
- **Evidence:** The absence of runtime validation on `a.provider` in `llm:setKey` (line 105) and `llm:setConfig` (line 118) is confirmed. TypeScript's `ProviderName` type is erased at runtime. An attacker with access to the main renderer frame can invoke `llm:setKey({ provider: '__proto__', key: 'x' })` and the handler passes it directly to `keyStore.setKey(a.provider, a.key)` (llmKeyStore.ts line 67: `data[provider] = encryptor.encryptString(key).toString('base64')`), writing an arbitrary key into the JSON key file. Confirmed via: (1) code-path trace — no guard between IPC argument and `data[provider]` write; (2) Node.js test confirms `JSON.parse('{"__proto__":"v"}')` returns an own property (not prototype pollution), so the file is written and read back with the arbitrary key intact; (3) no iteration over all keys occurs (verified via grep — `llmKeyStore.ts` has no `Object.keys/values/entries`), limiting the impact. However, two mitigations significantly reduce severity: (a) the foreign-sender frame guard (`isForeignSender`) blocks all non-main-frame callers including Browser board WebContentsViews — the channel is only reachable from the trusted main renderer, not a cross-origin page; (b) `readLlmConfig` sanitizes unknown providers back to `'openrouter'` on read, so `llm:setConfig` with an arbitrary provider self-heals. The actual impact is limited to key-file pollution (benign garbage entries) from a compromised/XSS'd main renderer. No path/shell interpolation of the provider string occurs.
- **Fix direction:** Add a `VALID_PROVIDERS` whitelist check (e.g. `if (!PROVIDERS.includes(a.provider)) return { ok: false, reason: 'invalid-provider' }`) at the top of `llm:setKey`, `llm:clearKey`, and `llm:setConfig` handlers, mirroring the validation already present in `readLlmConfig`. The `PROVIDERS` array is already derived from `DEFAULT_MODELS` in llmConfig.ts and can be exported and reused here.

### (2) llm:setKey accepts and passes an unbounded key string to the encryptor without length or character validation — `src/main/llmIpc.ts:105`

- **Where:** `src/main/llmIpc.ts:105`
- **Sev/cat:** Low / security
- **What:** The `llm:setKey` handler accepts `a.key` from the renderer and passes it directly to `keyStore.setKey(a.provider, a.key)`. There is no upper-bound length check, no character-set validation, and no check that `a.key` is actually a non-empty string. A renderer can send a megabyte-long string as the key. `createKeyStore.setKey` passes this to `encryptor.encryptString(key)` (which is a thin wrapper over `safeStorage.encryptString`), then writes the base64-encoded result to `userData/llm-keys.json`. Electron's `safeStorage.encryptString` does not impose a documented maximum, but at extreme sizes (e.g. 100 MB) this can allocate a large Buffer in MAIN, stall the main event loop during the synchronous `writeFileAtomic.sync`, and cause a denial-of-service. Additionally an empty string key (`a.key === ''`) is accepted by `setKey` but then `hasKey` returns `true` (the ciphertext for '' is a non-empty base64 string), leading `llm:status` to report `hasKey: true` even though no real key was stored — a misleading sentinel that prevents the UI from prompting the user to enter a key.
- **Trigger:** 1) Call `ipcRenderer.invoke('llm:setKey', { provider: 'openrouter', key: '' })` — status will report `hasKey: true` even though no real API key was provided. 2) Call with `key: 'A'.repeat(10_000_000)` — MAIN synchronously encrypts and writes a ~13 MB JSON file, stalling the event loop.
- **Evidence:** Code-path trace, confirmed in Z:/canvas-ade-context/src/main/llmKeyStore.ts and llmIpc.ts:

1. hasKey/hasProvider misalignment (primary defect): `llmKeyStore.ts` line 77-79 — `hasKey` checks only `enc.length > 0` where `enc` is the base64 ciphertext. `encryptString('')` produces `Buffer.from('ENC:', 'utf8')` (via fakeEncryptor) or a non-empty platform ciphertext (real safeStorage) → non-empty → `hasKey` returns `true`. Meanwhile `getProvider` in `llmService.ts` line 164-165: `key = keyForProvider(...)` → `getKey` decrypts to `''` → `!key` is `true` → returns `null`. Result: `llm:status` emits `{ hasKey: true, hasProvider: false }` for an empty-string stored key — Settings UI shows the key as configured but every summarize returns `no-provider`, with no re-prompt to the user. No existing test covers `setKey('', ...)`. The `KeyStore` interface JSDoc (line 28) says "true when a non-empty entry exists" — refers to ciphertext, not plaintext, so the contract is internally consistent but semantically broken for the UI.

2. Unbounded length (secondary, lower severity): `llmIpc.ts` line 105-110 passes `a.key` directly to `keyStore.setKey` with no length/type guard. With a multi-MB string, `encryptString` allocates a large Buffer and `writeFileAtomic.sync` (llmKeyStore.ts line 50) blocks the main event loop synchronously. Real DoS surface requires only an authenticated renderer call.

Severity adjusted to Low: both sub-bugs require the trusted renderer (frame-guarded, main-window only) to send a deliberately empty or huge key. There is no cross-origin exploit. The hasKey/hasProvider misalignment is a real UX sentinel defect but not a security issue.
- **Fix direction:** Add a guard in `setKey` (or in the `llm:setKey` IPC handler) that rejects empty strings and enforces a reasonable upper-bound length (e.g. 1024 chars). Update `hasKey` to also verify the decrypted key is non-empty (or rename the check to `hasCiphertext` to make the semantic gap explicit). This closes both sub-bugs: empty-string keys are rejected before encryption, and huge strings never reach the encryptor/disk.
