## Summary
On line 172â€“175, `result.text` (the raw string from the LLM provider) is interpolated directly into the board memory file: `` `# ${title}\n\n${result.text}\n` ``. There is no length cap, no character filtering, and no sanitization. A malicious or misbehaving LLM provider could return a very large string (up to the uncapped HTTP response body), a string containing null bytes, path-relevant sequences, or sequences that would corrupt downstream Markdown parsing or MCP reads. The `MAX_INPUT_CHARS = 4000` cap is applied only to the INPUT, not to the OUTPUT.

## Where
`src/main/summaryLoop.ts:174`

## How it triggers
A provider (especially `local` or `openrouter` pointing at an untrusted endpoint) returns a megabyte-long response or a response containing sequences like `../../../`, null bytes, or control characters. These are written verbatim into `board-<id>.md`.

## Verification evidence
Watertight code-path argument:

1. `summaryLoop.ts:174` writes `result.text` verbatim: no output-length cap, no character filter.

2. `llmService.ts:63` shows `max_tokens: SUMMARY_MAX_TOKENS` (1024) is only injected into the **Anthropic** provider request body. The OpenAI-compatible branch (`openrouter`/`openai`/`local`, lines 77â€“84) serializes `{ model, messages }` only â€” `max_tokens` is absent. For `local` pointing at e.g. Ollama (default `num_predict=-1`) or an untrusted `openrouter` model, the response can be arbitrarily large.

3. `parseResponse` (lines 88â€“100) extracts `choices[0].message.content` as a raw string with no length check and returns it directly.

4. `runSummarize` wraps it as `{ ok: true, text: <uncapped string> }` (line 217). No post-parse truncation exists anywhere in the chain.

5. `canvasMemory.writeBoard` (line 88â€“97) validates only the board ID via `safeBoardId`; it writes the `md` argument byte-for-byte via `writeFileAtomic.sync`. No content size guard.

6. Tests in `summaryLoop.test.ts` cover `MAX_INPUT_CHARS` (input cap) but no test covers output size.

The result: a `local` provider returning a megabyte-long completion writes that content in full to `.canvas/memory/board-<id>.md`. Prompt-injection payloads embedded in `result.text` land in a trusted-looking memory file read back by the MCP layer. Disk-fill is the immediate data-integrity risk. The Anthropic path is protected by `max_tokens`; `openai` is practically bounded by OpenAI's API; the real exposure is `local` and `openrouter` with an untrusted/unconstrained endpoint.

## Suggested fix direction
Add a `MAX_OUTPUT_CHARS` constant (e.g. 8000, ~4Ã— the 1024-token budget) and slice `result.text` before the template literal on line 174: `result.text.slice(0, MAX_OUTPUT_CHARS)`. Separately, add `max_tokens: SUMMARY_MAX_TOKENS` to the OpenAI-compatible request body in `buildRequest` so the server-side cap matches intent for all providers.
