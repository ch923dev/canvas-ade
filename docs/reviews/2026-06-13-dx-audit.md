# DX & Code-Quality Audit — 2026-06-13

Scope: code-review process · CI/git-hook performance · architecture foundation · testing
foundation. Goal: cut total PR cycle time without weakening the gates that have been catching
real bugs. Status: **assessment + plan; nothing implemented yet** (implementation belongs on a
`fix/dx-*` worktree branch per the parallel-session convention).

---

## 0. Executive summary

The two pain points — review nit-churn and the 3–5 min pre-push e2e — **multiply each other**.
Every review round forces a fix commit + push; every push re-pays the FULL two-OS e2e matrix
AND a FULL re-review of the whole diff, which finds 2–3 new ever-smaller nits (the PR #116
"8 rounds" pattern), which forces another push. A typical feature PR burns:

```
~5 review rounds × ( e2e matrix 3–5 min  +  bot latency ~5–10 min  +  author fix/reply ~10 min )
≈ 60–120 min of churn per PR, on top of the actual feature work
```

Three structural facts make most of that churn removable:

1. **The reviewer re-reads the full diff on every push** with no severity floor on inline
   comments ([nit] is an allowed inline severity), so each round can mint new findings on code
   that was already reviewed and unchanged.
2. **The pre-push matrix duplicates the merge gate.** CLAUDE.md's sequential-merge protocol
   already mandates the full gate + e2e after EACH merge into main — so the per-push full
   matrix is redundant insurance. Pre-push only needs to be a fast, scoped sanity tier.
3. **The Linux Docker leg has never been the leg that catches.** The matrix's documented
   catches (the `RangeError`/pid-reuse bug) were **Windows-only**; the Linux leg mostly
   mirrors the Windows run at roughly equal cost (image rebuild + in-container
   electron-vite build + xvfb run).

Headline plan: severity-floor + incremental-diff the reviewer (quick), drop the Linux leg to
once-per-PR (quick), then tag/scope e2e by changed paths (medium), and keep paying down the
god-file ratchet (long). Projected steady state: **~10–20 min of process overhead per PR**
(1–2 review rounds, ~1–2 min per push), full matrix paid exactly once at merge.

## 1. Code review process — assessment

**Setup today** (`.github/workflows/claude-code-review.yml`): on every PR `opened` +
`synchronize` (non-draft), claude-code-action reviews the **entire PR diff**, posts inline
comments with severities `[critical] / [warning] / [nit]`, a summary, labels, milestone. It is
already self-clearing and disposition-aware (#95, #109) — those were the right fixes for
*stale* noise. The remaining noise is *fresh* noise.

**Why it's noisy:**

- **No severity floor for inline comments.** The prompt instructs an inline comment for "EACH
  NEW or still-unaddressed issue" and explicitly blesses `[nit]` as an inline severity. An
  eager reviewer model will always find 2–3 nits in any nontrivial diff; the prompt currently
  *asks* for them.
- **No comment budget.** Nothing tells it "5 findings max, most important first."
- **Full-diff scope on every push.** A round-3 push that only fixes round-2 nits still gets
  the whole diff re-read — unchanged code gets re-litigated with slightly different nits
  (disposition-matching only protects *previously flagged* findings, not *new* nits on old
  lines).
- **The inline-reply mandate amplifies cost.** CLAUDE.md requires an inline reply on EVERY
  reviewer comment — correct for signal, but it means each nit costs author minutes even when
  declined. Fewer comments is the only real lever.
- **Doc-only PRs get full reviews.** The `paths:` filter is present but commented out.

**Verdict:** the workflow's hygiene machinery (self-clear, dispositions, allowlist) is mature;
the *review contract* is untuned — it reviews like a thorough first-pass reviewer on every
push instead of like a merge gatekeeper on the delta.

## 2. CI / git hooks — assessment

**Pipeline today:**

| Gate | Runs | Cost | Verdict |
|---|---|---|---|
| pre-commit | typecheck ×3 + lint + format:check (docs-only → format only) | seconds | ✅ right-sized |
| pre-push | **full e2e matrix**: Windows native (~100 tests, `workers:1`, serial) + Linux Docker (image rebuild + in-container build + xvfb run) | 3–5 min | ❌ over-scoped |
| CI `check` (pr.yml) | audit + typecheck + lint + format + **all 2 100+ vitest** + build | ~5 min, free to author | ✅ right-sized |
| merge protocol | full gate + e2e after each merge into main | once per PR | ✅ the real insurance |

**What's already good:** the hook is well-engineered (origin-only, docs-only skip, BUG-018/068
base-resolution guards, clear bypass). The app boots **once per worker** (worker-scoped
fixture), so per-spec boot overhead is already amortized. The problem is purely **scope**: all
~100 tests × 2 OSes on every push, when the diff usually touches one board type.

**Specific waste:**

- The **Linux leg ≈ half the wall time** and is the lower-yield leg locally (its unique value —
  POSIX kill paths, xvfb rendering — changes only when `src/main/**` / process-tree /
  packaging code changes). It also carries the Docker-Desktop-auto-update and global-image-tag
  failure modes (memory: docker-e2e-tag-race).
- **No test selection.** A `noteTint`-only diff still runs all terminal/preview/native specs.
- `workers:1` is correct (native WebContentsView + PTY contention) — parallelization is NOT
  the available lever; *selection* is.

## 3. Architecture foundation — assessment

Shape is genuinely good: strict main/preload/renderer separation, locked security model,
MessagePort data plane, stores vs components cleanly split, and a working **file-size ratchet**
(`max-lines` 700 global + 4 downward-only pins: TerminalBoard 631 · mcpSmoke 1000 ·
Canvas.tsx 779 · PlanningBoard 666). Wave-5 + the god-file lanes (#97–#105) already extracted
the worst seams.

**Remaining weight (raw lines, non-test):** usePreviewManager 1109 · Canvas.tsx 1035 ·
canvasStore 944 · preview.ts 897 · pty.ts 819 · TerminalBoard 781 · BoardFrame 773 ·
boardSchema 690 · useTerminalSpawn 685 · AppChrome 671. (Several are under the CODE-line cap
only thanks to comment density.)

**How this feeds the churn:** big hot files attract review comments (more surface per diff
hunk) and make diffs look cross-cutting, which defeats any path-based test scoping. The
ratchet prevents regrowth but has no *paydown* schedule — pins only move when someone happens
to shrink a file. The seam tables in `docs/research/2026-06-09-god-file-maintainability.md`
are the existing, correct backlog; it needs cadence, not redesign.

One inconsistency: `mcpSmoke.ts` (1221 raw / pinned 1000) is a smoke harness pinned like
production code; it's also the one surviving `CANVAS_SMOKE` exception with a tracked
Playwright port (`e2e/mcp.e2e.ts`) — finishing that port retires both oddities.

## 4. Testing foundation — assessment

Strategy on paper (Testing Trophy, `docs/testing/TESTING.md`) is excellent and the volume
split already matches it: **168 vitest files (~2 100+ tests, ~seconds–1 min)** vs **32 e2e
specs / ~100 e2e tests**. The pyramid/trophy is NOT inverted by count. The problem is that the
**e2e tier is positioned as a per-push gate**, so its cost is paid 5× per PR instead of 1×.

**Misallocation inside the e2e keep-set:** TESTING.md's own decision rule ("only what
reproduces ONLY in the booted app") is violated by several pure-renderer specs — `menu`,
`modal`, `noteTint`, `titleEdit`, `textToolbar`, parts of `boardKeyboard`/`planningKeyboard`/
`commandPalette`. Caveat that must be respected: several of these exist *because* jsdom
provably cannot catch the listener-removed-mid-dispatch / real-input-coordinate class (D1-B,
W4 paste, full-view add-note). The right move is not deletion but **thinning**: keep ONE
real-input sliver per interaction pattern per area, push the assertion bulk (variants, states,
edge values) down to jsdom integration. Realistic target: ~100 → ~60–70 e2e tests, with the
heavyweight native specs (terminal*, browser*, preview*, fullview, processTree, recovery)
untouched.

**Genuine gap (already known):** auto-update is the one e2e-only surface with zero coverage —
correctly deferred to Phase 5; nothing to do now.

## 5. The plan

### Tier 1 — Quick wins (this week, ~2–4 h total effort)

**QW-1 · Severity floor + comment budget in the reviewer prompt** — `claude-code-review.yml`
- Change step 4: inline comments ONLY for `[critical]` and `[warning]` ("would you block the
  merge / is this a probable bug?"); hard cap **5 inline comments**, most important first.
  `[nit]`s: at most a 2–3-bullet "Nits (non-blocking, no reply needed)" section in the step-5
  summary — never inline, never re-raised. Add an explicit do-not-comment list: anything the
  linter/prettier enforces, naming taste, comment phrasing, speculative "consider adding a
  test" without a concrete missed case.
- Add a **verification bar**: a behavior claim must cite the actual code path (file:line) it
  verified, not an inference from naming — this is Anthropic's own published guidance for
  tuning its managed code review (nit caps "report at most five, mention the rest as a count",
  skip rules for anything CI already enforces, verification bar) — see §7.
- Also amend CLAUDE.md's inline-reply rule to match: replies remain mandatory for inline
  (critical/warning) threads; summary-listed nits need no disposition.
- Optional action-level helper: `classify_inline_comments` (on by default in v1) buffers
  inline comments and Haiku-filters them before posting — verify it's active; consider
  `use_sticky_comment: true` if summary comments ever stack.
- *Why:* the prompt currently solicits nits; the disposition machinery then forces you to pay
  for each one. *Saving:* 2–3 comments/round → ~0–1; biggest single lever on round count.

**QW-2 · Incremental re-review on `synchronize`** — `claude-code-review.yml`
- Embed the reviewed head SHA in the summary marker:
  `<!-- claude-code-review:auto head:<sha> -->`. New step 2b: read the previous summary's
  `head:` SHA; if found, review **only** `git diff <last-reviewed-sha>..HEAD` (plus enough
  surrounding file context to verify) and say so in the summary ("incremental review of
  <short-sha>..<short-sha>"). Full-diff review only on the first run (or when the marker is
  missing / the base was rebased — detect via `git merge-base --is-ancestor`).
- Pair it with a **re-review convergence rule** in the prompt: "On any review after the first,
  post `[critical]`/`[warning]` findings only — suppress all new nits." This is verbatim the
  pattern Anthropic's Code Review docs recommend to stop a one-line fix reaching round seven,
  and matches industry practice (CodeRabbit auto-pauses incremental reviews after 5 reviewed
  commits per PR by default, for exactly this reason — see §7).
- Note: the action has NO built-in incremental mode (a requested feature, not shipped —
  claude-code issue #19618), so both pieces live in the prompt.
- *Why:* structurally ends the ever-smaller-nit loop — unchanged code can't grow new findings
  in round 3. *Saving:* rounds converge because each round's input is only what changed.

**QW-3 · Skip doc-only and trivial PRs** — `claude-code-review.yml`
- Uncomment/extend the `paths:` filter to `src/**`, `e2e/**`, `.github/**`, root configs —
  mirroring the pre-push docs-only skip semantics (`^docs/|\.md$` excluded).
- *Why/saving:* zero-value reviews disappear; the docs(history) commits that follow every
  merge stop triggering bot runs on stacked PRs.

**QW-4 · Demote the Linux Docker leg from per-push to per-PR** — `.githooks/pre-push`
- Default push gate = **Windows leg only** (`pnpm test:e2e`). Run the Linux leg when (a) the
  push's changed set touches `src/main/**`, `e2e/**`, `Dockerfile.e2e`, `package.json`,
  `pnpm-lock.yaml`, `playwright.config.ts`, or (b) `E2E_FULL_MATRIX=1` is set. Add the
  merge-protocol requirement (CLAUDE.md parallel-sessions section): **the full matrix runs
  once per PR — at the pre-merge gate** (which the sequential-merge protocol already mandates),
  so cross-OS coverage is never lost, just paid once.
- *Why:* the Linux leg ≈ half the push cost and its unique catch surface is main-process/POSIX
  code; the documented real catch (pid-reuse RangeError) was on the Windows leg, which stays.
  *Saving:* 3–5 min → ~1.5–2.5 min per push, immediately, with one shell-script edit.

**Tier-1 combined effect:** per-PR churn drops from ~60–120 min to roughly
`2 pushes × 2 min e2e + 1–2 lean review rounds ≈ 15–25 min`.

### Tier 2 — Medium term (next 2–3 weeks)

**MT-1 · Tag + scope the e2e suite (path-based selection)**
- Tag every spec title with an area: `@core` (boot, recovery, reset-isolation, placement),
  `@terminal`, `@preview` (browser*, preview*, fullview), `@planning` (whiteboard, text*,
  noteTint, planningKeyboard), `@chrome` (menu*, modal, commandPalette, wayfinding, titleEdit,
  boardKeyboard, groups, evidence, recap).
- Add `test:e2e:smoke` = `playwright test --grep @core` (~6–8 tests, well under a minute
  after the build).
- In pre-push, map changed paths → tags (a small `scripts/e2e-scope.sh` consumed by the hook):
  `src/renderer/src/canvas/boards/terminal/**` → `@core|@terminal`; preview/browser paths →
  `@core|@preview`; planning paths → `@core|@planning`; cross-cutting files (`canvasStore`,
  `boardSchema`, `Canvas.tsx`, `BoardFrame`, `AppChrome`, `e2e/fixtures.ts`, `src/main/**`,
  configs) → full suite. Unknown/mixed → full suite (fail open to full, never to skip).
- *Why:* `workers:1` makes parallelization unavailable; selection is the only scaling lever.
  *Saving:* a scoped feature push pays ~30–90 s instead of the full Windows leg. Keep the
  scoping logic conservative — the cross-cutting list errs toward "run everything".

**MT-2 · Thin the pure-renderer e2e specs down to slivers**
- Per §4: for `menu`, `modal`, `noteTint`, `titleEdit`, `textToolbar`, and the keyboard specs,
  keep exactly one real-input probe per interaction pattern (the mid-dispatch-listener and
  coordinate-transform classes stay e2e-proven); move variant/state assertions to
  `*.integration.test.tsx`. Update TESTING.md's keep-set table in the same PR.
- *Why:* honors the existing decision rule; shrinks the suite every future push pays for.
  *Saving:* ~30 % fewer e2e tests; also fewer flake retries (`retries:2` surface shrinks).

**MT-3 · Reviewer trigger de-escalation (only if QW-1/2 prove insufficient)**
- Fallback position: drop `synchronize`; review on `opened` + `ready_for_review` + on-demand
  re-review (a `needs-review` label or `@claude review` comment trigger). Decide after ~5 PRs
  of data with the tuned prompt.
- *Why second-line:* incremental review (QW-2) preserves the always-fresh-review property,
  which has real value in solo+agents development; going on-demand sacrifices it, so do it
  only if the data says the loop persists.

**MT-4 · Finish the `e2e/mcp.e2e.ts` port** (tracked follow-up in TESTING.md)
- Retires the last `CANVAS_SMOKE` harness and the 1221-line `mcpSmoke.ts` pin in one move;
  the new spec gets an `@mcp` tag and joins the scoped selection.

### Tier 3 — Long term (structural)

**LT-1 · God-file paydown with cadence, not just a ratchet**
- Adopt: every Nth feature PR in an area must land one Tier-1 seam extraction from the
  2026-06-09 seam tables for that area's pinned/near-cap file, lowering the pin in the same
  PR (the doctrine already says pins move downward-only — give it a schedule). Order by churn
  × size: Canvas.tsx (779 pin) → TerminalBoard (631) → PlanningBoard (666) → then the
  comment-dense near-cap set (usePreviewManager, canvasStore, preview.ts, pty.ts). Leave
  `useTerminalSpawn`/`mcpDispatchGate` last per the risk-tier table.
- *Why:* directly reduces review surface per diff and makes path-scoped e2e selection
  *stay* accurate (cross-cutting files are the scoping spoiler).

**LT-2 · Encode the testing decision rule into review + lint, not prose**
- Add to the reviewer prompt: "flag NEW e2e tests whose behavior is provable in
  jsdom/integration per docs/testing/TESTING.md — that is a `[warning]`, not a nit." Add a
  lightweight lint/CI check that a new `e2e/*.e2e.ts` file must carry an area tag.
- *Why:* keeps the thin-top from regrowing the way god files did before the ratchet.

**LT-3 · Phase 5 packaging unlocks the auto-update e2e**
- Unchanged from the roadmap; noted here only so the "one uncovered surface" isn't forgotten
  when packaging lands.

## 5½. Best-practice grounding (researched 2026-06-13)

The plan maps onto published guidance rather than inventing process:

- **Reviewer tuning** — Anthropic's managed Code Review docs recommend severity floors, nit
  caps ("at most five, rest as a count in the summary"), skip rules for CI-enforced concerns,
  a file:line verification bar, and post-first-review nit suppression; the local
  `/code-review` plugin guidance is "flag only significant bugs, ignore nitpicks and likely
  false positives". The claude-code-action itself exposes `paths` filtering (via standard
  workflow `on.paths`), `use_sticky_comment`, and `classify_inline_comments`, but no built-in
  incremental review — that's prompt-level (issue #19618 tracks it as a feature request).
- **Review philosophy** — Google eng-practices: approve once the change *improves overall
  code health*, `Nit:` = optional-by-definition, never block on taste. Our `[nit]`-inline
  current behavior violates this by making nits cost mandatory dispositions.
- **AI-reviewer norms** — CodeRabbit defaults to pausing incremental reviews after 5 reviewed
  commits/PR to avoid noise on near-identical diffs; per-PR comment budgets and
  delta-only re-review are the cross-tool convergent pattern.
- **Slow-e2e placement** — Playwright's documented tag/`--grep` workflow (`@smoke` subsets per
  push, full suite at merge/nightly) and test-impact analysis (Nx-affected-style path→test
  mapping) are the standard answers when parallelization is unavailable; "smoke on every
  push, full matrix at the merge gate" is the norm for CI-minute-constrained teams.

Key sources: code.claude.com/docs/en/code-review · github.com/anthropics/claude-code-action
(README, docs/usage.md, docs/solutions.md) · google.github.io/eng-practices/review ·
docs.coderabbit.ai/changelog · playwright tag guides (timdeschryver.dev, dev.to/playwright) ·
nx.dev affected docs.
