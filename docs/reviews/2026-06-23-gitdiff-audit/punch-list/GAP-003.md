# GAP-003 — `parseDiffStat` under-counts content lines whose body starts with `--` / `++`

**Severity:** Low · **Class:** real-bug (documented "approximate", but wrong on ordinary content) ·
**Rubric:** correctness vs a git/oracle reference; tests that assert real behavior.

## What's missing / wrong
`parseDiffStat` (`src/renderer/src/lib/diffStat.ts:24-36`) excludes any line that
`startsWith('+++') || startsWith('---')`, intending to skip the `+++ b/file` / `--- a/file` headers.
But a **content** line in a unified diff is `±` + the line body. A deleted line whose body begins
with `-` (e.g. removing a markdown `---` horizontal rule, a row of dashes, or another diff) renders
as `---…` and is wrongly skipped; an added line whose body begins with `++` renders as `+++…` and is
likewise skipped. Result: the `+N −M` chip silently undercounts.

The module documents itself as "approximate by design" (chip, not a ledger), so this is **Low** — but
it is a genuine miscount on real content, not just an edge of git's own format.

## Demonstrated (this audit)
A diff-shaped input with one real deletion line whose body is a `--`-prefixed dash rule:
```
parsed: { ins: 1, del: 0, files: 1 }   # del should be ≥ 1 — the `---…` deletion line was skipped
```

## Files
- `src/renderer/src/lib/diffStat.ts`
- `src/renderer/src/lib/diffStat.test.ts`

## Fix options
1. **Hunk-aware parse:** only treat `---`/`+++` as headers when NOT inside a hunk (i.e. they appear
   right after `diff --git`/`index`, before the first `@@`). Inside a hunk, classify by the **first
   char only** (`+`/`-`) regardless of the following chars.
2. **Exact ledger:** have MAIN return `git diff --numstat` alongside (or instead of) the raw text for
   the chip, keeping the raw only for the view-diff panel. (Larger; crosses into MAIN — coordinate
   with GAP-002.)

## Validation
- Add `diffStat.test.ts` cases: a deletion line `--- a rule`, an addition line `+++ marker`, and a
  multi-hunk diff; assert the counts match a hand-verified oracle.

## Collisions
