## Findings

### [PTY-2] Restart on a never-Started restored/duplicated terminal leaves the idle-on-mount flag set
- **Severity:** Low-Medium · `boards/TerminalBoard.tsx:458-488` (restart) + `:381-397` (idle branch) · `store/canvasStore.ts:122-133` (`idleOnMountIds`)
- **What:** `restart()` spawns a fresh PTY via `respawn()` but never calls `clearIdleOnMount(board.id)`. Only the idle **Start** button clears the flag. A restored/duplicated terminal Restarted without ever clicking Start stays in `idleOnMountIds` while its PTY runs.
- **Why:** On the next spawn-effect re-run (a config Apply — shell/launchCommand/cwd/projectDir), `adoptTerminal` returns `{adopted:false}` (session is live, not parked), then `isIdleOnMount` is still true → renders the idle "Start" overlay over a live PTY. Clicking Start then spawns a *second* PTY under the same id (the first is reaped) — an unintended kill+respawn.
- **Repro:** Open a project → on a never-Started terminal click **Restart** → open Configure, change cwd/shell, **Apply** → idle overlay appears over a running shell. (Same via Duplicate.)
- **Fix:** Call `clearIdleOnMount(board.id)` at the top of `restart()` (a Restart is explicit start intent), mirroring the Start button.

### [WB-1] Degenerate arrow/pen tap pushes a phantom undo snapshot
- **Severity:** Low-Medium · `boards/PlanningBoard.tsx:249-264` (down) + `:321-334` (up)
- **What:** Selecting `arrow`/`pen` and tapping the well without dragging calls `beginChange()` on pointer-down. On pointer-up the gesture is discarded as degenerate (arrow `|Δ|<=4`, pen `pts<4`) with no `commit`, so `boards` is unchanged — but `recordPast` already pushed an identical snapshot.
- **Why:** Next `undo()` is a visible no-op (present = identical past), and it arms a redo of that no-op; it also suppresses the next *real* edit's checkpoint via the `past[last] === boards` guard. The **move** path was explicitly hardened against exactly this (lazy-checkpoint comments at lines 210-211, 313-314); the draw tools were not.
- **Repro:** Select arrow (or pen) → single click on the whiteboard, no drag → Ctrl/Cmd-Z once → nothing changes.
- **Fix:** Defer `beginChange()` out of `onWellPointerDown` for arrow/pen; call it in `onWellPointerUp` only inside the commit branches (right before `commit(...)`), mirroring the move path.

### [SEC-1] PTY `cwd` accepted with no validation
- **Severity:** Low · `main/pty.ts:430` (`cwd: opts.cwd || os.homedir()`)
- **What:** `opts.cwd` flows from the renderer straight into `pty.spawn`. Unlike `shell` (hardened via `resolveShell` against the enumerated list) and `dir` (hardened via `isUnsafeProjectDir`), `cwd` gets no existence/dir check. A corrupt/hand-edited `canvas.json` with a missing or non-dir `cwd` reaches spawn.
- **Why:** An invalid `cwd` can throw inside `pty.spawn`; that path is already try/caught → `spawn-failed`, so it degrades rather than crashes. Trusted-user input by the locked design — this is hardening **parity**, not an exploit (`cwd` is the one spawn input that escaped the `shell` treatment).
- **Fix:** Before spawn, fall back to `os.homedir()` if `opts.cwd` isn't an existing directory (mirror the `shell` fallback).

### [PERSIST-A] Project-switch flush-save result ignored
- **Severity:** Low · `canvas/AppChrome.tsx:59`
- **What:** On a switch, step 1 does `await window.api.project.save(toObject())` but discards the returned boolean; `disposeLiveResources()` + `applyOpenResult(load)` proceed unconditionally.
- **Why:** `project:save` returns `false` on any write failure. The normal autosaver surfaces that via `onError` (SAVE-1), but this explicit switch-time flush bypasses the autosaver — and the debounced timer is gated out once status flips to `loading` — so a failed final flush of the *outgoing* project is swallowed and its tail edits are lost with no signal. Exactly the silent-loss class SAVE-1 closed.
- **Fix:** Check the result; surface failure via the autosaver `onError` path (optionally offer to abort the switch).

### [PERSIST-B] Pending autosave timer not cancelled across a project switch
- **Severity:** Low (benign) · `store/useAutosave.ts:46-49` + `canvas/AppChrome.tsx:56-65`
- **What:** `switchTo` can't `flush()`/`cancel()` the autosaver (the instance is closed over inside `useAutosave`). A debounce timer armed editing project A survives the switch.
- **Why:** After load flips status back to `open` and `currentDir` is now B, the leftover `dirty` timer fires `run()` → writes `toObject()` (now B's boards) to B's dir. **Data-correct** (no A→B cross-write — `toObject()` and `currentDir` both already point at B), just a redundant extra write right after load.
- **Fix:** Expose `flush`/`cancel` from `useAutosave` and `cancel()` at the top of `switchTo`. Low priority.

### [PREV-A] `evictLiveBoard` can resurrect a deleted board's runtime entry
- **Severity:** Low · `boards/BrowserPreviewLayer.tsx:464-484` (no-snapshot async branch)
- **What:** When a non-full-view board has no snapshot, `evictLiveBoard` does `await capturePreview(id)` then unconditionally `closeBoard(id)`, which `rec(id)` + `patchRuntime(id,{live:false})` (both create-if-absent) with no `recs.current.has(id)` recheck after the await.
- **Why:** If the board is deleted during the capture round-trip, the post-await `closeBoard` re-creates a stale `recs` row + an idle `previewStore` orphan — the resurrection class that `demoteToSnapshot`, `attachBoard`, and `beginMotion` all explicitly guard. `evictLiveBoard` is the lone async path missing the guard.
- **Repro:** Open full view on A (evicts a freshly-attached, never-snapshotted live board B via the async branch); delete B within the capture window → B leaks a `recs` row + previewStore entry.
- **Fix:** After the `await`, `if (!recs.current.has(id)) return` before `patchRuntime`/`closeBoard`.

### [PREV-B] Node RESIZE lacks the synchronous detach that node DRAG has
- **Severity:** Low · `canvas/BoardNode.tsx:246` vs `canvas/Canvas.tsx:331-338`
- **What:** `onNodeDragStart` fires `void window.api.detachAllPreviews()` synchronously to dodge the #43961 ghost. The NodeResizer path only `setNodeGesture(true)` on first `onResize`; the actual detach then goes through the async `beginMotion` (which `await`s `capturePreview` per view).
- **Why:** For the first ~tens of ms of a resize, the resized board's live `WebContentsView` is still attached and `flushBatch` repositions it every frame — the per-frame-setBounds-then-detach sequence the drag path was hardened against. The native layer can ghost at the old size until the snapshot lands.
- **Fix:** Add `void window.api.detachAllPreviews?.()` in NodeResizer `onResize`, matching `onNodeDragStart`.

### [PREV-C] Global `menuOpen` singleton
- **Severity:** Low (latent) · `canvas/BoardFrame.tsx:171-174` + `store/previewStore.ts:72,119`
- **What:** `menuOpen` is one global boolean, but every `BoardFrame` writes it from its own `open` state with cleanup `() => setMenuOpen(false)`. If two menus were ever open at once, the first to close clears the shared flag and prematurely reattaches live views under the still-open second menu (occluded by the always-above native layer).
- **Why (barely a bug):** The outside-pointerdown listener closes menu A when you open menu B, so two are rarely open — the invariant is held by incidental UI behavior, not the state model. Any future second popover reintroduces it.
- **Fix:** Make it a ref-count / `Set<boardId>` in previewStore so `menuOpen` is true while ANY menu is open.

### [STATE-1] Duplicate / push-spawn while focused leaves the new board dimmed
- **Severity:** Low (UX) · `canvas/Canvas.tsx:385-388` (duplicate) + `:358-379` (`applyPush` spawn)
- **What:** `addCentered` (dock/EmptyState add) calls `setFocusedId(null)` after adding (#14 fix) so a fresh board is never dimmed. The ⋯-menu **Duplicate** and **push-preview spawn** paths don't.
- **Why:** While focused, every other board is `dimmed: focusedId !== b.id`. The new board's id ≠ `focusedId`, so it mounts at 55% opacity though it's the `selectedId` (selection ring on a dimmed board), landing inside the focused frame as a faded ghost. Same defect #14 patched on `addCentered`.
- **Repro:** Double-click a board to focus → ⋯ → Duplicate → copy appears dimmed. Same via a Terminal Preview push that spawns a Browser while focused.
- **Fix:** `setFocusedId(null)` in `boardActions.duplicate` and in `applyPush`'s spawn branch.

### [STATE-2] `updateBoard` wipes the redo branch on a no-op value patch
- **Severity:** Low (latent) · `store/canvasStore.ts:285-298`, contract at `:347-352`
- **What:** `updateBoard` sets `changed=true` purely on id match (not value diff), then returns `future:[]` if `future.length` — clearing the armed redo branch and minting a new `boards` ref. `beginChange`'s comment promises the opposite ("clear `future` only when boards truly change"). `resizeBoard` has the same id-only logic.
- **Why:** A patch re-applying identical values clears the redo stack though nothing changed, and the fresh `boards` ref defeats downstream `boards !== before` ref checks (the `doUndo`/`doRedo` focus-clear guards). Narrowly reachable (RF emits position changes only on real movement), hence Low.
- **Fix:** Only set `changed` / return a new object when a patched key's value actually differs; leave `future` intact otherwise.

### [SEC-2] `pty:port` MessagePort re-post uses `'*'` targetOrigin
- **Severity:** Nit · `preload/index.ts:164`
- **What:** The data-plane port is re-posted into the main world with wildcard target origin.
- **Why:** Not exploitable today — single same-origin document, no iframes (`webviewTag:false`, nav guard pins origin). The documented Electron MessagePort-forwarding pattern. Flagged only because `'*'` becomes load-bearing if an iframe is ever added.
- **Fix:** Use `window.location.origin` instead of `'*'`.

### [PERSIST-C] `createProject` bypasses the `writeProject` envelope guard
- **Severity:** Info · `main/projectStore.ts:93-94`
- **What:** `createProject` writes the fresh `{schemaVersion:2, viewport:null, boards:[]}` via `writeFileAtomic.sync` directly, not through `writeProject`.
- **Why:** The doc is a hardcoded valid constant and there's no prior file to rotate — functionally correct. Flagged only because it's the one write path sidestepping the centralized guard, so a future edit to the fresh-doc shape wouldn't be caught.
- **Fix:** None required; optionally route through `writeProject` for single-path discipline.
