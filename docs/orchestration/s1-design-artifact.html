What S1 does and does not build

  • Region 1 · chat spine S1 — conversational orchestrator loop with run memory, on the Jarvis LLM provider/config machinery (Q9 default) with a distinct orchestrator system prompt encoding 07 §4 (four-field task spec, effort ladder, ≤2 concurrent writers disclosed). Its tools are the existing gated dispatch surface. Every write still pays the human confirm.
  • Region 2 · plan strip S1 read-only — a projection of the run's planning elements. Click opens the full Planning board. editing, mid-run task adds → S2
  • Region 3 · worker canvas S1 — worker terminals stay ordinary boards (Q11 lens model); the card is chrome over them with the 09 §4 Layer-A glance: state icon · activeForm line · timer · artifact chip, ordered needs-input → working → spawning → done. worktree/branch chips → S3
  • Region 4 · needs-you strip S1 minimal — surfaces the existing attention marks as a list with a jump. approve/deny inline, batched write confirms, stall + unverified-done watch queue → S2
  • Region 5 · header S1 — run title, run timer, pause-all. token/cost meter + budget cap → S4, gated on Q10 accounting

One open call worth naming: the header carries Pause all in S1 because 07 §6 lists it in the S1 region set, and it is cheap (stop dispatching; running workers finish their turn). It is not a kill — that stays the per-board control.

3Lead entry point — replacing the Settings picker

Today the wire-facing lead role is granted from Settings › Agent orchestration › Lead terminal: pick a terminal from a dropdown, click twice, then restart the agent in that terminal so the token mints. That surface shipped as an explicit proof-of-plumbing for Phase 1 and is now being removed.

Why the replacement is strictly better (not just prettier)

grantLead only designates; the token is minted by the existing spawn-time provisioning seam the next time that board's terminal spawns (mcp.ts routes the minter by lead.designated()). The Settings picker always fires after the terminal already spawned — which is exactly why it must tell you to restart. Granting at creation time designates before the first spawn, so the terminal boots holding a lead token on its first breath. The restart disappears because the ordering is finally right, not because we hid it.

3c · Every outcome, and the copy that carries it

Grant at creation Designation lands before the held spawn is released, so the terminal boots holding a lead token. No restart, no note, nothing to read. Silence is the success state.
Promote a live board Toast: “orchestrator holds the lead role. Restart its agent to pick up the token.” — with a Restart agent action in the toast. This restart is real and unavoidable (the token mints at spawn), so it is disclosed and made one click, not hidden.
Already-active Confirm: “builder-1 currently holds the orchestrator role. Revoke it and grant to orchestrator?” · Revoke & grant / Cancel. Single-active-lead (Q2) stays enforced in leadAuthority — the UI just stops making you go find the other board.
Revoke Immediate. Toast: “Orchestrator role revoked — its token is dead immediately.” Badge clears. Unchanged semantics from today.
Grant fails at creation Consent revoked mid-dialog, or the broker won't start. The dialog stays open with the reason inline and the board un-spawned — nothing is lost and nothing is silently downgraded to a plain terminal. You ticked a box; we do not quietly untick it.
Board closed / consent revoked Designation + token die with it — existing leadAuthority.onBoardClosed / revokeAllConnected behaviour, untouched.

3d · What is deleted, and what is deliberately not

  • Deleted: the entire Settings › Agent orchestration › Lead terminal block in OrchestrationPane.tsx (dropdown, two-step arm/confirm, note line) and its unit tests. The orchestration consent toggle, Sync button, and worker spawn-cap in that pane stay.
  • Untouched — this is load-bearing: leadAuthority.ts, orchestrationLead.ts, the three IPC handlers, the preload surface, the minter routing in mcp.ts, and the broker's lead tier. Phase 1's deliverable is the plumbing; S1 only replaces the surface that calls it. S4 docks onto the same plumbing.
  • Not deferred to S4: the swarm board's brain is app-resident in S1, so it needs no lead token — the picker's replacement is fully independent of it and buildable now. Deferring would leave a surface we have already called temporary sitting in Settings for three more phases while a strictly better path exists today.
