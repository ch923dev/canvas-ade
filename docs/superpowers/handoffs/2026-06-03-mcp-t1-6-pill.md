## What shipped

The board-chrome status indicator now derives from the SAME coarse bucket the MCP exposes
(`boardStatusBucket` → `canvas://boards` / `canvas://board-states`), so the human-visible dot and the
agent's view can't disagree (the M1-gate "one source of truth").

- `src/renderer/src/store/boardStatus.ts` — new **`bucketToPill(bucket)`**: the single
  bucket→`{dot,label}` pill map. `running` = `--ok` (pulses), `idle` = neutral `--text-3`,
  `awaiting-review`/`blocked` = `--warn`, `failed` = `--err`, `static` = no pill.
- `src/renderer/src/canvas/boards/BrowserBoard.tsx` — the title-bar pill is now
  `bucketToPill(boardStatusBucket('browser', { preview }))`, replacing an ad-hoc
  connected→green / connecting→amber map that **diverged** from the bucket. Visible change: a loaded
  preview now reads **idle** (neutral dot), matching what `canvas://boards` reports — not the old
  green "connected". (The old `statusPill` helper was removed.)
- `src/renderer/src/canvas/BoardNode.tsx` — the **LOD card** dot is now bucket-derived from the live
  terminal/preview stores too. The old `lodStatus(type)` was type-only and *lied* — a browser always
  showed green even when load-failed; a terminal always neutral.
- `src/renderer/src/smoke/e2eHooks.ts` — `boardBucket(id)` (the live bucket, same path
  `buildBoardSnapshot` feeds MCP) + `bucketPillDot(bucket)` harness accessors.
- `src/main/e2e/probes/status.ts` — new `board-status-pill` probe: asserts the rendered browser
  title-bar dot colour == `bucketPillDot(boardBucket(browserId))` for its live state. Added to the
  playlist after `menuChrome`.

## Decision: terminal pill left as its rich lifecycle pill
The Terminal board keeps `statusFor(state)` (agent identity + timer + braille spinner;
`spawning`/`awaiting-input`/`spawn-failed` dots). Those are **sub-states the coarse bucket
intentionally collapses** (the bucket only distinguishes `running` vs `idle` for terminals, from
`isRunning(state)`). In **steady** states the terminal pill and bucket agree exactly
(`running`→`--ok`, `idle`/`exited`→neutral); they only differ during the transient `spawning` (pill
`--ok`, bucket `idle`). The e2e probe therefore asserts exact dot-equality on the **browser** (the
surface T1.6 actually rewired, bucket-derived by construction), not the terminal.

- **Follow-up (optional, not gating):** to make the terminal pill *also* bucket-exact, enrich
  `boardStatusBucket` to map the full `TerminalState` (`spawning`→running, `awaiting-input`→
  awaiting-review, `spawn-failed`→failed) instead of the current `isRunning` boolean. That would also
  give MCP attention real terminal `failed`/`awaiting-review` coverage (today reserved for M8). It
  touches `terminalRuntimeStore` (store the state, keep the `running` boolean selector for the
  preview-edge), `boardStatus.ts`, `buildBoardSnapshot`, `useMcpPublish` — still renderer-only but it
  changes the buckets MCP sees, so it was left out of T1.6's tight scope.
