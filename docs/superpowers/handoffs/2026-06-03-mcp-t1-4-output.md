## What shipped

A read-only, size-capped, ANSI-stripped, paginated view of a board's PTY scrollback —
the first PTY-output exposure to agents. **No new buffer**: it reads the EXISTING 256 KB
replay ring (`buf.data`) that adopt-replay already maintains.

## Decisions resolved (from the kickoff's open questions)
- **D1 ANSI:** strip to plain text (server-side; control-seq injection surface). Raw kept as a
  possible future opt-in.
- **D2 page size:** 25k **chars** (UTF-16 code units, matching the ring's `.length`-based cap).
  Documented in both `MAX_OUTPUT_PAGE` constants.
- **D3 cursor:** tail-anchored = chars-from-end already consumed. Cursor 0 = newest page;
  `nextCursor` walks backward; absent at the front. Robust to ring shift (no head offsets).
- **D4 exited boards:** readable — `readPtyOutput` falls back to the parked ring, same as adopt-replay.
- **SDK gotcha (the load-bearing one):** the SDK's `UriTemplate.match` compiles `{?cursor}` into a
  **mandatory** `\?cursor=…` regex group, so a URI with no query won't match it. Fix: register TWO
  disjoint templates (`…/output` vs `…/output{?cursor}`) sharing one handler. Their regexes are
  mutually exclusive, so each concrete read routes to exactly one. (Verified by the live test.)

## 🔒 Security
Never returns the raw ring (capped at every read path: 256 KB ring × 25k page, re-capped in pkg).
Read-only, control-plane only — no write/echo to the PTY, no new renderer IPC. Honest truncation
(`droppedOlder` / `nextCursor`, never blind). ANSI stripped server-side.
