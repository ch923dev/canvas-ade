## T8 — what is half-done and the PROBLEM to fix FIRST

The stopped agent added the e2e probe and, when the real `webContents.paste()` did not reach the
focusable-but-not-editable `.pl-well`, applied a **contingency** (all in WIP `35999b9`):
- `src/main/e2e/probes/whiteboard.ts` — `whiteboardPasteImage` probe (4 parts: `whiteboard-paste-image`,
  `whiteboard-paste-reload`, `whiteboard-asset-dedup`, `whiteboard-asset-gc`) + diagnostics. It focuses
  `.pl-paste-proxy` then calls `ctx.win.webContents.paste()`.
- `src/main/e2e/index.ts` — probe registered in `PLAYLIST` before `seed`. (Good, keep.)
- `src/renderer/src/canvas/boards/PlanningBoard.tsx` — added a hidden `contentEditable`
  `.pl-paste-proxy` div + a native `addEventListener('paste')` → `onWellPaste`, and **REMOVED
  `onPaste={onWellPaste}` from `.pl-well`**.

**⛔ The problem (do not ship this):** routing paste ONLY through a hidden proxy that the e2e explicitly
focuses very likely makes the probe pass while **breaking real-user paste**. A real user focuses the
well (or a note), never the `pointer-events:none / opacity:0 / tabIndex:-1` proxy — and the well's
`onPaste` was removed. So real Ctrl+V of a screenshot would no longer create an image element. This is
a false-green (the exact trap the project warns about — memory `e2e-sendinputevent-vs-dispatchevent`).
`typecheck:web` passes, but the probe was **never confirmed green** (agent stopped mid-debug).

### Decide the paste mechanism (pick one), then make the e2e exercise the SAME path real users hit

Verify the premise first: in Chromium, a real Ctrl+V fires the `paste` DOM event on the focused element
even when it is NOT contentEditable (it bubbles to document), so React's `onPaste` on the focused
`.pl-well` almost certainly works for real users. Only the e2e's `webContents.paste()` is the odd one
out (it targets an editable element). Options, in recommended order:

1. **Keep `onPaste` on the well for real users; change the E2E driver to a real Ctrl+V keystroke.**
   Revert the PlanningBoard contingency (restore `onPaste={onWellPaste}` on `.pl-well`, drop the proxy +
   native listener). In the probe, replace `webContents.paste()` with a synthesized Ctrl+V to the
   focused well:
   `ctx.win.webContents.sendInputEvent({ type:'keyDown', keyCode:'V', modifiers:['control'] })` (+ char +
   keyUp). This drives the SAME path a user hits (paste on the focused well → `onPaste`). VERIFY it
   actually delivers the clipboard image; if Chromium won't fire `paste` with image data from a
   synthesized keystroke on a non-editable div, fall to option 2.

2. **Additive proxy (keep real-user path intact).** Restore `onPaste={onWellPaste}` on `.pl-well` AND
   keep the proxy + native listener as an ADDITIONAL path used only by the e2e. Real users → well
   onPaste; e2e → focuses proxy + `webContents.paste()`. No UX regression. Downside: the e2e exercises a
   slightly different element than real users (acceptable since the well path is also wired and the
   persistence/dedup/GC assertions are mechanism-agnostic). Add a brief comment explaining the proxy is
   an e2e affordance, not a user surface.

3. **Document-level paste listener gated on well-focus** (my originally-suggested contingency). Keep
   `onPaste` on the well; ALSO `window.addEventListener('paste', …)` in a `useEffect`, handling only when
   `wellRef.current?.contains(document.activeElement)`. Probe focuses the well, `webContents.paste()`.
   Verify it fires; if `webContents.paste()` still won't deliver to the focused non-editable well, this
   alone won't help the e2e (but real users keep working) → combine with option 1's keystroke for the e2e.

**Recommendation:** try **option 1** (cleanest, no proxy, real path, no UX surface added). If the
keystroke won't carry the image, use **option 2** (lowest-risk: real path preserved, proxy only for the
test). Whatever you pick, **the acceptance bar is: real-user paste demonstrably works AND the 4 probe
parts are `ok:true`.** If you cannot make a real-input e2e drive the genuine user path, prefer option 2
over weakening the user feature; do NOT remove the well `onPaste`.
