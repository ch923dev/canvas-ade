# Canvas ADE — Project State Report (origin/main @ 0ccfb9f4, 2026-06-20)

> Generated by a 10-agent workflow (9 code-grounded subsystem/roadmap readers + 1 synthesis pass)
> against `origin/main` (verified identical to local `main` @ `0ccfb9f4`, schema v12).

## 1. TL;DR

Canvas ADE is a mature, feature-complete spatial canvas IDE. All core subsystems — the React Flow v12 canvas engine, terminal (xterm + node-pty), OSR browser preview, planning whiteboard, board groups/connectors, persistence (schema v12), and the full MCP swarm + orchestration layer — are shipped on `main` and rated **Solid** by code-grounded audits, with no open correctness, security, or data-loss findings. The one thing between the product and release is **Phase 5 packaging/signing**: code signing, a live HTTPS auto-update feed, app icons, and a fix for the `electron-builder publish:null` bug, none of which are on `main` yet. The 2026-06-19 feature audit surfaced 43 polish findings (3 High / 17 Med / 23 Low), now organized into the "PA" remediation epic — **PA-1, PA-2, and PA-8 are merged**; the rest are file-disjoint slices ready to run in parallel. The genuinely urgent must-work-now items are small: the three High findings (CANVAS-01, PLAN-01, PLAN-02) are 1-line/S-effort perf and a11y fixes, two of which (CANVAS-01, PLAN-02) are already done. The real release work is infrastructure (certs, feed, icons), not code.

## 6. Release blockers (Phase 5)

Phase 5 packaging/signing is **the single thing between feature-complete and shippable.** All blockers are infrastructure, not code quality.

| Blocker | Severity | What's needed | Effort |
|---|---|---|---|
| Phase 5 infra not on main | High | `feat/phase5-packaging` (#161, base 502d859e) carries icons + electron-updater infra + CI signing stages; was 32 commits behind, rebased 2026-06-19, typecheck green. Must rebase + merge. | — |
| Code signing | High | macOS notarization + Windows Authenticode certs must be obtained and wired into CI (currently `CSC_IDENTITY_AUTO_DISCOVERY=false`, no certs). Unsigned installers fail OS security checks. | M |
| Auto-update HTTPS feed | High | `electron-builder.yml` `publish: null` must be replaced with a live HTTPS feed URL **before** activation. Security invariant: unsigned auto-update over non-HTTPS = RCE. electron-updater is dependency-present but dormant-by-design. | M |
| `publish: null` bug | Med | `publish: null` silently nullifies the release CI `--publish always` flag → workflow goes green with **zero uploaded assets** (traced to app-builder-lib PublishManager.js:63-64). Expected to resolve when feed URL is set, but **must be tested in the first production build**. | S |
| App icons | High | `build/` is empty (`.gitkeep` only). Must add PNG (Win/Linux) + ICNS (Mac); signed builds fail at packaging without them. | S |
| Packaged recap-hook verify | Med | BUG-003 fix landed (#107: `app.asar`→`app.asar.unpacked` path + `ELECTRON_RUN_AS_NODE=1`), but **has never run in a real packaged build** — no packaged smoke exists pre-Phase 5. Must verify end-to-end when the first installer ships. | S |
| Asar slimming | Med | Renderer-only deps (geist, react, react-dom, @xyflow/react, @xterm/*) are double-shipped (Vite-bundled into `out/renderer` AND packed raw into `app.asar`). Packaging efficiency, not in Phase 5's listed scope; reduces binary size. | M |
| node-pty stable watch | Low | Pinned to 1.2.0-beta.13 (winpty-free) due to spaced repo path. Monitor for a winpty-free stable 1.2.0, or plan repo relocation if staying on the beta long-term. | S |

**Sequencing risk:** if signing and feed are not **both** live before the first production release, users of that build can never receive future updates — the `publish: null` bug means a stale version could ship if the order is wrong. This is the one release-blocker class that is failure-by-omission, not failure-by-bug.
