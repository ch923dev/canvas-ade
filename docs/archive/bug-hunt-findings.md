## 1. [HIGH] üëÅ Dragging a board over a live Browser board: the native WebContentsView stays attached and occludes the dragged board

- **File:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:267 (detach wired only to useOnViewportChange); Canvas.tsx:212 (onNodeDragStart only calls beginChange, no node-drag detach)`  ¬∑  **Class:** occlusion  ¬∑  **Confidence:** high
- **Why real:** The detach+snapshot mechanism is wired ONLY to camera-viewport changes, and node drag does not change the viewport, so an overlapped Browser view is never detached during a node drag.

1) Detach is invoked only from beginMotion/endMotion, which are registered exclusively via useOnViewportChange({onStart: beginMotion, onChange: startPump, onEnd: endMotion}) at BrowserPreviewLayer.tsx:267. The only other detach paths (demoteToSnapshot:143, closeBoard:181) are called only from endMotion (253) or from reconcile's board-removal branch (278-284). None fires on overlap.

2) useOnViewportChange fires only on camera pan/zoom. In @xyflow/react v12 (node_modules/@xyflow/react/dist/esm/index.mjs:1310-1324) its callbacks (onViewportChangeStart/Change/End) are dispatched solely from XYPanZoom's onPanZoomStart/onPanZoom/onPanZoomEnd. Node dragging uses a separate system, XYDrag (index.mjs:1586+), which mutates node.position (1671) without touching the viewport transform ‚Äî so a node-only drag never triggers beginMotion/endMotion.

3) Canvas.tsx wires only onNodeDragStart={onNodeDragStart} (line 212), and that handler is `() => beginChange()` (line 146) ‚Äî undo-history bookkeeping only, no preview detach. There is no onNodeDrag/onNodeDragStop handler.

4) The one path that does run during drag is benign: RF emits `position` NodeChanges for the dragged board B ‚Üí nodeChangesToIntents maps to a 'move' intent (nodeChanges.ts:19-20) ‚Üí updateBoard (Canvas.tsx:107) ‚Üí store change ‚Üí BrowserPreviewLayer's store subscription (line 328) runs reconcile. But reconcile's toGeom (313-324) filters to type==='browser', so a dragged Terminal/Planning board B is excluded; the Browser board A's geometry is unchanged. For A, reconcile takes the r.exists && r.attached branch (299-305) and merely RE-pushes A's own bounds via attachPreview ‚Äî keeping A attached and on top. No branch detaches A because some other board overlaps it.

5) A native WebContentsView paints above ALL HTML and cannot be clipped (preview.ts:17-19; ADR 0002). So while B is dragged over A's device-stage rect, A's live page covers B. RF's HTML z-index on the dragged/selected node cannot raise an HTML node above a native OS layer.

Result matches the repro: B is occluded over A's preview rect during (and after) the drag, and only reappears once the camera pans/zooms ‚Äî which finally fires endMotion ‚Üí demoteToSnapshot/attach. The defect is real.

Severity: high rather than critical. It is a transient, recoverable visual/interaction artifact (no data loss, no crash; the board reappears on any camera move) and falls in the known WebContentsView-occlusion class documented in ADR 0002. But it directly breaks the core "drag a board" interaction whenever a live Browser board is on screen, hence high.
- **Repro:** 1) Add a Browser board A; ensure zoom > LOD_ZOOM (0.4) and it is on-screen and attached (status connected, native view live). 2) Add a Terminal or Planning board B nearby. 3) Grab B by its titlebar (.board-titlebar) and drag it so it overlaps A's device-stage area. EXPECTED: B slides over A. ACTUAL: the overlapping portion of B (titlebar/content/snapshot) is hidden behind A's live native page; B appears clipped/erased over A. Releasing the drag leaves it occluded; only a subsequent camera pan/zoom (which triggers useOnViewportChange ‚Üí endMotion) restores normal layering by detaching/snapshotting A.
- **Fix:** Trigger the same detach-during-motion behavior on node drag that already exists for camera motion. In Canvas.tsx, wire onNodeDragStart and onNodeDragStop to a callback the BrowserPreviewLayer exposes (e.g. via a shared ref/context or store flag): on drag start call beginMotion (capture‚Üísnapshot‚Üídetach all live Browser views, set gestureRef), on drag stop call endMotion (reattach eligible/under-cap, snapshot the rest). This makes any drag gesture demote live native views to clippable HTML snapshots so HTML z-ordering applies, exactly as a camera gesture does today.

Concretely: add an imperative handle to BrowserPreviewLayer (or move beginMotion/endMotion into a small module-level event bus) and call it from Canvas's onNodeDragStart/onNodeDragStop. A cheaper but coarser alternative: detach all live Browser views whenever ANY non-browser board's geometry changes within reconcile ‚Äî but the drag-start/stop hook is more precise and mirrors the existing camera-motion contract. Note onNodeDragStart currently only calls beginChange(); preserve that call and add the detach trigger alongside it.

## 2. [HIGH] üëÅ A Browser board's live native WebContentsView occludes and steals input from an overlapping board, making the underlying board unselectable/undeletable in the working-zoom band

- **File:** `src/renderer/src/main/preview.ts:166 (addChildView ‚Äî native view captures input); BrowserPreviewLayer.tsx:267`  ¬∑  **Class:** occlusion  ¬∑  **Confidence:** high
- **Why real:** The full causal chain is provable in code. (1) preview.ts:166 `owner.contentView.addChildView(e.view)` attaches an Electron WebContentsView ‚Äî a native OS layer that paints above ALL HTML and intercepts pointer input over its bounds (acknowledged as inherent in CLAUDE.md/ADR 0002: "paints above ALL HTML ‚Üí occludes boards/in-canvas chrome"). Clicks over it interact with the web page; they do NOT fall through to the React Flow HTML node beneath. (2) A view is attached/live whenever zoom ‚â• LOD_ZOOM(0.4): isLiveEligible (previewPlan.ts:18-22) only requires zoom ‚â• 0.4, positive size, and on-screen ‚Äî i.e. the entire working-zoom band. (3) Boards can freely overlap: canvasStore.ts has NO collision detection ‚Äî addBoard (line 77) places at the exact given position, updateBoard move/drag (line 97) has no overlap guard, and Canvas.addCentered (Canvas.tsx:123) always drops a new board at the current view center, so adding B over a centered live A overlaps them. (4) Selection/deletion is HTML-hit-test-only: the sole board-select path is React Flow pointer hit-testing ‚Üí onNodesChange select intent (Canvas.tsx:103-120 via nodeChanges.ts), and the sole UI delete path is deleteKeyCode=['Backspace','Delete'] (Canvas.tsx:219) which acts on the *selected* node. onNodeDoubleClick focus also needs the click to land on B's HTML. There is NO board list, context menu, or programmatic delete-by-id anywhere in the UI (AppChrome.tsx exposes only add-board and zoom/fit controls; canvasStore has selectBoard/removeBoard but they are only invoked from the click-driven onNodesChange and dock). Therefore, if B's selectable HTML (notably its `.board-titlebar` drag handle) sits under A's live native rect, B cannot be selected, focused, or deleted with nothing/A selected. Severity is high not critical because (a) A's native rect is only the device STAGE (browserLayout.deviceStageRect), not the whole board ‚Äî A's titlebar(34)/URL bar(30)/STAGE_PAD(14)/border are HTML and don't steal input, so only B's region under A's web-content area is dead; and (b) an escape hatch exists ‚Äî zooming below 40% detaches A (endMotion/demoteToSnapshot path) and re-enables clicking B, or the user can move/delete A first. B is recoverable but only via a non-obvious workaround.
- **Repro:** 1) Add a Browser board A; let its native view go live (zoom ‚â• 40%, A connected). 2) Click the dock '+ Browser' (or '+ Terminal') to add board B ‚Äî addCentered drops it at the view center, on top of A; or drag A over an existing B so A's device stage covers B's titlebar. 3) Try to click B where it sits under A's web-content stage to select it: the click instead lands on A's web page (selects/scrolls A's DOM), B never gets a React Flow select. 4) Press Backspace/Delete: nothing selected (or A selected) ‚Üí B is not removed. 5) Double-click to focus B: the dbl-click hits A's page, focusBoard never fires for B. B is stranded until you zoom out below 40% (A detaches) or move/delete A. Worse case: make B itself a Browser board fully behind A ‚Äî both native views overlap with arbitrary OS stacking and neither is reliably click-selectable.
- **Fix:** Provide at least one selection/delete path that does not depend on HTML hit-testing under the native layer. Concrete options (any one closes the strand): (a) Add a board-list/outline panel in AppChrome (it already reads useCanvasStore boards) where clicking a row calls selectBoard(id) and a row delete calls removeBoard(id) ‚Äî bypasses the native occlusion entirely. (b) When a board is selected via the list/keyboard, render a small HTML delete/focus affordance OUTSIDE the native stage rect (e.g. on the titlebar, which is HTML and not covered when only the stage overlaps) and ensure Backspace/Delete removes the store-selected board even if React Flow has no selected node (wire a renderer-level key handler in Canvas.tsx that calls removeBoard(useCanvasStore.getState().selectedId) so delete works off store selection, not just RF deleteKeyCode). (c) Mitigate the root occlusion: temporarily detach (preview:detach) the topmost live view on a modifier-held click or when a marquee/alt-click targets a lower board, then hit-test the HTML beneath; reattach on release. (d) Add z-order/"send to back / bring to front" and a click-cycling behavior so overlapping boards can be reordered, combined with detaching the front view to expose the one behind. The minimal high-value fix is (a)+(b): a list to reach any board plus a store-selection-driven Delete, so an occluded board is always reachable regardless of native stacking.

## 3. [MEDIUM] üëÅ Browser board stays 'idle' (view never opens, URL never loads) when created/loaded below LOD zoom or off-pane

- **File:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:286-290, 159-178`  ¬∑  **Class:** ?  ¬∑  **Confidence:** high
- **Why real:** The claim is confirmed by the code. `attachBoard` (BrowserPreviewLayer.tsx:159-178) is the ONLY place that calls `window.api.openPreview` (line 173) and sets `status:'connecting'` for a first open (line 172). For a brand-new board (`!r.exists && !r.attached`), `reconcile` only calls `attachBoard` when `liveEligible(g)` is true (lines 288-290). `liveEligible` -> `isLiveEligible` (previewPlan.ts:18-22) returns false when `zoom < LOD_ZOOM` (LOD_ZOOM = 0.4, canvasView.ts:16) or when `screenY < paneTop`. So a Browser board created while the camera is zoomed out below 0.4 (or with its stage above the pane top) is never attached, `openPreview` is never called, and the page never loads. The previewStore entry stays at DEFAULT_RUNTIME (`status:'idle'`, `snapshot:null`, previewStore.ts:33-41). I verified status is only ever set to 'connecting' at two sites (lines 172 and 295); line 295 is gated on `r.exists`, which is only set inside `attachBoard`, so it can never run for a board that was never attached. The two load-event handlers (lines 363/371) only fire after a view is opened. In BrowserBoard's `DeviceContent` (BrowserBoard.tsx:265-291), with status 'idle' and no snapshot, neither the load-failed branch nor the snapshot branch is taken, so it renders the placeholder whose title is `''` (because `runtime.status === 'connecting'` is false at line 288) and a sub line showing the bare URL, with a gray 'idle' status pill (statusPill default, lines 306-307). Recovery only happens when the user zooms/pans past LOD: `useOnViewportChange.onEnd` -> `endMotion` (line 253) -> `liveEligible` becomes true -> `attachBoard`. The most realistic trigger is the dock-add path: `addCentered` (Canvas.tsx:123-133) places a new board at the current view center; if the user adds a Browser board while zoomed out (a normal whole-canvas overview), zoom < 0.4 so it lands idle and dead until interaction. The Phase 3 canvas.json load path would hit this identically (and could also hit the off-pane `screenY < paneTop` sub-case). This is genuinely wrong behavior, not handled anywhere: there is no eager open or idle-state fallback.
- **Repro:** 1. Open the canvas and zoom the camera out below 0.4 (LOD_ZOOM) so boards show as LOD cards (or just be in a wide overview). 2. Click the dock '+ Browser' button. `addCentered` adds a Browser board at the view center; because zoom < 0.4, `liveEligible` is false in reconcile, so `attachBoard`/`openPreview` are never called. 3. Observe: the new Browser board's title pill shows a gray 'idle' dot, the device stage is empty (no snapshot, no 'Connecting‚Ä¶' text ‚Äî the title string is '' because status !== 'connecting'), only the URL text appears, and the page never loads. 4. Zoom in past 0.4: on move-end, endMotion -> liveEligible -> attachBoard fires, the view finally opens and the URL loads. Before that interaction it stays a dead idle frame.
- **Fix:** Make reconcile mark a not-yet-eligible new board as pending/connecting so the UI doesn't look dead, OR (cleaner) ensure such a board gets opened/loaded once it becomes eligible without requiring a zoom gesture. Concrete options: (a) In reconcile, in the `!r.exists && !r.attached` branch, when `!liveEligible(g)` set `patchRuntime(g.id, { status: 'connecting' })` so DeviceContent shows the 'Connecting‚Ä¶' placeholder, and ensure a subsequent viewport settle triggers attach (endMotion already does). (b) Better: eagerly `openPreview` the view (load the URL) even when below LOD/off-pane but keep it detached with a snapshot fallback, so the page is actually fetched and a snapshot can be captured; only gate the *attach* (native compositing) on `liveEligible`, not the open/load. (c) On board create, if not live-eligible, schedule the same `pickLive`/attach reconciliation that endMotion runs (e.g. run an eligibility pass after the initial reconcile) so a board created in the snapshot zone still loads in the background and shows a snapshot rather than a bare idle frame. Also update DeviceContent so the placeholder shows a sensible label for 'idle' (e.g. treat idle like connecting, or show 'Preview' / paused) instead of an empty title.
