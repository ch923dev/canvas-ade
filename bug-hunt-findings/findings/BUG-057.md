# BUG-057: Auto-connect detect path calls updateBoard without beginChange — background 1s timer silently destroys the user's redo branch

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/useBrowserAutoConnect.ts`, `src/renderer/src/store/canvasStore.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The detect branch runs `live.updateBoard(bid, { url: next })` with no `beginChange()`. canvasStore.updateBoard (canvasStore.ts:537-567) clears the redo branch (`future: []`) on any real change and records no undo checkpoint. So a machine-driven URL push from the always-on 1s interval (a) silently wipes an armed redo stack with zero user gesture — the user presses Ctrl+Z, the dev server comes up a second later, auto-push fires, and Ctrl+Shift+Z is dead; and (b) coalesces the URL change into the previous undo checkpoint, so undoing the user's last gesture also reverts the detected URL. This is the exact class fixed in BUG-020 (MCP configureBoard) and BUG-021 (applyPush) in the 2026-06-07 hunt; this call site was missed.

## Where
`src/renderer/src/canvas/boards/useBrowserAutoConnect.ts:104` — `useBrowserAutoConnect` (the `plan.kind === 'detect'` branch's async detectPorts resolver inside the 1s interval tick)

## How it triggers
A Browser board linked to a terminal (previewSourceId set) with no valid http url yet; user performs an undo (arming redo); within the backoff window detectPorts resolves a server URL and updateBoard fires.

## Verification evidence
Confirmed. useBrowserAutoConnect.ts:104 calls `live.updateBoard(bid, { url: next })` from the always-on 1s interval with no beginChange(). canvasStore.ts:537-567 proves updateBoard records NO undo checkpoint and, on any real diff, returns `s.future.length ? { boards, future: [] } : { boards }` (:566) — i.e. it wipes an armed redo branch. The detect path guards `url !== next` (:104), so when it fires it is always a real diff → future cleared. The convention it violates is documented in-repo: useMcpCommands.ts:60-65 ('beginChange() FIRST, mirroring every user-gesture call site … without it the change is non-undoable and a no-op edit would still clear an armed redo branch' — the BUG-020 fix), and BrowserBoard.tsx:156-158 does beginChange()+updateBoard for the same url field. Net: a background machine write (a) kills the user's redo stack with zero gesture, (b) is itself non-undoable and coalesces into the previous checkpoint, so undoing the last user gesture also reverts the detected URL. Low — matches the BUG-020/021 precedent grade.

## Suggested fix direction
Route the detect-push through the same beginChange()/trackedChange convention the MCP configureBoard fix established (beginChange dedups, so no phantom snapshots).
