# BUG-038: Configured caps above MAX_PERSISTED_CALLS (2000) are unenforceable — the persisted counter wraps to 0 through the corrupt-file rejection path, with a misleading corruption warning per call

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/llmBudget.ts`, `src/main/llmConfig.ts`, `src/main/llmIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
llmConfig accepts any finite maxCallsPerDay >= 0 (readLlmConfig floors it, no upper bound; llm:setConfig adds none either), but llmBudget.read() rejects any persisted `calls` > MAX_PERSISTED_CALLS (= 2000) as 'corrupt/tampered' and resets the day count to 0. With a legitimate cap of e.g. 5000: tryConsume writes calls=2001, the very next read sees 2001 > 2000, warns '[llmBudget] budget file has an invalid shape', and returns null → calls resets to 0 → the count cycles 0→2001→0 forever. The user's cap is never reached (effectively unlimited calls), and the console is spammed with a misleading corruption warning on every call past 2000.

## Where
`src/main/llmBudget.ts:24` — the module-level `MAX_PERSISTED_CALLS` constant, enforced in the module-private `read()` (consumed via `createBudgetStore`'s `current()`/`tryConsume`)

## How it triggers
Set maxCallsPerDay to any value > 2000 (hand-edit llm-config.json — readLlmConfig accepts it — or call llm:setConfig with 5000), then exceed 2000 summarize calls in one local day. The 2001st persisted count is discarded as corrupt and counting restarts at zero.

## Verification evidence
Mechanical wrap verified across the two modules. Config side: readLlmConfig (llmConfig.ts:82-87) accepts any finite maxCallsPerDay >= 0 — no upper bound — and llm:setConfig adds none (llmIpc.ts:195, see candidate 2). Budget side: llmBudget.ts:24 `MAX_PERSISTED_CALLS = DEFAULT_MAX_CALLS_PER_DAY * 10` = 2000; read() line 73 rejects `p.calls > 2000` → warn '[llmBudget] budget file has an invalid shape' (line 79) → null → current() (line 97) returns `{day, calls: 0}`. tryConsume (lines 102-110) has no upper clamp on write: with cap=5000 and stored calls=2000, `2000 >= 5000` is false → write(calls: 2001). The very next current() rejects 2001 as corrupt and restarts at 0, so the count cycles 0→2001→0 forever and the user's 5000 cap is never reached (effectively unlimited). Test gap confirmed: llmBudget.test.ts exercises caps 0-5 only — nothing covers cap > 2000. Low: requires a deliberately hand-set cap >2000 plus >2000 real calls in one local day.

## Suggested fix direction
Derive MAX_PERSISTED_CALLS from the configured cap (or clamp the cap at write/read time), and clamp tryConsume's write.
