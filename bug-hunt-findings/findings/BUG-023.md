# BUG-023: Restart respawns the PTY at default 80x24 for a terminal created-and-restarted entirely under LOD because restart() bypasses the #40 deferred-fit guard

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
TerminalBoard.restart() spawns the new PTY directly from term.cols/term.rows with no fit.fit() and no proposeDimensions/defer guard, unlike the #40-fixed initial launch() path. For a terminal created while the camera is below the 40% LOD threshold and restarted before ever being zoomed into detail view, term.cols/rows are still the xterm constructor default 80x24, so the respawned shell and any launchCommand TUI are written at 80 columns. Impact is transient and self-healing (the always-connected ResizeObserver reflows the PTY the moment the well becomes visible on zoom-in), and the trigger is near-unreachable in practice because the restart button is hidden behind the opaque LOD card.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:315-342 — restart() (cols/rows read at 333-334); contrast launch() defer guard at 237-241; LOD render at 368-412

## How it triggers
1) Zoom the canvas below 40% (LOD). 2) Dock '+ Terminal' to add a board with launchCommand set; addCentered (Canvas.tsx:133-140) keeps the LOD zoom, so the board mounts with its xterm well at display:none and fit.fit() no-ops → term.cols/rows stay at the constructor default 80x24, and the #40-deferred initial spawn waits. 3) Without ever zooming into detail view, click the Restart button (hit-testable because TerminalBoard always renders the full BoardFrame with actions and the LOD overlay at line 401 is pointerEvents:'none', so the click passes through to the title bar beneath the opaque LOD card). 4) restart() calls spawnTerminal({cols: term.cols=80, rows: term.rows=24}) and writes launchCommand at 80 columns. Self-corrects to true width only when the user later zooms into detail (RO fits → onResize → proc.resize → SIGWINCH).

## Verification evidence
Code-path trace against current code: (a) restart() at TerminalBoard.tsx:327-335 calls window.api.spawnTerminal with cols: term.cols, rows: term.rows (lines 333-334) and never calls fit.fit() nor checks fit.proposeDimensions() — unlike launch() (lines 237-241) which is the #40 fix that defers until proposeDimensions returns finite dims. (b) FitAddon no-ops on a display:none element: node_modules/@xterm/addon-fit/lib/addon-fit.js — proposeDimensions() returns undefined when cell.width/height===0 (hidden subtree) and fit() early-returns on undefined/NaN, so no resize occurs; the well is display:none under LOD via shellHidden (TerminalBoard.tsx:381,425) wired from lod (BoardNode.tsx:58,105). This is exactly the mechanism the CONFIRMED #40 finding (docs/bug-hunt-findings.md:585-590) verified, and xterm's default geometry is 80x24. (c) The ResizeObserver (lines 275-285) stays connected for the board lifetime, but its fit.fit() ALSO no-ops while the well is display:none, so it does not refresh term.cols/rows under LOD — it only corrects once the well is visible. (d) Restart is reachable at LOD: TerminalBoard always renders the full BoardFrame with actions (lines 370-399); the LOD overlay (line 401) sets pointerEvents:'none', which inherits to its BoardFrame child (no override in BoardFrame.tsx:135-153), so clicks pass through to the restart IconBtn beneath. (e) pty.ts:287-291 creates the PTY at opts.cols/rows (80/24 arrive defined, so the ?? fallback is moot) and pty.ts:349-350 writes launchCommand+'\r' at those spawn dims. The prior refutation at docs/bug-hunt-findings.md:719 is itself flawed: its LOD argument ('BoardFrame renders actions only in non-LOD branch, so you cannot click Restart while display:none') is wrong for the current code (TerminalBoard always renders the full chrome), and its ResizeObserver argument does not hold under LOD (fit.fit() no-ops on display:none). The build is current; this is the live main code.

## Suggested fix direction
Mirror the #40 launch() guard inside restart(): before reading dims, call fit.fit() and gate on fit.proposeDimensions() returning finite cols/rows; if the well is not yet fitted (LOD/display:none), defer the respawn until the ResizeObserver reports the first good fit (or omit cols/rows so MAIN's ?? 80/24 is a transient placeholder and defer the launchCommand write until the first real {t:'resize'} arrives). Simplest: have restart set a 'pending respawn' flag that the existing ResizeObserver's launch-style path consumes, reusing the same deferral the initial spawn already has. Severity is Low (not the candidate's Medium): the defect is transient, self-heals on zoom-in via the live ResizeObserver, only affects launch-time TUI width, and the trigger requires a blind pixel-precise click on an invisible 24px button behind the opaque LOD card on a terminal that was created under LOD and never zoomed into — a near-unreachable flow, matching the original #40 Low rating.
