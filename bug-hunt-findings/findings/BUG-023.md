# BUG-023: Spawn accepts cols>1000 that isValidResize forever rejects afterward — a >1000-col terminal board's PTY silently freezes at spawn dimensions (row changes dropped too)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/pty.ts`, `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
Two halves of one asymmetry. (1) `pty:spawn` passes `cols: opts.cols ?? 80, rows: opts.rows ?? 24` straight into `pty.spawn` with none of the integer/range validation that `isValidResize` enforces on the resize path — the doc on isValidResize claims absurd values 'must never reach' the PTY, but spawn is an unguarded second entry point. (2) More concretely: isValidResize rejects cols > 1000, but a terminal board can legitimately exceed 1000 columns (a wide board / full view on a 4K-8K display at font size 8); such a board spawns fine at e.g. 1200 cols, but every subsequent resize message — including pure ROW changes, because the whole message fails on the cols field — is silently dropped by attachPortInput's isValidResize gate, so the PTY stays frozen at spawn-time dimensions and any TUI (claude) misrenders with no error anywhere.

## Where
`src/main/pty.ts:341` — the `pty:spawn` handler inside `registerPtyHandlers` (the unvalidated `cols: opts.cols ?? 80, rows: opts.rows ?? 24` passed to `pty.spawn`; the rejecting gate is `attachPortInput`/`isValidResize` at pty.ts:33-42/62)

## How it triggers
Resize a terminal board (or full-view it on a high-DPI/4K+ display) so the xterm grid exceeds 1000 columns, then change the board's height — the resize message is silently swallowed and the PTY keeps the stale row count.

## Verification evidence
Both halves verified in current code. (1) Asymmetry: pty:spawn (pty.ts:341-342) passes `cols: opts.cols ?? 80, rows: opts.rows ?? 24` straight to pty.spawn with no isValidResize call, while attachPortInput (pty.ts:62) drops the ENTIRE resize message when `isValidResize(m.cols, m.rows)` fails, and isValidResize (pty.ts:33-42) hard-caps cols/rows at 1000 — so a pure row-only change is also swallowed once cols>1000. (2) The >1000-col state is reachable: the renderer sends raw fitted dimensions with no clamp — spawn uses `cols: term.cols, rows: term.rows` (TerminalBoard.tsx:296-297, 466-467) and term.onResize forwards `{t:'resize', cols, rows}` verbatim (TerminalBoard.tsx:375-376); the board NodeResizer has only minWidth/minHeight, NO max (BoardNode.tsx:267-269), so board width is unbounded; fontSize clamps to a floor of 8 (schema v8), giving ~4.8px cells → a board dragged past ~4800px grid width fits >1000 cols. xterm.js imposes no 1000-col ceiling. Net behavior: such a board spawns fine at e.g. 1200 cols, then EVERY subsequent resize (including height-only changes) is silently dropped at pty.ts:62 → the PTY keeps stale rows and a TUI misrenders, with no error anywhere. Severity Low as graded — the setup (multi-thousand-px board at minimum font, or an 8K-class full view) is rare, and the legitimate renderer always sends well-formed integers so the unvalidated-spawn half is doc-claim drift rather than an exploitable gap (the IPC is main-frame-guarded, pty.ts:303).

## Suggested fix direction
Apply the same validation/clamp at spawn (or raise/centralize the cap consistently on both paths); consider clamping cols and applying rows when only one field is out of band.
