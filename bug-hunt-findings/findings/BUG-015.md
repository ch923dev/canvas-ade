# BUG-015: beginMotion's post-detach forEach clobbers a board reattached by a concurrent endMotion/applyLiveness (no gestureRef/attached/attachSeq re-check after the detach await)

- **Severity:** Low
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
beginMotion's async capture/detach IIFE re-checks gestureRef only after its FIRST await (capturePreview, line 354) but its post-detach forEach (lines 365-370) re-checks only rec existence — not gestureRef, r.attached, or attachSeq. If endMotion fires during the second (detachPreview) await, applyLiveness->attachBoard re-attaches the board (sets r.attached=true / issues attachPreview IPC) and the IIFE's trailing r.attached=false then clobbers it. The result is a transient renderer<->main desync: the native view stays attached on MAIN but the renderer's flushBatch/beginMotion stop tracking it (a frozen/trailing native layer), OR the board is left stuck on a stale snapshot though it is live-eligible. Recoverable on the next gesture-end reconcile; no data loss/crash/leak.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:352-371 (beginMotion IIFE) — specifically the post-detach forEach at 365-370 vs the only re-check at 354; clobbered by endMotion (420-423) -> applyLiveness (379-408) -> attachBoard (263-297, sets r.attached=true / attachSeq++ at 286-287)

## How it triggers
Have a live Browser board (zoom >= LOD, on-pane). Start a camera pan/zoom or node drag/resize (beginMotion: gestureRef=true, IIFE captures then awaits). The capture await is the WIDE window and is guarded by line 354, so a gesture that ends during capture is safe. The defect needs the gesture to END (endMotion -> gestureRef=false -> applyLiveness -> attachBoard re-attaches the at-rest-eligible board) during the NARROWER detachPreview await (line 364). The IIFE then resumes and runs r.attached=false + patchRuntime(live:false) at lines 368-369, undoing the reattach. Easiest to observe by artificially delaying the main-side preview:detach handler and ending a quick gesture in that window.

## Verification evidence
Code-path trace on HEAD ee15e9b. JS is single-threaded; capturePreview/detachPreview are real ipcRenderer.invoke suspensions (preload/index.ts:70-71). (1) beginMotion line 354 re-checks gestureRef AFTER capture; lines 365-370 forEach after the detach await checks ONLY `if (!r) return` (git grep confirms gestureRef appears at 348/351/354/421/566 — never in 365-370). (2) endMotion (420-423) sets gestureRef=false then applyLiveness; for an at-rest live-eligible board liveIds.has(g.id) is true -> attachBoard (399), which synchronously sets r.attached=true + r.attachSeq++ (286-287) and, when bounds changed, issues window.api.attachPreview (289). (3) MAIN processes IPC FIFO: capture, then detach (#2), then attach (#3 if bounds changed) -> native view ends ATTACHED, but the renderer's IIFE continuation sets r.attached=false -> DESYNC. flushBatch line 320 (`if (!r.attached) continue`) then stops tracking the attached view, and beginMotion line 349 excludes it from future detach -> a frozen/trailing native layer over moving HTML. In the bounds-unchanged sub-case attachBoard diff-skips (274-283) so nothing reattaches, leaving the board stuck on its snapshot though eligible. Both self-heal on the next gesture-end (attachBoard sees r.attached=false -> reattaches). This is NOT the prior-fixed case: bug #26/#45 added the attachSeq reattach-guard to demoteToSnapshot (lines 242/250/253) and bug #37 added only the existence re-check to beginMotion (359/366) — neither added a reattach/gestureRef/attachSeq re-check to beginMotion's detach phase (docs/bug-hunt-findings.md:434-458, 536-541).

## Suggested fix direction
In beginMotion's post-detach forEach (lines 365-370), re-check that the board is still meant to be detached before writing r.attached=false — mirror demoteToSnapshot's guards: bail if !gestureRef.current (a new/ended gesture now owns liveness) and/or snapshot attachSeq per board before the detach await and skip the clobber if r.attachSeq changed (a concurrent attachBoard re-claimed it). Practically: capture `const seq = recs.current.get(g.id)?.attachSeq` per board before the detach, then in the forEach `if (!r || r.attachSeq !== seq || !gestureRef.current) return` before setting r.attached=false. This makes beginMotion symmetric with the demoteToSnapshot reattach-race fix.
