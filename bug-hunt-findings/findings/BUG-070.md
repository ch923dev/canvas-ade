# BUG-070: `prepare` script swallows hook-installation failure with an empty catch — the sole mechanism enabling the only e2e gate can no-op with zero signal

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `package.json`, `.githooks/pre-push`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`"prepare": "node -e \"try{require('child_process').execSync('git config core.hooksPath .githooks')}catch(e){}\""` — any failure (git not on PATH for the node child, not a git dir, config lock, install run with scripts disabled) is caught and discarded, so `core.hooksPath` silently stays unset/drifted and BOTH local gates (cheap pre-commit trio and the pre-push e2e matrix — which deliberately does NOT run in CI) are disabled without any signal. This exact failure mode has already occurred in this project (memory note `pre-push-gate-hookspath-bypass`: a fast push with no `[pre-push]` line = ungated push reached origin). The script should at minimum print a loud warning on failure, and ideally a guard (e.g. in pre-push CI-side or a `doctor` check) should verify `git config core.hooksPath` equals `.githooks`.

## Where
`package.json:38` — the `prepare` npm lifecycle script (`scripts.prepare`)

## How it triggers
Any environment where the prepare-time `git config` fails or where the config is later overwritten (other tooling, fresh clone with `--ignore-scripts`, pnpm install variants) — subsequent pushes skip the e2e gate with no indication.

## Verification evidence
package.json:38 verbatim: `"prepare": "node -e \"try{require('child_process').execSync('git config core.hooksPath .githooks')}catch(e){}\""` — execSync inherits no stdio config (errors are thrown, then discarded by the empty catch), so any failure (git absent from the node child's PATH, non-git dir, `--ignore-scripts` skipping prepare entirely, config lock) leaves `core.hooksPath` unset with no warning. Refutation attempt failed: a repo-wide grep for `hooksPath` finds NO other automated guard — only docs (TESTING.md:171, CLAUDE.md:197) and manual checklists (docs/superpowers/plans/2026-06-09 'must print .githooks'); neither hook self-verifies, and e2e deliberately does not run in Actions, so a silent prepare failure disables both the pre-commit trio and the only e2e gate with no indication. The project's own memory (`pre-push-gate-hookspath-bypass`) records an ungated push reaching origin from exactly this drift class. Note honestly: the catch itself is deliberate (install must not fail in CI/Docker where git config can fail) — the defect is strictly the absence of any warning/verification, which is why this stays Low (process-gate reliability, no user-facing impact).

## Suggested fix direction
Print a loud `console.warn` in the catch, and/or have the hooks' docs-adjacent doctor step (or pre-push itself, which CAN run when installed) assert `git config core.hooksPath` equals `.githooks`.
