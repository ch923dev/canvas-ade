# BUG-026: Empty note/free-text orphaned when its grip is pressed (no drag) then focus leaves — blur-prune permanently skipped

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/planning/NoteCard.tsx`, `src/renderer/src/canvas/boards/planning/FreeText.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
A freshly-dropped empty note/free-text auto-focuses so that blurring without typing prunes the orphan. Pressing the card's grip sets dragging.current=true to suppress the prune for the grip-induced textarea blur, then clears it via setTimeout(0). On a zero-movement grip press the blur fires synchronously while dragging.current is still true, so the empty element is NOT pruned, is never re-focused, and lingers permanently in board.elements — round-tripping through canvas.json with no schema-level cleanup.

## Where
- src/renderer/src/canvas/boards/planning/NoteCard.tsx:120-132 (grip onPointerDown sets dragging.current=true + setTimeout(0) reset), 146-148 (textarea onBlur prune guarded by !dragging.current); identical structure in src/renderer/src/canvas/boards/planning/FreeText.tsx:104-113, 134-136

## How it triggers
1) Select the note tool and click empty whiteboard well in a Planning board → makeNote creates {text:''} committed to store immediately (PlanningBoard.tsx:229-231), tool resets to 'select' so interactive=true. 2) The new note mounts and auto-focuses its textarea (NoteCard.tsx:69-72, mount-only effect, fires because interactive && note.text===''). 3) Instead of typing, single-click on the note's padding grip ring (event target = the non-focusable .pl-note-grip div, NOT the textarea). 4) The grip onPointerDown sets dragging.current=true and schedules setTimeout(()=>dragging.current=false,0). 5) Chromium's focus controller, finding no mouse-focusable ancestor of the grip div, blurs the textarea as the synchronous default action of the press — onBlur runs with dragging.current STILL true (the setTimeout macrotask cannot run until the input task ends), so the prune `if(!dragging.current && text.trim()==='')` is skipped. 6) No pointer movement → onWellPointerMove never sets dragPos; onWellPointerUp sees dragPos===null and commits nothing (PlanningBoard.tsx:301-311). 7) Click empty space / another board: the textarea is already blurred and is never re-focused, so no second blur ever fires. The empty note persists. Same flow for empty free-text via its left drag gutter.

## Verification evidence
Code-path trace against post-48c6500 code (both files last touched by 48c6500). The only auto-removal path for an untouched empty note/text is the textarea onBlur prune (NoteCard.tsx:146-148 / FreeText.tsx:134-136); KeyDown-Backspace prune (NoteCard.tsx:155) requires the textarea to be focused, and the X delete button is manual. The mount-only auto-focus effect ([] deps, lines 69-72 / 63-66) cannot re-run after the grip press, so no second blur occurs. Timing is decisive and holds in Chromium/Electron: the grip handler sets dragging.current=true synchronously; the focus shift (and its blur dispatch) is the synchronous default action of the pointerdown/mousedown and runs before the event loop yields, hence before the setTimeout(0) reset — confirmed by the code's own comments ('focus leaves when the grip is pressed') which are why the dragging guard was added. stopPropagation() on the grip does not suppress the focus default (only preventDefault would, and it is not called); wellRef.setPointerCapture in startElementDrag does not cancel the pending blur. makeNote/makeText create text:'' (elements.ts:51-67) and the note is committed to the store at creation (PlanningBoard.tsx:229-231), so the orphan is real store/persisted state, not transient. No schema/save/load filter removes empty elements (grep over src/renderer/src/lib found no empty-element pruning; boardSchema migration pipeline is empty). No test covers NoteCard/FreeText. This is an incomplete-remediation hole in the 48c6500 empty-element blur-prune (comments cite #29 for NoteCard, #36 for FreeText): the dragging guard added to stop mid-drag pruning over-suppresses the zero-movement grip-press path. (Note: the candidate's '#29/#42' label is imprecise vs docs/bug-hunt-findings.md numbering, but the defect itself is accurately described.)

## Suggested fix direction
Do not rely on a time-based dragging flag to gate the prune. Instead, only suppress the prune when an actual move was committed, or re-evaluate emptiness after the gesture settles. Options: (a) on pointer-up of a zero-movement grip grab (onWellPointerUp with dragPos===null for a 'move'), re-check and prune the still-empty element; (b) move the prune off blur into a deferred check (e.g. queueMicrotask/rAF after the gesture) that reads current text and whether a move committed, rather than reading dragging.current at blur time; (c) make the grip press re-focus the empty textarea so a later genuine blur can prune it. Apply the same fix symmetrically to FreeText.tsx. A regression test should simulate grip pointerdown (no move) + pointerup + focus-elsewhere and assert the empty element is removed.
