# BUG-026: Staged-image filenames collide across app restarts — module-level seq resets and writeFileSync silently overwrites a prior session's staged image for the same board

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/terminalImageStaging.ts`, `src/main/clipboardIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
Staged-image filenames are `paste-<boardId>-<seq>.png` where `seq` is a module-level counter (`let seq = 0`) that resets on every app launch, while board ids persist across sessions in canvas.json and staged files persist up to the 1h prune window. Paste an image into board X, quit, relaunch, paste another image into board X → the new stage writes `paste-X-1.png` over the old file via `writeFileSync`. Any consumer still holding the old path — e.g. a `claude --resume`d agent whose prior prompt references the staged path — silently reads a different image than the one the user pasted.

## Where
`src/main/terminalImageStaging.ts:41` — `stageClipboardImage()` (filename construction; `seq` declared at line 12)

## How it triggers
Paste image into a terminal board → quit the app within the prune window → relaunch the same project → paste again into the same board → first file is overwritten under the identical name; a resumed agent re-reading the old path gets the wrong image.

## Verification evidence
Code-path confirmed: `let seq = 0` is module-scoped with no persistence (src/main/terminalImageStaging.ts:12); the filename is `paste-<safeId(boardId)>-<seq>.png` with no timestamp/random component (line 41) and `writeFileSync(file, png)` overwrites silently (line 42). Board ids persist in canvas.json across sessions. Neither cleanup path prevents the collision: pruneOld only removes files older than 1h (lines 54-59), and cleanupStaged is invoked only from the renderer's TerminalBoard unmount cleanup (src/renderer/src/canvas/boards/TerminalBoard.tsx:533) — which never runs when the app quits (window destruction skips React effect cleanup). So: paste into board X → quit within 1h → relaunch → paste into X again → stageClipboardImage writes paste-X-1.png over the previous session's file. Any consumer still holding the old path (e.g. a `claude --resume`d agent re-reading a referenced image — resume is a first-class feature, PR #89) silently gets a different image. Impact is narrow (ephemeral tmp files, most consumers embed the image at paste time), hence Low.

## Suggested fix direction
Add a uniqueness component (timestamp or randomUUID slice) to the staged filename.

None — safe for parallel assignment.
