# BUG-013: applyOpenResult's async .bak retry has no epoch/generation check — a late recovery (or error-set) clobbers a concurrently opened project; AppChrome.switchTo lacks the BUG-008 busy guard and WelcomeScreen's per-mount guard doesn't cover the cross-surface race

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/store/canvasStore.ts`, `src/renderer/src/canvas/AppChrome.tsx`, `src/renderer/src/canvas/WelcomeScreen.tsx`, `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
applyOpenResult awaits `window.api.project.reopenFromBak(r.dir)` when the primary doc is deep-corrupt, then unconditionally applies the result — on .bak success it calls applyLoadedDoc with status:'open', on failure it sets status:'error' — with no generation/epoch token. If a second project open completes during that await, the first call's late continuation overwrites the newly-opened project's boards/viewport/project state with the stale project's .bak content (or stamps status:'error' over a successfully opened project). The concurrent-call path is reachable because AppChrome.switchTo has no busy guard (unlike WelcomeScreen's BUG-008 guard), and a second switchTo also fires `project.save(toObject())` while the renderer still holds the OLD project's boards but MAIN's currentDir already points at the first clicked project — a cross-project write.

## Where
`src/renderer/src/store/canvasStore.ts:779` — `applyOpenResult` (the awaited `reopenFromBak` continuation inside the canvasStore Zustand store)

## How it triggers
Open a project whose canvas.json is deep-corrupt (triggers the .bak retry IPC), and while it is in flight switch to another project from the AppChrome switcher; the first call's late .bak recovery or error-set lands after the second open and clobbers it.

## Verification evidence
Code-path proof. canvasStore.ts:779-797 — after `await window.api.project.reopenFromBak(r.dir)` the continuation applies UNCONDITIONALLY: applyLoadedDoc(set, d2, {dir: r.dir, ... status:'open'}) at 783 or the status:'error' set at 794; nothing re-checks project.dir or any token, so a second open completing during the await is overwritten by the stale project's .bak content (or stamped 'error'). Concurrency is reachable: (a) AppChrome.tsx:119-154 switchTo has NO busy guard (full read confirms — contrast WelcomeScreen.tsx:14-17's explicit BUG-008 guard documenting this exact 'last to resolve overwrites' hazard); (b) worse, switchTo's setProjectLoading (line 138) unmounts Canvas and mounts WelcomeScreen (App.tsx:84), whose `busy` is fresh per-mount local state (WelcomeScreen.tsx:17) — so during the first switch's awaited disposeLiveResources/open/.bak chain the user can click a recent there, firing a second concurrent open the guard never sees. Cross-project write hazard is real: project:reopenFromBak deliberately does NO setCurrentDir (projectIpc.ts:201-209), so after open(dir2) set currentDir=dir2, a late dir1 .bak apply leaves the renderer showing dir1's boards with status 'open' while project:save writes toObject() to getCurrentDir()=dir2 (projectIpc.ts:229-232) — autosave then writes project-1 content into project-2's canvas.json. Severity kept Medium (not High): the .bak-retry window requires a deep-corrupt-but-envelope-valid primary plus a concurrent open landing in the i

## Suggested fix direction
Capture a monotonically increasing open-epoch at applyOpenResult entry and bail before applying after each await if it changed; add the busy/in-flight guard to switchTo (module/store-level, not component-local).
