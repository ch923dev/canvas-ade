# BUG-013: Restart during the mount's adopt/deferred-launch window spawns a second PTY that overwrites the first session entry without reaping it (orphaned process tree)

- **Severity:** Medium
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`, `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The TerminalBoard mount has two independent spawn paths — the restart callback and the mount effect's adopt-gated/LOD-deferred launch() — with no shared in-flight guard (spawned/spawnAllowed are closure-local to the spawn effect, invisible to restart). If restart runs before the mount's adoptTerminal() resolves (or while launch() is deferred under LOD), restart spawns session A, then the mount's launch() later spawns session B. In main, pty:spawn does sessions.set(id, B) at pty.ts:343 with NO dedupe and NO kill of the displaced proc, so proc-A drops out of both the sessions and parked maps. cleanup() and disposeAllPtys() never reap it, leaking a live shell/agent child-process tree until the OS or natural exit reclaims it.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:315-342 (restart spawns directly, no in-flight guard) and :234-273 (launch/adopt path with closure-local spawned/spawnAllowed); src/main/pty.ts:343 (sessions.set overwrites a live session without reaping the displaced proc)

## How it triggers
Two reachable windows. (1) Normal zoom: mount runs spawn(), sends adoptTerminal(id) (async invoke) and has not yet registered any main-side session; the user clicks the visible Restart title-bar button within that round-trip (or during the brief initial-layout window where fit.fit() yields non-finite dims so the immediate launch() defers to the ResizeObserver). Restart's killTerminal no-ops (no session yet) and spawnTerminal creates proc-A. Then adoptTerminal resolves adopted:false → spawnAllowed=true → launch() spawns proc-B → sessions.set(id, B) overwrites A. (2) LOD mount: the well is display:none so proposeDimensions() is non-finite and launch() stays deferred indefinitely; Restart (reachable because the LOD overlay is pointerEvents:none over the still-mounted full chrome) spawns proc-A, then zooming back in fires the ResizeObserver's launch() which spawns proc-B and overwrites A. In both cases proc-A is in neither sessions nor parked, so neither killTerminal(id), cleanup(), nor disposeAllPtys() (app quit) ever kills it.

## Verification evidence
Code-path trace against current post-48c6500 code (pty.test.ts 12/12 green, line refs verified): (a) restart (TerminalBoard.tsx:315-342) is a separate useCallback that calls window.api.spawnTerminal directly and cannot read/set the spawn effect's closure-local spawned/spawnAllowed/disposed (declared at TerminalBoard.tsx:234-236, confirmed via grep — no ref/state escape). (b) launch() (TerminalBoard.tsx:237-261) does NOT call killTerminal before spawning; it just spawns. (c) The mount effect's adopt resolves adopted:false for a fresh board (pty.ts:117-119: parked.get(id) miss) → spawnAllowed=true; launch() (TerminalBoard.tsx:270-271). (d) pty:spawn (pty.ts:274-353) has no sessions.has(id) dedupe and unconditionally sessions.set(opts.id, {proc, port1, buf}) at pty.ts:343, dropping any prior entry's proc/port WITHOUT killTree. (e) cleanup(id, proc) (pty.ts:390-402) only reaps sessions.get(id) (now B); proc-A's own onExit calls cleanup(id, A) → isStaleExit(B, A)=true (pty.ts:379-381) → no-op. (f) disposeAllPtys (pty.ts:456-460) iterates only sessions+parked — proc-A is in neither. The prior REFUTED finding (docs/bug-hunt-findings.md:831-843) only covers restart racing an EXISTING session where killTerminal precedes spawnTerminal FIFO; it explicitly relies on a kill-before-spawn that does NOT exist on the mount's deferred-launch path, so it does not cover this case. This is a genuinely new defect, not one of the remediated 50.

## Suggested fix direction
Give the two spawn paths a shared in-flight guard so only one PTY can exist per board id at a time. Either (a) hoist the spawn-in-flight/spawned state to a useRef on the component so both restart and launch() observe and set it (restart should cancel/await the pending mount spawn, and a deferred launch() should bail if restart already spawned), or (b) make main authoritative: in pty:spawn, if sessions.has(opts.id), kill/reap the displaced proc (cleanup on the old entry, killTree) before sessions.set — turning the silent overwrite into a safe replace so no proc can be orphaned by a double-spawn under the same id.
