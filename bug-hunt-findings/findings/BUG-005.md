# BUG-005: Preview module state (views map / owner / attached) survives main-window destruction — after macOS window reopen, attach() skips addChildView so previews are blank (until a motion detach/reattach), and the docs-mandated 'closed' webContents cleanup is missing

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/main/index.ts`, `src/renderer/src/canvas/boards/usePreviewManager.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The preview module keeps `views`, `owner`, and per-entry `attached` in module scope, and nothing resets them when the main window is destroyed (no `'closed'` handler anywhere in src/main). On macOS — where `window-all-closed` does not quit and `activate` recreates the window — the fresh renderer's `preview:open` finds the STALE entry, and `attach()` sees `e.attached === true` so it skips `addChildView` on the NEW window's contentView, leaving the board blank (or rejecting on a destroyed webContents). Compounding it, `detach()` checks `!owner` but not `owner.isDestroyed()`, and `disposeOne()`'s single try block wraps both `detach(e)` and the mandatory `e.view.webContents.close()` — so a detach throw skips the close and leaks the preview renderer while the entry is deleted from the map.

## Where
`src/main/preview.ts:366` — `attach()` (the `if (!e.attached)` guard at lines 368-371 skips `addChildView` for the new window; `detach()` at line 379-380 lacks an `owner.isDestroyed()` check; `disposeOne()` at lines 391-401 wraps detach + close in one try)

## How it triggers
macOS: open a project with Browser boards → close the main window (app stays alive per window-all-closed) → click the dock icon ('activate' recreates the window) → previews never render again; any subsequent preview:close/closeAll leaks the webContents because detach throws on the destroyed old owner before close() runs.

## Verification evidence
Code-path proof: `views`/`owner` are module-level (src/main/preview.ts:47-48) and per-entry `attached` is only ever cleared by detach() (line 388), reached solely via the preview IPC handlers or disposeAll — nothing fires on window destruction (grep over src/main: zero `on('closed')` handlers; index.ts registers none, and the renderer's unmount cleanup `closeAllPreviews()` at usePreviewManager.ts:996-1003 is a React effect cleanup that does not run on window close). On macOS, window-all-closed does not quit (index.ts:512-517) and 'activate' recreates the window (index.ts:448-450). The fresh renderer calls openPreview for each persisted Browser board (usePreviewManager.ts:455-465) → preview:open → ensure() refreshes `owner` and returns the STALE entry (preview.ts:234-237) → attach() sees `e.attached === true` and provably skips `owner.contentView.addChildView(e.view)` for the NEW window (preview.ts:366-371) — the native view is never added to the new window: the board reports 'connected' (loadURL succeeds; Electron docs confirm child WebContentsView webContents are NOT destroyed automatically on window close and must be closed in a 'closed' handler — exactly the handler this app lacks, so the preview renderer processes also persist un-owned while the window is closed) yet paints nothing, and preview:capture returns blank for it. One correction to the claim: it is not strictly permanent — the first pan/zoom triggers the renderer's detach→reattach cycle, whose attach() then adds the view.

## Suggested fix direction
Register a window 'closed' (or destroyed) hook that either closes all preview webContents per the Electron docs or resets owner + every entry's attached flag; split detach() and webContents.close() into separate try blocks and add an owner.isDestroyed() check.
