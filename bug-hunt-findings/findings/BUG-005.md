# BUG-005: Browser board stranded on 'load-failed' after a successful Reload/Back/Forward (no "navigation restarted" signal clears the renderer status latch)

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
Once a Browser board reaches 'load-failed' (dead/refused URL, blocked scheme, or HTTP 4xx/5xx), a later SUCCESSFUL load via the Reload/Back/Forward URL-bar buttons never restores 'connected'. Main clears its internal `failed` latch on did-start-navigation and does emit did-finish-load on the good load, but there is no event type that tells the renderer "navigation restarted, clear load-failed", and the renderer's did-finish-load handler hard-returns while status is load-failed. The board permanently shows the red 'failed' dot / 'Couldn't load' overlay even though the live page is rendering, and on any LOD/motion demote the load-failed overlay masks the otherwise-valid snapshot. Recoverable only by editing the URL-bar text.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:601-602 (did-finish-load handler: `if (cur === 'load-failed') return`); src/main/preview.ts:164-166 (did-start-navigation clears e.failed but emits nothing); src/main/preview.ts:380-403 (preview:goBack/goForward/reload emit nothing); src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:604-609 (did-navigate handler never touches status)

## How it triggers
1) Point a Browser board at a URL whose dev server is down (connection refused) → did-fail-load fires → renderer status = 'load-failed' (red dot, "Couldn't load"). 2) Start the dev server. 3) Click the URL-bar Reload button (BrowserBoard.tsx:188 → reloadPreview → preview:reload → wc.reload()). The page now loads successfully, but the board stays on 'load-failed' indefinitely. Same outcome via Back/Forward to a good page, or an in-page link load. The ONLY escape is editing the URL-bar input text, because only reconcile's `g.url !== r.lastUrl` branch (BrowserPreviewLayer.tsx:489-492) resets status to 'connecting'; Reload/Back/Forward never mutate board.url.

## Verification evidence
Code-path trace on current HEAD (ee15e9b, post-48c6500, files unmodified per git status). Reload path: BrowserBoard.tsx:188 onClick=`window.api.reloadPreview(board.id)` → preload index.ts:84 invoke('preview:reload') → preview.ts:398-403 handler calls `e.view.webContents.reload()` and emits NOTHING. reload() fires did-start-navigation (main frame) → preview.ts:164-166 sets `e.failed = false` only, no emit. On success Chromium fires did-finish-load → preview.ts:172-179: re-applies zoom, and because `!e.failed` is now true, emits `{type:'did-finish-load'}`. Renderer receives it at BrowserPreviewLayer.tsx:595-603: `const cur = ...status; if (cur === 'load-failed') return` (lines 601-602) → promotion to 'connected' SUPPRESSED. did-navigate also fires (preview.ts:183-206, code 200 → isErrorResponseCode false, no fail emit) → renderer handler lines 604-609 patches only liveUrl/canGoBack/canGoForward, never status. The PreviewEvent union (preview.ts:10-13) has NO "navigation-started/restarted" type, so the renderer is never told to reset to 'connecting'. attachBoard sets 'connecting' only for a brand-new board (`!r.exists`, line 293); a reattach does not. Net: status is permanently pinned at 'load-failed'. Worse on demote: DeviceContent (BrowserBoard.tsx:272) checks load-failed BEFORE snapshot (282), so a valid post-recovery snapshot is hidden behind "Couldn't load". Verified Reload/Back/Forward never change board.url (BrowserBoard.tsx:177/183/188 call goBackPreview/goForwardPreview/reloadPreview, not updateBoard), so the reconcile URL-edit escape (BrowserPreviewLayer.tsx:489-492) does not fire. Prior findings (docs/bug-hunt-findings.md §8, lines 153-183) confirm the renderer guard pre-existed as a belt-and-suspenders against the error-page's OWN did-finish-load; it now over-applies to genuine recovery loads. preview.test.ts covers only the pure helpers, asserting nothing about this state machine.

## Suggested fix direction
Add a "navigation restarted" signal so the renderer leaves load-failed when a fresh main-frame navigation begins. Two non-exclusive options: (a) Main side — in the did-start-navigation handler (preview.ts:164-166), when details.isMainFrame, additionally emit a new PreviewEvent (e.g. {type:'did-start-navigation'} or reuse a status reset) so the renderer can set status back to 'connecting'; this naturally covers reload/back/forward/in-page since all go through did-start-navigation. (b) Renderer side — narrow the guard at BrowserPreviewLayer.tsx:601-602 so it only suppresses the error-page's immediate did-finish-load (e.g. track that a did-fail-load just latched and clear that one-shot suppression on the next navigation start) rather than blocking every did-finish-load while status==load-failed. Either way add an e2e/unit assertion for "fail then successful reload → connected". Be careful not to regress Bug #8/#5: the error page's own did-finish-load that immediately follows a did-fail-load must still be suppressed.

## Prior-hunt relation
#5: the connection-refused/error-page suppression latch (and its renderer mirror) was added without resetting the renderer status on a NEW navigation start — so the fix that stops the error page from clobbering load-failed now also stops a genuine recovery from clearing it.
