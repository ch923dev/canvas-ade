# BUG-044: A transiently-slow (>500ms) recent-project path is not just hidden but PERMANENTLY deleted from the MRU file by the next touchRecent

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/recentProjects.ts`, `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
listRecents treats any path whose access() check exceeds PATH_CHECK_TIMEOUT_MS (500ms) as gone (BUG-010 — reasonable for display). But touchRecent builds its persisted list from `await listRecents(...)` and writes the filtered result back to recent-projects.json. So a project on a momentarily-slow network share / spun-down drive / cold SMB session is silently and permanently dropped from the recents file the next time the user opens ANY project (every project:open and project:current calls touchRecent). The entry never comes back when the share wakes up. The intended BUG-010 behavior was a non-blocking display prune; persisting that prune turns a transient timeout into durable state loss.

## Where
`src/main/recentProjects.ts:90` — `touchRecent` (the `others` list built from the timeout-filtered `listRecents` output)

## How it triggers
Have a recent project on a UNC/SMB share or sleeping external drive; while its first access() probe takes >500ms (cold SMB handshake routinely does), open any other project — touchRecent persists the list without it.

## Verification evidence
src/main/recentProjects.ts — pathExists (lines 39-53) resolves false when access() exceeds PATH_CHECK_TIMEOUT_MS=500 (line 19, explicitly designed for stale UNC/SMB shares whose first probe routinely exceeds it on a cold session); listRecents filters those entries out (lines 78-79); touchRecent then builds its persisted list FROM that filtered output — line 90 `const others = (await listRecents(userDataDir)).filter((r) => r.path !== path)` — and writeFileAtomic.sync's the reduced list back to recent-projects.json (line 95). touchRecent runs on every project:open (projectIpc.ts:152), project:create (224 via remember) and project:current (274), so opening ANY project while one recent's probe is transiently slow permanently drops that entry from the MRU file; it never returns when the share wakes (only re-added if the user manually re-opens that project via the folder dialog). recentProjects.bug010.test.ts contains no touch-persistence assertion (grep 'touch' → no matches), so no coverage contradicts this. The display-time prune (BUG-010) was reasonable; persisting the raced-out result converts a transient timeout into durable state loss. Low: MRU-list convenience loss only, self-healing by manual re-open.

## Suggested fix direction
In touchRecent, persist from the raw shaped list and prune only entries whose access() definitively REJECTED (distinguish rejected from timed-out in pathExists).
