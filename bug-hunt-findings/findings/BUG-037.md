# BUG-037: llm:summarize validates input.text but not input.system — a truthy non-string system builds a malformed provider body that burns a non-refunded budget slot (BUG-011 class); no length bound on either field

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/llmIpc.ts`, `src/main/llmService.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
BUG-011 added a guard rejecting a missing/empty `text` specifically so 'no future renderer caller can leak a null-content request or waste the daily cap'. But `input.system` is still unchecked: any truthy non-string (number, object, array) flows into chatMessages() / the anthropic `system` field in llmService.buildRequest, producing a malformed body the provider rejects with HTTP 400 — AFTER deps.budget.tryConsume() has irrevocably consumed a daily slot (provider errors are deliberately not refunded). Additionally neither `text` nor `system` has any length bound, so a giant string is JSON.stringify'd synchronously in MAIN and egressed at API cost.

## Where
`src/main/llmIpc.ts:108` — the `llm:summarize` `ipcMain.handle` callback inside `registerLlmHandlers`

## How it triggers
Renderer invokes llm:summarize with `{ text: 'hi', system: { role: 'x' } }` (or system: 42). buildRequest serializes `content: {role:'x'}` / `system: 42`; provider returns 400; the call surfaces as provider-error and one budget slot is gone. Repeat to drain the 200/day cap with zero successful summaries.

## Verification evidence
llmIpc.ts:108-109 checks only `typeof input?.text !== 'string' || input.text.length === 0` — `input.system` is never type-checked, and neither field has a length cap. Path verified: llmService.ts:42 `if (input.system) msgs.push({ role: 'system', content: input.system })` and llmService.ts:65 `...(input.system ? { system: input.system } : {})` pass the raw value through; JSON.stringify (lines 62/88) serializes `content: {…}` / `system: 42` into the body → provider rejects (HTTP 400 → opaque throw at line 191). Budget ordering verified: runSummarize consumes the slot at llmService.ts:219-222 BEFORE the try at line 224, and the comment at line 221 states 'a later provider-error is NOT refunded'. So repeated calls with `{text:'hi', system:{...}}` drain the daily cap with zero successful summaries. The BUG-011 comment in this exact handler (llmIpc.ts:103-107) states the intent — reject malformed input 'before any provider/budget work' — system was missed. Channel renderer-reachable (preload/index.ts:261). Low: reachable only by a buggy/compromised renderer (same threat model as the accepted BUG-011).

## Suggested fix direction
Extend the line-108 guard: system must be undefined or a bounded non-empty string; add a length cap on text.
