# BUG-033: mcp:boards sender guard is a stale inline copy, not ipcGuard.isForeignSender — fails OPEN when the window is unresolved and throws on a destroyed window

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/boardRegistry.ts`, `src/main/ipcGuard.ts`, `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
registerBoardRegistryHandler guards with `const main = getWin()?.webContents.mainFrame; if (main && e.senderFrame && e.senderFrame !== main) return`. This deviates from the canonical isForeignSender in two documented ways the ipcGuard refactor specifically fixed: (1) when getWin() returns null (boot window before createWindow, or after window teardown) `main` is undefined and the guard is skipped — any sender's payload is accepted into applySnapshot (fail-open, vs ipcGuard's explicit DENY); (2) when the window exists but is destroyed, `getWin()?.webContents` throws "Object has been destroyed" inside an ipcMain.on listener, escaping to the process-level uncaughtException sink (which exits the app). The handler is registered before mainWindow is created, so the fail-open window is real.

## Where
`src/main/boardRegistry.ts:236` — `registerBoardRegistryHandler()` (the inline sender guard inside the `mcp:boards` ipcMain.on listener, lines 236-237)

## How it triggers
An mcp:boards IPC message arriving while mainWindow is null (boot) — accepted unguarded; or arriving between window destroy and app exit — throws into the uncaughtException sink.

## Verification evidence
Confirmed as a real deviation from the canonical guard with two inverted fail-modes; practical exploitability ~nil. boardRegistry.ts:235-237: `const main = getWin()?.webContents.mainFrame; if (main && e.senderFrame && e.senderFrame !== main) return` — (1) fail-OPEN when getWin() returns null: `main` is undefined, the condition short-circuits, and any payload reaches applySnapshot; ipcGuard.ts's documented rule (:13-14) is the opposite — 'Window unresolved, destroyed, or its webContents destroyed → DENY'. The window is real: registerBoardRegistryHandler is called at index.ts:237 while createWindow() runs at :421, and mainWindow is never nulled on 'closed' (no such handler in index.ts), so getWin() can also return a destroyed window. (2) Throw path: on a destroyed-but-non-null window, `getWin()?.webContents` throws 'Object has been destroyed' inside the ipcMain.on listener → process uncaughtException sink (index.ts:548) → crashShutdown(1) → app.exit(1). ipcGuard.ts's header (:7-9) documents this exact class ('three of those copies resolved the frame in the caller, which throws if the webContents is torn down mid-invoke') and guards it with win.isDestroyed()/webContents.isDestroyed() (:23) — boardRegistry kept the pre-refactor inline copy while recapConsent/pty/projectIpc/llmIpc/preview/clipboard all consume isForeignSender. Violates the stated 'IPC handlers must validate sender frame (ipcGuard)' invariant. Low is right: both windows require an IPC-capable sender while no trusted one exists.

## Suggested fix direction
Replace the inline check with `if (isForeignSender(e, getWin)) return` from ./ipcGuard.
