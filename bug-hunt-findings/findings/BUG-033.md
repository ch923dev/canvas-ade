# BUG-033: IPC handlers never validate event.senderFrame — PTY/preview isolation relies solely on the preview view having no preload (defense-in-depth gap)

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
All 16 ipcMain.handle handlers in pty.ts and preview.ts are registered on the global ipcMain bus and ignore the calling frame (every handler destructures `_e` and never inspects `event.senderFrame`/`event.sender`). ipcMain channels are shared by ALL webContents in the app, including the per-board preview WebContentsViews that load untrusted localhost content. The only thing keeping that untrusted content off powerful handlers like `pty:spawn` (arbitrary shell + cwd + full app env) is that the preview views are created with NO preload, so untrusted pages have no ipcRenderer. There is no IPC-layer allowlist as a second line of defense, so the PTY-isolation security invariant is incidental rather than enforced. Not exploitable today; impact is contingent on a future regression.

## Where
- src/main/preview.ts:289,315,327,334,340,346,351,360,380,389,398 (all preview handlers) and src/main/pty.ts:272,274,355,360,365 (all pty handlers) — none check event.senderFrame against the main window. Preview views created with no preload at src/main/preview.ts:131-141.

## How it triggers
No live trigger against current code: preview WebContentsViews have no preload (preview.ts:131-141 webPreferences has sandbox:true/contextIsolation:true/nodeIntegration:false and NO preload key), so untrusted previewed pages have no ipcRenderer/window.api and cannot invoke any handler. It becomes exploitable only if a future change adds any preload to the preview view (or otherwise exposes ipcRenderer to untrusted frames): a malicious/compromised previewed localhost page could then call window.api.spawnTerminal / ipcRenderer.invoke('pty:spawn', …) to run an arbitrary executable with the app's full process.env, or call preview:navigate/preview:capture on any board id. Because handlers don't check the sender, such a call would succeed regardless of which frame originated it.

## Verification evidence
Verified against current post-48c6500 code. (1) Grep of `ipcMain.(handle|on)` over src returns exactly 16 handlers (5 in pty.ts, 11 in preview.ts); reading each shows none reads event.senderFrame/event.sender — they destructure `_e` and ignore it (pty.ts:274 `pty:spawn (_e, opts)` spawns pty.spawn(shell,args,{cwd,env:{...process.env}}); preview.ts:360 `preview:navigate`, :327 `preview:capture`, etc.). (2) Grep for `senderFrame|e.sender|event.sender|IpcMainInvokeEvent` over src/ returns NO matches — confirms zero IPC-sender enforcement anywhere. (3) preview.ts:127-142 ensure() builds the WebContentsView with webPreferences {sandbox,contextIsolation,nodeIntegration:false,partition} and NO preload — so the no-ipcRenderer-on-untrusted-frame assumption is the sole protection. (4) preload/index.ts:108-109 exposes `api` only via contextBridge on the main window (index.ts:43 preload), confirming the main window is the only legitimate caller. (5) This specific concern is NOT among the prior 50 bugs: bug-hunt-findings.md #34 (main-window will-navigate guard) and #36 (preview scheme allowlist) are distinct and both already implemented in current code (index.ts:75-95, preview.ts:73-81/159-161/298/365). Verifying the sender of IPC messages is an explicit item on Electron's official security checklist, so this is a legitimate (if latent) hardening gap, accurately scoped by the candidate as "Low / not exploitable today."

## Suggested fix direction
Add an IPC-layer sender allowlist as defense-in-depth so the PTY/preview-isolation invariant is enforced rather than incidental. Pass a `getWin`-aware check into registerPtyHandlers/registerPreviewHandlers (both already receive a getWin closure) and, at the top of each handler, reject calls whose `event.senderFrame` is not the main window's main frame — e.g. `if (event.senderFrame !== getWin()?.webContents.mainFrame) throw new Error('forbidden')` (or return a benign no-op for the non-spawn handlers). A small shared `assertMainSender(event)` helper applied uniformly across all 16 handlers keeps it DRY. This is the documented Electron best practice ("verify the sender of all IPC messages") and closes the single-point-assumption that today only the absence of a preview preload prevents untrusted localhost content from reaching pty:spawn. Low priority for the current single-user, no-preview-preload state; best landed alongside any future change that touches preview webPreferences.
