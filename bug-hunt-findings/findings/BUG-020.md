# BUG-020: sanitizeDispatchText strips C0+DEL but passes C1 controls (U+0080–U+009F) — 8-bit CSI/OSC/DCS forms survive into the confirm modal and the PTY write

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/dispatchSanitize.ts`, `src/main/dispatchSanitize.test.ts`, `src/main/mcpOrchestrator.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The sanitizer's stated goal is that 'ESC and friends are a terminal-escape injection surface' and must be stripped, but the strip loop only removes code points <= 0x1F and 0x7F. The C1 range U+0080–U+009F — the single-byte forms of exactly those escapes (0x9B = CSI, 0x9D = OSC, 0x90 = DCS, 0x84/0x85 = IND/NEL) — passes through untouched, so a tainted relay/handoff payload can carry control characters that render invisibly in the human-confirm modal (what the human reads is not byte-identical to what is written) and reach the target PTY where a raw-mode TUI agent may interpret 8-bit C1 input as control sequences. The one-confirm-one-command invariant itself holds (U+0085 NEL is not an Enter for shells, and CR/LF are rejected), so impact is limited to display-spoofing/control-char smuggling rather than multi-command execution.

## Where
`src/main/dispatchSanitize.ts:39` — `sanitizeDispatchText` (the strip predicate `if (code <= 0x1f || code === 0x7f) continue` in the per-character loop)

## How it triggers
An MCP dispatch (handoff_prompt/assign_prompt/relay_prompt) whose prompt text contains C1 characters, e.g. relayed from a worker board's tainted output: a C1-form CSI/OSC sequence survives sanitizeDispatchText and is written to the target PTY after human confirm.

## Verification evidence
Confirmed by direct code reading + a mechanical demo. dispatchSanitize.ts:37-40: the strip loop is `if (code <= 0x1f || code === 0x7f) continue` — no branch covers 0x80-0x9f. Demo run of the identical loop logic over a string containing U+009B (CSI), U+009D (OSC), U+0090 (DCS), U+0084/U+0085: output length === input length (all C1 chars survive). dispatchSanitize.test.ts:23-30 exercises only C0 (\x00 \x07 \x1b \x7f) — no C1 coverage. The module's own doc-comment (dispatchSanitize.ts:14-15) declares the goal as stripping the 'terminal-escape injection surface'; the C1 range is the canonical 8-bit encoding of exactly those escapes, so this is an incomplete implementation of the declared invariant. The sanitized text is what the human confirms and what reaches writeToPty (mcpOrchestrator.ts:183-198 — sanitize runs BEFORE nonce/confirm precisely so the human sees what runs), and C1 chars render invisibly in the HTML confirm modal, so the read-vs-written byte identity claim is broken. Impact correctly bounded by the candidate: the one-confirm-one-command invariant holds (only \r\n are rejected line terminators; U+0085 NEL is not an Enter for shells/PTYs, and the JS string is UTF-8 encoded on write so C1 arrive as 2-byte sequences whose control interpretation is target-dependent) → Low.

## Suggested fix direction
Extend the strip predicate to `(code >= 0x80 && code <= 0x9f)` alongside C0+DEL, with a C1 test case.
