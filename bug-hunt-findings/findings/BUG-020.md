# BUG-020: fitZoomFactor's documented invariant (bounds.width / zoomFactor === presetW) is broken because the consumer rounds the native bounds width but derives the zoom factor from the un-rounded stage width

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`, `src/renderer/src/lib/cameraBounds.ts`, `src/renderer/src/lib/cameraBounds.test.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
cameraBounds.ts documents (lines 74-75) that setting zoomFactor = (nodeWorldW/presetW)*camZoom makes the native preview page lay out at exactly the preset CSS width W because bounds.width / zoomFactor === W. But the sole consumer, BrowserPreviewLayer, computes the native bounds with roundRect (boundsFor, line 138 → integer screen width) while deriving the zoom factor from the UN-rounded world stage width (zoomFor, line 147). Main then applies setBounds(round(bounds)) + setZoomFactor(zoomFactor), so the page's held CSS layout width = round(W_stage_world*camZoom)/((W_stage_world/presetW)*camZoom), which drifts from presetW by up to ~0.5*presetW/(W_stage_world*camZoom) px. Impact is purely cosmetic sub-pixel-to-~1-2px width error in the live (non-clamped) regime; it essentially never crosses a media-query breakpoint, hence Low.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:135-141 (boundsFor — rounds the screen rect) and 144-150 (zoomFor — feeds the un-rounded deviceStageRect width to fitZoomFactor); invariant asserted at src/renderer/src/lib/cameraBounds.ts:74-75,83-86; bounds+zoom applied at src/main/preview.ts:239-247 (attach) and 315-323 (setBoundsBatch)

## How it triggers
Any live Browser board whose native screen stage width (W_stage_world * camZoom) is non-integer — the common case after any camera zoom. boundsFor rounds that width to an integer for setBounds, but zoomFor computes zoomFactor from the un-rounded world stage width, so the two are mutually inconsistent and the responsive page lays out at slightly-off CSS width (presetW * round(S)/S) rather than exactly 390/834/1280. Concretely, a mobile-preset board ~371px world stage at camZoom 0.45 lays out at ~389.73 CSS px (drift -0.27px); a ~334px stage at camZoom 0.4 lays out at ~390.70 px (drift +0.70px).

## Verification evidence
Code-path trace on current code (cameraBounds.ts NOT in remediation commit 48c6500 — git show --stat confirms only preview.ts/BrowserPreviewLayer/previewPlan/previewStore were touched, not cameraBounds.ts, so this is a pre-existing property, not a broken remediation): boundsFor (BrowserPreviewLayer.tsx:138) = roundRect(worldRectToScreen(stage, vp, paneOffset)) → bounds.width = Math.round(W_stage_world*camZoom). zoomFor (line 147) = fitZoomFactor(deviceStageRect(...).width /* un-rounded world px */, presetW, camZoom) → (W_stage_world/presetW)*camZoom. Main (preview.ts:246 round(bounds) setBounds; 233/245 setZoomFactor) makes held CSS width = bounds.width/zoomFactor = presetW*round(S)/S, S=W_stage_world*camZoom. The module docstring (cameraBounds.ts:74-75) asserts bounds.width/zoomFactor===W; the existing unit test (cameraBounds.test.ts:163-170) only asserts it against the UN-rounded width (nodeWorldW*camZoom), so it never exercises the consumer's rounding and passes (verified: npx vitest run cameraBounds.test.ts → 24 passed). A throwaway node repro reproducing deviceStageRect/fitZoomFactor + the rounding showed drifts of -0.27px / +0.70px in realistic live (zoom>=LOD 0.4) non-clamped cases and confirmed the candidate's drift formula 0.5*presetW/(W_stage_world*camZoom); a parameter sweep capped the worst non-clamped rounding drift at ~1-2px. (Larger ~hundreds-of-px deviations seen at heavy zoom-out are the separate, documented setZoomFactor 0.25 clamp, NOT this rounding effect — the candidate correctly scopes itself to rounding only.)

## Suggested fix direction
Make the two paths consistent so the documented invariant holds: derive the zoom factor from the SAME width fed to setBounds. Either (a) compute the rounded screen bounds first and back out zoomFactor = roundedBoundsWidth / presetW (then bounds.width/zoomFactor === presetW exactly), or (b) accept that integer native bounds make exact presetW unattainable and soften the docstring/contract to "≈ presetW (±<1px from integer bounds rounding)" plus add a test asserting the consumer-path drift stays sub-pixel. Note the geometric ≤0.5px native-pixel error from integer setBounds is intrinsic and unavoidable; option (a) only re-attributes it so the responsive layout width snaps to an exact presetW. Given the sub-pixel impact and zero risk of crossing real breakpoints, lowest-effort acceptable fix is (b) documentation + a guard test.
