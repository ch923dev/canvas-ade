# BUG-028: strokePaths useMemo never hits: perfect-freehand outlines recompute for all committed strokes on every PlanningBoard re-render (drag / zoom frame)

- **Severity:** Low
- **Category:** performance
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/planning/WhiteboardSvg.tsx`, `src/renderer/src/canvas/boards/PlanningBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
WhiteboardSvg memoizes the heavy perfect-freehand outline math on the `strokes` array identity, but PlanningBoard derives `strokes` via `viewElements.filter(...)`, which returns a brand-new array reference on every render. With Object.is dependency comparison the memo dependency changes every render, so the memo never hits and getStroke (a two-pass O(n) outline computation per stroke) re-runs for every committed stroke on each re-render — including every pointer-move frame of an unrelated element drag and every frame of a continuous zoom gesture. Pure wasted CPU during gestures; output is identical, no data loss or crash.

## Where
- src/renderer/src/canvas/boards/planning/WhiteboardSvg.tsx:54 — strokePaths useMemo; src/renderer/src/canvas/boards/PlanningBoard.tsx:369 — strokes = viewElements.filter(...); driven by PlanningBoard.tsx:270 (setDragPos per move frame) and PlanningBoard.tsx:85 (raw-zoom subscription re-rendering per zoom frame). Heavy work: src/renderer/src/canvas/boards/planning/svgPaths.ts:46 strokeToPath -> getStroke (src/vendor/perfect-freehand/getStrokePoints.ts + getStrokeOutlinePoints.ts).

## How it triggers
On a Planning board with several committed pen strokes: (1) Drag a note/checklist/arrow with the select tool — onWellPointerMove fires setDragPos every pointer-move frame (PlanningBoard.tsx:270), PlanningBoard re-renders, viewElements = translateElement(...) (new array via .map), then strokes = viewElements.filter(...) (new array via .filter). The new `strokes` ref invalidates the useMemo dep, so strokeToPath/getStroke runs for ALL strokes every frame even though no stroke changed. (2) A continuous Ctrl+wheel/pinch zoom gesture: PlanningBoard subscribes to the raw zoom scalar (PlanningBoard.tsx:85) and re-renders every zoom frame, hitting the same recompute. Even a plain re-render misses the memo (elements.filter still yields a new ref), but that is cheap when re-renders are infrequent.

## Verification evidence
Code trace confirmed against current (post-48c6500) source. WhiteboardSvg.tsx:54: `const strokePaths = useMemo(() => strokes.map((s) => strokeToPath(s.points)), [strokes])`. PlanningBoard.tsx:368-369 derives arrows/strokes via `viewElements.filter(...)`; Array.prototype.filter always returns a NEW array object, so the [strokes] dependency changes identity every render (React useMemo uses Object.is). PlanningBoard.tsx:270 `setDragPos({...})` runs in onWellPointerMove every pointer-move frame -> PlanningBoard self-re-render (internal state, unaffected by any parent memo). PlanningBoard.tsx:85 `const zoom = useStore((s) => s.transform[2])` subscribes to the continuous zoom scalar, so PlanningBoard re-renders every frame of a zoom gesture (BoardNode's #45 fix to subscribe to the LOD boolean does NOT insulate this — the offending subscription is PlanningBoard's own). strokeToPath (svgPaths.ts:46) calls getStroke = getStrokePoints (O(n), lrp/dist per point) + getStrokeOutlinePoints (O(n), rotAround/vector math + array allocations per point); pushBoardPoint (lib/pen.ts:58) appends a point every draw frame with NO cap, so n is dozens-to-hundreds per stroke. The WhiteboardSvg.tsx:53 comment ("Memoize the (potentially heavy) outline math") states the exact intent the broken dep defeats. Not a duplicate of the 50: the only prior WhiteboardSvg finding (#4) was hit-test fall-through; #45 was BoardNode VDOM re-render (different component, different defect) and was remediated by subscribing to isLod(); this inner-memo defeat is new.

## Suggested fix direction
Make the memo key on something referentially stable instead of the filtered-array identity. Options: (a) memoize per-stroke by id+points (e.g. a useMemo/Map keyed on each stroke's id and points reference, since translateElement returns the SAME element object for unchanged strokes — only the moved one gets a new ref), so unchanged strokes reuse their cached path; (b) lift the arrows/strokes filtering out of the per-render path and memoize `strokes` in PlanningBoard on `elements` (and only re-derive the single dragged element), so the array identity is stable when nothing changed; (c) compute strokePaths keyed on a content signature rather than array identity. Approach (a) gives the most benefit since it survives the unavoidable new-array-per-drag-frame while still skipping getStroke for every stroke that did not move.
