# BUG-028: preview:screenshot reports 'copied to clipboard' even when the clipboard write threw and nothing was saved anywhere

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/previewScreenshot.ts`, `src/renderer/src/canvas/boards/BrowserBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The handler swallows `deps.writeImage(png)` failures (catch comment: "non-fatal, still try to save"), but the result type has no way to convey "clipboard failed". If no project is open the handler returns `{ ok: true, assetId: null }` — and the asset-save catch path does the same with a comment asserting "the clipboard copy already succeeded", which is false when writeImage threw. The renderer maps `{ok:true, assetId:null}` to the toast "Screenshot copied to clipboard", so on a clipboard failure with no project open (or a failed asset save) the user is told the screenshot was copied when nothing happened anywhere — a silent total failure presented as success.

## Where
`src/main/previewScreenshot.ts:47` — `registerPreviewScreenshotHandler()`, inside the `preview:screenshot` ipc.handle callback (the swallowed `deps.writeImage(png)` at lines 46-50)

## How it triggers
Clipboard unavailable (Wayland without the portal, locked session, clipboard contention) + no project open (or project asset save also fails) → press the Screenshot button on a live Browser board → toast says 'Screenshot copied to clipboard' but clipboard and disk both have nothing.

## Verification evidence
Straight-line code path confirmed: deps.writeImage(png) failure is swallowed with no flag (src/main/previewScreenshot.ts:46-50, comment 'non-fatal, still try to save'); the ScreenshotResult union (lines 24-26) has no way to convey clipboard failure; with no project open `deps.currentDir()` → null → `return { ok: true, assetId: null }` (lines 51-52); the saveAsset catch (lines 56-61) also returns { ok: true, assetId: null } with a comment asserting 'the clipboard copy already succeeded' — provably false when writeImage threw. The renderer maps {ok:true, assetId:null} to the toast 'Screenshot copied to clipboard' (src/renderer/src/canvas/boards/BrowserBoard.tsx:180-186, line 185). So under a clipboard failure (headless/Wayland/locked session — the code's own catch comment names these) combined with no-project-open or a failed asset save, a total silent failure is presented as success. Low: UX-level wrong feedback, no data corruption.

## Suggested fix direction
Track a clipboardOk flag through the handler and extend the result type / toast wording to reflect partial or total failure.
