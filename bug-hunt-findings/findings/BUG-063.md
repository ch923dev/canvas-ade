# BUG-063: digestTerminal emits only the first linked preview consumer (find, not filter) — multi-viewport compare canvases are under-reported

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/lib/digest.ts`, `src/renderer/src/lib/viewportCycle.ts`, `src/renderer/src/lib/tidyLayout.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
digestTerminal uses `d.boards.find(...)` to locate a browser whose previewSourceId === b.id and emits a single `Feeds preview "X"` line. Multiple browsers previewing one terminal is the app's flagship flow (Duplicate → nextViewport cycles mobile/tablet/desktop for side-by-side compare; tidyLayout smart mode is built around a terminal feeding SEVERAL browsers). For that canvas the terminal's digest names only the first browser in doc order and omits the rest, so the reopen panel / Tier-2 LLM context under-describes the link graph.

## Where
`src/renderer/src/lib/digest.ts:44` — `digestTerminal()` (the consumer lookup feeding the `Feeds preview` line)

## How it triggers
One terminal with 2+ linked browser boards (e.g. created via Duplicate-to-compare). buildDigest's terminal entry lists exactly one 'Feeds preview' line regardless of consumer count.

## Verification evidence
digest.ts:44-47: `const consumer = d.boards.find((o) => o.type === 'browser' && o.previewSourceId === b.id); if (consumer) lines.push(`Feeds preview "${consumer.title}"`)` — Array.find returns the first doc-order match, so a terminal with N>=2 linked browsers gets exactly one 'Feeds preview' line naming an arbitrary (doc-order-first) consumer. Multiple consumers per terminal is a first-class state in this codebase: canvasStore.ts:520-522 explicitly documents that a duplicated linked Browser KEEPS its previewSourceId (the duplicate-to-compare flow, with viewportCycle.ts cycling mobile/tablet/desktop), and tidyLayout.ts smartLayout (lines 133-143) is built around collecting ARRAYS of browsers per source terminal. The asymmetry is also internal to digest.ts itself: digestBrowser (lines 58-61) reports its source link, but the terminal side drops all but one reverse link. Coverage gap confirmed: digest.test.ts constructs at most one consumer per terminal (lines 90-106), so nothing pins the current single-line behavior as intended. Wrong-but-cosmetic info in the reopen panel / Tier-2 LLM context → Low.

## Suggested fix direction
Use filter and emit one 'Feeds preview' line per consumer (or one line naming all).
