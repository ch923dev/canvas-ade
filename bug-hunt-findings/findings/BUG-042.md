# BUG-042: createProject on a folder where BOTH canvas.json and .bak are unparseable silently overwrites the corrupt primary's bytes with a fresh empty doc (no rename-aside, no prompt)

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/projectStore.ts`, `src/main/projectIpc.ts`, `src/renderer/src/canvas/WelcomeScreen.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
createProject does reuse-if-exists via readProject; when BOTH canvas.json and canvas.json.bak fail to parse (exactly the state a user is in after 'Could not open project'), it falls through and writes a fresh empty doc via writeProject. writeProject's rotation is skipped because tryParse(primary) returns undefined for the corrupt file, so writeFileAtomic OVERWRITES the corrupt canvas.json with `{schemaVersion:8, viewport:null, boards:[], connectors:[]}` — irreversibly destroying bytes that are frequently hand-recoverable (truncated/garbled JSON). Secondary effect: on the next open, gcAssets quarantines every blob in assets/ into assets/.trash (recoverable, but alarming). The natural user flow hits this: open fails with an error, user retries via 'Create project…' on the same folder, and the broken-but-fixable project file is wiped without any confirmation.

## Where
`src/main/projectStore.ts:124` — `createProject` (the reuse check at line 125 falling through to `writeProject(dir, fresh)` at line 137)

## How it triggers
canvas.json AND canvas.json.bak both unparseable (e.g. truncation by a sync tool/crash predating the atomic writer, or hand-editing); user clicks 'Create project…' (WelcomeScreen.onCreate or ProjectSwitcher.createNew) and picks that same folder.

## Verification evidence
src/main/projectStore.ts:118-140 — createProject's reuse check is `readProject(dir)` (line 125), which returns ok:false when both canvas.json and canvas.json.bak fail tryParse; it then falls through to `writeProject(dir, fresh)` (line 137). Inside writeProject, `tryParse(primary) !== undefined` (line 107) is false for the unparseable primary, so rotation is skipped and writeFileAtomic (line 114) rename-replaces canvas.json with `{schemaVersion:8, viewport:null, boards:[], connectors:[]}` — the corrupt primary's bytes (often hand-recoverable truncated JSON) are irreversibly destroyed. No renderer confirmation exists: WelcomeScreen.onCreate / ProjectSwitcher.createNew go dialog → create with no prompt, and the natural retry-after-open-failure flow hits this. The PERSIST-C comment (lines 130-133) shows the corrupt-existing-file case was assumed away ('There is no prior file here'). Two corrections to the claim, driving the severity down to Low: (1) the .bak is NOT touched (rotation skipped), so its bytes survive — 'no copy anywhere' is overstated, only the primary's bytes are lost; (2) the trigger requires BOTH files unparseable AND the user explicitly choosing Create on that same folder; and gcAssets quarantines to assets/.trash (recoverable, line 213-235), not deletion.

## Suggested fix direction
When readProject fails but a canvas.json exists on disk, rename it aside (canvas.json.corrupt-<ts>) before the fresh write — or surface a confirm to the renderer.
