# BUG-022: Note/checklist placement is blocked over committed arrows/strokes (vector hit-test keyed on `drawing`, not any non-select tool)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/planning/WhiteboardSvg.tsx`, `src/renderer/src/canvas/boards/PlanningBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The #4 remediation disabled committed-vector hit-testing only while a draw tool is active (drawing = tool === 'arrow' || 'pen'), not for the note/check place tools. With note/check active a press directly on a committed arrow line or stroke fill is intercepted by the SVG path's onPointerDown (stopPropagation + select + arm-drag), so the well's onWellPointerDown — the only place note/checklist creation happens — never fires and no element is placed where the user clicked. The DOM cards were fixed to fall through for ALL non-select tools, so placement over a card works; placement over a vector does not — an incomplete realization of the #4 fix whose own note even flagged this case.

## Where
- src/renderer/src/canvas/boards/planning/WhiteboardSvg.tsx:95 (arrow pointerEvents: drawing ? 'none' : 'stroke') and :119 (stroke pointerEvents: drawing ? 'none' : 'auto'); vector onPointerDown at :96-100 and :120-124 (stopPropagation + onSelect + onDragStart). src/renderer/src/canvas/boards/PlanningBoard.tsx:373 (drawing = tool === 'arrow' || tool === 'pen', excludes note/check); :225-239 (note/check creation lives in onWellPointerDown, preempted by the vector's stopPropagation).

## How it triggers
On a selected Planning board: draw an arrow or a pen stroke. Pick the note (or check) tool and press directly on top of the existing arrow line (~1.5px hit target, rarer) or anywhere on the committed stroke's filled outline (whole region is a hit target, reliable). The vector receives the pointerdown because drawing is false in note/check mode (pointerEvents stays 'stroke'/'auto'); its handler stops propagation, selects the vector and arms a board-local move via wellRef.setPointerCapture. The well's onWellPointerDown never runs, so no note/checklist is created at the click point (and the vector may be nudged if the pointer moves before release).

## Verification evidence
Code-path trace against current post-48c6500 source. PlanningBoard.tsx:373 sets drawing = tool === 'arrow' || tool === 'pen', so for tool === 'note'|'check' drawing === false. WhiteboardSvg.tsx:95/119 therefore keep pointerEvents 'stroke'/'auto' on committed arrows/strokes. WhiteboardSvg.tsx:96-100 / :120-124 — the path onPointerDown calls e.stopPropagation() then onSelect(id) + onDragStart(e,id). onWellPointerDown (PlanningBoard.tsx:215-260) is a bubble-phase handler on the ancestor well (onPointerDown={onWellPointerDown}, :390), and note/check creation happens only there (:225-239); a descendant's React synthetic stopPropagation prevents that ancestor handler from firing — the exact mechanism the prior #4 entry relied on (bug-hunt-findings.md:102). onDragStart=startElementDrag (PlanningBoard.tsx:196-210) arms mode:'move' and captures the pointer on the well. Asymmetry confirmed: NoteCard.tsx:89-95/120-132/150-152 and ChecklistCard.tsx:138-140/184-186/240-242/267-269 all gate stopPropagation on `interactive` (select only), so in note/check mode card presses fall through to the well and placement over a card works — but vectors do not. The #4 fix note itself (bug-hunt-findings.md:108) claims "note/check tools place-and-return immediately, so they don't need the guard" — that claim is wrong because the place happens inside onWellPointerDown, which the vector's stopPropagation preempts. No planning component test exercises this routing (only svgPaths.test.ts / elements.test.ts, pure-logic).

## Suggested fix direction
Gate the vector hit-test disable on "any non-select tool" rather than just the draw tools, matching the card guards. Either pass a non-select flag (e.g. pass `drawing={tool !== 'select'}` from PlanningBoard.tsx:426, or add a separate `placing` prop and OR it in) so WhiteboardSvg lines 95/119 set pointerEvents:'none' whenever note/check/arrow/pen is active. Then a press over committed ink falls through to onWellPointerDown and the note/checklist is placed (or the new draft begins) as intended; selection/drag of vectors remains intact because it is only active in select mode.
