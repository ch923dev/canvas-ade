# BUG-022: Natural-exit cleanup unconditionally tree-kills the already-exited PID — narrow PID-reuse race can force-kill an unrelated same-user process tree

- **Severity:** Low
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
When a PTY process exits on its own, `proc.onExit` (pty.ts:395) calls `cleanup(opts.id, proc)` → `cleanupCore` → `deps.killTree(s.proc)` unconditionally; killTree then runs `taskkill /PID <pid> /T /F` (Windows) or `process.kill(-pid, 'SIGKILL')` (POSIX) against a PID that has ALREADY exited and been reaped. If the OS recycles that PID before taskkill resolves it (node-pty's process handle is released by the synchronous `proc.kill()` right after queuing the execFile), the force-kill lands on an innocent same-user process tree — e.g. another terminal board's agent or a build process. The window is tens of ms and requires exact PID reuse, hence Low.

## Where
`src/main/pty.ts:481` — `cleanupCore` (the unconditional `const done = deps.killTree(s.proc)` after the identity check; reached from `proc.onExit` via `cleanup`)

## How it triggers
Any natural shell/agent exit (user types `exit`, claude session ends) on a machine with heavy process churn (e.g. a parallel build spawning hundreds of short-lived processes — exactly the agentic-CLI workload), such that the exited PID is recycled within the ~30-100ms window before taskkill resolves it.

## Verification evidence
Code path confirmed unconditional: pty.ts:372-395 — proc.onExit marks the session 'exited' (line 385) then calls `cleanup(opts.id, proc)` (line 395) → cleanupCore (pty.ts:471-488) which, after the identity check passes (same proc), runs `deps.killTree(s.proc)` at line 481 with NO already-exited guard (despite `s.state === 'exited'` being set one frame earlier — the information to skip exists). killTree (pty.ts:536-558) issues `execFile('taskkill', ['/PID', pid, '/T', '/F'])` then synchronously `proc.kill()` (which releases node-pty's process handle, making the PID recyclable on Windows before the async taskkill resolves it) — so a force-kill is issued against a PID known to be dead, and if the OS recycles it inside that window the /F lands on an innocent process tree (classic CWE-362 pid-reuse TOCTOU); POSIX `process.kill(-pid)` (line 561) has the analogous pgid hazard. Two corrections to the candidate's reasoning: (a) Windows does NOT re-parent orphans — ToolHelp snapshots keep the stale ParentProcessId, so `taskkill /T` on a dead root CAN still find surviving children, meaning the call is intentional reaping (kill-the-tree policy), not 'pure downside'; (b) severity: the harm window is tens of ms and requires the exact PID to be re-issued to a new process within it — a rare probabilistic race, not a 'realistic edge' users will hit, so Medium is too high → Low. The prior pid-reuse fix was indeed test-side only (docs/testing/TESTING.md e2e hardening); no production guard exist

## Suggested fix direction
On the onExit path, skip the root-PID kill (or kill children-only) when the session state is 'exited'; any guard must preserve reaping of surviving children of the dead shell.
