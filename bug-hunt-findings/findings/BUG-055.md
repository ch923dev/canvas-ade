# BUG-055: Ctrl + zero-deltaY wheel events (tilt-wheel / horizontal trackpad pan) shrink the terminal font

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED

- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The Ctrl-wheel font-zoom handler maps deltaY to a step with `e.deltaY < 0 ? 1 : -1`, so any wheel event with ctrlKey and deltaY === 0 — a tilt-wheel horizontal scroll or a two-finger horizontal trackpad pan while Ctrl is held — is treated as zoom-out and decrements the font by 1px per event. A horizontal pan emits many events, so the font slams toward MIN_TERMINAL_FONT (8) and each step also runs the undo-checkpoint/persist path (updateBoard) via setFont.

## Where
`src/renderer/src/canvas/boards/TerminalBoard.tsx:653` — `onWheel` (the native ctrl-wheel font-zoom listener effect inside `TerminalBoard`)

## How it triggers
Hold Ctrl (e.g. intending to zoom) and scroll horizontally over a terminal well with a tilt wheel or trackpad: deltaY=0, deltaX≠0 events fire repeatedly; each one calls fontStepRef.current(-1).

## Verification evidence
Confirmed unconditional code path at TerminalBoard.tsx:646-657: the native wheel listener checks only `if (!e.ctrlKey) return`, then `e.preventDefault(); e.stopPropagation(); fontStepRef.current(e.deltaY < 0 ? 1 : -1)`. A wheel event with ctrlKey and deltaY === 0 (tilt-wheel horizontal scroll, two-finger horizontal trackpad pan with Ctrl held — these emit deltaX≠0, deltaY=0) falls into the -1 branch on every event, repeatedly calling nudgeFont(-1) → setFont (566-584), which steps the font toward MIN_TERMINAL_FONT and runs the beginChange/updateBoard persist path per burst. No e2e/unit coverage exercises a deltaY=0 ctrl-wheel. Direction-wrong behavior on real but uncommon input → Low.

## Suggested fix direction
Early-return when e.deltaY === 0 in the ctrl-wheel handler.
