# BUG-019: doUndo/doRedo clear focus unconditionally, dropping the focus dim on a no-op undo/redo (empty history)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Canvas.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
doUndo and doRedo call setFocusedId(null) unconditionally, decoupled from whether the store actually mutated. When the history stack for that direction is empty, undo()/redo() are true no-ops (return state unchanged) yet the focus dim is still cleared, so pressing Ctrl+Y / Ctrl+Shift+Z (or Ctrl+Z on an empty undo stack) with nothing to do silently exits focus mode for no reason. Cosmetic, transient, recoverable (re-double-click to refocus); it is a regression introduced by the #33/#44 focus-clear remediation.

## Where
- src/renderer/src/canvas/Canvas.tsx:174-181 — doUndo/doRedo useCallbacks; depends on canvasStore.ts:165-174 (undo/redo no-op return s) and history.ts:19,33 (applyUndo/applyRedo return null on empty stack)

## How it triggers
Most reliable (fully runtime-reachable) path is REDO, since `future` is empty by default: 1) Click a dock button to add a board (addBoard records past; future stays []). 2) Double-click that board to focus it (focusBoard sets focusedId=id and selects it; other boards dim to opacity 0.55 via the nodes memo `dimmed: focusedId!==null && focusedId!==b.id`). 3) Press Ctrl+Y or Ctrl+Shift+Z. The keydown handler (Canvas.tsx:202-212) calls doRedo(); redo() finds future empty so applyRedo returns null and the store returns s unchanged (true no-op), but doRedo still runs setFocusedId(null) (line 180). All boards return to full opacity — focus mode visually exits even though no redo happened. The symmetric UNDO case (empty past + a focusable board) is currently only reachable via loadObject, which is test-only until Phase 3 file I/O is wired, because addBoard always leaves at least one past entry while any board exists.

## Verification evidence
Code-path trace against current (post-48c6500) source: history.ts:19 `if (past.length === 0) return null` and :33 `if (future.length === 0) return null`. canvasStore.ts:165-169 undo() — `const r = applyUndo(...); return r ? {...} : s` (returns s unchanged on null); :170-174 redo() identical for redo. Canvas.tsx:174-177 `doUndo = useCallback(() => { undo(); setFocusedId(null) }, [undo])` and :178-181 doRedo likewise — setFocusedId(null) is unconditional, not gated on whether the store changed. Focus drives dimming: Canvas.tsx:86 `const [focusedId, setFocusedId] = useState(null)`; :98 `data: { board: b, dimmed: focusedId !== null && focusedId !== b.id }`. focusBoard (Canvas.tsx:149-156) sets focusedId via double-click, wired at :243 `onNodeDoubleClick={focusBoard}`. The heal effect (Canvas.tsx:188-191) only RE-clears focus when the board is gone; it never re-establishes focus, so once cleared it stays cleared until the user refocuses. Existing tests corroborate the no-op semantics: history.test.ts asserts `applyUndo([], 'present', [])` and `applyRedo([], 'present', [])` both return null. This is distinct from prior findings #33/#44 (which were the OPPOSITE defect — undo/redo LEAVING focus set); the remediation that added the unconditional setFocusedId(null) is exactly what overshoots here, per finding #33's own proposed fix snippet (`const doUndo = useCallback(() => { setFocusedId(null); undo() }, [undo])`).

## Suggested fix direction
Gate the focus clear on whether the store actually changed. E.g. have undo()/redo() return a boolean (or read the boards ref before/after) and only call setFocusedId(null) when a real undo/redo occurred: `const before = useCanvasStore.getState().boards; undo(); if (useCanvasStore.getState().boards !== before) setFocusedId(null)`. Or expose canRedo/canUndo and short-circuit the keydown handler before invoking doUndo/doRedo. The clear must stay coupled to an actual state transition so a no-op keypress no longer exits focus mode. Keep focus-state ownership in Canvas.tsx.
