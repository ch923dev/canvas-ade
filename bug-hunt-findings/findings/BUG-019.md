# BUG-019: flush reply wired with ipcMain.once — one ignored foreign/spurious message permanently consumes the frame-guarded listener, degrading the BUG-038 guard to the 1500ms timeout

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`, `src/main/flushChannel.ts`, `src/main/index.flushRenderer.test.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
flushRenderer wires the BUG-038 frame-guarded `finish` via `ipcMain.once(replyChannel, finish)`, but `once` removes the listener on FIRST delivery regardless of finish's outcome. If a foreign-frame message arrives first, isForeignSender correctly ignores it but the listener is already gone, so the legitimate renderer reply (sent after the awaited project:save) is never received and the flush resolves only via the 1500ms forceFinish timeout — performGuardedQuit then proceeds to shutdown/app.exit while the save may still be in flight, the exact last-second-edit loss BUG-M2/BUG-038 were fixed to prevent. The helper's own test encodes keep-waiting semantics that the `once` wiring cannot deliver.

## Where
`src/main/index.ts:503` — `flushRenderer` (the `ipcMain.once(replyChannel, finish)` registration; `finish` built by `makeFlushFinish`, flushChannel.ts)

## How it triggers
Any message on the reply channel from a non-main-frame sender (or any sender main fails the frame check for, e.g. a sender whose frame was torn down mid-send) arriving before the autosave flush reply during quit.

## Verification evidence
Mechanical defect confirmed by code reading. index.ts:503 registers `ipcMain.once(replyChannel, finish)`. Node's EventEmitter.once removes the wrapped listener on FIRST emission, before/regardless of the inner function's outcome. makeFlushFinish's `finish` (flushChannel.ts:71-76) handles a foreign sender by early-returning WITHOUT resolving — semantics that only make sense if the listener stays armed to receive the later legitimate reply. With `once`, after one foreign-frame (or destroyed-window-denied, ipcGuard.ts:23) message on the channel, the legitimate renderer reply (sent after its awaited project:save) is delivered to nobody; the flush resolves only via the 1500ms forceFinish timer (index.ts:502), and performGuardedQuit proceeds to shutdown/app.exit while the save may still be in flight. The helper's own test ('🔒 ignores a foreign-frame reply', index.flushRenderer.test.ts:88-105) encodes keep-waiting semantics the once-wiring cannot deliver, and onCleanup already calls removeAllListeners(replyChannel) (index.ts:496) so `ipcMain.on` is the natural correct wiring. Severity stays Low: the channel name is a 122-bit randomUUID delivered only to the main window's webContents, so a foreign sender reaching it requires the same compromise the guard is defense-in-depth against; no normal-use path emits a spurious message on that channel.

## Suggested fix direction
Use ipcMain.on(replyChannel, finish); the existing onCleanup removeAllListeners already handles removal on resolve.
