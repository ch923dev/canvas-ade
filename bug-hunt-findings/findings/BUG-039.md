# BUG-039: runSummarize breaks its documented never-throws contract when the budget persist throws — tryConsume runs outside the try/catch and the llm:summarize IPC promise rejects raw

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/llmService.ts`, `src/main/llmBudget.ts`, `src/main/llmIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
runSummarize's contract (doc comment, lines 207-211) is 'NEVER throws — callers fall back to Tier 1; the app never blocks on the brain'. But `deps.budget.tryConsume(cap)` is called BEFORE the try block, and tryConsume → write() → mkdirSync + writeFileAtomic.sync can throw (EPERM from a Windows AV/indexer lock on llm-budget.json, ENOSPC, read-only userData). The exception escapes runSummarize, rejects the `llm:summarize` ipcMain.handle promise, and surfaces to the renderer as a raw 'Error invoking remote method' rejection instead of the typed SummarizeResult union every caller is written against. summaryLoop happens to survive (its outer catch at summaryLoop.ts:562), but the IPC path violates the typed-result contract.

## Where
`src/main/llmService.ts:222` — `runSummarize` (the `deps.budget.tryConsume(cap)` reservation at lines 219-222, before the `try` at line 224)

## How it triggers
Any throw from the synchronous budget write while a renderer-initiated llm:summarize is in flight — e.g. antivirus briefly locking userData/llm-budget.json during writeFileAtomic's rename (a known-real Windows failure mode), or a full disk.

## Verification evidence
Direct path verified. llmService.ts:207-211 doc comment: 'NEVER throws — callers fall back to Tier 1'. But the budget reservation at lines 219-222 (`deps.budget.tryConsume(cap)`) executes BEFORE the try block at line 224. tryConsume → write() (llmBudget.ts:87-90): `mkdirSync` + `writeFileAtomic.sync` with NO try/catch — EPERM (AV/indexer lock on llm-budget.json during the atomic rename, a real Windows mode), ENOSPC, or read-only userData all throw and propagate out of runSummarize. The asymmetry the candidate cites is real: read() in the same file (llmBudget.ts:61-84) deliberately try/catches its fs ops, so fs failure here is an anticipated condition only the write side skips. Call site verified: llmIpc.ts:110-111 `return runSummarize(config, input, deps)` inside the async ipcMain.handle with no catch → the rejection surfaces to the renderer as a raw 'Error invoking remote method' instead of the typed SummarizeResult union (llmService.ts:201-205) every caller is written against. Low: needs a transient fs failure coinciding with a renderer-initiated summarize; summaryLoop's own path survives via its outer catch.

## Suggested fix direction
Move tryConsume inside the try (mapping a throw to provider-error), or try/catch the budget write itself.
