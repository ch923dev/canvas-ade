# BUG-021: injectCspMeta silently returns input on regex no-match and is never tested against the real index.html — a meta-tag reshape would silently ship the dev CSP (script-src 'unsafe-inline') in packaged builds

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/csp.ts`, `src/main/csp.test.ts`, `src/renderer/index.html`, `electron.vite.config.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
injectCspMeta replaces the CSP via the regex `/(<meta\s+http-equiv="Content-Security-Policy"\s+content=")[^"]*(")/` and returns the input unchanged on no match — no error, no log. The regex requires exactly this attribute order, double quotes, and no intervening attributes, while src/renderer/index.html's fallback meta content is the DEV policy including script-src 'unsafe-inline'. csp.test.ts tests the function only against a synthetic placeholder string and never reads the real index.html, so a routine edit (attribute reorder, single quotes, an added attribute) would make the build-time injection silently no-op and the packaged app would ship with the dev CSP — re-opening the inline-script XSS vector PROD_CSP exists to close — with every gate green.

## Where
`src/main/csp.ts:32` — `injectCspMeta` (the `html.replace(...)` at csp.ts:34-37 with no no-match detection)

## How it triggers
Any future edit to src/renderer/index.html's CSP meta tag that changes attribute order/quoting (e.g. prettier/plugin reformat, adding a nonce attribute), followed by a packaged build.

## Verification evidence
All factual claims verified; confirmed strictly as the LATENT silent-failure defect the candidate itself describes (no current misbehavior). (1) csp.ts:34-37 — `html.replace(regex, ...)` with no no-match detection: on no match it returns the input unchanged, no throw/log. (2) Demo run of the exact regex: it matches today's real src/renderer/index.html (the multi-line meta at lines 14-17 — \s+ spans the newlines), but returns FALSE (→ silent no-op) for an attribute-reordered meta (`content` before `http-equiv`), an extra-attribute meta, and a single-quoted meta. (3) index.html:16 — the fallback content that would then ship IS the dev policy including `script-src 'self' 'unsafe-inline'`, and the comment at lines 6-13 even names the scenario ('used if the plugin ever no-ops'). (4) csp.test.ts:26-31 tests injectCspMeta only against a synthetic one-line placeholder string; nothing reads the real index.html, and the vite plugin (electron.vite.config.ts:42-50) does not assert the transform changed the html — so the regression would be invisible to every gate. Severity Low and conditional: the named triggers are weaker than claimed (prettier does not reorder or re-quote HTML attributes), so this needs a deliberate human/agent edit to the meta tag; today's build is correct.

## Suggested fix direction
Make injectCspMeta throw (or the plugin assert output !== input in prod builds), and/or add a test that runs it against the real src/renderer/index.html.
