# BUG-021: FreeText drag-gutter swallows pen/arrow draw-through (stopPropagation runs before the !interactive guard)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/planning/FreeText.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The FreeText left-edge drag grip's onPointerDown calls e.stopPropagation() unconditionally at the top, then guards with `if (!interactive) return`. In a draw mode (pen/arrow), the stop runs first, so the press on the 6px grip band is consumed and never bubbles to the well's onWellPointerDown — no draft stroke/arrow begins. This is the lone Planning element surface that inverts the documented #6 draw-through pattern (NoteCard grip and ChecklistCard header both guard BEFORE stopping); the 48c6500 remediation fixed FreeText's root and textarea handlers but missed the grip.

## Where
- src/renderer/src/canvas/boards/planning/FreeText.tsx:104-106 — the pl-text-grip span onPointerDown handler

## How it triggers
Select a Planning board (tool cluster appears), pick the pen or arrow tool, then start a gesture with the pointer-down landing on the 6px-wide, full-height drag gutter on the left edge of an existing free-text element. The grip's onPointerDown runs e.stopPropagation() (line 105) before returning at the !interactive guard (line 106), so the well's onWellPointerDown (PlanningBoard.tsx:215, bubble-phase handler at line 390) never fires and no draft begins — the gesture is silently lost. Retrying a few px to the right (over empty well or the textarea, which correctly falls through) works.

## Verification evidence
Code-path trace on current committed code (git status shows FreeText.tsx clean). FreeText.tsx line 105 is `e.stopPropagation()` and line 106 is `if (!interactive) return` inside the grip span (lines 101-121); the grip is a 6px-wide alignSelf:stretch hit target with no pointerEvents override (lines 114-120). The grip is a descendant of the well (PlanningBoard.tsx renders FreeText at line 451 inside the pl-well div, lines 386-417). The well's onWellPointerDown is the ONLY place a pen/arrow draft starts (PlanningBoard.tsx:240-255) and runs in the bubble phase (onPointerDown={onWellPointerDown}, line 390); in draw mode it does NOT early-return at line 221 (`if (tool === 'select' && ...)` — tool is not 'select'), so it would start the draft if reached. A descendant React synthetic stopPropagation prevents that. Asymmetry confirmed against siblings: NoteCard.tsx grip lines 120-124 do `if (!interactive) return` THEN `e.stopPropagation()` (guard first); ChecklistCard.tsx header lines 169-174 gate with `if (interactive && e.target === e.currentTarget)` (guard first); FreeText inverts the order. git log -p on 48c6500 shows the remediation changed FreeText's root to `if (interactive) e.stopPropagation()` and the textarea similarly, but for the grip it only converted `if (interactive) onDragStart(...)` into `if (!interactive) return` while LEAVING `e.stopPropagation()` unconditional at the top — confirming an incomplete fix of finding #4/#6, whose own writeup (bug-hunt-findings.md:104) cited the cards as the correct guarded pattern.

## Suggested fix direction
Reorder the grip's onPointerDown to guard before stopping, mirroring NoteCard.tsx:123-124: move `if (!interactive) return` above `e.stopPropagation()` so the stop only runs in select mode. In a draw mode the press then bubbles to onWellPointerDown and the pen/arrow draft starts over the free-text grip, matching the other Planning element surfaces. No prop changes needed; one-line reorder.
