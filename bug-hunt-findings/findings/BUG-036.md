# BUG-036: llm:setConfig persists maxCallsPerDay with zero runtime validation (and leaves baseUrl length unbounded) — escapes the BUG-040 hardening in the same handler

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/llmIpc.ts`, `src/main/llmConfig.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The llm:setConfig handler validates provider, model (length-capped via MAX_MODEL_LEN) and baseUrl (loopback), but `a.maxCallsPerDay` is written to llm-config.json verbatim with zero runtime checks. Any non-number truthy value (string, object, array) is persisted; readLlmConfig later repairs it to undefined, so a previously configured cap is silently destroyed and the budget reverts to the 200 default. A multi-megabyte string passes `a.maxCallsPerDay ?? existing.maxCallsPerDay` and is JSON.stringify'd + writeFileAtomic.sync'd in MAIN — the exact synchronous event-loop-stall DoS class BUG-012 (key) and BUG-040 (model) were fixed for. The same handler also leaves `baseUrl` LENGTH unbounded: `http://127.0.0.1/` + megabytes of path/query passes isLoopbackBaseUrl (hostname check only) and is persisted synchronously and echoed back on every llm:status.

## Where
`src/main/llmIpc.ts:195` — `registerLlmHandlers()`, inside the `llm:setConfig` ipcMain.handle callback (the unvalidated `maxCallsPerDay: a.maxCallsPerDay ?? existing.maxCallsPerDay`)

## How it triggers
A renderer caller invokes llm:setConfig with `maxCallsPerDay: '<huge string>'` or any non-number, or with `baseUrl: 'http://127.0.0.1/' + 'a'.repeat(50_000_000)`. No UI path sends maxCallsPerDay today (Settings omits it), so the field is reachable only by a buggy/compromised renderer — which is precisely the threat model the existing MAX_KEY_LEN/MAX_MODEL_LEN guards in this same function were added for.

## Verification evidence
llmIpc.ts:164-199: the handler validates provider (line 177, VALID_PROVIDERS), model length (line 181, MAX_MODEL_LEN=256, with a comment citing the exact 'unbounded string reaches writeFileAtomic.sync synchronously in MAIN (event-loop stall / DoS surface)' rationale), and baseUrl loopback-ness (line 186) — but line 195 `maxCallsPerDay: a.maxCallsPerDay ?? existing.maxCallsPerDay` passes any non-null value (string/object/multi-MB string; `??` filters only null/undefined) straight into writeLlmConfig (llmConfig.ts:95-98, JSON.stringify + writeFileAtomic.sync in MAIN). isLoopbackBaseUrl (llmConfig.ts:48-58) checks only type/URL-parse/protocol/hostname — no length cap — so `http://127.0.0.1/` + megabytes of path passes line 186 and is persisted and echoed on every llm:status (llmIpc.ts:131). readLlmConfig's read-time repair (llmConfig.ts:82-87: typeof number + isFinite + >=0) confirms non-numeric caps are considered invalid, yet the write path accepts them, silently destroying a previously configured cap (repaired to undefined → 200 default). Channel is renderer-reachable: preload/index.ts:272. Same threat model and class as the accepted BUG-012/BUG-040 guards three lines above. Low: requires a buggy/compromised renderer; no UI path sends maxCallsPerDay today.

## Suggested fix direction
Mirror the BUG-040 guards: reject non-integer/negative/huge maxCallsPerDay and cap baseUrl length before persisting.
