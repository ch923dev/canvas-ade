# BUG-017: BUG-002 fix incomplete: applyLiveness-driven demoteToSnapshot has the same unguarded capture window — a concurrent attachBoard diff-skips and the board is left detached while eligible

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/usePreviewManager.ts`, `src/renderer/src/canvas/boards/usePreviewManager.integration.test.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
attachBoard's diff-skip guard `!demoting.current.has(g.id)` (the BUG-002 fix) only protects against beginMotion's demote, because `demoting` is populated solely in beginMotion. demoteToSnapshot, when invoked directly by applyLiveness (occlusion/LOD/focus demote), never registers in `demoting`, so during its capturePreview IPC round-trip a concurrent applyLiveness→attachBoard for the same board takes the diff-skip no-op path (patches live:true, bumps NO attachSeq, issues NO attachPreview); the demote then resumes, passes its seq guard, detaches the view in main, and sets attached=false/live:false — overriding the newer 'should be live' decision. The board sits at rest showing a frozen snapshot and nothing re-attaches it until the next camera move or selection change.

## Where
`src/renderer/src/canvas/boards/usePreviewManager.ts:442` — `attachBoard` (the diff-skip guard `!demoting.current.has(g.id)` inside the useCallback starting at :399; `demoteToSnapshot` at :365)

## How it triggers
Two applyLiveness passes within the capturePreview round-trip (~30–100 ms): e.g. click a board overlapping a live Browser preview (selection-occlusion demote starts), then immediately click empty canvas to deselect (attachBoard diff-skips), or rapid selection changes during marquee/Esc-unfocus sequences.

## Verification evidence
Code-path confirmed. grep proves `demoting.current.add` exists ONLY in beginMotion (usePreviewManager.ts:558; delete at :605) — demoteToSnapshot invoked directly by applyLiveness (:644, :679) never registers there. Faulty interleaving: (1) applyLiveness pass A demotes board B (e.g. selection-occlusion, :679) → demoteToSnapshot snapshots seq=r.attachSeq (:369) and awaits capturePreview (:376); r.attached stays true and lastSent/lastZoom unchanged. (2) applyLiveness pass B (deselect) within that window → attachBoard(B): the diff-skip at :439-449 passes (r.exists && r.attached && !demoting.has(B) && rectsEqual(lastSent,bounds) && lastZoom===zoomFactor) → patches live:true, returns WITHOUT bumping attachSeq or issuing attachPreview. (3) demote resumes: recs.has ✓ (:385), r.attached ✓, r.attachSeq===seq ✓ (:386) — the seq guard cannot see the newer attach intent — so it detaches on main (:388) and sets attached=false/live:false (:390-391). Final state: board eligible-but-detached at rest, frozen snapshot. No healer: reconcile's exists-branch re-push requires r.attached (:885), its new-board branch requires !r.exists (:833), and the subscription applyLiveness fires only on selChanged/full-view (:965). The integration test covers ONLY the beginMotion/endMotion variant (usePreviewManager.integration.test.tsx:121 'endMotion-during-detach race (BUG-002)'). Medium: same defect family/grade as the shipped BUG-002 fix; persists indefinitely at rest, heals only on the next camera/selection

## Suggested fix direction
Register direct demotes in `demoting` for their capture window (mirroring beginMotion), or have the diff-skip path bump attachSeq so the demote's seq guard detects the newer attach intent.
