# BUG-049: Explicit push of an unchanged URL to an already-linked browser never consumes the reloadNonce bump (reconcile is boards-ref-gated) — no immediate reload, plus a deferred surprise re-navigate on the next unrelated boards mutation; load-failed recovery itself is masked by useBrowserAutoConnect

- **Status:** CONFIRMED

- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
applyPush's 'existing' branch bumps previewStore.requestReload(target.id) and relies on the following updateBoard to trigger usePreviewManager's reconcile, which reads the nonce. But when the target browser already has the same `url` AND the same `previewSourceId` (precisely the Bug #44 recovery scenario the nonce was built for: re-push the same dev-server URL to recover a load-failed view after the server restarts), updateBoard's diff guard makes it a pure no-op — the `boards` array ref is unchanged, and the manager's store subscription (usePreviewManager.ts:952-957) gates reconcile on `s.boards !== prevBoards`. selectBoard only triggers applyLiveness, which never navigates. The nonce sits unread until some unrelated future boards mutation, so the push silently does nothing: the native view stays on its stale Chromium error page, defeating reloadNonce's documented purpose ('an explicit push always reloads and can recover a load-failed board', previewStore.ts:32-38).

## Where
`src/renderer/src/lib/canvasDecisions.ts:99` — `applyPush` (the `target.kind === 'existing'` branch's `requestReload` call)

## How it triggers
Browser board already linked to a terminal (previewSourceId set) showing load-failed; dev server comes back on the same URL; terminal re-detects the port (or the user explicitly pushes) → applyPush 'existing' branch with identical url+previewSourceId → no reload happens.
