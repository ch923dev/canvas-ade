# BUG-065: Settings recap toggle is optimistic fire-and-forget: ignores {ok:false} and lets a recap:setConsent rejection float unhandled, showing a false-saved state

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/SettingsModal.tsx`, `src/main/recapConsent.ts`, `src/renderer/src/canvas/RecapConsentModal.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The checkbox onChange does `setRecapConsent(next)` optimistically and then `void window.api.recap.setConsent(next)` with no ok-check and no .catch. recap:setConsent can (a) return a non-throwing {ok:false} (MAIN currentDir null / frame-guard, recapConsent.ts:78-85) or (b) REJECT when writeConsent throws (e.g. ENOSPC in userData - writeFileAtomic.sync at recapConsent.ts:50 throws out of the handler). On (b) the void'd promise produces an unhandledRejection - the exact BUG-031 class this same file fixed for llm.status() - and in both cases the checkbox shows the new state while nothing was persisted and the SessionStart hook was not installed/removed. The hook-removal direction is privacy-relevant: user unticks 'Agent recaps', UI shows disabled, but the transcript-recording hook stays installed in .claude/settings.local.json.

## Where
`src/renderer/src/canvas/SettingsModal.tsx:278` — `SettingsModal` component, the "Agent recaps (this project)" checkbox `onChange` handler (`void window.api.recap.setConsent(next)`)

## How it triggers
Toggle the recap checkbox while the userData disk is full / permission-denied (writeConsent throws), or during an IPC teardown race; also any {ok:false} reply.

## Verification evidence
SettingsModal.tsx:275-279: onChange does `setRecapConsent(next)` (optimistic UI) then `void window.api.recap.setConsent(next)` — no ok-check, no .catch. Both failure modes are real: (a) non-throwing {ok:false} — recapConsent.ts:78-81 returns { ok: false } on the foreign-sender guard or when getCurrentDir() is null/desynced (unit-proven in recapConsent.test.ts:111-115); (b) genuine REJECTION — recapConsent.ts:82-84 calls writeConsent → writeFileAtomic.sync (line 50, throws on ENOSPC/EPERM in userData) and then onDecision (hook install/remove fs writes) with NO try/catch in the handler, so ipcMain.handle propagates the throw and the renderer invoke rejects; the void'd promise becomes an unhandledRejection — the same class this file's llm calls were fixed for (BUG-031 tests at SettingsModal.test.tsx:266-308 with explicit unhandledrejection listeners), while the recap-toggle tests (lines 312-372) only mock resolved setConsent and never cover failure. In both cases the checkbox shows the new state while nothing persisted, and on the untick direction the SessionStart transcript-recording hook stays installed in .claude/settings.local.json while the UI claims recaps are off — the privacy-relevant lie. Contrast: RecapConsentModal handles the rejection path of the SAME IPC (keeps modal open, shows error). Low: requires a disk failure or dir desync.

## Suggested fix direction
Mirror the llm-call pattern: await, check .ok, revert the optimistic state and surface an error on {ok:false} or rejection.
