# BUG-007: First save after T5 .bak recovery rotates the deep-corrupt primary OVER the last-good .bak before the recovered doc is durable

- **Severity:** Medium
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/projectStore.ts`, `src/renderer/src/store/canvasStore.ts`, `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`writeProject` rotates canvas.json → canvas.json.bak whenever the primary merely passes the ENVELOPE check — and in the T5 recovery flow the primary is by definition envelope-valid but deep-corrupt, while the .bak holds the only good snapshot. The first autosave after a .bak recovery synchronously copies the known-corrupt primary over the good .bak and only then starts the async atomic write of the recovered doc. A crash or write failure in that window leaves BOTH files corrupt — total on-disk project loss; even without a crash, the on-disk recovery snapshot is destroyed until the second save.

## Where
`src/main/projectStore.ts:107` — `writeProject()` (the `if (tryParse(primary) !== undefined)` guard that calls `rotateBakAtomic` at lines 107-113 before the `writeFileAtomic` at line 114)

## How it triggers
1) canvas.json becomes envelope-valid but deep-corrupt (e.g. a board with non-finite x — fromObject throws); 2) app opens it, renderer recovers from canvas.json.bak (T5 path, status 'open'); 3) user makes any edit, autosave fires writeProject; 4) crash/power-loss in the ms-window between rotateBakAtomic and writeFileAtomic completing → both files corrupt.

## Verification evidence
Code path verified: src/main/projectStore.ts:96-115 — writeProject does `if (tryParse(primary) !== undefined) rotateBakAtomic(primary, bak)` (sync, line 107-109) and only THEN `await writeFileAtomic(primary, ...)` (line 114). tryParse (lines 53-61) checks only JSON.parse + isEnvelope (schemaVersion number + boards array), so the T5 deep-corrupt primary — which by construction IS envelope-valid (canvasStore.ts applyOpenResult lines 762-784 only reaches project:reopenFromBak after MAIN returned the envelope-valid primary and renderer fromObject threw) — passes the rotation guard and is copied over the last-good .bak. project:save (projectIpc.ts:229-250) is the same path autosave uses post-recovery. If the process dies (or writeFileAtomic rejects, e.g. ENOSPC — the small .bak rotation can succeed where the larger primary temp-write fails) between line 109 and line 114 completing, BOTH canvas.json and canvas.json.bak hold the deep-corrupt content and the next open fails primary→T5→bak→error: total on-disk project loss. PERSIST-1 (lines 97-104) guards only the OUTGOING doc, not the rotation of a deep-corrupt primary IN. projectStore.test.ts has rotation-atomicity and readBak tests but no rotate-after-T5-recovery sequence test. Severity kept Medium: catastrophic outcome (whole-project loss) but compound trigger (deep-corrupt-yet-envelope-valid primary + crash/write-failure inside the first post-recovery save window).

## Suggested fix direction
Write the new primary to a temp+rename FIRST and rotate the old primary into .bak only after the new primary is durable; or skip rotation when the primary fails deep validation (renderer could flag the recovery via the save IPC).
