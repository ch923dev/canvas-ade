## Summary
The store's only beginChange() no-op guard, `if (s.past[s.past.length - 1] === s.boards) return s` (canvasStore.ts:162), holds right after a recordPast checkpoint but NOT right after an undo: applyUndo sets present = past[last] then slices past, so the new past[last] is a different object than the restored boards. Any beginChange() fired at gesture-start that then commits nothing therefore proceeds to recordPast + `future: []`, destroying the armed redo branch and making Ctrl+Shift+Z a no-op. The prior fix for bug #12 only moved the checkpoint out of the Planning element-MOVE path; the weak store guard and four other no-op-prone beginChange callers remain live redo-wipe vectors.

## How it triggers
Add a board; move it once (beginChange -> updateBoard, checkpoint A, future empty); Ctrl+Z (undo -> boards back to pre-move, future=[A] armed). Then fire ANY beginChange-at-gesture-start that commits nothing: (1) a zero-movement titlebar drag-start (RF d3-drag fires onNodeDragStart -> beginChange on a plain click, then emits no position change), (2) a click on a resize handle with no drag (onResizeStart -> beginChange; onResizeEnd does not check for change), (3) a degenerate arrow click on a Planning well (beginChange at pointer-down; pointer-up discards the <=4px arrow without commit), or (4) a quick pen tap (beginChange at pointer-down; pointer-up discards the <4-point stroke). Each runs beginChange while boards !== past[last], so future is cleared. Ctrl+Shift+Z is now a no-op; the undone move cannot be re-applied.

## Suggested fix direction
Strengthen the store guard so beginChange is a no-op whenever boards has not changed since the last checkpoint OR undo/redo: track a `lastCheckpoint: Board[] | null` set on every recordPast AND on undo/redo's resulting present, and skip beginChange when s.boards === lastCheckpoint (bug #12's Option B). This is the systemic fix that closes all callers at once. Alternatively (more invasive) move beginChange to commit-time in every no-op-prone caller (onNodeDragStart -> only on onNodeDrag/Stop with real delta; onResizeStart -> only when size actually changed; arrow/pen -> beginChange just before the commit, inside the non-degenerate branch on pointer-up, mirroring the move path already fixed for #12). Prefer the store-level fix.

Consolidated site (beginchange-downtime-checkpoint-wipes-redo): Mirror the PlanningBoard #11/#12 pattern: do not checkpoint at resize pointer-down. Move beginChange out of onResizeStart and take it lazily only when a resize actually commits (e.g. in onResize/onResizeEnd, guarded by an actual size delta), so a zero-movement handle click is a true no-op. Belt-and-suspenders: harden the store guard so beginChange is a no-op whenever boards is unchanged since the last checkpoint OR undo/redo — e.g. track a lastSnapshot ref set on recordPast AND on undo/redo's resulting present, and skip beginChange when s.boards === lastSnapshot (this also covers any other down-time beginChange callers). Add a regression test asserting that a bare beginChange() after undo leaves future intact.

Consolidated site (beginchange-postundo-guard-miss-redo-wipe): Mirror the PlanningBoard #12 fix at the Canvas board-drag level: defer the undo checkpoint to drag-stop and only take it when the board actually moved (capture start position in onNodeDragStart, compare to end in onNodeDragStop, snapshot only on non-zero delta) instead of calling beginChange() at drag-start. Alternatively/additionally adopt finding #12's Option B in the store: track a lastCheckpoint ref set on every recordPast AND on undo/redo's resulting present, and make beginChange a no-op when s.boards === lastCheckpoint so a no-op gesture cannot wipe the redo branch post-undo. The store-guard fix is preferable as it closes the whole class (any beginChange-before-possibly-no-op-edit call site), not just this one.
