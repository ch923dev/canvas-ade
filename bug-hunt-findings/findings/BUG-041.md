# BUG-041: A trailing slash on the local baseUrl produces '…/v1//chat/completions' — no normalization anywhere on the write/read/use path, and the opaque 'local HTTP 404' error makes it undiagnosable

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/llmService.ts`, `src/main/llmConfig.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
buildRequest joins the local base with a bare template: `` url: `${base}/chat/completions` ``. A baseUrl of `http://127.0.0.1:1234/v1/` (trailing slash — the natural thing to paste from LM Studio's UI or a browser address bar) is fully valid per isLoopbackBaseUrl and is persisted as-is, producing `http://127.0.0.1:1234/v1//chat/completions`. Express/path-to-regexp-style routers (LM Studio, many OpenAI-compatible shims) do not collapse the double slash and return 404, so every summarize becomes `{ok:false, reason:'provider-error', message:'local HTTP 404'}` — and, per the no-refund policy, each attempt still burns a daily budget slot. Nothing in the chain (Settings modal, llm:setConfig, readLlmConfig, buildRequest) normalizes or strips the trailing slash, and llmService.test.ts:58-66 only covers the no-trailing-slash form.

## Where
`src/main/llmService.ts:83` — `buildRequest` (the OpenAI-compatible-shape URL join)

## How it triggers
User enters `http://127.0.0.1:1234/v1/` in Settings → Save → any summarize (digest refresh, summary loop). Opaque 'local HTTP 404' provider errors with no hint that the slash is the cause; Tier-1 fallback silently engages.

## Verification evidence
All three layers verified to pass the trailing slash through untouched: (1) write — SettingsModal.tsx:127 sends the raw input value and llmIpc.ts:186 only runs isLoopbackBaseUrl; (2) validation/read — isLoopbackBaseUrl (llmConfig.ts:48-58) checks URL-parseability + protocol + hostname only, so 'http://127.0.0.1:1234/v1/' is fully valid and readLlmConfig (llmConfig.ts:81) re-applies the same check without normalizing; (3) use — buildRequest (llmService.ts:83) does bare concatenation `` url: `${base}/chat/completions` `` → 'http://127.0.0.1:1234/v1//chat/completions'. fetch/URL do not collapse double slashes in a path, and path-to-regexp/Express-style routers (LM Studio and most OpenAI-compatible shims) do not match '//' against '/v1/chat/completions' → 404. The failure presents exactly as claimed: BUG-003 (llmService.ts:189-191) deliberately surfaces only `local HTTP 404` (no URL), so the misconfiguration is opaque from the UI, and per the no-refund policy (llmService.ts:221) every attempt still burns a daily budget slot. Test gap confirmed: llmService.test.ts only covers the no-trailing-slash base. Low: single-character misconfiguration with a Tier-1 fallback, though the natural copy-paste from a browser/LM Studio makes it a realistic footgun.

## Suggested fix direction
Strip trailing slashes from baseUrl once (e.g. `base.replace(/\/+$/, '')` at buildRequest, or normalize at write time).
