# BUG-071: Renderer-only deps in `dependencies` are double-shipped: Vite-bundled into out/renderer AND packed as raw node_modules in app.asar

- **Severity:** Low
- **Category:** performance
- **Status:** CONFIRMED
- **Files touched:** `package.json`, `electron-builder.yml`, `electron.vite.config.ts`
- **Roadmap:** partially covered by Phase 5 — Packaging & release (see cross-ref)
- **Assigned:** _(left blank for the dev team)_

## Summary
react, react-dom, @xyflow/react, @xterm/xterm, @xterm/addon-fit, @xterm/addon-webgl, and zustand are renderer-side deps that vite bundles into `out/renderer` at build time, yet they sit in `dependencies`. electron-builder packs production `dependencies` (plus their transitive trees) into app.asar automatically, independent of the `files: [out/**/*, package.json]` allowlist — so every one of these is shipped twice: once inside the renderer bundle and again as raw node_modules in the asar (react-dom alone is multiple MB; @xyflow/react pulls d3 packages). CLAUDE.md's own convention states 'Renderer deps bundled by Vite; native/runtime deps stay in dependencies', i.e. these belong in devDependencies. Only main-process runtime deps (node-pty, simple-git, write-file-atomic, @expanse-ade/mcp, @electron-toolkit/*, electron-updater) need to remain in `dependencies` because `externalizeDepsPlugin()` leaves them as runtime requires.

## Where
`package.json:44` — the `dependencies` map (renderer-only entries: `@xterm/*` lines 48-50, `@xyflow/react` line 51, `react`/`react-dom` lines 54-55, `zustand` line 58)

## How it triggers
Any packaged build (`pnpm pack:dir` / staging or production CI matrix) — inspect app.asar: the renderer libraries appear both in out/renderer assets and under node_modules.

## Verification evidence
Empirically confirmed against the actual packaged artifact: listing Z:\Canvas ADE\release\win-unpacked\resources\app.asar (built with the same `files: [out/**/*, package.json]` config, electron-builder.yml:9-11) shows node_modules inside the asar containing react, react-dom, scheduler, loose-envify, @xyflow/react + @xyflow/system + the full d3-* tree (d3-color/dispatch/drag/ease/interpolate/selection/timer/transition/zoom), @xterm/xterm, @xterm/addon-fit, @xterm/addon-webgl, zustand, use-sync-external-store (1624 asar entries total) — proving electron-builder's production-dependency collection runs independent of the `files` allowlist. Every one of these is renderer-only and already compiled into the out/renderer bundle (electron.vite.config.ts:69-74 — the renderer config has no externalize plugin; `externalizeDepsPlugin()` is applied only to main (line 54) and preload (line 62), so main/preload never `require` them at runtime). This directly contradicts CLAUDE.md's own convention ('Renderer deps bundled by Vite; native/runtime deps stay in dependencies'). Main-process runtime deps (node-pty, simple-git, write-file-atomic, electron-updater, @electron-toolkit/*) legitimately remain. Pure installer-bloat/maintenance-surface issue, no functional break — Low stands.

## Suggested fix direction
Move react, react-dom, @xyflow/react, @xterm/* and zustand to devDependencies (Vite bundles them regardless); verify with a pack:dir + asar listing that only main-process deps remain.
