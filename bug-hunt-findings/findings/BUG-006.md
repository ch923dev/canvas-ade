# BUG-006: Stale output from a dying old PTY bleeds into a freshly-restarted session because proc.onData forwards to the live port by dynamic id lookup with no proc-identity guard

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
On Terminal Restart / config-respawn the old proc is killed and a new proc is registered under the same board id, but the OLD proc's onData listener forwards each chunk to whatever session currently owns the id (sessions.get(opts.id)) instead of checking it still owns this proc. node-pty's Windows conout worker keeps draining and emitting data for up to ~1s after kill() (FLUSH_DATA_INTERVAL drain timeout), so late/in-flight bytes from the dying old ConPTY are posted to the NEW session's port and term.write-n into the freshly term.reset() xterm, corrupting the new shell's prompt/output. Recoverable (clears on next redraw) but a genuine display-corruption bug distinct from the onExit case the prior pass refuted.

## Where
- src/main/pty.ts:303-315 — proc.onData handler in the pty:spawn IPC handler (the `const live = sessions.get(opts.id); if (live) live.port.postMessage({t:'data',d})` path, missing a `live.proc === proc` guard that onExit/cleanup has via isStaleExit)

## How it triggers
A Terminal board is actively emitting output (e.g. a build log, `ping -t`, `tail -f`, or an agent streaming tokens). The user clicks Restart (TerminalBoard.tsx:315 restart()), or changes a spawn dep so the spawn effect respawns. The renderer issues pty:kill then pty:spawn over the same IPC channel (FIFO). pty:kill → cleanup(id) → sessions.delete(id) + killTree(oldProc): killTree calls oldProc.kill() but the Windows ConoutConnection.dispose() defers the actual worker terminate behind a setTimeout(1000ms) drain, and setting _outSocket.readable=false does not stop a flowing net.Socket from emitting buffered data. pty:spawn then runs and sessions.set(id, {newProc, newPort1, newBuf}). Any old-socket data event that fires on a later event-loop tick now resolves sessions.get(id) to the NEW session and posts {t:'data', oldChunk} to newPort1 → newPort2 → renderer onWinMsg/port.onmessage → term.write(oldChunk) into the just-reset xterm, corrupting the new prompt/output (worse if the stale chunk carries cursor/clear/color control sequences).

## Verification evidence
Code-path trace on current (post-48c6500) code. pty.ts:303-315: proc.onData appends to its own buf (correct) then does `const live = sessions.get(opts.id); if (live) live.port.postMessage({t:'data',d})` — the live session is resolved AT FIRE TIME and there is NO `live.proc === proc` check. Contrast onExit (pty.ts:316-334) which posts over the captured `port1` (its OWN port) and calls cleanup(opts.id, proc) guarded by isStaleExit(s.proc, proc) (pty.ts:379-381, 390-393) — that is the guard the prior pass relied on. The prior verification in docs/bug-hunt-findings.md:826 (item #3) explicitly only refuted the onExit `[process exited]` bleed via the captured-port argument and at :822/:831-843 only validated isStaleExit for onExit/cleanup; it never addressed onData, which dynamically re-resolves the live port. Restart ordering confirmed: TerminalBoard.tsx:318 killTerminal then :327 spawnTerminal (same component instance, term.reset() at :325, NEW port arrives over the existing onWinMsg listener at :202-217 which writes any {t:'data'} into the SAME term via term.write). killTree (pty.ts:412-449) calls proc.kill() but never disposes the onData EventEmitter2 disposable nor removeAllListeners, so the listener stays live. node-pty proof the old socket keeps emitting after kill: opt.useConptyDll is unset in pty.ts:287-293 → legacy path; windowsConoutConnection.js:92-121 dispose()→_drainDataAndClose() sets setTimeout(_destroySocket, FLUSH_DATA_INTERVAL=1000ms) before _worker.terminate(); windowsPtyAgent.js:128-148 kill() merely sets _outSocket.readable=false (does not pause/destroy the socket). Test gap: pty.test.ts covers isStaleExit (onExit) but has NO onData identity test; the e2e respawn check (e2eSmoke.ts:166-179, "Fix #1") only asserts the NEW sentinel appears on an idle shell — it does not assert ABSENCE of stale old-proc data and would not catch this.

## Suggested fix direction
In the proc.onData handler add a proc-identity guard symmetric to the onExit/cleanup path: forward only when the live session still owns THIS proc, e.g. `const live = sessions.get(opts.id); if (live && live.proc === proc) live.port.postMessage({t:'data', d})`. (Still keep appending to the closure's own boxed buf so park/adopt scrollback remains correct.) This mirrors the existing isStaleExit reference-identity check used for stale exits, closing the equivalent stale-DATA path. Optionally add a unit/e2e assertion that a respawn under the same id while the old proc is still streaming does not deliver old-proc bytes to the new port.

## Prior-hunt relation
#16: the restart/respawn-under-same-id case was hardened with the isStaleExit identity guard on onExit/cleanup, but the equivalent identity guard was NOT applied to onData forwarding, leaving a stale-DATA (not stale-exit) bleed path that the prior verification overlooked.
