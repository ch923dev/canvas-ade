# BUG-006: Transcript watcher lifecycle is prune-only on project switch: switching away and back leaves recap auto-refresh permanently dead until a NEW claude session rewrites the map file

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`, `src/main/agentRecapWatcher.ts`, `src/main/agentRecapMap.ts`, `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`recapWatcher.track()` is called from exactly one place — `watchRecapMap`'s onChange (fs events on session-map.jsonl, plus one boot prime) — while `retain()`, driven by every project open/save/switch, only UNTRACKS. Switching project A → B → A untracks A's transcript watchers and nothing re-tracks them, so the hands-free "transcript write → 25s debounce → recap refresh" loop is silently dead for A until a brand-new claude session appends to the map file. The inverse defect also exists: the boot prime and every map append re-track ALL historical entries across ALL projects.

## Where
`src/main/index.ts:363` — the `onBoardsObserved` callback `(liveBoardIds) => recapWatcher?.retain(liveBoardIds)` passed to `registerProjectHandlers` (the only `track()` call site is `watchRecapMap`'s onChange at index.ts:402-404)

## How it triggers
With recap enabled and a live claude session in project A: switch A -> B -> A. Subsequent transcript writes in A no longer refresh the recap; only board-content changes (memoryEngine) or a manual refresh do.

## Verification evidence
Confirmed. recapWatcher.track() has exactly ONE call site: index.ts:402-404, inside watchRecapMap's onChange — which fires only on fs events on session-map.jsonl in userData (agentRecapMap.ts:148-149 directory watch filtered to the map filename) plus the single boot prime (:154). A project switch never touches that file. The retain() side is wired at index.ts:363 and driven from projectIpc's onBoardsObserved at projectIpc.ts:191 (project:open), :241 (project:save), :296 (project:current) — and retain (agentRecapWatcher.ts:119-123) is untrack-only by construction (`if (!liveBoardIds.has(id)) untrackId(id)`). So A → B: retain(B-ids) disposes A's transcript fs.watch handles; B → A: retain(A-ids) disposes B's, and NOTHING re-tracks A's (the in-memory recapMap still holds A's paths, but no watcher exists) — the hands-free transcript-write → 25s-debounce → recap-refresh loop is silently dead for A until a new claude session appends to the map file or an app restart re-primes. The inverse defect is also real: the prime/onChange loop at index.ts:402 tracks ALL map entries across ALL projects unconditionally. Test coverage confirms the gap: agentRecapWatcher.test.ts covers retain-prunes (:90, :114) but no re-track-after-switch-back. Medium: silent death of the feature's core auto-refresh in a perfectly normal flow (project switching); recap still updates via memoryEngine intents and manual refresh, so not High.

## Suggested fix direction
Re-track from the in-memory recapMap on project open/current (e.g. have onBoardsObserved pass live ids through a re-arm that calls track for ids present in recapMap), or make retain accept the map and re-arm missing watchers.
