# BUG-056: pasteIntoTerminal's disposal guard (`term.element === undefined`) never fires — xterm keeps `element` defined after dispose, so a paste resolving after teardown writes into a disposed Terminal

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`, `src/renderer/src/canvas/boards/TerminalBoard.paste.test.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
pasteIntoTerminal guards both await boundaries with `if (term.element === undefined) return`, commented 'disposed during the await'. But xterm's Terminal.dispose() does not clear the element reference (verified: the bundled @xterm/xterm lib contains no `element = void 0` / element-nulling in any dispose; `element` is a getter over `_core.element`, which stays set after dispose). So if the board unmounts or the spawn effect re-runs (Apply & restart) while stageClipboardImage/readText is in flight, the guard passes and term.paste() executes against the disposed terminal — the user's paste is silently delivered to a dead instance (lost), and xterm's write path (no `_isDisposed` guard in write, verified in the bundle) feeds the disposed core's write buffer.

## Where
`src/renderer/src/canvas/boards/TerminalBoard.tsx:112` — `pasteIntoTerminal` (module-level exported async function; the post-await disposal guard)

## How it triggers
Press Ctrl+V in a terminal (image staging IPC adds latency), then have the board respawn/unmount before the IPC resolves — e.g. config Apply, board delete, or project switch during the await.

## Verification evidence
Demonstrated by live repro against the repo's actual bundled @xterm/xterm (jsdom + require('node_modules/@xterm/xterm')): new Terminal() → element undefined; after open(host) → element defined; after dispose() → element STILL DEFINED; term.paste('hello') after dispose → does not throw (silently swallowed by the disposed core). The bundle contains zero element-nulling (grep: 0 hits for element=void 0/undefined). Therefore the guards at TerminalBoard.tsx:112 and :118 ('disposed during the await') only ever fire for a never-opened terminal — they can NEVER detect disposal during the stageClipboardImage/readText awaits, so a paste resolving after board unmount/respawn ('Apply & restart' recreates the Terminal; pasteIntoTerminal holds the old instance — TerminalBoard.tsx:401, :873) executes paste() against the disposed Terminal. The suite cannot catch this: TerminalBoard.paste.test.ts:23-27 hard-codes `element: {}` in the fake. Honest impact note: the demonstrated outcome (paste silently dropped, no throw) is behaviorally identical to what the intended early-return would do, so this is an ineffective/dead guard whose protected race remains unhandled — Low is the right grade.

## Suggested fix direction
Check disposal via a real signal — e.g. compare against the live termRef.current identity (as the respawn path does) or track an explicit disposed flag set in the effect cleanup — instead of term.element.
