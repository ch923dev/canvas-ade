# BUG-001: Unguarded proc.write/proc.resize in MessagePort message handlers throw on an exited pty, escaping to uncaughtException and exiting the whole app

- **Severity:** High
- **Category:** error-handling
- **Status:** CONFIRMED · double-verified (2 independent refuters)
- **Files touched:** `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
Both the spawn-time and adopt-time MessagePort 'message' handlers in pty.ts call proc.write/proc.resize with no try/catch. node-pty's WindowsPtyAgent.resize throws 'Cannot resize a pty that has already exited' once the native process exits, and this throw escapes the EventEmitter-based MessagePortMain listener as an uncaughtException, which index.ts wires to crashShutdown -> app.exit(1), tearing down every other terminal and Browser preview in the app.

## Where
- src/main/pty.ts:338-339 (spawn-time port1.on('message') handler — proc.write / proc.resize); src/main/pty.ts:126-127 (adopt-time port1.on('message') handler — p.proc.write / p.proc.resize)

## How it triggers
A Terminal board's shell exits (user types `exit`, the agent CLI crashes, or the process dies) while a resize message is in flight over the still-open port. The renderer's xterm onResize (TerminalBoard.tsx:189-191; TerminalSmoke.tsx:61) fires from a ResizeObserver-driven fit.fit() on ANY layout change — LOD transitions, board resize, panel toggles — and posts {t:'resize',cols,rows}. On Windows node-pty 1.2.0-beta.13 with useConptyDll=false (pty.ts never sets it), the native _$onProcessExit sets _exitCode immediately, but proc.onExit (which triggers cleanup -> port.close) does not fire until the _outSocket 'close' event, delayed by FLUSH_DATA_INTERVAL=1000ms + socket destroy. In that ~1s window the port is still open and the session still in `sessions`, so a resize message reaches proc.resize -> WindowsPtyAgent.resize -> throw. The throw propagates out of the MessagePortMain EventEmitter listener (Electron does not wrap listener dispatch; only the special 'error' event is special-cased) to process 'uncaughtException' -> index.ts:209 -> crashShutdown(1) -> app.exit(1), killing the entire app.

## Verification evidence
Code-path trace on installed node-pty 1.2.0-beta.13 (version confirmed in node_modules/node-pty/package.json:7): (1) Renderer posts resize: src/renderer/src/canvas/boards/TerminalBoard.tsx:189-191 term.onResize -> portRef.current?.postMessage({t:'resize',cols,rows}); driven by ResizeObserver+fit.fit() at TerminalBoard.tsx:275-280. (2) Main handler is UNGUARDED: src/main/pty.ts:336-340 port1.on('message', e => { ... else if (m.t==='resize'...) proc.resize(m.cols,m.rows) }) — no try/catch, unlike the adjacent onData (309-314) and onExit (319-324) which ARE wrapped. Same unguarded pattern in adopt at pty.ts:124-128. (3) node-pty resize throws on exited pty: node_modules/node-pty/lib/windowsPtyAgent.js:119-124 `resize = function(cols,rows){ if (this._exitCode !== undefined) throw new Error('Cannot resize a pty that has already exited'); ...}`. _exitCode is set immediately on native exit in _$onProcessExit (windowsPtyAgent.js:189-191). (4) onExit/port-close is DELAYED: with _useConptyDll=false, _$onProcessExit -> _flushDataAndCleanUp schedules _cleanUpProcess after FLUSH_DATA_INTERVAL=1000ms (windowsPtyAgent.js:22,197-214) -> _outSocket.destroy() -> windowsTerminal.js:101-104 'close' -> emit('exit') (fires proc.onExit) + _close(). So _exitCode is set ~1s BEFORE proc.onExit -> cleanup(opts.id,proc) -> s.port.close() runs (pty.ts:316-327, 390-402). (5) Crash sink: src/main/index.ts:209 process.on('uncaughtException', err => crashShutdown(1,err)); crashShutdown calls app.exit(exitCode) at index.ts:202-208. Handler is live shipping code: registerPtyHandlers wired at index.ts:142. The write path is partially mitigated because Terminal.prototype._close no-ops write (terminal.js:169-175), but resize is NOT no-op'd, and on *nix UnixTerminal.resize calls native pty.resize on a possibly-dead fd (unixTerminal.js:251-261) which can also throw — so resize is the clear cross-platform throw. Not among the 50 prior findings (docs/bug-hunt-findings.md has no entry for unguarded port write/resize -> uncaughtException; #40 only notes the resize round-trip exists for geometry, not a crash).

[2nd refuter confirmed] Code-path trace on live HEAD (ee15e9b, both files unmodified): (1) Throw site: node_modules/node-pty/lib/windowsPtyAgent.js:119-122 `resize = function(cols,rows){ if(this._exitCode!==undefined) throw new Error('Cannot resize a pty that has already exited'); ...}`. (2) _exitCode set on exit: windowsPtyAgent.js:189-191 _$onProcessExit sets this._exitCode = exitCode (native callback). (3) Close/no-op delayed: outSocket 'close' -> WindowsTerminal._close (terminal.js:169-175 replaces write with a no-op and emits 'exit') only fires after _cleanUpProcess destroys outSocket, which is scheduled FLUSH_DATA_INTERVAL=1000ms after exit (windowsPtyAgent.js:22,205,207-214). So resize() throws during the >=1s gap. (4) Unguarded call: pty.ts:336-340 `port1.on('message', e => { ... else if (m.t==='resize' && m.cols && m.rows) proc.resize(m.cols,m.rows) })` has no try/catch; same at pty.ts:124-128 for the adopt path (p.proc.resize). Contrast with the post-side calls onData (pty.ts:309-314) and onExit (pty.ts:319-324) which ARE try/catch-wrapped — the author guarded the post side but not the inbound proc-call side. (5) Escape to crash: MessagePortMain extends NodeEventEmitter (electron.d.ts:8846); a synchronous throw in a non-'error' EventEmitter listener is not swallowed by emit and, dispatched from Electron's native port-message callback with no surrounding try/catch, surfaces as process 'uncaughtException' -> index.ts:209 crashShutdown(1,err) -> index.ts:207 app.exit(1). (6) Trigger source verified: src/renderer/src/canvas/boards/TerminalBoard.tsx:189-190 posts {t:'resize',cols,rows} on term.onResize, driven by a ResizeObserver at line 275. (7) Not a prior finding: bug-hunt-findings.md #28 (shell dedupe, pty.ts:124), #39 (killTree), #47 (crash hook) are unrelated; this proc-call-throw path is new. Note: the secondary `write`/EPIPE claim is weaker (write becomes a no-op after _close, and a pre-close socket.write usually emits an async 'error' rather than throwing synchronously), but the resize path alone is a clean, reachable synchronous-throw -> app-crash.

## Suggested fix direction
Wrap the proc.write/proc.resize calls inside both MessagePort 'message' handlers (spawn-time pty.ts:336-340 and adopt-time pty.ts:124-128) in try/catch that swallows/logs the error, mirroring the existing onData (309-314) and onExit (319-324) guards. Optionally also guard against a known-exited proc by tracking exit state, but the try/catch is the minimal robust fix that prevents a single dead-pty write/resize from becoming a whole-app uncaughtException crash.
