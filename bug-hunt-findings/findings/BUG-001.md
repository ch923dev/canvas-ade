# BUG-001: flushRenderer touches webContents of the destroyed main window — every window-close quit on Win/Linux throws into the crash sink, killing the guarded-quit chain

- **Severity:** High
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`, `src/main/quit.ts`, `src/main/ipcGuard.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`flushRenderer()` does `const win = mainWindow; const wc = win?.webContents` with no `win.isDestroyed()` check, and index.ts never nulls `mainWindow` on close (no `'closed'` handler). On Windows/Linux the only quit path is closing the window, which runs `'closed'` → `'window-all-closed'` → `app.quit()` → `'before-quit'` → `performGuardedQuit` → `flushRenderer` — and accessing `.webContents` on a destroyed BrowserWindow throws "Object has been destroyed". The synchronous throw escapes `performGuardedQuit` (quit.ts's `.catch` only handles promise rejections), lands in `process.on('uncaughtException')` → `crashShutdown(1)`, demoting the #49 awaited PTY-tree drain to a fire-and-forget race against `app.exit`, killing the BUG-M2 flush handshake, and exiting 1 on every normal quit.

## Where
`src/main/index.ts:488` — `flushRenderer()` (the `const wc = win?.webContents` access; `win.isDestroyed()` is never checked before dereferencing the destroyed window)

## How it triggers
User closes the main window on Windows or Linux (Alt+F4 / titlebar X / taskbar close) — i.e. every normal quit. Worst case: agents with deep child trees are running; the unawaited tree-kill races process exit.

## Verification evidence
Code path verified and empirically reproduced on this repo's Electron 42. (1) index.ts:64/89 — `mainWindow` is set in createWindow and NEVER nulled (grep for `mainWindow =`/`on('closed'` finds only line 89; no 'closed' handler). (2) index.ts:512-517 routes window-all-closed → app.quit(); index.ts:525-537 before-quit → performGuardedQuit({flush: flushRenderer,...}). (3) flushRenderer (index.ts:486-489) does `const wc = win?.webContents` with no `win.isDestroyed()` check. Probe 1 (repo electron.exe, temp-dir script): accessing `.webContents` on a closed BrowserWindow → `THREW: Object has been destroyed`. Probe 2 replicated index.ts's exact wiring (window-all-closed→quit, before-quit→performGuardedQuit-shaped chain with the same `deps.flush().catch(...)` shape from quit.ts:17-23) and closed the window: output was `UNCAUGHT_EXCEPTION reached crash sink: Object has been destroyed` and the guarded `shutdown()`/`exit(0)` log lines NEVER printed — the sync throw escapes before `.catch` attaches, exits the before-quit listener, and lands in process.on('uncaughtException') → crashShutdown(1) (index.ts:548), i.e. fire-and-forget `void shutdown()` + app.exit(1). On Win/Linux there is no other quit affordance (Menu.setApplicationMenu(null), index.ts:186-190), so this is EVERY normal quit. Mitigating color for severity honesty: the renderer's beforeunload autosave still fires on a real window close, and crashShutdown's killTree execFile spawns are issued synchronously before app.exit.

## Suggested fix direction
Guard like ipcGuard.ts:23 — check `!win || win.isDestroyed()` before touching `.webContents` in flushRenderer (and/or null mainWindow in a 'closed' handler); optionally make performGuardedQuit sync-throw-safe (wrap flush in Promise.resolve().then()).
