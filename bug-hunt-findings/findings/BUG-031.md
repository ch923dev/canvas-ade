# BUG-031: Normal quit paths (before-quit / window-all-closed) fire shutdown() without awaiting, so the awaitable PTY tree-kill added for #49 is bypassed on the common quit path

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The #49 remediation made disposeAllPtys()/shutdown() return an awaitable Promise that resolves only once each session's taskkill /T /F child reaps the agent tree (bounded by a 2s timeout), but it wired `await shutdown()` ONLY to the abrupt e2e app.exit path (index.ts:158). The two real-world quit paths still call `void shutdown()` fire-and-forget: window-all-closed (index.ts:188) then `app.quit()`, and before-quit (index.ts:193). On a normal user quit Electron does not wait for that floating promise, so the process can exit while taskkill is still walking a deep child tree, leaving an orphan window. Impact is narrow/Low: node-pty's own ConPTY/conout handles are disposed synchronously via proc.kill() inside killTree before exit, and the spawned taskkill.exe is an independent OS process that typically survives main's death to finish reaping descendants.

## Where
- src/main/index.ts:188 (window-all-closed handler — `void shutdown(); app.quit()`); src/main/index.ts:193 (before-quit handler — `void shutdown()`); contrast src/main/index.ts:158 (e2e abrupt path — the only `await shutdown()`)

## How it triggers
Open a Terminal board running an agent that spawns a deep/slow child process tree, then quit normally: close the last window (window-all-closed → `void shutdown(); app.quit()`) or trigger app quit (before-quit → `void shutdown()`). Neither handler awaits the returned Promise, so Electron proceeds through will-quit/quit and terminates the main process while the async `taskkill /PID <pid> /T /F` is still reaping descendants — orphaning grandchild processes for the residual window before taskkill (or OS reparenting) completes.

## Verification evidence
Read src/main/index.ts: shutdown() (179-185) returns `disposeAllPtys()`'s Promise. Only the e2e branch awaits it: line 158 `await shutdown()` before `app.exit(code)` (159). The two normal handlers do not: line 188 `void shutdown()` then line 189 `app.quit()`; line 193 `void shutdown()`. The code comment at 176-177 explicitly admits "the sync paths (before-quit / window-all-closed / crash hooks) fire it best-effort without awaiting." disposeAllPtys() (pty.ts:456-460) maps cleanup()/reapParked() → killTree(), whose win32 branch (pty.ts:412-436) resolves only on the execFile('taskkill','/PID',pid,'/T','/F') callback or a 2s timeout — i.e. genuinely async work that `void` discards. Grep of index.ts confirms NO event.preventDefault()/`quitting`-flag guarded await-then-quit pattern anywhere (only the nav-guard preventDefaults at 82/86). No test covers the shutdown/quit paths (grep of *.test.ts for shutdown/before-quit/disposeAllPtys: no matches). This is the unfixed remainder of prior #39 claim (b): #49 made teardown awaitable + awaited it on app.exit only. Severity downgraded from the candidate's Medium to Low: killTree synchronously calls proc.kill() (pty.ts:431-435) disposing node-pty's ConPTY/conout Worker before exit, and the detached taskkill.exe is not in an Electron job object so it normally outlives main and still reaps the tree — matching #39's own Low grading of the identical race.

## Suggested fix direction
Apply the standard guarded async-quit pattern on the normal paths, mirroring the e2e branch. In before-quit, on first entry call event.preventDefault(), set a module-level `quitting` flag, `await shutdown()` (bounded by its existing 2s timeout), then `app.exit(0)` (or app.quit()); on re-entry with the flag set, do nothing so the second pass proceeds. For window-all-closed (non-darwin) route through the same `app.quit()` so the guarded before-quit handles the drain rather than the current fire-and-forget `void shutdown()`. This makes the common quit path reap the agent child tree before exit instead of racing it, closing the residual orphan window. (Crash hooks at 206 remain best-effort since an uncaughtException handler cannot await before exit — that is acceptable and already documented.)
