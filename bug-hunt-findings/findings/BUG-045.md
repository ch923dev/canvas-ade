# BUG-045: BoardMenu's ⋯ trigger lacks a pointerdown stopPropagation — clicking the open trigger closes then instantly reopens the menu (toggle is dead)

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/BoardFrame.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
While the menu is open, a document-level pointerdown listener closes it (line 213: document.addEventListener('pointerdown', close)). The menu ITEMS guard against this ordering by stopPropagation on pointerdown (lines 252, 284), but the ⋯ trigger button does not: IconBtn only stops propagation on mousedown (line 137), and the close listener is on pointerdown, which fires first and bubbles to document. Sequence when clicking the open trigger: pointerdown -> close() -> setOpen(false) commits; click -> openMenu -> setOpen(v => !v) with v now false -> true. Net effect: the toggle (active={open}, setOpen(v=>!v)) is dead — clicking the active ⋯ button closes and immediately reopens the menu (with a setMenuOpen detach/reattach churn on any live Browser preview each time, since the open effect re-registers the popover token). The only ways to dismiss are Esc or clicking elsewhere.

## Where
`src/renderer/src/canvas/BoardFrame.tsx:243` — `openMenu` in `BoardMenu` (the ⋯ trigger `IconBtn` at lines 268-276; the document pointerdown close listener at line 213)

## How it triggers
Open any board's ⋯ menu, then click the ⋯ button again expecting it to close.

## Verification evidence
Code-path proof. BoardFrame.tsx:213 — while open, `document.addEventListener('pointerdown', close)` closes on ANY outside pointerdown. The menu ITEMS defend against exactly this ordering with `onPointerDown={(e)=>e.stopPropagation()}` (lines 252, 284), but the ⋯ trigger does not: IconBtn only stops `mousedown` (line 137-138), and its `onPointerDown` PROP is invoked from inside the mousedown handler (line 140) — there is no real pointerdown handler on the button, so the native pointerdown bubbles to document (no ancestor stops it; BoardNode.tsx has zero stopPropagation, RF node drag is d3 mousedown-based). Sequence on clicking the open trigger: pointerdown → close() → setOpen(false) flushed before the discrete click; click → openMenu (line 243-246) → setOpen(v=>!v) reads false → true → reopens (with a setMenuOpen detach/reattach cycle on live previews via the line 201-204 effect). Decisive corroboration: AppChrome.tsx:181-183 fixed this exact race on the ProjectSwitcher trigger — `onPointerDown={(e)=>e.stopPropagation()}` with comment 'or re-clicking to close would close-then-reopen (BoardMenu parity)' — proving the race class is real in this codebase while BoardMenu itself never got the guard. No contradicting test: BoardMenu.integration.test.tsx:11 covers only the ITEM-side ordering; the trigger-while-open case is untested (test line 104 even re-opens via a bare click without simulating the preceding pointerdown).

## Suggested fix direction
Add onPointerDown stopPropagation to the ⋯ trigger (the ProjectSwitcher pattern at AppChrome.tsx:183), or have IconBtn stop the real pointerdown.
