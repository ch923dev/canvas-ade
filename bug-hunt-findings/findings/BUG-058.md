# BUG-058: reconcile's attached-board bounds re-push lacks the full-view guard — pushes the camera-scaled canvas rect to a full-view board's native view

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/usePreviewManager.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
attachBoard (line 402-421) and flushBatch (509-524) both branch on `fullViewIdRef.current === g.id` and use fullViewBoundsFor (the modal rect), with an explicit HOLD when the portal hasn't landed — comments call falling back to the canvas rect 'the full-view native-bounds bug'. reconcile's re-push branch (885-891) has no such check: for the attached full-view board it computes `previewGeom.boundsAndZoom(g, vp, paneOffset)` — the camera-scaled CANVAS rect — and fires attachPreview with it, snapping the always-above native view from the modal frame to its canvas position over the scrim. It also clobbers r.lastSent with the canvas rect. The subscription callback's trailing applyLiveness (line 965, fullViewIdRef !== null) immediately re-attaches at the modal rect, so the wrong bounds are visible for at most a frame or two — a transient native-view flash at the canvas position during full view.

## Where
`src/renderer/src/canvas/boards/usePreviewManager.ts:885` — `reconcile` (the `r.attached && !gestureRef.current` bounds re-push branch)

## How it triggers
Any canvasStore.boards mutation while a Browser board is in full view and its view is attached to the modal frame: editing the URL in the full-view URL bar, an auto-connect detect updateBoard, an MCP board patch, or any other board's geometry change (reconcile iterates all browser boards).

## Verification evidence
Confirmed by direct comparison of the three bounds producers in usePreviewManager.ts. attachBoard (:402-421) and flushBatch (:509-524) both branch on `fullViewIdRef.current === g.id` and HOLD rather than fall back to the canvas rect ('never fall back to the camera-scaled canvas rect'). reconcile's attached re-push (:885-891) has no full-view check: for the full-view board r.attached is true (attached at the modal rect) and gestureRef is false at rest, so any boards mutation (URL edit in the full-view bar, auto-connect detect updateBoard, any board's geometry change — reconcile fires on every `s.boards !== prevBoards`, :954) computes previewGeom.boundsAndZoom = the camera-scaled CANVAS rect, which ≠ lastSent (modal rect) → not diff-skipped → fires attachPreview with the canvas rect over the scrim and clobbers r.lastSent. The same subscriber then runs applyLiveness (:965, fullViewIdRef !== null) → full-view branch → attachBoard pushes the correct modal rect (lastSent now differs, so no diff-skip), healing it. Both IPCs are issued synchronously in order within one subscriber tick, so the wrong bounds are visible for at most a frame or two — a transient native-view flash plus a wasted IPC. Low is correct.

## Suggested fix direction
Give reconcile's re-push branch the same isFullView/fv guard (skip or use fullViewBoundsFor) that attachBoard and flushBatch carry.
