# BUG-010: startLocalServer registers no server 'error' listener and its Promise has no reject path — any listen() failure crashes app startup with no diagnostic

- **Severity:** Medium
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/localServer.ts`, `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

> **Consolidates 2 discovery findings** (same root cause, multiple call sites): `localserver-listen-no-error-handler-no-reject`, `localserver-listen-error-unhandled`

## Summary
startLocalServer (src/main/localServer.ts:57-73) creates an http.Server and calls server.listen(0,'127.0.0.1', cb) but never registers a 'error' listener, and the returned Promise is constructed with only `resolve` (no reject), settled solely inside the listen success callback. If listen ever emits 'error' (EACCES from an AV/sandbox/firewall loopback-bind denial, EMFILE/ENFILE under fd exhaustion, ENETDOWN), Node re-throws the unhandled emitter error synchronously; it propagates to process.on('uncaughtException') (index.ts:209) → crashShutdown(1) → app.exit(1), tearing the whole app down at startup with no user-facing diagnostic. The Promise itself can never reject, so there is no path to surface a clear error or fall back gracefully. The entire window/PTY/preview stack is gated on `await startLocalServer()` (index.ts:141).

## Where
- src/main/localServer.ts:57-73 — startLocalServer (Promise has no reject; no server.on('error')); reached via src/main/index.ts:141 `localServer = await startLocalServer()` inside app.whenReady, which gates createWindow() at index.ts:145; crash sink at src/main/index.ts:202-209 (crashShutdown / process.on('uncaughtException'))
- src/main/localServer.ts:57-72 — startLocalServer (no server.on('error') listener; Promise executor captures only resolve; close:() => server.close() also discards close errors). Boot consumer: src/main/index.ts:141 — `localServer = await startLocalServer()`.

## How it triggers
At app start, listen(0,'127.0.0.1') fails on the loopback interface. Realistic causes: EACCES (security software / sandbox / firewall blocking a loopback listen), EMFILE/ENFILE (file-descriptor exhaustion from many open boards/PTYs/views), or ENETDOWN. (EADDRINUSE is effectively impossible here because port 0 = OS-chosen ephemeral port — used in the repro only to deterministically force the same 'error' code path.) On any such error, net.Server emits 'error' with no listener attached → Node throws it as an uncaughtException → crashShutdown(1) exits the app with code 1 and zero diagnostic about a port-bind failure. The await never settles, so createWindow() is never reached on the off chance the exit were deferred.

## Suggested fix direction
In startLocalServer, switch the Promise to `(resolve, reject)` and attach `server.once('error', reject)` before/at listen so a bind failure rejects with the underlying error instead of being re-thrown as an uncaughtException. At the call site (index.ts:141), wrap the await in try/catch (or add .catch to whenReady().then) to surface a clear "could not bind local preview server on 127.0.0.1" diagnostic and degrade gracefully (e.g. continue without the dev/preview local server, or show a fatal-dialog) rather than a silent exit(1). Consider also removing the now-settled 'error' listener after successful listen to avoid leaking a handler.

Consolidated site (localserver-listen-error-unhandled): Attach a `server.on('error', reject)` and accept the reject param in the Promise executor so a listen failure rejects rather than escalating to an uncaughtException; remove the error listener once 'listening' fires. At the call site (index.ts:141) wrap `await startLocalServer()` in try/catch (or add a .catch to whenReady) to surface a graceful 'preview server unavailable' state and continue booting the window instead of aborting. Optionally make `close` accept/log a callback error for symmetry. Severity Low: trigger is rare (loopback ephemeral-port bind is highly reliable) but the failure is fatal-to-startup with no user-facing message; no data loss or security impact.
