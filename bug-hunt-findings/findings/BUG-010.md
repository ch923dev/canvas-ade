# BUG-010: Selecting a board does not clear selectedConnectorId — one Delete press deletes BOTH the selected board and the still-selected connector

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Canvas.tsx`, `src/renderer/src/canvas/hooks/useCanvasKeybindings.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`onEdgeClick` clears the board selection when a connector is selected, but the inverse direction is missing: clicking a board re-selects it through React Flow's select changes → `foldSelectionIntents` → `setSelection`, and nothing clears `selectedConnectorId` (only Esc/pane-click, `startConnect`, and the connector-delete handler do). After clicking a connector then a board, both render as selected, and one Delete/Backspace keydown fires BOTH paths — the keybinding removes the connector (its `preventDefault` does not stop RF, whose `useKeyPress` ignores `defaultPrevented`) and RF's `deleteKeyCode` removes the selected board. The user silently loses an unrelated orchestration connector (or board); undo restores them only as two separate steps.

## Where
`src/renderer/src/canvas/Canvas.tsx:818` — the inline `onEdgeClick` handler on `<ReactFlow>` inside the `Canvas` component (`setSelectedConnectorId(edge.id)` at line 818; the missing inverse clear is the `foldSelectionIntents`/`setSelection` path, with the double-fire in `useCanvasKeybindings.ts:116-130`)

## How it triggers
Click an orchestration connector (selects it), then click any board's title bar (board selected, connector selection silently retained), then press Delete or Backspace.

## Verification evidence
Full path verified. (1) Selecting a board never clears selectedConnectorId: grep over the repo shows the ONLY clear sites are Canvas.tsx:565 (clearSelection — Esc/pane-click), Canvas.tsx:621 (startConnect), useCanvasKeybindings.ts:125 (the connector-delete handler itself) and the e2e hook; a board click flows through RF select changes → onNodesChange → foldSelectionIntents → setSelection (Canvas.tsx:436-465), which never touches it, and onPaneClick does not fire on node clicks. The edges memo (Canvas.tsx:318-335) keeps rendering that connector with selected:true (bright stroke + ✕). (2) Double-fire on one Delete: useCanvasKeybindings.ts:116-130 unconditionally removeConnector()s while selectedConnectorId is set — its e.preventDefault() (line 123) does NOT stop React Flow, because the vendored @xyflow/react useKeyPress downHandler (dist/esm/index.js:~430-450) never checks event.defaultPrevented (it tracks pressedKeys and calls its own preventDefault), and useGlobalKeyHandler (dist/esm/index.js:1225-1236) then deleteElements()s every selected node — the just-clicked board, selected via setSelection. So a user who clicks a connector, then a board, then presses Delete loses BOTH on one keypress, directly contradicting the mutual-exclusivity invariant the comment at useCanvasKeybindings.ts:113-115 asserts. No test covers this (the only selectedConnectorId test reference is a null prop in EscFullViewConfirm.integration.test.tsx).

## Suggested fix direction
Clear selectedConnectorId whenever a node-select intent lands (in the foldSelectionIntents/setSelection path or an onNodeClick), restoring the mutual-exclusivity invariant.
