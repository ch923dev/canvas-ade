## Summary
The cleanup at lines 94-107 unconditionally decrements assetUrlCache's refcount if an entry exists, but two paths reach cleanup WITHOUT this component having claimed a ref: (a) cache-miss mount whose async asset.read is still in flight at unmount — `cancelled` makes the .then return before the refs++ / refs:1 claim, yet cleanup still decrements; (b) the entry is created by a sibling between this card's useLayoutEffect (saw miss → no claim) and its useEffect (sees hit → comment says 'needs nothing here', claims nothing) — render then serves the cache URL via line 113 but no ref was ever taken. In both cases the decrement steals another live ImageCard's ref: refs hits 0, the URL is revoked and the entry deleted while that sibling still renders it → permanent 'broken image' until remount, plus the sibling's own cleanup decrements a missing entry. The inverse window also exists: layout effect claims on a cache hit, the entry dies before the passive effect, the passive effect's read then claims again (refs++ or refs:1) — two claims, one decrement → the object URL is never revoked (leak).

## Where
`src/renderer/src/canvas/boards/planning/ImageCard.tsx:94` — `useAssetUrl` (the passive effect's cleanup function)

## How it triggers
Two ImageCards sharing one assetId (Duplicate / alt-drag of an image element) where one mounts and unmounts within the window of an in-flight asset.read — e.g. duplicate an image and immediately undo, or rapid LOD/full-view remounts of a planning board containing duplicated images.

## Verification evidence
Confirmed by claim/release accounting on ImageCard.tsx. Claims happen at exactly three sites: useLayoutEffect cache-hit refs++ (line 53), resolve-path again.refs++ (80-81), resolve-path set refs:1 (87). The cleanup (94-107) unconditionally decrements ANY existing entry. Path (a): mount on cache miss (layout effect 51-53 sees no entry → no claim; passive effect 63-93 starts asset.read), then unmount while the read is in flight — `cancelled` (72-73) makes the resolver return before claiming, yet the cleanup at 99-104 still decrements an entry a sibling's resolved read created (87) in the meantime → refs hit 0, URL revoked + entry deleted while the sibling still renders it (line 113 then returns null for the sibling on its next render → permanent 'missing image' until remount). Path (b): sibling's in-flight read resolves (IPC reply task) BETWEEN this card's synchronous layout effect (saw miss → no claim) and its later passive-effect flush — the passive effect's `if (!cached)` (64) is then false and per its own comment (68-69, which wrongly assumes 'useLayoutEffect already claimed') does nothing; render serves the cache URL via line 113 with ZERO refs claimed, and this card's cleanup later steals the sibling's ref. The BUG-036 useLayoutEffect claim (38-59) only covers the cache-hit-at-layout case, leaving these two interleavings asymmetric. Narrow timing windows (one asset.read IPC round trip) with duplicated-image siblings → Low.

## Suggested fix direction
Track whether THIS hook instance claimed a ref (a boolean per effect run) and only decrement when it did; claim in the passive effect when layout saw a miss but the entry now exists.
