# BUG-029: xterm WebGL context lost while a terminal stays in detail view is never re-acquired (#11 fix implemented only LOD-release, not budget-freed re-attach)

- **Severity:** Low
- **Category:** performance
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
In TerminalBoard.tsx the WebGL addon's onContextLoss handler only disposes the addon and nulls webglRef; it adds no re-acquire path. The sole re-acquire calls are the initial mount (line 174) and the LOD true->false transition effect (line 141). A terminal that loses its GL context to a newer one under Chromium's ~16-context ceiling while remaining in detail view (lod=false) has no recovery hook and stays on the slower DOM renderer for the rest of its life unless the global camera zoom crosses the LOD boundary and back. Impact is purely perf/quality degradation, never a crash/leak/blank/data-loss.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:115-118 (onContextLoss only frees), :137-142 (LOD effect — only re-acquire path, gated on lod change), :174 (mount-only attach)

## How it triggers
Open enough detail-view terminals (lod=false, camera zoom >= 0.4) plus Browser boards to cross Chromium's ~16 live WebGL2-context-per-renderer ceiling. A newer context creation evicts the oldest detail-view terminal: it fires webglcontextlost, the addon preventDefaults and waits 3000ms, no webglcontextrestored arrives (newer contexts hold budget), so at T+3s onContextLoss fires and TerminalBoard disposes the addon + nulls webglRef -> that terminal silently downgrades to the DOM renderer. Later closing boards frees Chromium budget, but the downgraded terminal's lod never changed (it stayed in detail view), so the LOD effect does not re-run, the spawn effect does not re-run, and nothing calls attachWebgl again. The board is stranded on the DOM renderer. It only self-heals if the user zooms the whole camera below 0.4 and back above it (a global LOD round-trip detaches and re-attaches every terminal).

## Verification evidence
Code-path trace on current (post-48c6500) code. attachWebgl (lines 111-124) registers webgl.onContextLoss(() => { webgl.dispose(); webglRef.current = null }) — frees only, no re-add and no webglcontextrestored re-attach. detachWebgl (126-133) likewise only frees. The ONLY callers of attachWebgl are line 141 (LOD effect else-branch) and line 174 (spawn effect, once per mount; gated by lodRef). The LOD effect deps are [lod, attachWebgl, detachWebgl]; attachWebgl/detachWebgl are useCallback([]) so their identity is stable, meaning the effect re-runs ONLY when lod changes. The spawn effect deps [board.id, board.shell, board.cwd, board.launchCommand, attachWebgl, detachWebgl] are unchanged by a context loss, so no remount. onContextLoss mutates only a ref (no setState) so it triggers no re-render. Therefore a context loss with lod unchanged produces no re-acquire of any kind. Confirmed via node_modules/@xterm/addon-webgl/src/WebglRenderer.ts:105-126 that webglcontextlost is preventDefault'd, waits 3000ms then fires _onContextLoss, and that the addon's own webglcontextrestored handler is internal to the (now-disposed) renderer — so once TerminalBoard disposes the addon, no later restored event can be handled. Grep confirms no global WebGL registry/budget manager in src/renderer (only TerminalBoard.tsx manages production WebGL lifecycle; the smoke files are independent). lod is a GLOBAL camera flag (BoardNode.tsx:58 isLod(s.transform[2]); canvasView.ts:24-25 LOD_ZOOM=0.4), confirming the self-heal requires a whole-camera LOD round-trip. This is exactly option (c) of finding #11's fix list ("add a WebglAddon back when budget frees, instead of leaving evicted terminals permanently on the DOM path"), which the remediation did NOT implement — it implemented only option (b) (release at LOD). So it is a genuine incomplete remediation of #11, not a re-report of #11 as-was.

## Suggested fix direction
Add a recovery hook so an evicted detail-view terminal re-upgrades when budget frees. Lightweight option: in attachWebgl's onContextLoss, in addition to freeing, schedule a re-attach attempt (e.g. a short retry/backoff or a flag re-checked when any board unmounts) gated by !lod && termRef.current, since attachWebgl already no-ops if a context exists. More robust: introduce the module-level WebGL registry/budget tracker that #11's fix options (a)/(c) called for — track active addons across boards, and when a board frees its context (LOD detach, unmount, or context loss) signal the least-recently-upgraded DOM-fallback terminal to re-attach. Either restores the missing budget-freed re-acquisition path that LOD-only release left uncovered.
