# BUG-035: boardResults Map is never cleared on project switch or board deletion — stale (and id-colliding cross-project) results served via canvas://board/{id}/result

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/main/boardResults.ts`, `src/main/index.ts`, `src/main/projectIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The module-level `results` Map only ever grows (recordBoardResult sets; __resetBoardResults is test-only). Nothing clears it when a board is deleted or when the project switches — unlike memoryEngine.reset() and recapWatcher.retain(), which the same lifecycle paths drive. After switching from project A to project B, an MCP agent reading canvas://board/{id}/result for a B board whose id collides with an A board receives project A's stale verdict/summary as if it were B's. Even without collisions, results for long-deleted boards accumulate for the app's lifetime.

## Where
`src/main/boardResults.ts:13` — module-level `results` Map (read/written by `readBoardResult()` / `recordBoardResult()`)

## How it triggers
Record a result for board X in project A (MCP write_result), switch to project B containing a board with the same id (copied/fixture canvas.json), have an agent read canvas://board/X/result — A's result is returned as current.

## Verification evidence
boardResults.ts:13-28 is the whole module: a module-level Map with set (recordBoardResult), get (readBoardResult), and a test-only __resetBoardResults. Grep across src/ confirms the ONLY non-test call sites are index.ts:252 (`readResult: readBoardResult` wired into startMcpServer — agent-visible via canvas://board/{id}/result), index.ts:272 (recordResult), and mcpSmoke.ts. No production code ever clears or prunes it. The contrast claimed is real: the same project-open/switch paths DO reset sibling per-project state — projectIpc.ts:185 and :292 `memoryEngine.reset()` and :191/:241/:296 `onBoardsObserved(...)` (wired to recapWatcher.retain to drop watchers for boards of the project switched away from) — but neither path touches boardResults. So after switching A→B, a B board whose id collides with an A board reads A's stale verdict/summary as current, and results for deleted boards accumulate for the app lifetime. Low: collision requires duplicated nanoid ids (copied/fixture canvas.json), and the no-collision case is just unbounded-but-tiny memory growth.

## Suggested fix direction
Hook boardResults pruning into the same onBoardsObserved/retain lifecycle (retain live ids; clear on project switch).
