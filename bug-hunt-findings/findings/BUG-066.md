# BUG-066: RecapConsentModal.decide() handles only rejections — a non-throwing {ok:false} reply closes the modal as if the decision persisted

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/RecapConsentModal.tsx`, `src/main/recapConsent.ts`, `src/renderer/src/canvas/AppChrome.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
decide() does `await window.api.recap.setConsent(decision); onClose()`. The try/catch handles only REJECTIONS, but recap:setConsent's documented failure mode for a MAIN-side dir desync is a non-throwing `{ ok: false }` (recapConsent.ts:78-81: guard hit or getCurrentDir() === null). The code comment 'close ONLY once the decision is durably persisted' is therefore violated: on {ok:false} the modal closes with no error, the user believes recaps are enabled, no hook is installed, and the undecided state re-prompts on the next launch. AppChrome's own render guard acknowledges this exact desync ('MAIN/renderer dir desync can leave askRecap=true with no project open') but only gates the RENDER, not the write result.

## Where
`src/renderer/src/canvas/RecapConsentModal.tsx:22` — `RecapConsentModal` component, `decide()` (the `await window.api.recap.setConsent(decision)` line)

## How it triggers
User clicks 'Enable recaps' while MAIN's currentDir is null or desynced from the renderer's project.dir (the desync window AppChrome's line-77 comment documents).

## Verification evidence
RecapConsentModal.tsx:18-32: `await window.api.recap.setConsent(decision); onClose() // close ONLY once the decision is durably persisted` inside try/catch. But recap:setConsent's documented MAIN-side failure mode is a RESOLVED `{ ok: false }`, not a throw: recapConsent.ts:78-81 returns { ok: false } when the frame guard trips OR getCurrentDir() === null (nothing written, onDecision never called, no hook installed). This exact return is pinned by recapConsent.test.ts:111-115 ('setConsent returns {ok:false} when no project dir is open') and :127-134 (foreign sender). The preload types it as Promise<{ ok: boolean }> (preload/index.ts:278-279) and decide() never reads the result, so the catch cannot see the failure: the modal closes with no error, the user believes recaps are enabled, the consent stays undecided and re-prompts next launch — violating the invariant stated in the modal's own line-23 comment. The MAIN/renderer dir-desync window is acknowledged in AppChrome.tsx:77 ('MAIN/renderer dir desync can leave askRecap=true with no project open'), but that guard gates only the RENDER (line 78, projectDir !== null), not the write result — a renderer-open/MAIN-null desync still reaches decide(). Narrow trigger → Low.

## Suggested fix direction
Check the reply: `const r = await setConsent(...); if (!r.ok) { setError(true); setBusy(false); return } onClose()`.
