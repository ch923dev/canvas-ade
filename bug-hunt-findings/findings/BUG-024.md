# BUG-024: adoptCore replaces a live same-id session without reaping it — the displaced proc escapes both maps and survives disposeAllPtys/quit (missing the symmetric Bug #13 guard)

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/main/pty.ts`, `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`adoptCore` does `sessionsMap.set(id, {proc: p.proc, ...})` without checking whether `sessions` already holds a live entry for that id. If both maps ever hold the same id (live session + parked session), an adopt silently replaces the live entry: the displaced proc is then referenced by NEITHER `sessions` NOR `parked`, so its own onExit cleanup no-ops (identity guard), `pty:kill` can't find it, and `disposeAllPtys` (which only drains the two maps) misses it — the child process tree is orphaned even past app quit. This is exactly the invariant the Bug #13 fix added for the spawn path (`if (sessions.has(opts.id)) void cleanup(opts.id)` at pty.ts:315), but the symmetric guard is absent on the adopt path.

## Where
`src/main/pty.ts:235` — `adoptCore` (the unguarded `sessionsMap.set(id, { proc: p.proc, ... })`)

## How it triggers
Requires sessions and parked to simultaneously hold one id: duplicate board ids in a hand-edited canvas.json (two mounted boards sharing an id, one deleted/parked while the other respawns), an MCP/e2e-driven `pty:adopt` against a live id, or any future caller that adopts without the renderer's adopt-before-spawn discipline.

## Verification evidence
Confirmed as the missing map-invariant guard the candidate describes. adoptCore (pty.ts:218-249) does `sessionsMap.set(id, {proc: p.proc, ...})` at line 235 with no `sessionsMap.has(id)` check — directly asymmetric to the spawn path's Bug #13 guard at pty.ts:315 (`if (sessions.has(opts.id)) void cleanup(opts.id)` with an explicit comment that an unguarded overwrite drops the proc out of BOTH maps so neither cleanup() nor disposeAllPtys() ever kills it). If sessions[id] is live when adopt fires, the displaced proc is then referenced by neither map: its own onExit teardown no-ops via the identity guards (onData pty.ts:362-363, onExit 383-384, cleanupCore's isStaleExit at 479), pty:kill→cleanup finds only the adopted entry, and disposeAllPtysCore (pty.ts:499-507) only drains the two maps — so the child tree outlives app quit. Reachability requires the abnormal dual-map state (sessions[id] AND parked[id]): the normal UI serializes adopt-before-spawn (TerminalBoard.tsx:495-505 spawns only when adopt resolves false), and fresh boards get new uuids — but duplicate board ids from a hand-edited canvas.json are NOT deduped on load (no board-id dedup exists in canvasStore/boardSchema; the 'dedup' helpers there cover groups/connectors/selection), and a delete(park)→restart-of-the-duplicate(spawn)→undo(adopt) sequence then lands an adopt onto a live id. Latent invariant gap, abnormal-data trigger, worst-leak-class consequence → Low as graded.

## Suggested fix direction
Mirror the #13 guard in adoptCore: if sessionsMap.has(id), cleanup the live entry before the set (one line plus a unit test against the injectable core).
