# BUG-024: Checklist auto-grow rewrites board height on mount as an untracked (non-beginChange) store mutation

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/PlanningBoard.tsx`, `src/renderer/src/canvas/boards/planning/ChecklistCard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
growForChecklist writes board.h via updateBoard with no beginChange(), driven by ChecklistCard's ResizeObserver report() that runs synchronously on mount. So rendering/loading a planning board whose stored h is smaller than the checklist's measured content height silently mutates the persisted height outside the undo system, unlike every sibling planning handler. The defect is real, but current impact is Low: autosave/persistence is not built yet (Phase 3), the write is guarded by needed > board.h (not "every mount"), and it only-grows so it is bounded and one-shot per load.

## Where
- src/renderer/src/canvas/boards/PlanningBoard.tsx:187-193 — growForChecklist (updateBoard with no beginChange); src/renderer/src/canvas/boards/planning/ChecklistCard.tsx:108-116 — ResizeObserver effect calls report() synchronously on mount; src/renderer/src/store/canvasStore.ts:120-142 — updateBoard records no past and clears future if armed

## How it triggers
Render/load (or HMR remount) a planning board that contains a checklist whose measured bottom (element.y + card offsetHeight) + 48px exceeds the stored board.h — e.g. a board manually resized below the checklist content, or whose offsetHeight measures larger than at save time due to font/DPI/zoom-independent layout drift. On that mount, ChecklistCard's effect runs report() immediately -> onMeasureBottom -> growForChecklist computes needed = ceil(bottom+34+14); since needed > board.h it calls updateBoard(board.id,{h:needed}). No beginChange() is taken, so the height change is invisible to undo; updateBoard also clears any armed redo branch. If the board was previously grown-to-fit and reloaded with identical offsetHeight, needed <= board.h and NO write occurs (idempotent in the common case), so it is not literally every mount.

## Verification evidence
Code-path trace on current (post-48c6500) code: ChecklistCard.tsx:108-116 — useEffect creates `const report = () => onMeasureBottom(element.id, element.y + el.offsetHeight)`, calls `report()` synchronously, then attaches `new ResizeObserver(report)`. PlanningBoard.tsx:476 passes `onMeasureBottom={growForChecklist}`. growForChecklist (187-193): `const needed = Math.ceil(bottom + TITLEBAR_H + WELL_PAD); if (needed > board.h) updateBoard(board.id, { h: needed })` — no beginChange(), confirmed by contrast with deleteEl (143), toggle (151), appendItem (168), dropItem (176), onWellDoubleClick (291) which all call beginChange() first. canvasStore.ts updateBoard (120-142) does NOT push to `past` (untracked by undo) and returns `{ boards, future: [] }` when future is armed (redo-branch wipe); it produces a new boards array ref (a real state change). git log -S confirms growForChecklist was ADDED by remediation commit 48c6500 as the fix for prior finding #13 (clip/auto-grow), so this is a new concern about that fix, not one of the original 50, and is distinct from the refuted ChecklistElement.h:0 finding (docs/bug-hunt-findings.md:805, which concerns the unused element-level h field, not top-level board.h). Refutation of overstated claims: (1) NO autosave/persistence exists — Grep for autosave|saveProject|write-file-atomic|subscribe in src/**/*.ts(x) found only BrowserPreviewLayer's view-sync subscribe and main-process before-quit cleanup; CLAUDE.md states file I/O is Phase 3 / unbuilt — so "schedules an autosave the user never initiated" is a future, not current, harm. (2) Not "every mount" — guarded by `needed > board.h`. boardSchema persists/round-trips h (toObject/fromObject, geometry validated 305-307), so the issue does escalate to a save side-effect once Phase 3 lands. No DOM/jsdom test covers this path (no PlanningBoard/ChecklistCard render test; canvasStore.test.ts has no growForChecklist/board.h/idempotent assertions).

## Suggested fix direction
Make the auto-grow height write a non-persisting / layout-only concern rather than a tracked-document mutation: either (a) route the grow through a dedicated store action that updates h WITHOUT touching past/future and WITHOUT marking the doc dirty (so future Phase-3 autosave ignores it), or (b) treat checklist height as derived layout (let the well auto-size to content / use min-height / scroll) so board.h is never rewritten from a measure. If the explicit h write is kept, debounce/coalesce it, only-grow, and ensure it never clears the redo branch (updateBoard currently wipes future). Revisit before Phase 3 persistence so load stays idempotent and the height bump never schedules an autosave or pollutes undo/redo.
