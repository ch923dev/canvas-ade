# BUG-016: TerminalBoard.restart() resolves its async spawn into a captured term with no disposed-guard, so a same-window unmount/respawn makes the spawn-failed branch write to a disposed xterm

- **Severity:** Low
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
restart() (TerminalBoard.tsx:315-342) captures `const term = termRef.current` and fires `spawnTerminal` as an async ipcRenderer.invoke round-trip; its .then() failure branch calls term.write(...) and setState('spawn-failed') with no closure-local `disposed` flag and no .catch(). If the component unmounts (board delete) or the spawn effect re-runs (config Apply changing shell/cwd/launchCommand) during the in-flight IPC, the spawn-effect cleanup disposes that exact captured term (line 305) and nulls termRef, after which the restart .then() calls write() on a disposed xterm Terminal whose core write is unguarded in @xterm/xterm 5.5. Real but narrow: only the spawn-failed branch touches term, and setState-after-unmount is a benign no-op in React 18.3, so impact is a swallowed async exception / no-op write to a torn-down buffer, not data loss/leak/crash on the happy path.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:315-342 — restart() useCallback; specifically the spawnTerminal().then() at 336-341 calling term.write (339) and setState (338) on the captured local `term` (316)

## How it triggers
1) Click Restart on a Terminal board. restart() runs synchronously: captures `term`, fires killTerminal, term.reset(), setState('spawning'), then fires `spawnTerminal` (ipcRenderer.invoke('pty:spawn') — a real MAIN round-trip, preload/index.ts:46) whose .then() is now pending. 2) During that window perform either (a) open Config and Apply a shell/cwd/launchCommand change — this flips the spawn-effect deps [board.id,board.shell,board.cwd,board.launchCommand] (line 309), so React runs the spawn-effect cleanup which executes term.dispose() (line 305) and termRef.current=null (306) and builds a fresh term; or (b) delete the board, unmounting TerminalBoard and running the same cleanup. 3) The pending restart spawn resolves with res.state==='spawn-failed' (reachable when pty.spawn throws on an invalid shell/cwd or transient failure, pty.ts:294-298). 4) The .then() then runs setState('spawn-failed') (no-op, unmounted) and term.write('\x1b[31mspawn failed...') on the now-disposed term — xterm core write (write(e,t){this._writeBuffer.write(e,t)}) has no _isDisposed guard, so it schedules an async _inputHandler.parse on torn-down services. Note: on a successful restart spawn the .then() does not touch term, so the happy path is unaffected.

## Verification evidence
Code-path trace against current code: (1) restart (TerminalBoard.tsx:315-342) captures term at 316, has NO `disposed` flag and NO .catch(); its .then() at 336-341 calls setState('spawn-failed') and term.write only when res.state==='spawn-failed'. (2) The spawn effect by contrast was deliberately hardened for this exact hazard: closure-local `let disposed=false` (236), `if (disposed) return` guarding its adoptTerminal().then() (266), and cleanup disposing+nulling term (304-306). restart was not mirrored. (3) The async window is genuine: window.api.spawnTerminal = ipcRenderer.invoke('pty:spawn',...) (preload/index.ts:46), a full round-trip to MAIN's pty:spawn handler (pty.ts:274) which runs pty.spawn (real process create) before returning — far longer than one tick, so a follow-up Config-Apply or delete easily lands mid-flight. (4) spawn-failed is reachable: pty.ts:294-298 returns state:'spawn-failed' when pty.spawn throws (bad shell path / cwd). (5) Config-Apply re-runs the spawn effect because board.shell/cwd/launchCommand are spawn-effect deps (line 309) wired via useEffect(()=>spawn(),[spawn]) (311); React runs the prior cleanup (term.dispose, 305) before the new effect — disposing the very term the in-flight restart .then() holds. (6) Verified in the bundled @xterm/xterm 5.5.0 (node_modules/@xterm/xterm/lib/xterm.js): public Terminal.write = write(e,t){this._core.write(e,t)} and CoreTerminal write = write(e,t){this._writeBuffer.write(e,t)} — NEITHER checks _isDisposed; Disposable.dispose only sets _isDisposed=!0 and disposes children, it does not replace write with a noop or null _writeBuffer, so write-after-dispose schedules _innerWrite→_inputHandler.parse on disposed services. Bounding facts: only the spawn-failed branch is harmful; React is 18.3.1 where setState-after-unmount is a silent no-op (warning removed), so that half is benign; net impact is a swallowed async exception/no-op write, hence Low. No prior bug-hunt entry covers this renderer-side write-after-dispose angle — the prior 'Terminal Restart respawns under same id' entries (docs/bug-hunt-findings.md:831-843, refuted) concern only the MAIN-process session-reap race, not the disposed-term .then().

## Suggested fix direction
Mirror the spawn effect's hardening onto restart(): either (a) drive Restart through the spawn-effect lifecycle rather than an ad-hoc useCallback, or (b) capture a per-invocation disposed/cancel flag (or re-read termRef.current === term) inside the .then() and bail if the captured term was disposed or the component unmounted, and add a .catch() symmetric to the spawn effect. A lightweight guard: before term.write/setState in the .then(), check `if (termRef.current !== term) return` so a respawn/unmount that replaced or nulled the ref aborts the stale write.
