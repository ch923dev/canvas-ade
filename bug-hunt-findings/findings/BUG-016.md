# BUG-016: endMotion ignores menuOpen — camera wheel-zoom while a popover is open reattaches native views over the still-open popover

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/usePreviewManager.ts`, `src/renderer/src/canvas/BoardFrame.tsx`, `src/renderer/src/store/previewStore.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
endMotion guards only `usePreviewStore.getState().nodeGesture` before clearing gestureRef and running applyLiveness. While a board ⋯ menu / GroupNamePopover / GroupFocusPicker / Tidy picker is open, the menuOpen flag drives beginMotion (views detached so HTML popovers are visible — the PREV-C invariant), but a wheel-zoom over the canvas fires useOnViewportChange onEnd → endMotion, which sees nodeGesture=false and reattaches every eligible live WebContentsView — painting the always-above native layer over the still-open popover and stealing input where it overlaps. It also clears gestureRef while the menu is still registered, so the store-subscription applyLiveness guard `!gestureRef.current` no longer holds for the rest of the menu session.

## Where
`src/renderer/src/canvas/boards/usePreviewManager.ts:701` — `endMotion` (inside the `usePreviewManager` hook; the `if (usePreviewStore.getState().nodeGesture) return` guard)

## How it triggers
Open a board ⋯ menu (or group name popover / focus picker) that overlaps a live Browser board's device stage, move the pointer onto empty canvas, and wheel-zoom. The menu closes only on pointerdown/Escape/window-resize (BoardFrame.tsx:206-222), not on wheel, so it stays open; React Flow's wheel zoom fires onStart/onEnd; onEnd reattaches the native view over the open menu.

## Verification evidence
Code-path confirmed in current source. usePreviewManager.ts:693-704: endMotion guards ONLY `usePreviewStore.getState().nodeGesture` before clearing gestureRef and running applyLiveness; menuOpen is read nowhere on that path (its only uses are the hook selector at :170 and the begin/end effect at :718-725, which treats `nodeGesture || menuOpen` as one motion source). previewStore.ts:128-136 confirms menuOpen is a ref-counted token set that stays true while ANY popover is open. BoardFrame.tsx:206-222 confirms the menu closes only on document pointerdown, Escape, or window resize — a wheel event closes nothing. So: open ⋯ menu → menuOpen=true → effect beginMotion detaches live views (the PREV-C invariant); wheel-zoom over the canvas → React Flow d3-zoom fires useOnViewportChange onEnd (:706) → endMotion sees nodeGesture=false → clears gestureRef → applyLiveness reattaches every eligible WebContentsView — the always-above native layer paints over (and steals input under) the still-open body-portaled popover, and chromeExclusionZones (:290-313) contains no menu rect to demote it. Nothing re-demotes until the menu closes. Severity Medium: visible occlusion + input steal in a perfectly ordinary interaction (menu open + wheel zoom), a regression of the PREV-C class.

## Suggested fix direction
Make endMotion yield to menuOpen the same way it yields to nodeGesture (or fold menuOpen into the condition that owns gestureRef), so only the menu-close path reattaches.
