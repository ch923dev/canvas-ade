# BUG-030: attachBoard's post-openPreview patchRuntime resurrects a cleared previewStore entry (with live:true) for a board deleted during the open await — the #37/#48 orphan-leak guard was not applied here

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

> **Consolidates 2 discovery findings** (same root cause, multiple call sites): `attachboard-post-await-patch-no-existence-recheck`, `preview-patchruntime-after-await-no-existence-recheck`

## Summary
The first-open branch of attachBoard (BrowserPreviewLayer.tsx) awaits window.api.openPreview, then unconditionally calls patchRuntime(g.id,{live:true}) with no recs.current.has(g.id) existence re-check. If the board is deleted during that IPC await, reconcile's removal loop runs closeBoard + recs.delete + clearRuntime, but the post-await patch re-creates the deleted id's byId entry (patch is create-if-absent). No reconcile branch ever cleans it again (its rec is gone), so it leaks. Because the resurrected entry carries live:true, it also permanently inflates the DiagOverlay live-view count via selectLiveCount — a strictly-worse-than-#37 symptom (which left live:false). This is the same orphan class the #37/#48 remediation fixed in beginMotion (line 359) and demoteToSnapshot (line 249) via recs.current.has(), but the guard was missed on attachBoard's open await.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:294-296 — attachBoard (the `await window.api.openPreview` at 294 followed by the unguarded `patchRuntime(g.id,{live:true})` at 296)
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:288-296 — attachBoard NEW-board branch (await openPreview at 294, unconditional patchRuntime({live:true}) at 296); leak surfaces against reconcile removal loop at lines 466-471 (closeBoard + recs.current.delete + clearRuntime)

## How it triggers
1) Create a Browser board while it is live-eligible (zoom ≥ LOD_ZOOM 0.4, on-pane, not occluding a selected board/chrome) so reconcile's `!r.exists && !r.attached` branch fires `void attachBoard(g)` (line 479). 2) attachBoard enters the `else` (!r.exists) branch: sets r.exists=true, r.lastUrl=g.url, patches {status:'connecting',live:true}, then suspends on `await window.api.openPreview(...)` — a real IPC round-trip to main (which actually creates the WebContentsView and loads the URL), i.e. a wide window. 3) During that await, press Backspace/Delete → removeBoard (canvasStore.ts:112) runs Zustand `set`, synchronously notifying the imperative subscription (line 559) → reconcile. The id is in recs.current but not in `seen`, so the removal loop (lines 466-472) runs closeBoard(id) (sets r.attached=false/r.exists=false on the shared rec object + IPC closePreview + patch {live:false}), recs.current.delete(id), clearRuntime(id) (deletes byId[id]). 4) openPreview resolves → attachBoard resumes at line 296 → patchRuntime(g.id,{live:true}) recreates byId[id] = {...DEFAULT_RUNTIME, live:true}. The rec was already deleted, so reconcile's removal loop never re-fires for this id and the entry leaks for the session, also incrementing selectLiveCount permanently.

## Verification evidence
Code-path trace against current (post-48c6500) code: attachBoard is lines 263-299; the open branch is 290-294 and the unconditional post-await patch is line 296 — there is NO recs.current.has(g.id) / geomRef re-check between the await (294) and the patch (296). Contrast: the sibling awaits added exactly this guard during the #37/#48 remediation — demoteToSnapshot line 249 `if (!recs.current.has(g.id)) return` and beginMotion line 359 `if (shots[i] && recs.current.has(g.id))` + line 367 `if (!r) return`. previewStore.patch (previewStore.ts:66-69) is create-if-absent: `byId:{...s.byId,[id]:{...DEFAULT_RUNTIME,...s.byId[id],...patch}}`, so patching a cleared id resurrects it. clearRuntime (previewStore.ts:70-76) deletes the key. reconcile's removal loop (lines 466-472) only iterates `recs.current.keys()` and only fires when id ∉ seen — after recs.delete it never sees the id again, so the resurrected byId entry is never cleaned. removeBoard (canvasStore.ts:112-118) uses Zustand set → synchronous subscriber notification; the subscription (lines 559-567) calls reconcile unconditionally (no gesture guard around reconcile). openPreview is `ipcRenderer.invoke('preview:open',args)` (preload/index.ts:58-63) → a genuine async suspension. selectLiveCount (previewStore.ts:86-90) counts live===true and feeds Canvas.tsx:81 → DiagOverlay, so the live:true ghost inflates that metric — unlike #37, whose ghost was live:false (per docs/bug-hunt-findings.md line 539, which explicitly notes #37 does NOT inflate the live count). docs/bug-hunt-findings.md #37 (line 536-541) is the matching class and its fix recommendation names only beginMotion + demoteToSnapshot, never attachBoard — confirming attachBoard was overlooked.

— also confirmed (consolidated finding preview-patchruntime-after-await-no-existence-recheck): Code-path trace verified against current source: (1) previewStore.ts:66-69 patch is create-if-absent (spreads DEFAULT_RUNTIME for an absent id); clear (70-76) deletes the entry. (2) preload index.ts:58-63 openPreview = ipcRenderer.invoke('preview:open') → genuine async suspension. (3) canvasStore.ts:112-118 removeBoard is a synchronous set, so the subscription at BrowserPreviewLayer.tsx:559 runs synchronously during the await window. (4) reconcile removal loop (466-471) gated on recs.current.keys(): closeBoard sets r.attached/r.exists=false, then recs.current.delete(id), then clearRuntime(id).

## Suggested fix direction
Mirror the #37/#48 guard already present in demoteToSnapshot (line 249) and beginMotion (line 359): after the `await window.api.openPreview(...)` in attachBoard's open branch, re-check existence before any further mutation — e.g. `if (!recs.current.has(g.id)) return` immediately after the await, guarding the trailing `patchRuntime(g.id,{live:true})` at line 296 (and ideally only patching live:true once, not both before and after the await). Optionally also capture `const r = rec(g.id)` and bail if `!r.exists`/the rec was swapped, since closeBoard mutates the shared rec during the await. (Separately note for the dev: the open IPC was issued before closePreview on the same FIFO invoke channel, so the main-side native view may also need a close-ordering check — but the renderer-side orphaned previewStore entry is the confirmed defect here.)

Consolidated site (preview-patchruntime-after-await-no-existence-recheck): Mirror the sibling guard: after `await window.api.openPreview(...)` in attachBoard's NEW-board branch, re-check `if (!recs.current.has(g.id)) return` (and/or that r is still the live rec) before the final patchRuntime(g.id,{live:true}) at line 296, so a board removed during the open IPC round-trip is not resurrected. Alternatively make previewStore.patch a no-op for an absent id (track existence), which would close this entire defect class at the store boundary rather than at each await site. Low priority: the only observable effect is an orphaned in-memory previewStore.byId entry with live:true on an uncommon delete-while-opening interleaving, which slightly inflates the dev-only DiagOverlay live count; no functional misbehavior, crash, or data loss, and crypto.randomUUID ids are never reused so no live board ever reads the ghost.
