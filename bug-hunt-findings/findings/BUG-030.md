# BUG-030: bus.once + foreign-frame early-return cannot 'ignore and keep waiting' — one foreign event on the reply channel consumes the listener and forces the backstop deny/timeout

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/mcpConfirm.ts`, `src/main/mcpCommand.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
requestConfirm registers the reply listener with `bus.once(replyChannel, onReply)` (mcpConfirm.ts:105) and onReply early-returns on a foreign sender WITHOUT re-registering (line 89). Because `once` removes the listener on the FIRST event regardless of what the handler did, any event delivered on the reply channel from a non-main frame consumes the one-shot listener; the genuine human reply that follows is never handled, so the confirm falls through to the backstop timeout and resolves DENIED. The comment's intent ("a foreign frame can't decide — ignore") is not achieved. The identical pattern exists in mcpCommand.ts (bus.once line 75 + foreign early-return line 67), where it yields a spurious {ok:false,error:'timeout'} ack.

## Where
`src/main/mcpConfirm.ts:105` — `requestConfirm()` (the `bus.once(replyChannel, onReply)` registration; mirror pattern in `sendMcpCommand()`, mcpCommand.ts:75)

## How it triggers
Any non-main-frame IPC event arriving on the per-request reply channel before the human's real reply. Requires a sender that can emit ipcRenderer.send to that channel.

## Verification evidence
Mechanism confirmed by code + live repro. mcpConfirm.ts:105 registers `bus.once(replyChannel, onReply)`; onReply (:89) early-returns on `isForeignSender` WITHOUT re-registering. Node EventEmitter `once` removes the wrapper before invoking the handler — demonstrated with a minimal repro: after emitting one foreign event into a `once` handler that early-returns, `listenerCount === 0` and a subsequent genuine emit is never handled (output: 'listeners after foreign event: 0 / result: pending'). So a single foreign-frame event makes the genuine human reply unreachable; the confirm resolves DENIED only via the 10-min backstop (:101-103), contradicting the comment's 'a foreign frame can't decide — ignore' intent. Identical pattern in mcpCommand.ts (:75 bus.once, :67 early-return) → spurious {ok:false,'timeout'} ack. The test suite does NOT contradict: mcpCommand.integration.test.ts's fakeBus `once` (lines 31-40) does not auto-remove the handler (masking real EventEmitter semantics), and the only foreign-frame test (:68-76) never sends a genuine reply after the foreign one. Severity Low is correct: fail mode is fail-closed (deny, never approve), and no foreign IPC emitter exists today — preview WebContentsViews have no preload/ipcRenderer (preview.ts:424 states this explicitly) and the confirm channel is an unguessable randomUUID (:74).

## Suggested fix direction
Use bus.on + explicit removeListener in finish() so a foreign event is genuinely ignored while the wait stays armed (both mcpConfirm.ts and mcpCommand.ts).
