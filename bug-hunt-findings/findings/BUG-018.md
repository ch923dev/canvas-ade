# BUG-018: Pre-push hook skips the entire e2e matrix on a branch's FIRST push — `git diff <single-sha>` diffs against the working tree, not the pushed commits

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `.githooks/pre-push`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
When the remote ref doesn't exist yet (`remote_sha` is all-zeros), the hook sets `range="$local_sha"` and then runs `git diff --name-only --diff-filter=ACMR "$range"`. With a single commit-ish, `git diff` compares that commit against the WORKING TREE, not against any base — on a clean tree at the pushed tip the diff is empty, so `changed` is empty and the hook prints 'no new commits to test — skipping e2e' and exits 0. Since e2e was deliberately removed from GitHub Actions, the very push that creates a feature branch on origin (the standard PR flow here) reaches origin completely e2e-ungated; on a dirty tree the docs-only classification is computed from the wrong file set in both directions.

## Where
`.githooks/pre-push:24` — the new-branch case in the stdin ref loop (`range="$local_sha"` inside the `while read -r local_ref local_sha remote_ref remote_sha` loop; consumed by the `git diff` at line 28)

## How it triggers
First `git push -u origin feat/x` of any new branch with a clean working tree — the hook self-skips the matrix; later pushes to the same branch gate normally.

## Verification evidence
Code path: .githooks/pre-push:23-24 — `if [ "$remote_sha" = "$z40" ]; then range="$local_sha"` — then line 28 runs `git diff --name-only --diff-filter=ACMR "$range"`. Demonstrated in a temp repo (Z:\ not touched): with two commits and a CLEAN tree, `git diff --name-only --diff-filter=ACMR <HEAD-sha>` printed NOTHING (output `[]`), so `changed` is empty and lines 38-41 print 'no new commits to test — skipping e2e' and exit 0. Also demonstrated the dirty-tree corollary: modifying a tracked file made the same command print the working-tree file (`[src.ts]`) — i.e. classification runs against uncommitted local edits, not the pushed commits. Since e2e was deliberately removed from GitHub Actions (CLAUDE.md Status: 'e2e does NOT run in GitHub Actions'), the first `git push -u origin feat/x` of every feature branch — the standard PR flow in this repo — reaches origin completely e2e-ungated. The repo's own memory note (`pre-push-skips-e2e-new-branch`) records this firing in practice, and no fix exists on main. Correct intent for a new branch is e.g. `origin/main..$local_sha` or `$local_sha --not --remotes=origin`.

## Suggested fix direction
For the all-zeros remote case, diff against a real base: `git merge-base origin/main $local_sha` (fallback: `git rev-list $local_sha --not --remotes=origin` for commit-list semantics), never a bare single sha.
