# BUG-018: did-finish-load can flip an over-cap-evicted (closed, non-live) Browser board to 'connected' though no native view is attached

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The renderer's did-finish-load handler in BrowserPreviewLayer only guards against the 'load-failed' status; it never reconciles against the live/attached/exists lifecycle. When a still-loading Browser board is evicted over MAX_LIVE (closeBoard sets live:false and exists:false but leaves status at 'connecting'), a did-finish-load that fires on main before/just as the view is closed is delivered to the renderer and flips the board's status to 'connected'. The board then shows a green 'connected' status pill + conn-dot over a detached snapshot with no live view, and the wrong status persists until the board is re-attached or removed.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:595-603 (onPreviewEvent did-finish-load handler) and 302-313 (closeBoard); main emit at src/main/preview.ts:172-179

## How it triggers
Have 5+ live-eligible Browser boards (zoom >= LOD, on-pane) so MAX_LIVE=4 eviction applies. A board that was attached in a prior applyLiveness pass (status set to 'connecting' in attachBoard:293, openPreview issued, page still loading) is later evicted by applyLiveness when the user pans/selects/focuses so a closer board takes its slot: liveEligible && !occludesProtected true but pickLive (nearest 4 to viewport center) drops it, so the else branch runs closeBoard(g.id) (BrowserPreviewLayer.tsx:406). closeBoard synchronously sets r.exists=false/r.attached=false, fires void window.api.closePreview(id), and patchRuntime(id,{live:false}) — it does NOT change status (still 'connecting'). If the page's did-finish-load fires on main in the window after the renderer ran closeBoard but before/around when disposeOne actually closes the webContents (or while the emitted preview:event push is in-flight), the renderer handler runs with cur==='connecting' (not 'load-failed') and patchRuntime(id,{status:'connected'}). The board now reads 'connected' with live:false and only a (possibly stale) snapshot visible.

## Verification evidence
Code-path trace on current post-48c6500 code. (1) Handler BrowserPreviewLayer.tsx:601-603: `const cur = usePreviewStore.getState().byId[ev.id]?.status; if (cur === 'load-failed') return; patchRuntime(ev.id, { status: 'connected', liveUrl: ev.url, error: null })` — the ONLY guard is load-failed; there is no check of runtime.live, r.attached, or r.exists. (2) closeBoard (302-313) sets attached/exists false + patchRuntime(id,{live:false}) but never writes status, so status stays at the 'connecting' set by attachBoard:293. (3) preload onPreviewEvent (src/preload/index.ts:91-95) is a raw pass-through — every preview:event reaches the handler regardless of view existence. (4) main disposeOne (preview.ts:255-265) does detach + webContents.close() with no listener removal and no in-flight-event suppression; did-finish-load (preview.ts:172-179) emits whenever `!e.failed`, i.e. for a successful load. (5) Status pill/conn-dot in BrowserBoard.tsx:295-317 are driven purely by runtime.status, independent of runtime.live, so a connected status with live:false renders a green --ok dot+pill over the DeviceContent snapshot fallback. (6) Self-heal is limited: reconcile (474-486) only resets status when it is 'idle' (`(status ?? 'idle') === 'idle'`), so a board stuck on 'connected' while over-cap/detached is NOT corrected by reconcile; it stays wrong until it wins a live slot again (re-attach → correct connecting→connected) or is removed. The did-finish-load handler is a React component callback and is not unit-tested (preview.test.ts covers only pure functions; previewPlan.test.ts covers pickLive). Prior Bug #5 remediation (docs/bug-hunt-findings.md:153-183) added only the load-failed latch (main `!e.failed` + renderer belt-and-suspenders) and Bug #9 added pickLive visibility ranking — neither reconciled did-finish-load with the live/attached lifecycle, so the close-vs-load race is a genuine gap, not a re-report.

## Suggested fix direction
In the did-finish-load handler, additionally reconcile against the lifecycle before promoting to 'connected': skip the promotion when the board has no live native view (e.g. guard on `recs.current.get(ev.id)?.exists` / `?.attached`, or on the current runtime `live` flag, or ignore the event when there is no rec entry). Symmetrically this also prevents resurrecting a cleared runtime entry for a just-deleted board (closeBoard+clearRuntime path in reconcile:466-472), aligning the handler with the existing Bug #45/#48 recs-existence guards used in demoteToSnapshot/beginMotion.

## Prior-hunt relation
#5/#8: the prior status-machine remediation latched did-finish-load only against load-failed (Bug #5) but never reconciled did-finish-load with the live/attached lifecycle, so a connected status can be applied to an evicted/closed board — the terminal-status fix is incomplete for the close-vs-load race.
