- **Severity:** Medium
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/main/preview.test.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

> **Consolidates 3 discovery findings** (same root cause, multiple call sites): `preview-will-redirect-scheme-guard-missing`, `preview-will-redirect-not-guarded`, `preview-will-navigate-misses-subframes`

## Summary
The native Browser-preview view installs an in-page navigation scheme guard on `will-navigate` only (preview.ts:159-161). Per Electron 33's documented event model, `will-navigate` does NOT fire for the redirect leg of a navigation; server-side (302) redirects fire the separate cancellable `will-redirect` event, which is not guarded on the preview webContents. So a previewed localhost origin that responds to a navigation with `302 Location: file:///...` (or `data:`/`myapp://`) loads that disallowed scheme into the sandboxed preview view, defeating the "preview is http(s)-only / never a file viewer or general browser" trust boundary. This is an INCOMPLETE remediation of prior bug #36, which added only the `will-navigate` guard.

## Where
- src/main/preview.ts:159-161 — ensure(), the sole `wc.on('will-navigate', ...)` scheme guard on the preview WebContentsView (no `will-redirect`/`will-frame-navigate` listener); contrast src/main/index.ts:89-90 which guards both `will-navigate` and `will-redirect` on the main window.
- src/main/preview.ts:159-161 — ensure(): wc.on('will-navigate', ...) is the only navigation-scheme guard registered on the preview webContents; no wc.on('will-redirect', ...) or wc.on('will-frame-navigate', ...) handler exists anywhere in the file. Compare src/main/index.ts:89-95 (main window guards will-navigate + will-redirect + will-frame-navigate).
- src/main/preview.ts:159-161 — ensure(), the wc.on('will-navigate', ...) scheme guard on the preview WebContentsView's webContents

## How it triggers
A Browser board previews a localhost dev server (or any page served on that origin). The page (or a malicious/compromised dependency serving it, or a malicious page already on that origin) initiates a top-level navigation whose server response is `HTTP 302 Location: file:///C:/Users/.../secret` (or `data:text/html,...`, or a registered custom protocol). The initial http(s) URL passes the `will-navigate` guard (allowed); the 302 then fires `will-redirect`, which has NO listener, so the navigation commits and the preview view renders local-filesystem / data-URL content. The renderer-initiated load paths (preview:open line 298, preview:navigate line 365) are not involved — they only gate `loadURL`, so the in-page redirect never passes through `isAllowedPreviewUrl`.

## Suggested fix direction
In ensure() (preview.ts, alongside the existing will-navigate listener at 159-161), also attach `wc.on('will-redirect', (ev, url) => { if (!isAllowedPreviewUrl(url)) ev.preventDefault() })`, and for completeness `wc.on('will-frame-navigate', (details) => { if (!isAllowedPreviewUrl(details.url)) details.preventDefault() })` to cover subframe redirects — mirroring the main-window guard pattern at index.ts:89-95. Add a wiring test (or at minimum keep the isAllowedPreviewUrl unit coverage) that asserts a disallowed-scheme redirect URL is rejected. Severity is Medium not High because the preview view is sandboxed (sandbox/contextIsolation/nodeIntegration off, per-board partition) so there is no Node reach; the realistic harm is a file:/data: page loaded via a 302 from the user's own (or a compromised/dependency-served) localhost origin reading + exfiltrating local files reachable from the file: origin — a same-origin disclosure / contract violation, not a remote drive-by, and board.url is user-typed today (Phase 3 untrusted import not yet built, which would raise the impact).

Consolidated site (preview-will-redirect-not-guarded): Mirror the main-window hardening (index.ts:89-95) onto the preview view: in ensure(), extract the will-navigate body into a shared handler and also register it on wc.on('will-redirect', ...) and wc.on('will-frame-navigate', ...) (gated on isMainFrame for the latter, matching index.ts:93-95) so isAllowedPreviewUrl is enforced on server-initiated redirects and subframe navigations too. Because preventDefault() on a blocked redirect aborts with ERR_ABORTED (-3, already filtered at preview.ts:221), consider emitting an explicit terminal did-fail-load (errorCode -1, 'blocked scheme', mirroring preview:open/preview:navigate at lines 301-308 / 365-374) when a redirect is blocked, so the board resolves to "load-failed" instead of potentially hanging on "connecting". Add a small wiring test (or extend preview.test.ts) that drives a fake webContents emitter through will-redirect to assert preventDefault is called for a disallowed scheme.

Consolidated site (preview-will-navigate-misses-subframes): In ensure() (preview.ts), add a will-frame-navigate handler mirroring the main window's index.ts:93-95: `wc.on('will-frame-navigate', (details) => { if (!isAllowedPreviewUrl(details.url)) details.preventDefault() })`. This covers both main-frame and subframe cross-document navigations via the single richer event (it also subsumes the existing will-navigate for the main frame, though keeping both is harmless). Add a test alongside the existing isAllowedPreviewUrl cases asserting the handler is registered / that a blocked subframe URL is prevented. Consider also documenting that programmatic loadURL is already gated at the IPC boundary so the remaining surface is page-driven cross-document subframe navigation only.
