# BUG-014: No primary-button guard in the whiteboard pointer machine — right-click erases/creates/moves elements and wipes selection

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/planning/usePlanningPointer.ts`, `src/renderer/src/canvas/boards/planning/NoteCard.tsx`, `src/renderer/src/canvas/boards/planning/FreeText.tsx`, `src/renderer/src/canvas/boards/planning/ChecklistCard.tsx`, `src/renderer/src/canvas/boards/planning/ImageCard.tsx`, `src/renderer/src/canvas/boards/planning/WhiteboardSvg.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
onWellPointerDown, onWellPointerMove, onWellPointerUp, and startElementDrag never check e.button, and neither do any of the element-card pointerdown handlers (NoteCard grip, FreeText grip, ChecklistCard body, ImageCard, WhiteboardSvg vector paths), so every tool action fires for right- and middle-button presses. With the erase tool a right-press hit-tests and marks elements and the right-button pointerup COMMITS the deletion; note/check tools create-and-commit on right-press; pen/arrow/text commit on right-drag; and in select mode a right-press starts a captured move drag while a right-press on the empty well arms a marquee whose no-move pointerup calls clearSel(), wiping the selection. That last path undermines the BUG-013 context-menu fix: onWellContextMenu deliberately preserves a partial group selection on empty-space right-click, but the marquee click path triggered by the same right-click clears it anyway.

## Where
`src/renderer/src/canvas/boards/planning/usePlanningPointer.ts:187` — `onWellPointerDown` (inside the `usePlanningPointer` hook; also `startElementDrag` at :151, `onWellPointerMove`, `onWellPointerUp`)

## How it triggers
With the erase tool active, right-click an element (element is committed-deleted on button release). Or right-button-drag an element in select mode (it moves and commits). Or right-click empty well space while holding a partial selection (selection cleared). Or right-click with note/check tool armed (element created).

## Verification evidence
Confirmed by code-path trace. usePlanningPointer.ts onWellPointerDown (187-261), onWellPointerMove (263-320), onWellPointerUp (377-499), and startElementDrag (151-184) contain zero e.button/e.buttons checks; same for the card pointerdown handlers (NoteCard.tsx:93-99/112-152, ChecklistCard.tsx:150-155, ImageCard.tsx:151-158). A repo-wide grep for '.button' confirms the ONLY primary-button guard in the renderer is terminalSelection.ts:106 ('right/middle: let xterm + the React onContextMenu handle it'). Concrete faulty paths: (a) erase tool — a right-press hit-tests + marks (241-244) and the right-button pointerup commits the delete (426-433) before the contextmenu event even fires (Windows fires contextmenu after pointerup), so right-clicking an element with the eraser deletes it; (b) note/check tools — right-press immediately beginChange+commits a new element (207-221); (c) select tool — a right-press on a card runs onSelect+onDragStart (e.g. ChecklistCard.tsx:150-155, no button check), so a right-drag moves and commits; (d) select tool on empty well — right-press arms a marquee (198-204) and the no-move pointerup calls clearSel() (448-451), wiping the partial selection that onWellContextMenu (357-364, BUG-013) deliberately preserves. The existing BUG-013 test (PlanningBoard.interaction.test.tsx:180-183) dispatches only a synthetic 'contextmenu' MouseEvent, never a right pointerdown/up, so the path is uncovered.

## Suggested fix direction
Early-return on e.button !== 0 in onWellPointerDown/startElementDrag and the element-card pointerdown handlers (the terminalSelection.ts:106 convention).
