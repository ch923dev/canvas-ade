# BUG-008: A rejected capturePage() aborts the renderer detach pass — native preview keeps painting above all HTML (demoteToSnapshot + beginMotion)

- **Roadmap:** none

> **Consolidates 2 discovery findings** (same root cause, multiple call sites): `preview-capturepage-reject-aborts-demote-detach`, `beginmotion-promiseall-capture-no-catch-skips-detach`

## Summary
demoteToSnapshot awaits window.api.capturePreview(g.id) at BrowserPreviewLayer.tsx:243 with no try/catch, and the main-side preview:capture handler (preview.ts:330) calls webContents.capturePage() unguarded. On a GPU-contended / headless / multi-Electron-instance host (a condition the codebase itself documents and try/catch-guards in debugCaptureView, preview.ts:422-431), capturePage rejects, the rejected IPC throws in the renderer, and execution unwinds past the detach — so detachPreview is never sent, r.attached stays true, and the live flag is never cleared. The native WebContentsView the demote was meant to remove stays attached as a child of contentView, painting above all HTML and re-introducing the occlusion the demote path exists to prevent. The rejection is a renderer-side unhandled rejection that the main-process unhandledRejection hook cannot catch.

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:243 (await capturePreview, no try/catch) — demoteToSnapshot; caller at :407 uses `void demoteToSnapshot(g)`; main handler at src/main/preview.ts:330 (await capturePage, unguarded)
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:353 — beginMotion async IIFE (the unguarded `await Promise.all`); main side: src/main/preview.ts:330 — `preview:capture` handler's `await e.view.webContents.capturePage()` with no try/catch (contrast the guarded debugCaptureView at preview.ts:419-432)

## How it triggers
A camera gesture end, node drag/resize end, or selection/focus change runs applyLiveness(), which for an LOD/over-cap/occluding/unfocused attached Browser board calls `void demoteToSnapshot(g)` (line 407). Inside, line 243 awaits capturePreview. If the host cannot composite a surface — headless, GPU lost/contended, or several Electron instances at once (e.g. the parallel-agent/e2e setup, exactly the condition preview.ts:425-431 names) — main's capturePage() rejects, the IPC promise rejects, and the await throws. Lines 249-255 (the Bug #45/#48 re-checks, await detachPreview, r.attached=false, patchRuntime live:false) are all skipped. The native view is never removeChildView'd, so it stays attached over the board/other boards/in-canvas chrome. Transient rejections self-heal on the next gesture; a persistent capture-failure condition leaves the view attached and occluding indefinitely, and the previewStore live flag stays stuck at true.

## Suggested fix direction
Make the demote path robust to a capture rejection so the detach always runs. Wrap the line 243 await in try/catch (or treat a rejected/empty capture as url=null) and proceed to the detach + state-clear regardless of capture success — a missing snapshot is acceptable (the prior snapshot fallback stays), but the native view MUST still be detached. Apply the same to beginMotion's Promise.all(capturePreview) at line 353 (use Promise.allSettled or per-item try/catch) so a single capture failure does not abort the whole detach loop. Defensively, also guard capturePage() in the main preview:capture handler (preview.ts:330) in a try/catch returning null on reject (mirror debugCaptureView at 422-431) so the IPC resolves to null instead of rejecting. Optionally add a renderer-side .catch on the `void demoteToSnapshot(g)` call at line 407 to avoid the unhandled-rejection noise.

Consolidated site (beginmotion-promiseall-capture-no-catch-skips-detach): Make the capture pass resilient so one failure can't block the others' detach: replace `Promise.all` with `Promise.allSettled` (or per-board `capturePreview(...).catch(() => null)`) in beginMotion so a null/failed capture leaves that board live while the rest still snapshot+detach, mirroring the existing 'never strand on the bare cutout' intent. Independently/defensively, wrap `await e.view.webContents.capturePage()` in preview:capture (preview.ts:330) in a try/catch that returns null on rejection — exactly as debugCaptureView already does — so a capture failure degrades to 'no snapshot' instead of an unhandled renderer rejection. Note that detaching without a fresh snapshot risks a blank frame, so the preferred behavior is: keep the board live when its capture fails (don't detach it) rather than detaching it blind; allSettled with a per-board guard achieves this. Apply the same allSettled hardening to FlowSmoke.tsx:319 if that path is still exercised.
