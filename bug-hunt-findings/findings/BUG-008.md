# BUG-008: Autosaver clears `dirty` before the save and never re-arms it on failure — a failed autosave's edits are silently unrecoverable by flush

- **Severity:** Medium
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/store/useAutosave.ts`, `src/renderer/src/store/useAutosave.test.ts`, `src/renderer/src/canvas/AppChrome.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`createAutosaver`'s `run()` sets `dirty = false` BEFORE invoking `save()`, and on rejection or a `false` result it only calls `onError` (console.error) without restoring `dirty = true`. After one failed save (transient AV/file lock, ENOSPC), every subsequent flush — window blur, beforeunload, and critically MAIN's `project:flush` quit handshake — hits the `if (!dirty …) return` guard and writes nothing. If the user makes no further store mutation before quitting, the tail edits are permanently lost even though the disk may have recovered seconds later; SAVE-1 made the failure visible but not retryable.

## Where
`src/renderer/src/store/useAutosave.ts:35` — `run()` inside `createAutosaver()` (the `dirty = false` at line 35 precedes `opts.save()`; the failure paths at lines 41 and 43 never restore it)

## How it triggers
Edit boards → autosave fires while the disk transiently fails (locked canvas.json, ENOSPC) → project:save returns false / rejects → make no further edits → quit (or blur). The quit flush no-ops because dirty is false; last edits lost.

## Verification evidence
src/renderer/src/store/useAutosave.ts:32-44 — run() sets `dirty = false` (line 35) BEFORE `opts.save()`, and both failure paths (`ok === false` line 41, `.catch` line 43) only call onError; nothing restores dirty=true. flush() (lines 51-54) calls run(), which returns at line 34 `if (!dirty || status !== 'open') return Promise.resolve()` — so after one failed save with no subsequent store mutation, every later flush (blur line 114, beforeunload line 115, and MAIN's project:flush quit handshake line 123 / BUG-M2) writes nothing, even if the disk recovered. useAutosave.test.ts:70-88 (the two SAVE-1 tests) assert only that onError is called once — there is no test that a later flush retries after a failure, confirming no retry path exists. The quit path has no other writer: index.ts's flush handshake relies solely on saver.flush(). Transient write failures (AV/sync-tool file lock on canvas.json, ENOSPC) are realistic on Windows; the tail edits since the last good save are then silently lost on quit.

## Suggested fix direction
Restore dirty=true in the failure paths (or only clear dirty after the save resolves true), optionally with backoff to avoid hot-looping on a persistently failing disk.
