# BUG-062: stackCenteredRows retains the Math.max(0, ...widths) spread — residual site of the spread-RangeError class already fixed twice in the same file

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/lib/tidyLayout.ts`, `src/renderer/src/lib/groupReflow.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The 2026-06-07 hunt's BUG-015 ('tidyLayout spread-based Math.min/max throws RangeError on very large board counts, disabling all tidy modes', status: fixed) replaced the spreads in tidyLayout()'s origin scan and gridLayout's maxW scan with linear loops — both sites carry comments saying so. But stackCenteredRows (the row-centering core of smart mode, also reached via groupReflow) still does `Math.max(0, ...widths)` where widths.length scales with the number of preview clusters + fixed rows, i.e. O(board count). On an extreme canvas the same 'RangeError: Maximum call stack size exceeded' returns, disabling smart tidy — the exact failure BUG-015 was filed for.

## Where
`src/renderer/src/lib/tidyLayout.ts:98` — `stackCenteredRows()` (the row-centering core used by `smartLayout`)

## How it triggers
Smart tidy (or group reflow) on a canvas with enough linked terminal→browser clusters to exceed V8's argument ceiling (~65k clusters ≈ 130k boards) — roughly 2x the board count BUG-015 needed, since each cluster contributes one row.

## Verification evidence
tidyLayout.ts:98 is `const maxW = Math.max(0, ...widths)` where widths.length === rows.length, and smartLayout (lines 141-154) pushes one row PER linked preview cluster — O(board count). The same file carries two deliberate fixes of exactly this class with comments: gridLayout lines 201-203 ('Linear scan instead of Math.max(...spread) — avoids V8's argument-count ceiling on large sets') and the origin scan lines 241-248 (same comment). Reachability: smart is the DEFAULT mode (tidyLayout.ts:232, 256-258), reached from the store's tidyBoards (canvasStore.ts:591) and from groupReflow.ts:16 (`tidyLayout(members, { mode: 'smart' })`). Mechanism demonstrated: `node -e "Math.max(0, ...new Array(130000).fill(1))"` throws RangeError: Maximum call stack size exceeded (65k args still passes), so ~130k clusters (~260k boards) reproduce the exact failure the fixed sites prevent — and the threshold drops inside a deep React/Zustand call stack. Confirmed as an incomplete fix / residual same-class hazard; severity stays Low (the trigger board count is extreme, consistent with how the project graded the fixed siblings).

## Suggested fix direction
Replace the spread with the same linear max scan used at lines 201-203 / 241-248.
