# BUG-027: fromObject/migrate return the input doc by reference, so loadObject's store boards alias the caller's input (load-path clone asymmetry vs toObject)

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/lib/boardSchema.ts`, `src/renderer/src/store/canvasStore.ts`
- **Roadmap:** partially covered by Phase 3: Project create / open — canvas.json load + migrations + full-reopen-fidelity test (see cross-ref)
- **Assigned:** _(left blank for the dev team)_

## Summary
migrate() at SCHEMA_VERSION=1 with an empty MIGRATIONS map returns the exact input object by reference, fromObject returns that same object, and canvasStore.loadObject assigns fromObject(doc).boards straight into Zustand state. The store therefore aliases the caller's input array and element objects, so any out-of-band mutation of the source doc mutates live store state, bypassing set(). This is the documented inverse of toObject(), which deliberately structuredClones. The mechanism is real and reproducible, but no production caller of loadObject exists yet (file I/O is deferred to Phase 3), and the only realistic future caller — loadObject(JSON.parse(fileText)) — produces a freshly-parsed doc nobody else retains, so the aliasing is presently harmless. Genuine latent contract asymmetry, not currently triggerable in production.

## Where
- src/renderer/src/lib/boardSchema.ts:209-216 (migrate — `let d = doc` returned unchanged when schemaVersion === SCHEMA_VERSION), src/renderer/src/lib/boardSchema.ts:336-342 (fromObject — returns migrate(doc) with no clone; assertBoard only reads), src/renderer/src/store/canvasStore.ts:177-178 (loadObject — set({ boards: fromObject(doc).boards }))

## How it triggers
A caller must (a) invoke loadObject(doc) with a doc it still holds a reference to AND (b) later mutate doc.boards[i] in place. Then the store's live board object mutates out-of-band with no Zustand set()/re-render. Today there is NO such caller: loadObject is referenced only by canvasStore.test.ts:165 and the discard-only e2eHooks.roundTripOk(); no MAIN-side canvas.json read/IPC load path exists. The natural Phase-3 caller loadObject(JSON.parse(text)) does not retain the parsed object, so it would not trigger this. Triggering requires future code that reuses the parsed doc (in-memory hand-off, undo-snapshot reuse) — speculative at present.

## Verification evidence
Code trace: boardSchema.ts:209 `let d = doc`; the `while (d.schemaVersion < SCHEMA_VERSION)` loop body never runs at v1 (1 < 1 is false) and MIGRATIONS is `{}`, so migrate returns the original `doc` by reference (line 215). fromObject (336-342) validates via assertBoard (read-only, no copy) then returns migrate(doc) — same reference. loadObject (canvasStore.ts:178) assigns `.boards` of that reference into state. Asymmetry is intentional-by-contract: toObject (boardSchema.ts:187) `structuredClone(boards)` with comment 'so the doc owns its data' and a dedicated test (boardSchema.test.ts:157-162 'deep-clones boards so mutating the doc does not touch the source'); the load path has no equivalent clone and no equivalent test. Throwaway temp-dir repro (deleted) of the migrate/fromObject logic printed: `result === doc` true, `result.boards === doc.boards` true, `result.boards[0] === doc.boards[0]` true, and mutating doc.boards[0].x=9999 changed result.boards[0].x to 9999 (expected 0 if isolated). Grep confirms no production loadObject caller and no canvas.json/JSON.parse/readFile load wiring in src/main. Not present in docs/bug-hunt-findings.md (prior structuredClone refs concern toObject and round-trip only), so not a duplicate of the remediated 50.

## Suggested fix direction
Make the load path own its data symmetrically with toObject: structuredClone on the load path — either clone in loadObject (`set({ boards: structuredClone(fromObject(doc).boards) })`) or, more robustly, have fromObject/migrate return a cloned document so the no-migration (already-current) case still deep-copies (clone `doc` once at the top of migrate or in fromObject before returning). Add a regression test mirroring boardSchema.test.ts:157-162 asserting that mutating the input doc after loadObject/fromObject does not alter store/returned boards. Low priority until a Phase-3 caller that retains the parsed doc actually exists.
