# BUG-027: localServer removes its only 'error' listener after listen — a runtime accept error (EMFILE/ENFILE) becomes an unhandled 'error' event that exits the whole app via the uncaughtException sink

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/localServer.ts`, `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
startLocalServer attaches `server.once('error', onError)` to cover listen()-time failures, then removes it inside the listening callback — leaving the server with ZERO 'error' listeners for the rest of the app's lifetime. Node's net.Server emits 'error' on the server object for accept-time failures (classically EMFILE/ENFILE under fd pressure); an 'error' event with no listener throws synchronously into the process-level uncaughtException sink, which calls crashShutdown(1) → app.exit(1). The exact failure class the code's own comment defends against at bind time crashes the entire app if it occurs one accept later, killing live PTY agent sessions and unsaved work.

## Where
`src/main/localServer.ts:70` — `startLocalServer()` (the `server.removeListener('error', onError)` inside the listening callback)

## How it triggers
fd exhaustion at runtime (many PTY boards + watchers + sockets) while any process pokes the loopback preview port → accept fails with EMFILE → server emits 'error' with no listener → uncaughtException → crashShutdown(1) → entire app exits 1.

## Verification evidence
Code-path confirmed and deterministic given the trigger: `server.once('error', onError)` (src/main/localServer.ts:68) is removed inside the listening callback (`server.removeListener('error', onError)`, line 70), and the returned handle exposes only {url, port, close} (lines 73-77) — no other 'error' listener is ever attached, so the server runs for the app's lifetime with ZERO 'error' listeners. Node's net.Server emits accept-time failures on the server object ('error' from onconnection in lib/net.js — classically EMFILE/ENFILE), and an EventEmitter 'error' with no listener throws synchronously → process 'uncaughtException' → crashShutdown(1) → app.exit(1) (index.ts:543-548). The in-file comment (lines 63-66) names EMFILE/ENFILE explicitly as the defended failure class, but the once-listener covers only the listen() call — the identical error one accept later kills every live PTY agent session and the window. Low is the right grade: the mechanism is airtight but the trigger requires fd exhaustion.

## Suggested fix direction
Keep a permanent server.on('error') logger instead of removing the listener (or re-attach a no-crash handler after listen succeeds).
