# BUG-004: First tracked edit after undo/redo skips its checkpoint (lastRecorded === present) — the undone-to state lands on neither rail and the next undo jumps two steps, permanently losing it

- **Severity:** High
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/store/canvasStore.ts`, `src/renderer/src/store/canvasStore.test.ts`, `src/renderer/src/canvas/Canvas.tsx`, `src/renderer/src/canvas/boards/PlanningBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`beginChange()` skips its snapshot when the present matches the module-scoped `lastRecorded`, and `undo()`/`redo()` set `lastRecorded = r.present` — but after an undo the present was POPPED off `past` and is on neither rail. So the first real gesture after any undo/redo (drag, resize, planning edit, URL commit, font change) silently skips its pre-edit checkpoint, and the subsequent `updateBoard` clears `future`, leaving the undone-to state unreachable from both rails: the next undo jumps two logical steps and the intermediate state is permanently lost. Empirically reproduced against the real store: addBoard → move → undo → move → undo leaves `boards = []` — the board vanishes instead of returning to its pre-drag position.

## Where
`src/renderer/src/store/canvasStore.ts:677` — `beginChange()` (the `sameSnapshot(lastRecorded, s)` early-return; `lastRecorded` is set from the popped present in `undo()` at line 698 and `redo()` at line 724)

## How it triggers
Any sequence: Ctrl+Z (or Ctrl+Y), then immediately perform any tracked gesture (drag a board, resize, edit a planning element, commit a URL), then Ctrl+Z again. The state you undid to is skipped and lost.

## Verification evidence
Empirically reproduced against the real store module (esbuild-bundled canvasStore.ts, run in node from the system temp dir): addBoard → beginChange+updateBoard(x=200) → undo ⇒ 'C4 after undo: x=0 past.len=1'; then beginChange ⇒ 'past grew? false (past.len=1)' — the checkpoint was SKIPPED; then updateBoard(x=50) → undo ⇒ 'boards.len=0 (BOARD GONE — expected x=0)'. Mechanism: undo() sets `lastRecorded = r.present` (canvasStore.ts:698) where r.present is the snapshot POPPED off past (history.ts:21-22) — so it is no longer on any rail; the next beginChange hits `sameSnapshot(lastRecorded, s)` (canvasStore.ts:677) with identical refs and returns early, and the following updateBoard clears `future` (line 566). The undone-to state is then unreachable from both rails. The test suite covers only the adjacent edges — 'a bare beginChange() after an undo' (canvasStore.test.ts:375, no mutation) and 'updateBoard after an undo discards the redo branch' (line 343, no beginChange and no final undo) — never 'undo → beginChange+real edit → undo', so no existing coverage contradicts this. Affects every beginChange call site (drag, resize, planning edits, URL commit, font change). Graded High: undo is a core flow, the trigger (undo, then edit, then undo) is everyday usage, and in the minimal case the board vanishes entirely.

## Suggested fix direction
On undo/redo, don't reuse lastRecorded as a skip token for FUTURE edits — e.g. clear it and instead suppress the phantom via an explicit 'present equals what undo restored' marker consumed only by a mutation-free gesture, or move to a gesture-layer lazy checkpoint (record at first real mutation, not gesture start).
