# BUG-004: Adopted PTY session never delivers exit: proc.onExit posts to the spawn-time port closed by park(), not the live adopt port

- **Severity:** Medium
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
After a Terminal board delete (park) + undo (adopt) within the 120s TTL, the still-running process is re-bound to a NEW MessagePort, but proc.onExit (registered once at spawn time) still posts the {state:'exited'} and {exit} lifecycle messages to the ORIGINAL spawn-time port1 — which park() already closed. The post throws into a swallowing try/catch and cleanup() then closes the live adopt port without sending any exit message, so when the adopted shell/agent exits the renderer is never told: the board is stuck in 'running' forever (Interrupt button shown, run-timer keeps ticking, no '[process exited]' line). Impact is a misleading UI state-desync on a secondary path; the process itself is correctly reaped (no leak), hence Medium not High.

## Where
- src/main/pty.ts:316-334 (proc.onExit closes over the spawn-time port1 and posts to it at 320-321); src/main/pty.ts:96-108 (park() closes that same port1 at line 101); src/main/pty.ts:117-141 (adopt() creates a fresh port1' and only re-binds the input/resize 'message' handler at 124-129 — never re-wires onExit/onData); src/main/pty.ts:390-402 (cleanup() closes the live adopt port without emitting any exit message); src/renderer/src/canvas/boards/TerminalBoard.tsx:207-216 (renderer only transitions to 'exited' on m.t==='state'|'exit' control messages, never on data or port-close)

## How it triggers
1) Add a Terminal board (spawns proc; sessions.set(id,{proc,port:spawnPort1,buf}); proc.onExit captures spawnPort1). 2) Delete the board within the app — TerminalBoard unmount calls killTerminal, but the seeded path here is pty:park which calls park(id): sessions.delete + spawnPort1.close(); proc keeps running. 3) Undo within 120s — pty:adopt → adopt() makes a fresh MessageChannel (adoptPort1'/adoptPort2'), sessions.set(id,{proc,port:adoptPort1',buf}), posts adoptPort2' to the renderer (portRef now points at it), replays scrollback, re-emits running. 4) Let the adopted shell exit (type 'exit', or the agent finishes). proc.onExit fires and posts {state:'exited'}+{exit} to the CLOSED spawnPort1 → throws → swallowed by the try/catch at 319-324. cleanup(id,proc) (proc identity matches) then closes adoptPort1' silently. The renderer's adoptPort2' receives nothing and just goes dead, so state stays 'running': Interrupt button still rendered (TerminalBoard:353), the run-timer interval keeps ticking (useEffect on `running`), pill reads 'name · mm:ss', and no '[process exited]' line is written.

## Verification evidence
Code-path trace against current code (post-48c6500): pty.ts line 300 `const { port1, port2 } = new MessageChannelMain()` is the spawn channel; proc.onExit at 316 closes over THAT port1 and posts to it directly at 320-321 (NOT a dynamic sessions.get(id).port lookup — contrast onData at 303-315 which DOES look up `sessions.get(opts.id)` at fire time and thus follows the adopt). park() at 101 calls `s.port.close()` on the spawn port1. adopt() at 123 creates a brand-new `port1` and at 124-129 attaches ONLY an input/resize 'message' listener; grep across src/main confirms exactly ONE `proc.onExit` registration in pty.ts (line 316) with no re-registration in adopt(). cleanup() at 390-402 closes s.port (the adopt port) but emits no exit/state message before closing, and the renderer (TerminalBoard.tsx:207-216) transitions to 'exited' only on control messages t==='state'/'exit', never on a silent port close or on data — confirmed by terminalState.ts isRunning()=== state==='running' gating the Interrupt button and the elapsed-timer effect. The e2e 'terminal-adopt' block (e2eSmoke.ts:188-227) asserts only pid identity + scrollback replay and never lets the adopted process exit, so the gap is untested; pty.test.ts covers only appendRing/isStaleExit/canonicalizeShellPath, none of which exercise exit-after-adopt. The candidate's exact-line citations (onExit ~316-324, adopt rebinds only input/resize ~134-138) match. Note: the process IS reaped by cleanup→killTree on exit, so there is no process/handle leak — the defect is purely a renderer-visible state-machine desync, which down-rates severity to Medium.

## Suggested fix direction
Make exit delivery follow the adopt the same way onData already does: in proc.onExit, post the {state:'exited'} and {exit} messages to the CURRENT live port via `sessions.get(opts.id)?.port` (looked up at fire time) instead of the captured spawn-time port1, mirroring the dynamic lookup pattern at lines 307-314. Alternatively, have adopt() re-emit/redirect lifecycle by binding the new port before the proc exits, but the minimal, symmetric fix is the dynamic-lookup post in onExit. Add an e2e assertion that lets an adopted session exit and verifies the renderer reaches state 'exited' (and that the '[process exited]' line is written), since the current 'terminal-adopt' smoke never exercises exit after adopt.
