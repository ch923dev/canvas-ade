# BUG-002: Recap transcript egress is not consent-gated — revoking consent does not stop transcript reads + LLM egress

- **Severity:** High
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/index.ts`, `src/main/summaryLoop.ts`, `src/main/recapConsent.ts`, `src/main/agentRecapWatcher.ts`, `src/main/agentRecapMap.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The recap consent decision is consulted only at new PTY spawns (index.ts:215) and the project-open hook re-install (index.ts:351) — the actual egress path, `getAgentMilestones` → `buildRecapInput` → `runSummarize`, never checks `readConsent`. Revoking consent (`recap:setConsent('declined')`) only removes the hook: it does not untrack the transcript watcher, purge the in-memory recapMap, or delete the persistent session-map.jsonl, so a still-running claude session keeps debounce-firing the summary loop and transcript-derived milestones keep egressing to the LLM provider indefinitely — even across app restarts, since `watchRecapMap`'s prime re-tracks every persisted entry. This directly violates the consent modal's contract that scrubbed transcript content leaves only while recaps are enabled.

## Where
`src/main/index.ts:316` — the `getAgentMilestones` callback supplied to the summary loop (app bootstrap); gates only on `isTrustedTranscriptPath` + `existsSync`, never on consent

## How it triggers
1) Enable recap consent for a project; start a claude session in a terminal board (map learns the transcript path, watcher tracks it). 2) Revoke consent (recap:setConsent 'declined'). 3) The claude session (still running, or any later activity on that transcript / any meaningful board change feeding memoryEngine) fires summaryLoop.onIntent -> recap branch -> buildRecapInput egress. Persists across app restarts because session-map.jsonl is never purged.

## Verification evidence
Confirmed end-to-end. (1) Consent is consulted in exactly two places: index.ts:215 (PTY env provider, NEW spawns only) and index.ts:351 (hook re-install on project open) — verified by grep: readConsent has no other call sites in main. (2) The egress path never checks it: getAgentMilestones (index.ts:316-326) gates only on isTrustedTranscriptPath + existsSync, reading `recapMap.get(boardId)`; summaryLoop.doIntent's recap branch is taken purely on `milestones.length > 0` (summaryLoop.ts:494-508: `useRecap = !!milestones && milestones.length > 0` → buildRecapInput → runSummarize, i.e. provider fetch). summaryLoop.ts imports no consent module. (3) Revocation: recap:setConsent 'declined' (recapConsent.ts:78-84) calls onDecision, whose decline branch (index.ts:385-387) calls ONLY removeRecapHook — recapWatcher.untrack is never called (untrack is reachable only via retain(), wired to live-board pruning at index.ts:363, and the board is still live), the in-memory recapMap is never purged (its only assignment is watchRecapMap's onChange, index.ts:396-397), and session-map.jsonl is never deleted (recapMapPath is only ever defined/passed/watched: index.ts:200/356/383/396). (4) Persistence across restart: watchRecapMap's prime fire (agentRecapMap.ts:154) re-parses the map and index.ts:402-404 re-tracks EVERY entry unconditionally. So after revocation, the still-running claude session's transcript writes keep debounce-firing summaryLoop.onIntent (and memoryEngine save intents do too).

## Suggested fix direction
Gate the recap branch (getAgentMilestones or summaryLoop's useRecap) on readConsent==='enabled'; on decline, also recapWatcher.untrack the project's boards, drop their recapMap entries, and purge/rewrite session-map.jsonl.
