# BUG-002: Auto-pan during a node drag clears the shared gestureRef and reattaches all live native Browser views mid-drag (occlusion returns)

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`, `src/renderer/src/canvas/Canvas.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
beginMotion/endMotion share a single gestureRef flag across BOTH the camera path (useOnViewportChange) and the node-drag path (the nodeGesture effect). React Flow's default autoPanOnNodeDrag=true pans the viewport when a node is dragged to the pane edge; that programmatic pan fires useOnViewportChange.onEnd -> endMotion, which unconditionally sets gestureRef=false and runs applyLiveness() to reattach eligible always-above WebContentsViews while the node drag is still in progress. The reattached native layer re-occludes the dragged board until onNodeDragStop, defeating the very detach the node-drag path exists to provide (the incomplete remediation of prior bug #6).

## Where
- src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:420-423 (endMotion clears gestureRef + applyLiveness with no nodeGesture guard); :425 (useOnViewportChange onEnd=endMotion); :435-442 (nodeGesture effect routes node-drag through the same begin/endMotion + shared gestureRef); :379-417 (applyLiveness reattaches via attachBoard, never consults nodeGesture). Caller: src/renderer/src/canvas/Canvas.tsx:238-255 (ReactFlow with no autoPanOnNodeDrag override → default true; panOnScroll enabled).

## How it triggers
1) Add a Browser board, zoom in past LOD so its native view goes live (attached). 2) Grab a DIFFERENT board by its titlebar (onNodeDragStart → setNodeGesture(true) → nodeGesture effect → beginMotion sets gestureRef=true and captures+detaches the live Browser view). 3) Drag toward a pane edge so React Flow auto-pans the camera (autoPanOnNodeDrag default true). Each auto-pan rAF tick calls panBy → setViewportConstrained → d3ZoomInstance.transform(selection,t), which synchronously emits d3-zoom start/zoom/end with sourceEvent:null. 4) The end event → onPanZoomEnd (debounced 150ms because panOnScroll=true) → onViewportChangeEnd → useOnViewportChange.onEnd → endMotion. endMotion sets gestureRef=false and calls applyLiveness(), which reattaches the eligible Browser view via attachBoard while the drag is still ongoing (no onNodeDragStop yet). The always-above native view re-paints over the dragged HTML board. It is not torn down again until onNodeDragStop (or the next auto-pan tick's start re-runs beginMotion), so the occlusion persists for the rest of a held drag once the pointer leaves the edge zone.

This is the incomplete fix of prior findings #6/#1 (docs/bug-hunt-findings.md:139-142, 81-83), whose remediation assumed "a node drag never moves the camera" — true only for a non-auto-pan drag; the edge auto-pan case was not considered, and both gesture sources were composed onto one shared gestureRef with no refcount/co-active guard.

## Suggested fix direction
Compose the two gesture sources instead of sharing one boolean. Cheapest: in endMotion, do not clear gestureRef / reattach while a node gesture is active — guard with `if (usePreviewStore.getState().nodeGesture) { applyLiveness(); return }` (or skip applyLiveness entirely) so an auto-pan onEnd during a node drag cannot reattach; the node-drag effect's endMotion (driven by onNodeDragStop) remains the authority that finally clears the flag and reattaches. More robust: replace gestureRef with a refcount / separate cameraGesture and nodeGesture flags and only reattach when BOTH are clear. Alternatively set autoPanOnNodeDrag={false} on the ReactFlow in Canvas.tsx to remove the auto-pan trigger, though that changes drag UX and does not fix the general composition gap.
