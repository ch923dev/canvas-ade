# BUG-048: Connector rubber-band drag has no Esc abort and no blur/pointercancel teardown — a stale pointerup over a board silently commits an orchestration connector

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Canvas.tsx`, `src/renderer/src/lib/resolveConnectTarget.ts`, `src/renderer/src/store/slices/connectorSlice.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The M2 connector-drag effect (Canvas.tsx:684-700) arms window pointermove/pointerup listeners while connectFromId is set. There is no Esc abort (unlike useBoardPlacement, which explicitly aborts on Escape) and no blur/pointercancel teardown: pressing Esc mid connector drag leaves the rubber-band live and the gesture armed; Alt+Tab mid-drag (pointerup delivered to the other app) leaves the listeners installed across the focus loss. The next pointerup anywhere in the window — e.g. the click the user makes to re-focus the app, landing on a board — resolves through resolveConnectTarget and silently calls addConnector(connectFromId, target, 'orchestration'), creating a connector the user never intended (one tracked undo step is burned). Esc during the stale state also doesn't help: clearSelection clears selectedConnectorId but not connectFromId.

## Where
`src/renderer/src/canvas/Canvas.tsx:684` — the M2 connector-drag `useEffect` in `CanvasInner` (armed by `connectFromId`; `startConnect` set at line 620-624, `clearSelection` at lines 562-566)

## How it triggers
Press the title-bar connector handle to start a drag, then either press Esc (nothing cancels; releasing over a board still links it) or Alt+Tab away and back, then click on any board (the re-focusing click's pointerup creates the connector).

## Verification evidence
Code-path proof. Canvas.tsx:684-700 — the effect armed by connectFromId installs ONLY window pointermove/pointerup (694-695); cleanup runs solely on connectFromId change/unmount. Grep across src/renderer/src/canvas confirms setConnectFromId is written in exactly two places: startConnect (Canvas.tsx:623) and onUp (Canvas.tsx:691) — no keydown/blur path clears it. Esc maps to 'clearSelection' (useCanvasKeybindings.ts:62), and clearSelection (Canvas.tsx:562-566) calls selectBoard(null)/setFocusedId(null)/setSelectedConnectorId(null) — it never touches connectFromId, so Esc mid-drag leaves the gesture fully armed; releasing over any non-source board still resolves via resolveConnectTarget and calls addConnector(connectFromId, target, 'orchestration') (689-690), burning one tracked undo step. After a focus loss that swallows the pointerup, the listeners persist (connectFromId state survives blur), so the user's re-focusing click on a board commits an unintended connector. The asymmetry is an oversight, not a design choice: the sibling gesture's header (useBoardPlacement.ts:7-8) explicitly treats 'a late pointerup can't create a phantom board after cancel' as a requirement and even says its pointer model 'mirrors Canvas.tsx's connector rubber-band' (line 11) — but only the placement side got the abort.

## Suggested fix direction
Add an Escape handler + pointercancel/blur teardown to the connectFromId effect that nulls connectFromId/connectPointer without committing.
