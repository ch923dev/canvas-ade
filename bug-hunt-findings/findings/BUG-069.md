# BUG-069: Review workflow allowlists unrestricted `Bash(gh api:*)` while the agent processes PR-head-controlled content — prompt injection can drive arbitrary repo API writes

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `.github/workflows/claude-code-review.yml`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The claude-review job checks out the PR head (`ref: github.event.pull_request.head.sha`), instructs the model to read the diff, 'surrounding files in the working directory' and the repository's CLAUDE.md — all attacker-influenceable by whoever authors the branch — and grants `Bash(gh api:*)`, i.e. ANY GitHub API endpoint with ANY method under the job token's scopes (contents:read, pull-requests:write, issues:write). A prompt-injection payload embedded in changed files/comments/CLAUDE.md on the PR branch can therefore make the reviewer issue e.g. `gh api -X POST .../pulls/N/reviews -f event=APPROVE` (self-approving a malicious PR for branch-protection purposes), close/edit arbitrary issues, strip labels, or delete other actors' review comments — far beyond the intended self-clear DELETEs. The comment on lines 53-55 shows the allowlist was broadened from `gh api repos:*` to `gh api:*` solely because the self-clear pipeline starts with `gh api -X DELETE`; a least-privilege prefix (e.g. `Bash(gh api -X DELETE repos/<owner>/<repo>/pulls/comments:*)` plus the read endpoints) would cover the legitimate use.

## Where
`.github/workflows/claude-code-review.yml:57` — `claude-review` job, "Run Claude Code Review" step (`claude_args: --allowedTools "...Bash(gh api:*)"`)

## How it triggers
Any same-repo branch PR (fork PRs lack the OAuth secret) whose files/markdown contain instructions targeting the reviewer agent; the workflow runs automatically on open/synchronize.

## Verification evidence
All elements verified in .github/workflows/claude-code-review.yml: line 43 checks out `github.event.pull_request.head.sha`; lines 117-128 instruct the model to read the diff, 'open the surrounding files in the working directory', and 'Read the repository's CLAUDE.md' — all branch-author-controlled; line 57 allowlists `Bash(gh api:*)`, and the workflow's own comment (lines 53-55) confirms prefix-match semantics and that it was broadened from `gh api repos:*` solely so the self-clear's `gh api -X DELETE` pipeline matches; the job grants `pull-requests: write` + `issues: write` (lines 34-37). Capability chain: any command starting `gh api ` is permitted → any method on any endpoint within the token's scopes, e.g. `gh api -X POST repos/O/R/pulls/N/reviews -f event=APPROVE`, `gh api -X PATCH repos/O/R/issues/K -f state=closed`, or deleting other actors' review comments — all far beyond the intended DELETE-own-comments use. Caveats that keep this Low rather than higher: fork PRs never get the OAuth secret/job token path; PR self-approval specifically may additionally be blocked by the repo's 'Allow GitHub Actions to create and approve pull requests' setting (unverifiable from here) — but the broader arbitrary-write surface (issue close/edit, label/milestone manipulation, comment deletion) needs only the granted `issues: write`/`pull-requests: write`; and the repo is single-maintainer with same-repo agent-authored branches. Structural least-privilege gap, confirmed.

## Suggested fix direction
Narrow to the exact prefixes used: `Bash(gh api repos/<owner>/<repo>/pulls:*)`-style read endpoints plus `Bash(gh api -X DELETE repos/<owner>/<repo>/pulls/comments:*)` and the issues-comments DELETE — the prompt's own pipelines (lines 84-96) enumerate everything legitimately needed.
