# BUG-011: Deleting the dragged board mid-drag latches previewStore.nodeGesture true forever — all Browser previews freeze as snapshots; stale alignment guides stay painted

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/Canvas.tsx`, `src/renderer/src/canvas/boards/usePreviewManager.ts`, `src/renderer/src/store/previewStore.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`onNodeDragStart` sets `setNodeGesture(true)` and detaches every live native preview, and the matching `setNodeGesture(false)` + guides/overlaps/dropTarget cleanup live ONLY in `onNodeDragStop` — but @xyflow/system's XYDrag explicitly ABORTS the gesture when the dragged node disappears mid-drag, so React Flow never calls the stop handler. `nodeGesture` then latches true: every Browser board's WebContentsView stays demoted to a static snapshot indefinitely (until the user happens to complete another full drag/resize gesture), and the last snap frame's alignment guide lines, overlap tints, and lit drop-target group box remain painted.

## Where
`src/renderer/src/canvas/Canvas.tsx:549` — `onNodeDragStop` callback in the `Canvas` component (the only reset of `setNodeGesture(false)` / guides / overlaps / dropTarget, at lines 551-555; the latch is set in `onNodeDragStart` at line 530)

## How it triggers
While dragging a board by its title bar, press Delete/Backspace (the dragged node is selected, RF's deleteKeyCode removes it mid-drag), or have the MCP orchestration layer / an agent remove the board while the user is dragging it. The next 'drag' tick sets abortDrag and the end callback is skipped.

## Verification evidence
Verified in the vendored dependency AND the app. node_modules/@xyflow/system/dist/esm/index.js lines 2219-2225: on each d3 'drag' tick, `(nodeId && !nodeLookup.has(nodeId))` sets abortDrag = true (comment: 'if user deletes a node while dragging, we need to abort the drag to prevent errors'), and the 'end' handler (lines 2248-2250) does `if (!dragStarted || abortDrag) return` — so onNodeDragStop is never invoked. App side: setNodeGesture(true) is set in onNodeDragStart (Canvas.tsx:530) and setNodeGesture(false) exists ONLY in onNodeDragStop (Canvas.tsx:551) and NodeResizer onResizeEnd (BoardNode.tsx:287) — confirmed by repo-wide grep (the only other writer is the e2e hook). Delete mid-drag is reachable: the dragged node is selected, RF's useGlobalKeyHandler deleteElements()s it (dist/esm/index.js:1230-1234) → onNodesChange remove intent → removeBoard → nodeLookup loses the id → next pointer move (inevitable with the button held) sets abortDrag → 'end' skipped. Consequences confirmed: usePreviewManager.ts:723 `if (nodeGesture || menuOpen) beginMotion()` keeps every Browser board's WebContentsView demoted to a snapshot while the flag is latched (endMotion at :699-701 also explicitly returns while nodeGesture), until the user completes another full drag/resize gesture; the stale guides/overlaps set by the snap pass are cleared only in onNodeDragStop (Canvas.tsx:552-553) or by a later active-drag branch — the remove-change batch hits neither (`active` is empty).

## Suggested fix direction
Reset nodeGesture/guides/overlaps/dropTarget when the dragged board disappears — e.g. in the onNodesChange remove path or an effect keyed on boards that clears gesture state if the gesture's node id is gone (track the dragged id in a ref from onNodeDragStart).
