# BUG-011: Terminal Restart's spawnTerminal promise has no .catch(); a rejected pty:spawn invoke leaves the board stuck on 'spawning' with a dead session and an unhandled rejection

- **Severity:** Medium
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
restart() in TerminalBoard.tsx does setState('spawning') + term.reset(), then chains only .then() on window.api.spawnTerminal(...) with no .catch() — unlike the initial spawn path which handles both legs. The pty:spawn main handler throws (not resolves) when getWin() returns null (pty.ts:276), so a rejected invoke is swallowed: the board sits on 'spawning' forever over a blank/dead terminal with no error written and an unhandled promise rejection. Impact is a recoverable-but-confusing dead UI state (no data loss/crash/leak), matching Medium.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:327-341 — restart() callback (.then at 336, no .catch); contrast initial spawn .then+.catch at TerminalBoard.tsx:251-260; reject source at src/main/pty.ts:276 (if (!win) throw new Error('pty:spawn — no window'))

## How it triggers
User clicks the Restart action (IconBtn name="restart" onClick={restart}, line 359) at a moment when the pty:spawn invoke rejects rather than resolves. The reachable reject source is pty.ts:276: getWin() (() => mainWindow, index.ts:142) returns null — e.g. during a window recreate/teardown window — causing a synchronous throw in the IPC handler that ipcRenderer.invoke surfaces as a rejected renderer promise. Because restart() already ran setState('spawning') (line 326) and term.reset() (line 325), the board is left on a blank terminal showing the starting/spawning status with no recovery path other than deleting/recreating the board, and the rejection is unhandled (no renderer-side unhandledrejection handler exists).

## Verification evidence
Direct read of current code (last commit to the file is the remediation 48c6500; working tree clean). TerminalBoard.tsx:327-341 chains only .then((res)=>{ if(res.state==='spawn-failed'){ setState('spawn-failed'); term.write(...) } }) — no .catch — so a REJECTED invoke (vs a RESOLVED {state:'spawn-failed'}) is never observed and state stays 'spawning' (set at line 326). The initial spawn at lines 251-260 deliberately has BOTH .then (resolved spawn-failed) AND .catch((err)=>{ setState('spawn-failed'); term.write('spawn failed: '+err.message) }), proving the omission is an asymmetry, not intent. Reject path verified: preload index.ts:46-47 spawnTerminal is a thin ipcRenderer.invoke('pty:spawn', opts); pty.ts handler is synchronous and the only throw is line 276 (no-window) — the node-pty spawn() failure at 294 is caught and RETURNED as a value (line 297), so it resolves, but the no-window case throws and rejects. getWin = () => mainWindow (index.ts:142). No renderer-side unhandledrejection handler found (grep). Caveat lowering realism: mainWindow is never reassigned to null after first creation in index.ts (no 'closed'→null handler), so the !win branch is a narrow lifecycle race, not a common Restart path; combined with the recoverable visual nature this keeps severity at Medium. Not among the prior 50 findings — the prior restart finding (bug-hunt-findings.md:831) was the session-reaping race (UNCONFIRMED); this .catch gap is a distinct, still-present omission left by the remediation that added .catch only to the initial spawn.

## Suggested fix direction
Mirror the initial-spawn handling on the restart path: add a .catch((err) => { setState('spawn-failed'); term.write(`\x1b[31mspawn failed: ${err.message}\x1b[0m\r\n`) }) to the spawnTerminal(...) chain in restart() (TerminalBoard.tsx:327-341) so a rejected pty:spawn invoke transitions out of 'spawning' and surfaces the error, identical to lines 257-260. Optionally hoist the shared {res}/{err} handler into one helper used by both call sites to prevent future drift.
