# BUG-009: preview:capture handler has no try/catch — a rejected capturePage() throws into the renderer's await, skipping the detach so the always-above native view stays attached (occlusion + unhandled rejection)

- **Severity:** Medium
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/preview.ts`, `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

> **Consolidates 2 discovery findings** (same root cause, multiple call sites): `preview-capture-unguarded-rejection-blocks-detach`, `preview-capture-handler-uncaught-capturepage-rejection`

## Summary
The production preview:capture IPC handler (src/main/preview.ts:330) awaits webContents.capturePage() with no try/catch, even though the E2E debugCaptureView accessor (lines 422-431) was explicitly hardened against capturePage rejecting on headless/GPU-contended hosts. When capturePage() rejects, ipcMain.handle's promise rejects, so the renderer's `await window.api.capturePreview(id)` throws inside demoteToSnapshot (BrowserPreviewLayer.tsx:243) and beginMotion (line 353). Neither caller wraps the await, so the detach step that follows (line 252 / line 364) never runs: the native WebContentsView, which paints above all HTML, stays attached over the board, occluding and stealing input from whatever it overlaps (the ADR 0002 motion/LOD mitigation it is supposed to perform). It self-heals on the next onMoveEnd/applyLiveness retry only if the contention was transient; while contention persists every retry also rejects and the view stays attached. It also produces an unhandled promise rejection in the renderer.

## Where
- src/main/preview.ts:330 — ipcMain.handle('preview:capture') (unguarded `await e.view.webContents.capturePage()`); renderer impact at src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:243 (demoteToSnapshot) and :353 (beginMotion Promise.all)
- src/main/preview.ts:327-332 — ipcMain.handle('preview:capture') (missing try/catch around capturePage at line 330); contrast with src/main/preview.ts:419-432 debugCaptureView (has the guard). Renderer victims: src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:243 (demoteToSnapshot) and :353 (beginMotion).

## How it triggers
During a pan/zoom (useOnViewportChange.onStart → beginMotion) or an LOD/over-cap/occlusion demote (applyLiveness:407 → demoteToSnapshot), the renderer calls window.api.capturePreview → preview:capture → capturePage(). capturePage() rejects when the GPU/compositor cannot produce a frame: multiple Electron instances contending for the GPU (the same condition debugCaptureView guards against), a headless/remote-desktop session, or transient GPU loss. There is also a TOCTOU: preview:capture is async and yields at its await, so a concurrently-handled preview:close → disposeOne → webContents.close() (preview.ts:260) can destroy the webContents between the `!e.attached` check (line 329) and the await (line 330), making the in-flight capturePage reject on a destroyed webContents. In demoteToSnapshot the throw skips lines 249-255 so r.attached stays true and detachPreview is never called; in beginMotion the Promise.all rejects wholesale so NO live board detaches and the whole capture/detach continuation (lines 354-370) is skipped.

## Verification evidence
Read src/main/preview.ts: line 330 `const img = await e.view.webContents.capturePage()` is unguarded; by contrast debugCaptureView (lines 422-431) wraps the identical capturePage() call in try/catch with the comment "No composited display surface (headless / GPU-contended host …): capturePage rejects" — the author's own acknowledgement that capturePage can reject, and a clear asymmetry vs the production handler. Read src/preload/index.ts:70: capturePreview is a thin `ipcRenderer.invoke('preview:capture', id)` pass-through (no catch), so a main-side rejection propagates to the renderer await. Read src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx: demoteToSnapshot (238-258) does `const url = await window.api.capturePreview(g.id)` at 243 BEFORE the `await window.api.detachPreview(g.id)` at 252 — a throw at 243 skips the detach and the `r.attached = false` at 254; the function is invoked as `void demoteToSnapshot(g)` at 407 with no .catch. beginMotion (346-372) does `await Promise.all(live.map(... capturePreview ...))` at 353 inside a bare `void (async () => {…})()` (352) with no try/catch; a single rejected capture rejects the whole Promise.all so the detach loop at 364 never runs and gestureRef stays true until onEnd→endMotion (420) resets it. Grep confirms no `.catch(` and no try/catch anywhere around these awaits in the file. The renderer has no window.onunhandledrejection handler (grep found only a MAIN-process process.on('unhandledRejection') at src/main/index.ts:210, which does not fire for renderer-side rejections), so the rejection is logged by Chromium but the behavioral defect (view stays attached → occlusion) is the real impact. This is NOT one of the prior 50 (docs/bug-hunt-findings.md #2/#24/#26/#37/#45/#48 cover input-occlusion semantics, empty frames, post-await re-checks and orphan resurrection — none guards the capturePage rejection path).

— also confirmed (consolidated finding preview-capture-handler-uncaught-capturepage-rejection): Direct read of src/main/preview.ts: line 330 `const img = await e.view.webContents.capturePage()` has no try/catch — the only failure handling is `if (!e || !e.attached) return null` at line 329 (detached case only). The identical call at line 423 inside debugCaptureView IS wrapped in try/catch (lines 422-431) with the comment "No composited display surface (headless / GPU-contended host...): capturePage rejects. Report empty rather than letting the rejection abort..." — proving the author knows the call rejects and chose to guard only the debug path. Renderer trace: BrowserPreviewLayer.tsx:23

## Suggested fix direction
Mirror the debugCaptureView hardening on the production handler: wrap the capturePage() await in preview:capture (preview.ts:327-332) in try/catch and return null on rejection (treat a failed capture like an empty image). Defensively also wrap the renderer awaits — guard `await window.api.capturePreview(...)` in demoteToSnapshot and the Promise.all in beginMotion (or use Promise.allSettled) so a single capture failure cannot block the detach of the always-above native views; ensure r.attached/gestureRef are reset and detachPreview still runs on the capture-failure branch so the occlusion mitigation completes even when no snapshot pixels were obtained.

Consolidated site (preview-capture-handler-uncaught-capturepage-rejection): Mirror debugCaptureView's guard in the production handler: wrap line 330's capturePage() in try/catch and `return null` on rejection, so a failed capture is treated as "no snapshot" (the same sentinel demoteToSnapshot/beginMotion already handle via their `if (url)` / `if (shots[i])` checks) instead of rejecting the invoke. That single-site fix removes the unhandled renderer rejection AND lets the detach paths (detachPreview at BrowserPreviewLayer.tsx:252/364) still run, so native views are correctly demoted even when capture is unavailable. Optionally also defensively try/catch the renderer awaits, but the main-side guard is the minimal correct fix and restores parity between the two capture sites.
