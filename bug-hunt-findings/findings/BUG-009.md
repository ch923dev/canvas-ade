# BUG-009: ProjectSwitcher.switchTo has no in-flight guard (BUG-008 fixed only in WelcomeScreen) — concurrent switches can desync renderer boards vs MAIN currentDir, then autosave cross-writes projects

- **Severity:** Medium
- **Category:** concurrency
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/AppChrome.tsx`, `src/main/projectIpc.ts`, `src/renderer/src/canvas/WelcomeScreen.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
WelcomeScreen got a `busy` guard against concurrent `project:open` calls, but `ProjectSwitcher.switchTo` has no equivalent: the user can click recent B, re-open the dropdown during the awaited flush/dispose/open, and click recent C, interleaving two switch pipelines. MAIN's `currentDir` is set synchronously at each open handler's start, so it ends at the LAST-STARTED open (C), while the renderer's final board state is whichever `applyOpenResult` resolved LAST — which can be B. Because `project:save` writes the posted doc to `getCurrentDir()` unconditionally, the next autosave then writes B's canvas into C's canvas.json — cross-project data corruption.

## Where
`src/renderer/src/canvas/AppChrome.tsx:119` — `switchTo()` inside the `ProjectSwitcher` component (no busy/in-flight guard; only `setOpen(false)` at line 120)

## How it triggers
With project A open: click recent B in the switcher, immediately re-open the dropdown and click recent C while the first switch is still awaiting its flush/dispose/open; have B's open settle after C's (slow recents pathExists or a B-side reopenFromBak retry).

## Verification evidence
src/renderer/src/canvas/AppChrome.tsx:119-154 — switchTo has no busy/in-flight guard (only setOpen(false)); the trigger (toggle, lines 114-117) stays enabled, so during switchTo(B)'s `await window.api.project.save(...)` (line 131, status still 'open' → switcher still mounted) the dropdown can be reopened and recent C clicked, starting a second pipeline. The window is even wider than claimed: once setProjectLoading() flips status to 'loading', App.tsx:84 renders `status === 'open' ? <Canvas/> : <WelcomeScreen/>` — WelcomeScreen mounts FRESH with local busy=false (WelcomeScreen.tsx:17), so its enabled recent buttons allow a concurrent openDir(C) during the entire dispose/load phase too (the BUG-008 guard is per-mount local state and cannot see the in-flight switchTo). MAIN: each project:open sets currentDir synchronously at handler start (projectIpc.ts:145-157 remember → setCurrentDir before awaiting touchRecent) so currentDir ends at the LAST-ARRIVED open, while the renderer's final board state is the LAST applyOpenResult to settle — which can be the other project (B's handler stalling in touchRecent's up-to-500ms-per-entry pathExists races, or B's renderer-side T5 reopenFromBak retry adding an extra IPC round-trip, makes B commit after C while C set currentDir last). The amplifier is real: project:save (projectIpc.ts:229-238) writes the posted doc to getCurrentDir() unconditionally with no doc↔dir association, so the desync becomes a silent cross-project canvas.json overwrite.

## Suggested fix direction
Module/app-level in-flight switch guard (shared by ProjectSwitcher and WelcomeScreen), and/or have project:save take an expected dir (or generation token) that MAIN checks against currentDir before writing.
