# BUG-068: Pre-push hook dies silently (set -e + 2>/dev/null) when the advertised remote tip object is not in the local store

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `.githooks/pre-push`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
`changed="$changed$(git diff --name-only --diff-filter=ACMR \"$range\" 2>/dev/null)"` — if `$remote_sha` (the remote's advertised tip) does not exist in the local object store (push from a clone that hasn't fetched the remote's current tip, e.g. a force-push over a branch another machine/worktree advanced, or a pruned/shallow history), `git diff` exits non-zero. The exit status of a plain variable assignment is the command substitution's status, so `set -e` (line 13) kills the hook mid-loop. Because stderr is redirected to /dev/null, the user gets no message at all — just a failed push with no `[pre-push]` output, which looks like an inexplicable git failure. The `2>/dev/null` was clearly meant to tolerate diff errors, but `set -e` defeats that intent; it fails closed (blocks the push) yet undiagnosably.

## Where
`.githooks/pre-push:28` — the per-ref `while read` loop's command-substitution assignment under `set -e` (line 13)

## How it triggers
Push (typically --force / lease) to an origin branch whose current remote tip commit is not present locally; the hook exits non-zero before printing anything.

## Verification evidence
Demonstrated with a faithful simulation of .githooks/pre-push lines 13+21-30 (set -e; while read loop; `changed="$changed$(git diff --name-only --diff-filter=ACMR "$range" 2>/dev/null)`): feeding a stdin ref line whose remote sha is an object not present locally made the script exit 128 (git diff's 'bad object' status propagates through the command-substitution assignment, which `set -e` treats as fatal) with ZERO lines of output — the trailing `echo` never ran and git's own error went to /dev/null. In the real hook the user sees only git's generic push-hook failure with no `[pre-push]` diagnostic at all. The `2>/dev/null` on line 28 shows the author expected git diff to be fallible there, so the set -e abort defeats the intended tolerance: it fails closed (blocks the push) but undiagnosably. Realistic trigger: `git push --force` over a branch another machine/worktree advanced without first fetching (the advertised remote tip is then not in the local object store). Severity Low is right — rare trigger, fails safe (blocks rather than skips), but with zero diagnostics.

## Suggested fix direction
Guard the substitution (`|| true` inside it, or compute under an `if git diff ...; then` with an explicit fallback to 'unknown range → run full matrix').
