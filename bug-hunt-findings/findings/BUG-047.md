# BUG-047: Placement drag has no blur/pointercancel teardown — a stale window pointerup later commits a phantom board spanning from the abandoned drag origin

- **Severity:** Low
- **Category:** correctness
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/hooks/useBoardPlacement.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The in-flight drag's window pointermove/pointerup listeners are removed only by pointerup itself, Esc, unmount, or a new pointerdown ON the overlay (BUG-035 path). If the gesture is interrupted without a pointerup reaching the window — Alt+Tab / OS focus steal mid-drag (the pointerup goes to the other app), or a pointercancel (touch/pen) — dragCleanupRef stays armed and the tool stays armed. The next pointerup the window sees, even one originating on the app chrome ABOVE the overlay (the dock sits at z-50 over the z-40 capture layer, explicitly kept clickable per the Canvas.tsx comment at line 860), fires the stale onUp: it computes a rect from the ORIGINAL sx/sy to the new release point and commits addBoard — a phantom board spanning from the abandoned drag start to wherever the user clicked (e.g. the dock's Select button they pressed to cancel).

## Where
`src/renderer/src/canvas/hooks/useBoardPlacement.ts:106` — the window pointermove/pointerup pair armed inside `startPlacement` (`useBoardPlacement`), torn down only via `dragCleanupRef`

## How it triggers
Arm a board type, press and drag on the canvas, Alt+Tab away mid-drag (pointerup delivered to the other window), return, then click the dock's Select button (or any chrome above the overlay) — the click's pointerup creates the phantom board.

## Verification evidence
Code-path proof from the full hook read (useBoardPlacement.ts:1-117). The window pointermove/pointerup pair (lines 106-107) is removed ONLY via dragCleanupRef (lines 108-111), which is invoked from exactly three places: onUp itself (line 86), abortDrag on Escape (lines 57-61) / unmount (line 68), and a fresh startPlacement on the overlay (line 76, BUG-035). There is no 'blur', 'visibilitychange', or 'pointercancel' handler anywhere in the file — contrast useCanvasKeybindings.ts:212-215 which explicitly handles blur for the Ctrl-latch, showing the focus-loss class is a known hazard elsewhere. Interrupt paths that skip window pointerup: pointercancel (touch/pen, spec-guaranteed) and OS focus steal mid-drag (pointerup delivered to the other app). The tool stays armed and listeners stay live; the next pointerup the window sees — including one on chrome ABOVE the z-40 overlay (Canvas.tsx:863-864 comment: 'Chrome (z-50) stays above this (z-40), so the Select button / dock buttons remain clickable') — fires the stale onUp, which computes a rect from the ORIGINAL sx/sy (lines 93-102) and commits addBoard with `exact:true`. A dock pointerdown never touches the overlay, so the line-76 self-heal cannot run; only its pointerup bubbles to window.

## Suggested fix direction
Add window 'pointercancel' + 'blur' listeners to the in-flight drag that call abortDrag (mirroring the Esc path).
