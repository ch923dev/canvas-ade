# BUG-025: Deep load validation accepts non-positive / sub-minimum board and element geometry (MIN_BOARD_SIZE not enforced on load)

- **Severity:** Low
- **Category:** data-integrity
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/lib/boardSchema.ts`
- **Roadmap:** partially covered by Phase 3: Project create / open — canvas.json load + migrations + full-reopen-fidelity test (see cross-ref)
- **Assigned:** _(left blank for the dev team)_

## Summary
fromObject's deep guards (assertBoard / assertPlanningElement) validate only that board/element x/y/w/h are finite, never that w/h are positive or >= the documented MIN_BOARD_SIZE. A canvas.json with w:0, h:-50, or w:5 passes validation; boards feed b.w/b.h straight into the React Flow node style (Canvas.tsx:97), so a degenerate-but-finite size loads and renders zero-size / inverted / below the documented floor. Impact is a recoverable visual/data-quality defect that requires already-corrupt input, not a crash or data loss.

## Where
- src/renderer/src/lib/boardSchema.ts:305-307 (assertBoard geometry check); :259 (note w/h); :281 (checklist w/h); enforcement gap at src/renderer/src/store/canvasStore.ts:177-178 (loadObject, no clamp) vs :150 (resizeBoard clamps); consumer at src/renderer/src/canvas/Canvas.tsx:97 (node style width/height = b.w/b.h)

## How it triggers
Call loadObject (or the future Phase-3 canvas.json/.bak load path these guards exist to protect) with a board or note/checklist element whose w<=0 or h<=0, or a finite w/h below MIN_BOARD_SIZE {w:240,h:160} — e.g. from a truncated/corrupted write or a hand edit. fromObject returns it as valid; the board node then renders width:0 / height:-50 (invisible/degenerate) or smaller than the documented minimum.

## Verification evidence
Code trace: isFiniteNum (boardSchema.ts:239-241) returns true for 0, -50, and 5 (Number.isFinite of all three is true). assertBoard (305-307) only calls isFiniteNum on x/y/w/h with no positivity/min check; note (259) and checklist (281) likewise. MIN_BOARD_SIZE (125) is referenced only by resizeBoard (canvasStore.ts:150, Math.max clamp) and the NodeResizer minWidth/minHeight (BoardNode.tsx:88-89) — both interactive-resize-only; loadObject (canvasStore.ts:177-178) applies no clamp. Board w/h ARE consumed by the renderer: Canvas.tsx:97 sets node style {width:b.w,height:b.h}. git log -S confirms these guards were introduced by commit 48c6500 as 'fix #5' (comment lines 227-233: 'reject any board or element that does not match the schema'), so this is an incomplete remediation of that fix. A throwaway vitest repro (written to OS temp, run against the real module, then deleted) PASSED 3/3: fromObject accepted a {w:0,h:-50} terminal board, a {w:5,h:5} browser board (below MIN), and a note element {w:-10,h:0}. The existing boardSchema.test.ts:215 only tests w:-Infinity (non-finite, correctly caught) — no test covers finite degenerate values. This is DISTINCT from the prior REFUTED finding (bug-hunt-findings.md:805-817, 'checklist persists h:0'), which was scoped to a checklist h field that NO consumer reads; here the gap also covers board w/h which the renderer DOES read.

## Suggested fix direction
In assertBoard and assertPlanningElement (note/checklist), add a positive-size guard for w/h (>0). For the documented floor, prefer clamp-on-load (Math.max(MIN_BOARD_SIZE.w/h, value) inside fromObject or loadObject) over a hard reject, because checklist h:0 is legitimately app-produced (elements.ts seeds h:0, content-driven height) — a strict reject would wrongly fail valid data and trip the .bak fallback. Either exempt checklist h from the min-floor or normalize it on load.
