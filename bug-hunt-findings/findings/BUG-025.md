# BUG-025: stripAnsi misses 8-bit C1 introducers and leaks 7-bit DCS payloads into the 'plain text' MCP scrollback page

- **Severity:** Low
- **Category:** security
- **Status:** CONFIRMED
- **Files touched:** `src/main/ptyOutput.ts`, `src/main/pty.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
The module's own header states ANSI control sequences are "an injection surface for whatever renders the agent's view", and readPtyOutput promises ANSI-stripped plain text — but the ANSI regex only matches 7-bit ESC-introduced sequences. 8-bit C1 controls (CSI 0x9B, OSC 0x9D, DCS 0x90, ST 0x9C) pass through entirely unstripped, so on POSIX (no ConPTY re-render) any program emitting raw C1 bytes delivers live terminal control sequences into the page handed to the MCP agent/renderer. For 7-bit DCS (ESC P … ESC \\) the regex `\x1b[@-Z\\-_]` strips only the 2-byte introducer, so the entire DCS payload (sixel data, tmux passthrough, XTGETTCAP responses) leaks into the "plain text" output and can smuggle content past the strip.

## Where
`src/main/ptyOutput.ts:30` — module-level `ANSI` regex constant consumed by `stripAnsi()`

## How it triggers
On Linux/macOS, run any command in a terminal board whose output contains 8-bit C1 escape bytes (or DCS payloads), then read the board via the MCP read_board_output path or any consumer of readPtyOutput — the control bytes arrive in `text`.

## Verification evidence
Confirmed by direct repro against the exact regex (src/main/ptyOutput.ts:28-30). Ran the regex from line 30 in node: input '31mRED0m hi' (8-bit C1 CSI) came back as '31mRED0m hi' — completely unstripped (a 7-bit '\x1b[31m' equivalent strips to 'RED hi'); '0;titledone' (C1 OSC..ST) also passes through; '\x1bPq#0;2;0;0;0SIXELDATA\x1b\\' (7-bit DCS) strips only the 2-byte introducer and ST, leaving the payload 'q#0;2;0;0;0SIXELDATA' in the 'plain text'. The module's own header (lines 4-9) states ANSI sequences are 'an injection surface for whatever renders the agent's view' and promises plain text, so this is a defect against the stated contract. C1 codepoints are deliverable on POSIX: node-pty UTF-8-decodes the byte stream, so a program emitting 0xC2 0x9B yields U+009B in the ring string (Windows is shielded by ConPTY's 7-bit re-render, which is why the gap survives the Windows-heavy matrix). The unit tests (ptyOutput.test.ts:4-23) exercise only ESC-introduced forms — no C1/DCS coverage contradicts the claim.

## Suggested fix direction
Extend the regex with [\x80-\x9f]-introduced C1 forms (0x9B CSI, 0x9D OSC, 0x90 DCS through ST) and a DCS-payload-through-ST rule for the 7-bit form.
