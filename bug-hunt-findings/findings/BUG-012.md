# BUG-012: Terminal WebGL "pooling" fix never caps on-screen contexts — many detail-view terminals still exhaust Chromium's ~16-context budget (#11 remediation incomplete)

- **Severity:** Medium
- **Category:** performance
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/canvas/boards/TerminalBoard.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
Finding #11 was titled "No cap/pooling on per-terminal WebGL contexts." The 48c6500 remediation added only LOD-based release (option b of the finding's fix), not the cap/counter (option a) the title named. Because `lod` is a global zoom-only boolean, the release fires only when the WHOLE canvas zooms below 40% (where no terminal is visible anyway), giving zero relief for the real scenario: many terminals visible at working zoom, each unconditionally holding its own WebGL2 context with no aggregate ceiling. With enough boards the shared ~16-context budget is exceeded and the oldest terminals silently degrade to the slower DOM renderer.

## Where
- src/renderer/src/canvas/boards/TerminalBoard.tsx:111-142 (attachWebgl/detachWebgl + the lod effect); BoardNode.tsx:58 (global zoom-derived `lod`); canvasView.ts:24 (isLod / LOD_ZOOM=0.4)

## How it triggers
Open ~12+ Terminal boards (plus a couple of Browser boards consuming WebContentsView/GL budget) on a single canvas and keep the camera in the normal working band (zoom > 0.4). Every terminal has lod=false, so each calls attachWebgl and holds one live WebGL2 context. Once the renderer-wide live-context count crosses Chromium's ~16 ceiling, the oldest context is evicted; ~3s later (no restore arrives because newer contexts hold the budget) xterm's WebglRenderer fires onContextLoss, TerminalBoard disposes that addon, and that terminal runs on the DOM renderer for the rest of its life. Adding more terminals cascades the loss.

## Verification evidence
Code-path trace on current post-48c6500 code: (1) attachWebgl (TerminalBoard.tsx:111-124) short-circuits ONLY on this component's own webglRef.current (line 112) — there is no module-level counter, registry, or cap. Grep across src/renderer/src/canvas for webgl|cap|pool|counter|registry|MAX_LIVE found the ONLY cap is MAX_LIVE=4 in BrowserPreviewLayer.tsx:48, which bounds WebContentsViews, a different GPU consumer; terminals have no analogue. (2) The release effect (TerminalBoard.tsx:137-142) keys on `lod`. `lod` is supplied by BoardNode.tsx:58 as `useStore((s) => isLod(s.transform[2]))` — derived from camera zoom ALONE, not per-board visibility/occlusion — so it is the same boolean for every board. canvasView.ts:24 `isLod = zoom < 0.4`. Therefore detachWebgl only ever runs when the entire canvas zooms below 40%, at which point all terminals release together and none are on-screen; in the working band (>40%) the number of live terminal contexts equals the number of terminal boards, uncapped. (3) The remediation commit 48c6500's own changelog states the fix as "#10 TerminalBoard: pool the WebGL context — release at LOD, re-acquire on [detail]" — i.e. only option (b). Finding #11's Fix text explicitly says "Combine (a)+(b) for a deterministic ceiling that scales with the many-board premise"; (a), the cap/counter named by the finding TITLE ("No cap/pooling"), was never implemented. No test asserts a WebGL cap (grep of **/*.test.* for webgl/attachWebgl/context returned no matches). Severity ceiling: xterm WebglRenderer.ts:105-126 preventDefaults context loss, waits 3000ms for restore, and on real loss falls back to the DOM renderer — graceful perf/quality degradation, no crash/blank/data loss — so Medium, matching #11.

## Suggested fix direction
Implement the missing cap (finding #11 option a) in addition to the existing LOD release. Add a renderer-wide budget for terminal WebGL contexts (a small module-level registry/counter keyed by board id, capped well under 16 with headroom for Browser views + React Flow); attachWebgl should only acquire when under cap and otherwise leave the terminal on the DOM renderer, and detachWebgl should free the slot. Optionally prioritize visible/selected/recently-focused terminals and add LRU eviction (option c) so newly focused terminals can reclaim a GL slot from a stale one. Note the existing `lod`-based release is near-useless on its own because `lod` is global zoom-only (BoardNode.tsx:58) — a true cap, not the zoom gate, is what bounds the many-visible-terminals case.
