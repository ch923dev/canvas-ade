# BUG-032: removeRecapHook throws on malformed settings.local.json after consent is persisted — 'declined' recorded but the SessionStart hook stays installed (no retry path)

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/agentRecapMap.ts`, `src/main/recapConsent.ts`, `src/main/index.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
removeRecapHook does `blocks.map((b) => ({...b, hooks: b.hooks.filter(...)}))` with no guard that `blocks` is an array or that each block has an array `hooks` (contrast isRecapHookInstalled, which uses `b.hooks?.some`). A SessionStart block missing `hooks`, or hooks.SessionStart being a non-array (hand-edited file, or another tool writing the same file), throws TypeError. In recap:setConsent, writeConsent runs FIRST, then onDecision → removeRecapHook throws uncaught out of the handler: consent is already persisted as "declined" while the SessionStart hook remains installed, and nothing ever retries the removal (project open only re-installs on "enabled", never removes on "declined"). installRecapHook has the mirror-image hazard, and a corrupt-JSON settings file is read as {} then clobbered wholesale by writeSettings.

## Where
`src/main/agentRecapMap.ts:83` — `removeRecapHook()` (the unguarded `b.hooks.filter` inside `blocks.map`)

## How it triggers
Project's .claude/settings.local.json contains hooks.SessionStart as a non-array, or a SessionStart block without a `hooks` array (hand edit or third-party tool), then the user toggles recap consent off.

## Verification evidence
Mechanism confirmed; severity downgraded Medium→Low. agentRecapMap.ts:82-83: `blocks.map((b) => ({...b, hooks: b.hooks.filter(...)}))` — unguarded: a SessionStart block lacking a `hooks` array throws TypeError on `b.hooks.filter`, and a truthy non-array `hooks.SessionStart` throws on `blocks.map` (only `if (!blocks) return` at :81 guards, no Array.isArray). The contrast is in the same file: isRecapHookInstalled (:52) defensively uses `b.hooks?.some`. readSettings' cast (:37) gives no runtime guarantee. Ordering makes it sticky: recapConsent.ts:82-83 runs writeConsent FIRST, then onDecision with no try/catch — the throw escapes the ipcMain.handle callback (renderer invoke rejects; main does not crash), leaving consent persisted 'declined' while the hook remains installed, and only the 'enabled' branch is ever replayed at project open (index.ts:351), so nothing retries the removal. The install mirror (:65 `cfg.hooks.SessionStart.push` on a non-array) also throws after 'enabled' is persisted, though that side IS retried on every project open. Downgraded to Low because (1) the trigger requires a malformed/hand-edited or third-party-mangled settings.local.json (Claude Code and this app both only write well-formed hooks arrays), and (2) the impact is the metadata-mapping hook staying installed — new PTY spawns no longer get CANVAS_RECAP_BOARD (env provider checks consent, index.ts:215), and the transcript-egress continuation is finding #3's issue, not this one.

## Suggested fix direction
Mirror isRecapHookInstalled's defensiveness (Array.isArray on blocks, b.hooks?.filter ?? []) and wrap onDecision in try/catch (or remove the hook BEFORE persisting the decision).
