# BUG-032: previewStore.patch (create-if-absent) lets the unguarded onPreviewEvent handlers resurrect a cleared runtime entry for a just-removed Browser board — incomplete remediation of prior bug #37

- **Severity:** Low
- **Category:** resource-leak
- **Status:** CONFIRMED
- **Files touched:** `src/renderer/src/store/previewStore.ts`, `src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
previewStore.patch merges `{ ...DEFAULT_RUNTIME, ...s.byId[id], ...patch }`, so it unconditionally re-creates a byId entry when one is absent (previewStore.ts:66-69). The prior remediation of this resurrection defect (findings #37, code-comment "Bug #48") only added per-call-site `recs.current.has()` guards to the two capture-await sites (beginMotion at BrowserPreviewLayer.tsx:359/367 and demoteToSnapshot at :249) and left the store primitive create-if-absent. The three onPreviewEvent handlers (did-finish-load :603, did-navigate :605, did-fail-load :611) still call `patchRuntime(ev.id, ...)` with no existence guard, so a load/nav lifecycle event that arrives after the board was deleted (clearRuntime already ran) re-creates an orphan byId entry that is never cleared again. Impact matches #37: a benign in-memory orphan holding stale status/liveUrl/error strings; live stays false so selectLiveCount is not inflated and no mounted component reads it.

## Where
- src/renderer/src/store/previewStore.ts:66-69 (patch create-if-absent); src/renderer/src/canvas/boards/BrowserPreviewLayer.tsx:603/605/611 (onPreviewEvent handlers patchRuntime(ev.id,...) with no existence guard); cleared at BrowserPreviewLayer.tsx:471 (reconcile clearRuntime on removal)

## How it triggers
A Browser board has a load/navigation in flight on its native WebContentsView. The user deletes the board: removeBoard -> store subscription -> reconcile -> closeBoard(id) (dispatches async preview:close IPC) + clearRuntime(id) (synchronous, deletes byId[id], recs.delete). On main there is a real window between the renderer issuing preview:close and main actually running disposeOne->webContents.close(); if the in-flight load completes (did-finish-load) or navigates (did-navigate) or fails on its own during that window, emit() send()s a preview:event for that id to the renderer. The renderer processes that queued event AFTER clearRuntime already ran, so the onPreviewEvent handler calls patchRuntime(id, {status,...}) which re-creates byId[id]. Because the rec was already deleted (recs.delete at :470), reconcile's removal branch (:467) never fires for that id again, so the orphan is never cleared. (Close-induced aborts fire did-fail-load errorCode -3 which is filtered at preview.ts:221, so the reliable trigger is an organic in-flight completion, not the close abort itself.)

## Verification evidence
Code trace on the current post-48c6500 tree (both files last touched by 48c6500): (1) previewStore.ts:66-69 patch = `byId: { ...s.byId, [id]: { ...DEFAULT_RUNTIME, ...s.byId[id], ...patch } }` — unconditional create; there is no patch-only-if-exists variant, and clear() (lines 70-76) is the only deleter, fired solely from reconcile (:471). (2) The remediation diff `git show 48c6500 -- BrowserPreviewLayer.tsx` shows the resurrection fix was applied ONLY at the await sites: beginMotion got `if (shots[i] && recs.current.has(g.id))` (:359) and `if (!r) return` (:367), demoteToSnapshot got `if (!recs.current.has(g.id)) return` (:249), both citing "Bug #48 ... patching here would resurrect an orphan." (3) The same diff shows the did-finish-load handler WAS edited in 48c6500 — it gained a `cur === 'load-failed'` status-latch guard (Bug #5, :601-602) — yet no existence guard was added; did-navigate (:605) and did-fail-load (:611) call patchRuntime(ev.id,...) with no guard at all. (4) docs/bug-hunt-findings.md #37 explicitly names this root cause ("previewStore patch is create-if-absent ... RE-CREATES a byId entry ... never cleaned again") and offers the alternative "make patchRuntime no-op for an absent id" but recommends "the localized re-check at the two await sites is the minimal correct fix" — confirming the team scoped the fix to those two sites and left the event-handler path and the store primitive unaddressed. (5) Harm bound: all three event patches omit `live`, so DEFAULT_RUNTIME.live=false (previewStore.ts:39) persists; selectLiveCount (:86-90) counts only r.live===true so it is NOT inflated. The only iterating consumer is the DiagOverlay metric (Canvas.tsx:81); selectRuntime (BrowserBoard.tsx:117) reads by the live board's own id and a deleted board's component is unmounted. So the orphan is never read into UI — purely a benign leaked byId record.

## Suggested fix direction
Fix the root cause once at the store primitive rather than per call-site: add a `patchIfPresent`/existence-aware variant (no-op when `!(id in s.byId)`) and route the lifecycle-event handlers (and ideally the capture-await sites) through it, OR add an existence guard in onPreviewEvent (`if (!(ev.id in usePreviewStore.getState().byId)) return`, mirroring the existing did-finish-load status read at :601). Minor doc correction: the orphan from the event path holds only status/liveUrl/error strings, not a base64 snapshot data URL (snapshots are only patched by beginMotion/demoteToSnapshot), so the candidate's "if a snapshot patch lands, a base64 data URL" is an overstatement for this specific path. Low priority: no live-count inflation, no UI mis-read, no security impact — a small never-cleared in-memory record on an uncommon delete-during-in-flight-load interleaving.
