# BUG-043: audit:read limit is unvalidated: limit:0 returns the ENTIRE log (slice(-0) === slice(0)), and every read loads the whole unbounded JSONL into memory

- **Severity:** Low
- **Category:** error-handling
- **Status:** CONFIRMED
- **Files touched:** `src/main/auditLog.ts`, `src/main/auditIpc.ts`
- **Roadmap:** none
- **Assigned:** _(left blank for the dev team)_

## Summary
AuditLog.read computes `entries.slice(-limit)`. A caller-supplied limit of 0 yields slice(-0) = slice(0) → ALL entries (the exact opposite of 'cap to 0'); a negative limit (e.g. -5) yields slice(5) → everything but the first 5; a NaN/non-number yields slice(NaN) → all entries. auditIpc's `audit:read` handler passes the renderer's opts object straight through with no validation/clamping, violating the repo's IPC-inputs-must-be-validated invariant. Combined with the append-only, never-truncated mcp-audit.jsonl (each entry up to ~200KB of capped prompt/outputs), readRaw() reads the full file into a string and parseEntries materializes every entry on EVERY read — so a renderer bug or compromised renderer can pull the entire multi-hundred-MB trail over IPC and stall MAIN's memory, when the intent of the limit was to bound exactly that.

## Where
`src/main/auditLog.ts:167` — the `read` method of the AuditLog returned by `createAuditLog` (limit defaulted at line 166, `entries.slice(-limit)` at line 168)

## How it triggers
Renderer invokes audit:read with {limit: 0} (or any non-positive/non-numeric limit), or simply uses the viewer after months of MCP dispatch growth — read() parses the whole file regardless of limit.

## Verification evidence
src/main/auditLog.ts:165-169 — `const limit = readOpts?.limit ?? DEFAULT_READ_LIMIT; return entries.slice(-limit).reverse()` with no clamp. Verified semantics with node: [1,2,3,4].slice(-0) → [1,2,3,4] (limit:0 returns EVERYTHING, the opposite of a cap), slice(-(-2)) → [3,4] (negative limit drops the first n instead of capping), slice(-NaN) → all. src/main/auditIpc.ts:30-36 forwards the renderer's opts object straight to log.read() — only the sender frame is checked (isForeignSender), the limit value is never validated, contrary to the repo's IPC-input-validation invariant. readRaw (auditLog.ts:115-120) reads the entire mcp-audit.jsonl into a string and parseEntries materializes every entry on every read regardless of limit, and the log is append-only/never truncated with entries up to ~200KB each (MAX_LONG=100_000 × 2 fields + shorts, lines 56-58), so the limit never bounds MAIN's memory anyway. auditLog.test.ts:167-170 exercises a positive limit only. Low: requires a buggy/compromised renderer or months of growth; impact is memory pressure/IPC bloat, no integrity loss.

## Suggested fix direction
Clamp limit to a positive integer ceiling at the IPC boundary (Number.isInteger + Math.min/Math.max) and consider tail-reading the JSONL.
